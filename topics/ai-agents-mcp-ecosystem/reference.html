Quick Reference

Everything you need to design, build, and operate agent systems — organized for fast lookup when you're in the middle of building.

This reference is organized by the questions you ask in practice, not by how the technology is organized in documentation. Expand the section most relevant to what you're working on right now.

Section 1: Key Concepts Glossary

Agent

An autonomous software entity that uses an LLM as its reasoning engine, operating in a loop: observe the current state, think about what action to take, act by calling a tool, observe the result, repeat until the goal is achieved or a stopping condition is reached. Agents are defined by their loop, not their intelligence.

Tool

A discrete, atomic capability that an agent can invoke. A tool is defined by its name, description (what it does and when to use it), input schema (JSON Schema), and response format. Tools do one thing — they don't orchestrate other tools. Examples: read_file, send_email, search_web, execute_query.

Skill

A reusable, multi-step workflow that chains tools together to accomplish a meaningful goal. Skills encode the "how" of complex tasks: what tools to call, in what order, with what decision logic at each step. Skills can be human-defined (explicit, predictable) or agent-composed (flexible, emergent). Hospital analogy: surgical procedures are skills; surgical instruments are tools.

MCP (Model Context Protocol)

An open standard protocol published by Anthropic that defines how AI models communicate with external tools and data sources. MCP enables any MCP-compatible model to use any MCP-compatible tool without custom integration code. Analogous to HTTP for the web — a universal communication standard that enables an ecosystem.

MCP Server

A process that exposes capabilities (tools, resources, prompts) to MCP clients. The server handles all integration complexity: authentication with external services, data transformation, error handling, rate limiting. Servers speak the MCP protocol — the client never needs to know the implementation details of the underlying service.

MCP Client

A process (typically an AI application or agent framework) that connects to MCP servers and uses their capabilities. The client initiates the connection, performs capability negotiation, sends tool call requests, and receives results. Claude Desktop, Claude Code, Cursor, and Windsurf are all MCP clients.

Transport

The mechanism by which MCP messages are physically transmitted between client and server. Three options: stdio (subprocess, messages over stdin/stdout — best for local servers), HTTP+SSE (network-accessible, server-sent events for streaming — best for remote servers), Streamable HTTP (HTTP with streaming support — best for long-running operations). Transport choice affects latency, deployment model, and operational complexity.

JSON-RPC

The message format used by MCP. JSON-RPC 2.0 defines a standard way to encode method calls (requests) and their results (responses) as JSON. Every MCP message is a JSON-RPC 2.0 message. This standardization means any JSON-RPC library can be used to implement MCP clients and servers, reducing the implementation burden.

SSE (Server-Sent Events)

A one-way HTTP streaming mechanism used in MCP's HTTP+SSE transport for the server-to-client message channel. The client makes a GET request to an SSE endpoint; the server keeps the connection open and sends messages as events when they're ready. SSE is simpler than WebSockets (one direction, HTTP-native) but sufficient for MCP's message patterns.

Resources

MCP's mechanism for exposing read-only data to clients. Resources are identified by URIs (e.g., file:///path/to/doc or db://table/row/id). Unlike tools (which perform actions), resources are data sources that clients can subscribe to and retrieve. Clients can list available resources, read specific resources, and subscribe to notifications when resources change.

Prompts

Reusable, parameterized prompt templates that MCP servers can expose to clients. Servers with domain expertise can codify that expertise as prompts — "analyze_security_vulnerability" might be a prompt template that structures a thorough security review. Prompts enable server-defined best practices to be used consistently across different agents and use cases.

Sampling

An MCP capability that allows servers to request LLM completions from the client's model. This enables servers to perform AI-powered tasks without needing their own model access — they delegate generation back to the client's model. Sampling is how MCP servers can be "AI-powered" without each server needing its own API keys or model configuration.

Roots

MCP's mechanism for defining the scope of what a server can access. A client declares "roots" — specific file system paths, URLs, or repository references that the server is authorized to access. Roots enable fine-grained permission scoping: an MCP server can access ~/projects/myapp but not ~/projects/other-app. This is the primary safety boundary for file system servers.

Section 2: MCP Architecture

Capability Negotiation Process

Every MCP connection begins with a handshake: (1) Client sends initialize with its capabilities. (2) Server responds with its capabilities — what tools it exposes, whether it supports resources, whether it supports prompts, whether it accepts sampling requests. (3) Client sends initialized to confirm. Only after this handshake can tool calls begin. This ensures both parties know exactly what the other supports before any work begins.

Section 3: Tool Design Checklist

Naming Conventions

  • Use verb_noun format: read_file, send_email, search_database
  • Be specific enough to disambiguate: get_user_by_id, not get_user
  • Use consistent verb vocabulary: get/list/search/create/update/delete
  • Avoid abbreviations: get_repository, not get_repo
  • Namespace for multi-server systems: github_get_issue, jira_get_issue

Parameter Design

  • Mark parameters required vs. optional explicitly in the schema
  • Include descriptions for every parameter — the model reads these
  • Use enums for parameters with a fixed set of valid values
  • Provide sensible defaults for optional parameters
  • Include format hints for strings (e.g., "ISO 8601 date", "GitHub username")
  • Keep parameter count below 8 — more than this signals a tool is doing too much

Response Format

  • Use structured, consistent response shapes — not variable formats
  • Always include a success indicator — never infer success from absence of error
  • Return only what the model needs — not entire API responses
  • Truncate long text fields at a sensible limit (e.g., 2000 chars)
  • Include metadata the model may need: timestamps, source IDs, pagination tokens

Error Handling

  • Return structured errors: {error_code, message, suggestion}
  • Never return raw exception stack traces to the model
  • Distinguish recoverable errors (retry) from terminal errors (stop)
  • Include the failed input in the error response for debugging
  • Rate limit errors must include retry_after_seconds

Idempotency

  • Read operations should always be idempotent
  • Write operations should accept an idempotency_key parameter
  • Document whether the operation is idempotent in the tool description
  • Non-idempotent operations (send_email) require explicit confirmation patterns

Description Quality

  • State what the tool does in the first sentence
  • State when to use this tool vs. similar tools
  • State the expected outcome of a successful call
  • Include an example if the use case is non-obvious
  • State any important limitations or constraints
  • Test the description by asking: "Would a new developer know exactly when to use this?"
Section 4: Agent Loop Essentials

Context Management

  • Summarize accumulated tool results every 5-7 tool calls
  • Track what has been tried — agents loop when they forget
  • Include only what's needed for the current step in the context
  • For large file contents, pass only the relevant sections
  • Monitor context window usage — plan for summarization before hitting limits

Exit Conditions (Define All Four)

  • Success: What does "done" look like? Be precise.
  • Max iterations: Hard ceiling — enforce in code, not prompt
  • Repeated action: Same tool + same params = stuck. Exit and report.
  • Error terminal: Some errors are unrecoverable — stop trying

Retry Policies

  • Exponential backoff for rate limit errors: 1s, 2s, 4s, 8s
  • Max 3 retries for transient failures before surfacing to user
  • No retry for auth failures — they won't resolve without user action
  • No retry for validation errors — the schema needs fixing
  • Distinguish server errors (retry) from client errors (don't retry)

Cost Controls

  • Set per-task token budgets and enforce them
  • Alert when a task exceeds 2x the expected token cost
  • Log cost per tool call type to identify expensive operations
  • Use smaller models for simple classification steps within a loop
  • Cache intermediate results to avoid re-computing
Section 5: Performance Tuning

Token Optimization

  • Trim whitespace and formatting from tool responses
  • Return field subsets — only what the agent actually needs
  • Use structured formats (not prose) in tool responses
  • Compress conversation history with summaries before it bloats
  • Profile which tools return the most tokens — target those first

Parallel Tool Calls

  • Independent reads can always be parallelized
  • Model modern APIs return multiple tool_use blocks — handle them all
  • Don't serialize when you don't need to — latency adds up
  • Check for tool dependencies before assuming parallelism is safe
  • Implement a fan-out pattern for multi-source research tasks

Caching Strategies

  • Cache tool results at the MCP server layer, not the agent layer
  • Use TTLs appropriate to data volatility — schemas: hours; content: minutes
  • Semantic caching for search queries (embedding similarity)
  • Prompt caching for common system prompts (saves input token costs)
  • Implement cache invalidation on write operations

Streaming Responses

  • Stream final responses to users — don't buffer to completion
  • Stream tool call construction for real-time visibility
  • Use SSE or WebSockets for long-running agent tasks
  • Show progress indicators during multi-step tool chains
  • Implement heartbeats for connections that may timeout during long tool calls

Connection Pooling

  • Reuse MCP connections across requests — don't reconnect per call
  • Pool database connections within MCP servers
  • Implement connection health checks and automatic reconnection
  • Size connection pools based on observed concurrency, not guesses

Latency Targets

  • Local tools (filesystem, memory): target under 50ms per call
  • Network tools (APIs): target under 500ms per call
  • AI-powered tools (sub-agent calls): plan for 1-5 seconds
  • End-to-end task: set user expectations based on tool call count
Section 6: Security Checklist

Security for agent systems is a superset of API security. All the standard API security principles apply, plus agent-specific considerations around prompt injection, tool scope, and autonomous action blast radius.

Permission Scoping

  • Each agent gets only the tools it needs for its specific task
  • Use MCP Roots to restrict filesystem access to specific directories
  • Read-only tools for read-only tasks — never provide write access speculatively
  • Scope API permissions to minimum required scopes (GitHub: repo:read, not repo:write)
  • Use separate credentials for production vs. staging environments

Input Validation

  • Validate all tool parameters at the MCP server layer, not just in the schema
  • Sanitize path parameters to prevent directory traversal
  • Validate URL parameters to prevent server-side request forgery
  • Limit string parameter lengths to prevent context injection
  • Reject SQL-like strings in parameters that become database queries

Output Sanitization

  • Strip execution instructions from tool responses (prompt injection defense)
  • Truncate responses that may contain adversarial content at unsafe lengths
  • Flag tool responses that contain instructions to the model as suspicious
  • Never pass raw user-provided content directly to privileged tools

Rate Limiting

  • Rate limit at the MCP server level by client identity
  • Apply separate limits for read and write operations
  • Implement circuit breakers to prevent cascade failures
  • Alert on unusual spike patterns — may indicate runaway loops

Audit Logging

  • Log every tool call with: timestamp, agent identity, tool name, parameters, response summary
  • Log is immutable — write to append-only store
  • Retain audit logs for minimum 90 days (longer for regulated industries)
  • Alert on high-impact operations (deletes, deploys, sends) in real time

Sandboxing

  • Run MCP servers in isolated containers with minimal system permissions
  • Network-restrict servers that don't need external access
  • Code execution tools must run in sandboxed environments (no host access)
  • Use read-only mounts for filesystem servers that don't need write access
Section 7: Decision Matrix

The most important engineering judgment in this space is knowing when NOT to use agents. Not every task benefits from agentic architecture. Over-engineering is as costly as under-engineering.

When to Use What

Use Agents When:

  • The task requires multiple sequential decisions that depend on intermediate results
  • The task domain is ambiguous — the right approach can't be fully specified in advance
  • The task involves synthesizing information from multiple heterogeneous sources
  • Human judgment is the current bottleneck and the cost of occasional errors is acceptable
  • The task is high-value enough to justify per-task LLM inference costs

Use Workflows When:

  • The task sequence is known and deterministic — always the same steps
  • Reliability and auditability are more important than flexibility
  • The task involves regulated processes (financial, medical, legal) where deviation is not acceptable
  • Latency requirements are tight and LLM inference latency is a bottleneck
  • The task will run at high volume where per-call LLM costs add up significantly

Use Simple API Calls When:

  • A single API call answers the question completely
  • The input and output formats are both fully determined
  • You're prototyping and haven't yet identified where the reasoning complexity lives
  • The integration is one-off and won't be reused across models or agents

Build Custom vs. Use Ecosystem

  • Use ecosystem MCP servers when: They cover 80%+ of your use case, they are actively maintained, and the operational cost of running them fits your model
  • Build custom when: You have proprietary data sources not covered by existing servers, security requirements preclude external dependencies, or you need fine-grained control over the tool surface exposed to the model
  • Fork and customize when: An existing server covers most of your use case but needs modification — don't build from scratch what exists
