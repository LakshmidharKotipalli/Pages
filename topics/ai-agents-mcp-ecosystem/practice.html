Practice Exercises

Reading about agents produces familiarity. Struggling with these questions produces understanding.

These exercises are sequenced by difficulty. Don't skip to advanced questions until you can answer the warm-up questions without hesitation. The warm-up questions test vocabulary; the core questions test design thinking; the advanced questions test systems thinking; the expert questions test judgment under ambiguity.

How to use these: Click a question to reveal a detailed model answer. Attempt your own answer first — write it down or say it out loud. Compare your answer to the model. Where did you miss nuance? Where did you add it? The gap is your learning.

Warm-Up Questions (Vocabulary & Fundamentals)

Q1: What's the difference between a tool and a skill? Name an example of each.

Model Answer

A tool is a single, atomic capability — it does exactly one thing and returns a result. It has no internal decision logic; it just executes. Example: read_file — given a file path, returns the file contents. No reasoning, no branching, no side effects beyond the read.

A skill is an orchestrated, multi-step workflow that chains tools together with decision logic to accomplish a meaningful goal. Example: refactor_module — this skill reads the target file, analyzes its structure, identifies refactoring opportunities, writes the improved version, runs the test suite, and commits if tests pass. It uses multiple tools and includes branching logic (what to do if tests fail).

The hospital analogy: a scalpel is a tool. A laparoscopic appendectomy is a skill. Skills use tools, but skills are defined by their orchestration logic, not their individual steps.

Key distinction to remember: Tools are stateless and idempotent (calling them twice with the same input produces the same result). Skills are stateful workflows — their outcome depends on state accumulated across multiple steps.

Q2: Explain what MCP does in one sentence that a non-technical PM would understand.

Model Answer

"MCP is a universal standard that lets any AI assistant connect to any external tool or service — so instead of building a custom connection for every AI-tool pair, you build it once and it works everywhere."

Why this works for a non-technical PM:

  • It maps to a concept they already understand (standards like USB, Bluetooth)
  • It frames the value in terms of engineering efficiency (build once)
  • It implies why it matters (reduces vendor lock-in, speeds up integration)

What to avoid: Don't use "JSON-RPC," "transport layer," "stdio," or "client-server architecture" with a PM. These terms signal you don't understand your audience. The MCP technical details are irrelevant to a product discussion; the economic and architectural benefits are everything.

Bonus follow-up if they ask "so what?": "It means our team can add a new data source to our AI assistant in an afternoon instead of a week, and it works with Claude, GPT-4, or whatever model we switch to next year."

Q3: Why can't you just use direct API calls instead of MCP?

Model Answer

You can, and for simple integrations, direct API calls are often the right choice. MCP adds value when you have these specific needs:

1. Multiple model support: If your application needs to work with Claude today and GPT-4 tomorrow, direct API calls require writing model-specific integration code twice. MCP abstracts the tool layer so the same server works with any MCP-compatible model.

2. Ecosystem reuse: A GitHub MCP server built by someone else can be dropped into your agent immediately. A direct API integration must be built from scratch. As the MCP ecosystem grows, the advantage of ecosystem reuse compounds.

3. Capability negotiation: MCP includes a handshake protocol where the client and server negotiate capabilities — what tools are available, what resource types are supported, what prompts are registered. Direct API calls have no equivalent mechanism; you have to hardcode what the API supports.

4. Standardized error handling and observability: Every MCP server speaks the same protocol, so your observability infrastructure can be generic. With direct API calls, each integration requires custom error handling and logging.

When to use direct API calls instead: When you have a single model, a single integration, and the overhead of running an MCP server is not justified. Premature abstraction is a real cost. Start simple; add MCP when you're integrating a third tool or a second model.

Core Questions (Design Thinking)

Q1: Design the tool schema for a "web search" tool. What parameters? What response format?

Model Answer

Parameters (what the agent sends to the tool):

  • query (string, required): The search query. Keep this a free-form string — don't try to structure it. The model is good at writing search queries.
  • num_results (integer, optional, default: 5): How many results to return. Bounded between 1-20 to control token cost.
  • recency (enum, optional): "any" | "day" | "week" | "month" — filters by publication date when time-sensitivity matters.
  • safe_search (boolean, optional, default: true): Production agents should default to safe search on.

Response format (what the tool returns to the agent):

Return an array of result objects. Each result contains: title (string), url (string), snippet (string, 150-300 chars), published_date (ISO 8601 or null). Do NOT return the full page content — snippets are sufficient for the agent to decide which result to fetch with a separate read_url tool.

Error handling: If the search fails, return a structured error: {error: "RATE_LIMIT_EXCEEDED", retry_after_seconds: 30, suggestion: "Reduce query frequency or cache results"}. Never return an empty array on error — the agent cannot distinguish "no results found" from "search failed."

Tool description for the agent: "Search the web for current information on any topic. Use this when your training data may be outdated or when you need specific facts, recent news, or real-time data. Returns a list of results with titles and snippets — use the read_url tool to fetch the full content of any result."

Key design insight: The agent should read before writing. The ratio of read_file calls to write_code calls in a well-designed code agent is typically 3:1 or higher. Agents that write immediately without reading context are guessing. They'll appear to fix the bug while introducing new ones.

The test step is non-negotiable. An agent that fixes code without verifying the fix is not a reliable tool — it's a liability. Include test execution in the tool definition and make it part of the skill's exit condition: "only declare success if tests pass."

Q3: Your MCP server is returning 500 errors intermittently. Walk through your debugging approach.

Model Answer

Step 1 — Characterize "intermittent." Is it time-based (fails every N minutes), load-based (fails under concurrent requests), or input-based (fails on specific parameter patterns)? Pull logs for the last 24 hours and find the temporal pattern before touching any code.

Step 2 — Check the process health. Is the MCP server process consuming excessive memory and being killed by OOM? Is it leaving zombie connections that deplete a connection pool? Look at system metrics, not just application logs.

Step 3 — Isolate the failing tool. Are all tools returning 500s, or only specific ones? If only specific tools fail, the bug is in those tool implementations. If all tools fail, the bug is in shared infrastructure (auth middleware, connection pooling, request routing).

Step 4 — Replay failing requests. Extract the exact request payload from a failed call. Replay it in isolation, outside the agent loop. Can you reproduce the 500 on demand? A reproducible failure is a solvable failure. A non-reproducible failure in isolation is a race condition or concurrency bug.

Step 5 — Check external dependencies. If the tool calls an external API, check the rate limits, response time SLAs, and error rate of that dependency. A 500 from your MCP server is often a 503 from the upstream API that wasn't handled gracefully.

Step 6 — Add structured error handling. Before the next fix, add structured error logging at every boundary: request received, external call made, response received, response returned. The next intermittent failure will be much faster to diagnose with this in place.

Q4: Compare building a custom Slack integration vs. using an MCP Slack server. Trade-offs?

Model Answer

Custom Slack Integration (Direct API):

  • Pro: Exactly the API surface you need, nothing more. If you only need to post messages, you build exactly that.
  • Pro: No additional process to run. The integration is just your agent code making HTTP calls.
  • Pro: You control the error handling, retry logic, and response format exactly as your agent expects it.
  • Con: Only works with the model you've integrated it with. Switching models means rewriting the integration.
  • Con: You own the maintenance burden entirely — Slack API changes, auth token rotation, deprecations.
  • Con: Cannot be shared across teams or projects without duplicating code.

MCP Slack Server:

  • Pro: Works with any MCP-compatible model. Team switches from Claude to GPT-4? The Slack integration works unchanged.
  • Pro: If using a well-maintained open source server, you inherit improvements and bug fixes from the community.
  • Pro: Reusable across all agents in your organization. One deployment serves all.
  • Con: Additional operational overhead — another server process to deploy, monitor, and keep alive.
  • Con: The server's tool design may not match your agent's needs exactly, requiring adaptation work.
  • Con: Open source MCP servers vary enormously in quality — some are production-ready, others are demos.

Recommendation: For a prototype or single-agent system, direct API. For a multi-agent or multi-model organization, MCP. The cross-over point is typically when you have three or more agents that need the same integration.

Q5: How would you prevent an agent from accidentally deleting production data through a database tool?

Model Answer

Layer 1 — Tool design (most important): Do not give the agent a "delete" tool unless it absolutely requires one. Provide a "soft_delete" tool that sets a deleted_at timestamp and moves records to a trash table. Hard deletes require a second tool call to empty trash, which adds an explicit intent step the agent must choose to take.

Layer 2 — Environment separation at the connection level: The database tool's connection string should be injected at runtime based on context. Production agents connect to the production replica (read-only) by default; write access to production requires an explicit elevation that is logged. Staging agents connect to staging. This is enforced at the MCP server level, not the agent prompt level — the agent cannot override it.

Layer 3 — Dry-run mode: Every write operation should support a dry_run parameter. When dry_run is true, the tool returns what it would do without executing. The agent should be prompted to use dry_run first for any destructive operation and confirm the plan before executing.

Layer 4 — Human approval queue for high-blast-radius operations: Operations affecting more than N rows should route to a human approval queue rather than executing immediately. N depends on your data domain — for user records, N might be 10. For log records, N might be 1,000,000.

Layer 5 — Audit logging with rollback capability: Every database write operation must be logged with the full before-state, the agent's stated intent, and the after-state. For the first 30 days after deploying any agent with write access, maintain a point-in-time recovery capability. The ability to undo is the safety net that makes the other layers less catastrophic if they fail.

Advanced Questions (Systems Thinking)

Q1: Design a multi-agent system for automated code review. What agents? What tools? How do they communicate?

Model Answer

Agents in the system:

  • Orchestrator Agent: Receives the PR, breaks down the review into specialized tasks, routes to specialist agents, aggregates results, produces the final review comment. Does not do any analysis itself — it coordinates.
  • Style Agent: Specializes in formatting, naming conventions, and style guide compliance. Operates entirely statically — no code execution required.
  • Logic Agent: Analyzes correctness, edge cases, and algorithmic issues. Has access to the code search tool to understand the codebase context for the changed code.
  • Security Agent: Specializes in security vulnerabilities, injection risks, auth issues, and data exposure. Has access to a vulnerability pattern database tool.
  • Test Coverage Agent: Analyzes whether the change is adequately tested. Can execute tests and report coverage deltas.

Communication pattern: Orchestrator creates a shared review context object (PR diff, file list, repository metadata) and passes it to each specialist agent in parallel. Each specialist returns a structured report: {severity, findings[], suggested_fixes[]}. Orchestrator aggregates, deduplicates overlapping findings, and formats the final review.

Key design choice: Specialists run in parallel, not in series. A 4-agent parallel review takes roughly the same time as a single-agent review. Running them in series would be 4x slower with no quality benefit.

The failure mode to design against: Conflicting recommendations from different specialists. Logic Agent recommends refactoring a method; Style Agent says the refactored version violates naming conventions. The Orchestrator needs to surface conflicts, not silently suppress them.

Q2: Your agent system costs $50/user/day in API calls. How do you reduce it to $5 without losing capability?

Model Answer

A 10x cost reduction without capability loss requires finding where cost is concentrated and eliminating waste — not just prompting a cheaper model for everything.

Step 1 — Profile before optimizing: Break down costs by tool call type, model, and prompt length. You'll almost always find that 80% of the cost comes from 20% of the operations. The expensive operations are where you focus.

Step 2 — Cache aggressively: Tool results that are stable should be cached. A database schema lookup should be cached for hours. A search result can be cached for minutes. Semantic caching (embedding-based) can return cached results for queries that are semantically similar even if not lexically identical. Caching alone often achieves 40-60% cost reduction in knowledge-retrieval-heavy agents.

Step 3 — Route by complexity: Not every task requires the most capable (most expensive) model. Simple classification and routing decisions can be done with a smaller, cheaper model. Reserve the frontier model for complex reasoning steps. A hybrid routing approach — small model for triage, large model for execution — can cut costs by 30-50% with careful task decomposition.

Step 4 — Compress context aggressively: Every unnecessary token in the context window costs money. Summarize tool results instead of passing raw responses. After every 5-7 tool calls, compress the accumulated context into a summary. Strip whitespace and formatting from tool responses — they're for human readers, not models.

Step 5 — Batch and parallelize: Separate reads from writes. Batch multiple read operations into a single LLM call where possible. The overhead of multiple API round trips (latency + per-request fixed cost) adds up significantly at scale.

Q3: Architect an MCP server that provides access to a company's internal knowledge base. Consider security, caching, and freshness.

Model Answer

Security architecture (most critical layer):

  • The MCP server authenticates requests using the calling agent's identity token, not a shared service credential. This ensures every knowledge base query is attributable to a specific user/agent.
  • The server enforces document-level permissions from the source system — if a user can't see a document in Confluence, they can't retrieve it through the MCP server either. Permission checks happen at query time, not at index time.
  • All queries are logged with the requesting identity, query text, and documents returned. This creates an audit trail that satisfies compliance requirements and enables anomaly detection.

Caching architecture:

  • Three-tier cache: in-memory LRU cache for the last 1,000 query results (sub-millisecond), Redis distributed cache for the last 100,000 results (1-5ms), cold storage for everything older (10-50ms).
  • Cache keys include the requesting user's permission hash — the same query from a user with different permissions returns different (permission-filtered) results. Do not share caches across permission boundaries.
  • TTL varies by document type: volatile documents (Slack messages, Jira tickets) cache for 15 minutes; stable documents (engineering runbooks, architecture docs) cache for 4 hours.

Freshness architecture:

  • Webhook-based cache invalidation: when a document is updated in the source system, a webhook fires that invalidates the relevant cache entries immediately.
  • Expose a freshness timestamp with every result. The agent's prompt should instruct it to mention when information may be stale if the retrieved timestamp is older than N hours relative to the task context.
Q4: An agent is supposed to deploy code but sometimes deploys to the wrong environment. Design the safety architecture.

Model Answer

Root cause first: "Sometimes deploys to wrong environment" is almost always an ambiguity problem in the agent's context, not a model intelligence problem. The agent doesn't know with certainty which environment to target — it infers from context. The fix is to eliminate the inference requirement.

Layer 1 — Explicit environment in every tool call: The deploy tool signature must require an explicit environment parameter: {environment: "staging" | "production"}. No default. If the agent doesn't specify, the tool returns an error. This forces the agent to make a conscious, observable choice rather than silently defaulting.

Layer 2 — Confirmation gate for production: When environment is "production," the tool returns a confirmation object rather than executing: {plan: "Deploy commit abc123 to production cluster us-east-1", estimated_downtime: "30s", rollback_procedure: "deploy-rollback --commit abc122", confirm: false}. The agent must make a second tool call with confirm: true to execute. This forces a two-step process that is visible in the conversation log.

Layer 3 — Environment as ambient context: The agent's system prompt should be dynamically populated with the current context: "You are operating in [ENVIRONMENT]. Deployments to environments other than [ENVIRONMENT] require explicit user approval." This leverages the model's instruction-following to catch mismatches before the tool call happens.

Layer 4 — Anomaly detection on tool call patterns: A deploy-to-production call that was not preceded by a deploy-to-staging call in the same session should trigger an alert and a human review hold. The expected pattern is staging → production. Pattern violations are early warning signals.

Expert Questions (Judgment Under Ambiguity)

Q1: Design an agent-to-agent protocol for a system where specialized agents collaborate on a software release.

Model Answer

The key insight: agents do not coordinate by talking to each other directly in a chat interface. They coordinate through a shared state object and explicit handoff contracts. Think of it as a relay race, not a roundtable discussion.

The shared release state object: A structured document that every agent reads and writes to. Contains: {release_id, code_commit, environment, stage (enum: code|test|deploy|monitor), artifacts[], validations[], blockers[], audit_log[]}. Every agent reads the current state before acting and writes its results to the state after acting.

The handoff contract: Each agent has a defined trigger condition (when to start) and a defined completion condition (when to hand off). Code Agent: starts when a PR is merged, completes when it has written a changelog and tagged the commit. Test Agent: starts when code stage is COMPLETE, runs the test suite, fails fast if tests fail, marks test stage COMPLETE when green. Deploy Agent: starts when test stage is COMPLETE, runs the deployment, waits for health check, marks deploy stage COMPLETE. Monitor Agent: starts when deploy stage is COMPLETE, watches error rates and latency for 15 minutes, escalates if degradation detected.

The failure propagation contract: Any agent can write a blocker to the shared state. A blocker halts the pipeline and routes to the human approval queue. Agents must check for existing blockers before starting their stage. Blockers must include: {blocker_id, created_by_agent, severity, description, resolution_options[]}.

The philosophy: This is not an autonomous release pipeline — it's a human-supervised pipeline with agents doing the tedious work. The human remains in the loop at every blocker and at the final deploy-to-production confirmation. The agents handle the speed; the humans handle the judgment calls.

Q2: How would you build an MCP server marketplace with trust, versioning, and discoverability?

Model Answer

This is fundamentally a package registry problem, not a novel AI problem. The lessons of npm, PyPI, and Docker Hub apply directly.

Trust model: Three tiers. Tier 1 (Verified): First-party servers published by the service owner (Stripe's official MCP server, GitHub's official MCP server). Cryptographically signed by the publisher. Tier 2 (Community): Third-party servers with a minimum usage threshold, automated security scanning, and manual review for the top 100 by usage. Tier 3 (Unverified): Any published server with no review — clearly labeled as such. The UI must make these trust levels unmissable, not buried in metadata.

Versioning: Semantic versioning (major.minor.patch) with the same semantics as software libraries: major version bumps signal breaking changes to the tool schema or authentication requirements. Clients specify minimum and maximum compatible versions in their configuration. The marketplace must host all published versions — allowing a patch to fix the current major without breaking clients pinned to the previous major.

Discoverability: Two search modes. Intent-based ("I need a server that can access GitHub pull requests") routes to an LLM-powered semantic search over the server tool descriptions. Identifier-based ("find me the official Jira server") routes to an exact name/organization search. Server pages must show: compatible client list, tool inventory with descriptions, auth requirements, rate limits, and a "try it" sandbox that lets developers test tool calls before installing.

The hard problem — quality drift: A server that was excellent at publication may become buggy as upstream APIs change. Implement automated canary testing: the marketplace periodically exercises a representative set of tool calls against each registered server. Servers that start failing are automatically downgraded in search ranking and flagged for review. This is the governance mechanism the npm registry still lacks and suffers for.

Q3: Compare the agent architectures of Claude Code, Cursor, and GitHub Copilot. What trade-offs did each make?

Model Answer

These three products represent three distinct positions on the autonomy-safety spectrum, each making coherent trade-offs for their target user.

Claude Code (high autonomy, terminal-native): Designed for engineers who trust the agent to operate at a high level of autonomy in a terminal environment. The agent has broad file system access, can execute commands, run tests, and interact with git. The safety architecture relies on the engineer's ability to review changes in the terminal and use standard version control as the undo mechanism. Trade-off: high productivity ceiling, higher risk floor. Best for: senior engineers on tasks where speed matters more than oversight granularity.

Cursor (medium autonomy, IDE-integrated): Designed to extend the IDE experience rather than replace it. The agent operates within the context of what's visible in the editor. Change suggestions are presented as diffs before application — the human is always shown the change before it's made. The IDE integration provides rich context (open files, cursor position, selected text) that reduces hallucination rates. Trade-off: more guardrails reduce raw throughput but increase trust and reduce error rates. Best for: mixed teams where some members are skeptical of high-autonomy agents.

GitHub Copilot (low autonomy, inline suggestions): Originally designed around the inline completion paradigm — the agent suggests, the human accepts or rejects at every keystroke. The most conservative autonomy model of the three. Recent additions (Copilot Workspace) have moved toward higher autonomy, but the core product still optimizes for the human staying in control of every line. Trade-off: lowest productivity ceiling, lowest risk floor, widest market fit. Best for: organizations with compliance requirements, regulated industries, or teams early in their AI adoption journey.

The meta-observation: All three products are converging toward higher autonomy over time, but from different starting points and with different trust architectures. The winner in each market segment will be determined not by which product is "smarter" but by which product's trust model matches its users' risk tolerance and workflow integration depth.
