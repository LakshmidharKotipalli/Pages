What's Next

You understand agents, tools, skills, and MCP. Now the question is: which direction do you go deeper?

Each topic below builds on the foundation you've built here. They are ordered by their natural dependency — topics earlier in the sequence unlock topics later. The difficulty ratings reflect the conceptual and practical complexity, not the importance. All seven topics are worth studying; the sequence recommendation tells you the highest-leverage order.

Topic Connection Map

Related Topics

RAG (Retrieval Augmented Generation) — Medium

Connection to This Topic

RAG is the "how agents access knowledge" layer. You've learned that agents use tools to interact with the world — RAG is the specialized tooling pattern for accessing large, dynamic knowledge bases. A knowledge base MCP server is a RAG system packaged for agent consumption. Understanding RAG deeply will make you dramatically better at building agents that work with proprietary information.

What You'll Learn

  • How embeddings work and why semantic similarity enables knowledge retrieval
  • Vector database design: indexing strategies, distance metrics, filtering
  • Chunking strategies: fixed-size, semantic, hierarchical
  • Retrieval quality: precision vs. recall trade-offs
  • Hybrid search: combining vector similarity with keyword matching
  • Re-ranking retrieved results for relevance

Prerequisites

  • Understanding of how LLMs process context (you have this)
  • Basic familiarity with embeddings as vector representations
  • Comfort with database concepts (indexes, queries)
Recommended starting point: Study the concepts of semantic search and vector similarity before diving into specific vector databases. The concept is more important than the implementation choice.
Prompt Engineering for Agents — Medium

Connection to This Topic

You've learned what the agent loop is and how tools work. Prompt engineering for agents is the discipline of writing the system prompts and task descriptions that make the loop reliable. This is the highest-leverage skill improvement available to anyone who has just learned the basics — better prompts immediately improve the performance of every agent you build.

What You'll Learn

  • System prompt architecture: persona, capabilities, constraints, output format
  • Tool descriptions that guide selection without over-constraining
  • Chain-of-thought patterns for complex reasoning tasks
  • Few-shot examples for tool use patterns
  • Exit condition specification that prevents loops
  • Prompt testing and evaluation methodologies

Prerequisites

  • Basic prompt engineering knowledge (complete this topic first)
  • Understanding of tool schemas (covered here)
  • Hands-on experience with at least one agent loop failure — debugging a broken prompt is the best teacher
Multi-Agent Orchestration — Hard

Connection to This Topic

You've learned single-agent architecture with tools. Multi-agent orchestration is the next order of complexity — how do you coordinate multiple agents with different specializations, tools, and contexts to accomplish tasks that no single agent could handle reliably? This topic is where the hospital analogy becomes most precise: the attending physician coordinating a trauma team is the orchestrator agent.

What You'll Learn

  • Orchestrator vs. worker agent patterns
  • Shared state management across agent boundaries
  • Agent communication: direct vs. event-based vs. shared state
  • Failure handling in multi-agent systems
  • Avoiding circular dependencies between agents
  • Cost management at scale — multi-agent systems can be expensive

Prerequisites

  • Single agent proficiency — you should have built and debugged at least 3 single-agent systems
  • Distributed systems fundamentals: consistency models, failure modes, eventual consistency
  • Experience with message queues or event-driven architectures (helpful but not required)
Warning: Don't reach for multi-agent architecture until you've exhausted single-agent solutions. Multi-agent systems multiply both capability and complexity. Start with the simplest architecture that could work.
Production Observability for AI — Medium

Connection to This Topic

The debugging section of this course focused on how to diagnose failures when they happen. Production observability is the discipline of building the infrastructure so that failures are visible before they become user-impacting incidents. You cannot run agents in production reliably without observability — you're flying blind.

What You'll Learn

  • Distributed tracing for multi-step agent workflows
  • Metrics: latency, token costs, tool success rates, loop iteration counts
  • Logging: structured event logs vs. conversation transcripts
  • Alerting: what thresholds indicate a problem vs. normal variance
  • LLM-specific observability tools: LangSmith, Helicone, Arize
  • Cost attribution: which agents, tasks, and tools are driving spend

Prerequisites

  • Familiarity with standard observability tools (Datadog, Grafana, Prometheus)
  • Understanding of distributed tracing concepts (spans, traces, correlation IDs)
  • Experience operating at least one agent in a non-trivial context
Agent Safety and Alignment — Hard

Connection to This Topic

The security checklist and safety architecture patterns in this course are the operational layer of agent safety. This topic goes deeper — the theoretical and practical foundations of ensuring agents act within intended boundaries across a range of operating conditions, including adversarial ones. As agents gain more autonomy and higher-stakes access, safety becomes the non-negotiable constraint.

What You'll Learn

  • Formal specification of agent goals and constraints
  • Prompt injection: attack vectors, defenses, and detection
  • Tool use safety: blast radius analysis, reversibility requirements
  • Human oversight patterns: when and how to require human approval
  • Constitutional AI and policy-based constraints
  • Red-teaming agent systems: adversarial testing methodologies

Prerequisites

  • Understanding of failure modes (covered here in debugging and pitfalls)
  • Security fundamentals: threat modeling, defense in depth
  • Experience with at least one agent system that had a safety incident — the lessons are most salient after a real failure
LLM Fine-tuning for Tool Use — Hard

Connection to This Topic

This course focused on prompting general-purpose models to use tools reliably. Fine-tuning takes a different approach: training a model specifically on tool-use data for your domain, producing a model that is faster, cheaper, and more reliable on your specific tool set. This is a specialist track — most teams won't need it, but for high-volume production deployments, the economics can be compelling.

What You'll Learn

  • Supervised fine-tuning on tool use examples
  • Dataset construction: curating tool call examples from production logs
  • Evaluation: how to measure tool selection accuracy and parameter correctness
  • LoRA and PEFT: parameter-efficient fine-tuning approaches
  • Deployment: serving fine-tuned models alongside standard models
  • Trade-offs: fine-tuned specialists vs. general-purpose frontier models

Prerequisites

  • ML fundamentals: training loops, loss functions, gradient descent
  • Training infrastructure: GPU cluster access or managed training services
  • Production agent deployment experience — you need real usage data to build a good fine-tuning dataset
Edge AI and Local Agents — Hard

Connection to This Topic

This course focused on cloud-hosted models accessed via API. Edge AI is the emerging frontier where agents run on-device — no cloud API call, no latency, no data leaving the device. This matters for latency-critical applications (real-time agent assistance), privacy-critical applications (healthcare, legal), and connectivity-limited deployments (industrial, remote locations). MCP servers that work with local models follow the same protocol — the transport layer changes, but the architecture is identical.

What You'll Learn

  • Local model options: Ollama, llama.cpp, MLX for Apple Silicon
  • Quantization: reducing model size for edge deployment without unacceptable quality loss
  • Model selection: which models are capable of tool use at small sizes?
  • Local MCP server patterns: stdio transport for zero-network-overhead tool calls
  • Hybrid architectures: local model for fast operations, cloud for complex reasoning
  • Hardware considerations: which edge hardware enables useful agent capabilities?

Prerequisites

  • Understanding of LLM architectures: transformer basics, model size trade-offs
  • Systems knowledge: memory constraints, compute budgets, thermal management
  • Practical experience: having run at least one local model (Ollama is the fastest starting point)

Recommended Learning Sequence

Based on dependency relationships and typical learner profiles, here are three learning paths from this foundation:

Path A: Production Engineer

Goal: Deploy reliable agents in production

  1. Prompt Engineering for Agents — Immediately improves everything you build
  2. RAG — Most agents need knowledge access
  3. Production Observability — Can't operate what you can't see
  4. Agent Safety — Required before high-stakes access
  5. Multi-Agent Orchestration — When single agents aren't enough

Path B: Research and Exploration

Goal: Understand the frontier of agent capabilities

  1. Multi-Agent Orchestration — Frontier capability area
  2. Agent Safety and Alignment — Necessary constraint on capability
  3. LLM Fine-tuning for Tool Use — Specialized capability optimization
  4. Edge AI and Local Agents — Emerging deployment paradigm
  5. RAG — Foundation for knowledge-intensive agents

Path C: Platform Builder

Goal: Build infrastructure other teams use

  1. Production Observability — Platform teams own the monitoring layer
  2. Agent Safety and Alignment — Platform teams set safety standards
  3. Multi-Agent Orchestration — Platform teams enable complex workflows
  4. RAG — Platform teams build the knowledge layer
  5. Edge AI — Platform teams enable new deployment models

The One Thing to Do Next

Regardless of which path fits you: build something real before studying the next topic.

The pattern that distinguishes engineers who compound their learning from those who plateau: they implement each concept before moving to the next. You cannot understand prompt engineering for agents until you've written a prompt that failed and iterated to fix it. You cannot understand production observability until you've been surprised by an agent failure in production.

The hospital analogy applies here too: a medical student who reads every textbook without clinical rotations knows the vocabulary but not the medicine. The rotations are not supplementary to the education — they are the education. Build. Debug. Observe. Then study the next topic with real experience to anchor the new concepts to.
