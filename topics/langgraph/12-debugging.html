The Debugging Process

When your agent does something unexpected, you need a systematic approach — not intuition, not print statements, and definitely not panic. Here is the complete playbook: a 5-step framework, every common error pattern, and the full tools arsenal.

The Fundamental Principle

Agent failures are always one of three things: the LLM made a bad decision, the state evolved incorrectly, or the graph topology is wrong. The 5-step framework below systematically rules out each category. Start with Step 1 every time, even when you are sure you know the answer — the state often surprises you.

1

Get the State

First thing, always — before any other debugging action

The single most important first action when debugging any LangGraph agent: inspect the current state snapshot. You cannot reason about an agent's behavior without knowing exactly where it is and what it knows.

Python — Get current state
config = {"configurable": {"thread_id": "your-thread-id"}} # Synchronous state = graph.get_state(config) # Async state = await graph.aget_state(config) # State snapshot contains: print(state.values) # The full state dict print(state.next) # Tuple of next nodes to execute print(state.config) # Config including checkpoint_id print(state.metadata) # Step number, source, writes print(state.created_at) # Timestamp of this checkpoint print(state.parent_config) # Config of the previous checkpoint
What to Look For
  • What is state.next? Which node is the agent about to execute (or stuck waiting on)?
  • What are the last 3 messages? state.values["messages"][-3:] — what did the LLM just see and decide?
  • Is there a pending interrupt? state.next == ("__interrupt__",) means it is waiting for human input
  • What is the iteration count? Check your iteration_count field — is it near the limit?
  • Are any fields unexpectedly None or empty that should have been populated by a previous node?
✈
Air Traffic Control Analogy

Asking for the state is like pulling up the transponder data on a tracked aircraft: Where is the plane right now? What is its current heading and altitude? What is in the flight data recorder? You would never try to diagnose an in-flight problem without knowing the aircraft's current position first.

2

Get the History

Trace the state evolution — find where things went wrong

The current state tells you where you are. The history tells you how you got there. Every checkpoint is a full snapshot of state at a point in time. Find the transition where the behavior deviated from what you expected.

Python — Get full state history
config = {"configurable": {"thread_id": "your-thread-id"}} # Returns an iterator of StateSnapshot objects, most recent first history = list(graph.get_state_history(config)) # Inspect each checkpoint for checkpoint in history: print(f"Step: {checkpoint.metadata.get('step')}") print(f"Node that just ran: {checkpoint.metadata.get('source')}") print(f"Next nodes: {checkpoint.next}") print(f"Checkpoint ID: {checkpoint.config['configurable']['checkpoint_id']}") print(f"Created at: {checkpoint.created_at}") print("---") # To find where a field changed: for checkpoint in history: val = checkpoint.values.get("your_field") print(f"Step {checkpoint.metadata.get('step')}: {val}")
What to Look For
  • At which checkpoint did a state field change unexpectedly? Compare consecutive checkpoints
  • Which node's output caused the state to diverge from the expected path?
  • Is there a checkpoint where state.next routed to an unexpected node?
  • What was the state immediately before the problematic node ran? That is your debug window
  • Save the checkpoint_id of the last known-good state — you will need it for Step 4
✈
Air Traffic Control Analogy

The state history is the flight data recorder and cockpit voice recorder combined. Every checkpoint is a timestamped entry showing exactly what the aircraft knew, decided, and did at that moment. When investigating an incident, investigators review the full recording from departure, not just the final seconds.

3

Check the LangSmith Trace

See exactly what the LLM was told and what it decided

State history tells you how the state changed. LangSmith tells you why — specifically, what prompt was sent to the LLM, what it responded with, and which tool calls it made. This is the layer where 80% of agent bugs live.

Python — Connect run to LangSmith trace
# Ensure env vars are set (do this at startup): # LANGSMITH_API_KEY=your-key # LANGSMITH_PROJECT=your-project # LANGCHAIN_TRACING_V2=true (older versions) # Invoke with run_name for easy LangSmith search: config = { "configurable": {"thread_id": "your-thread-id"}, "run_name": "debug-session-user-123", "metadata": {"user_id": "user-123", "debug": True} } result = await graph.ainvoke(input_state, config=config) # The run will appear in LangSmith under your project # Search by run_name or metadata to find it quickly # To get the run_id programmatically (for API access): # Use the LangSmith client: # from langsmith import Client # client = Client() # runs = client.list_runs(project_name="your-project", run_name="debug-session-user-123")
What to Look For in LangSmith
  • Expand each LLM call node: what was the exact system prompt? What was the human message? What did the LLM respond?
  • Tool calls: what arguments did the LLM construct for each tool call? Were they correct?
  • Tool outputs: what did each tool actually return? Did it match what the LLM expected?
  • Latency per node: which node is slowest? Is a tool call timing out and returning empty?
  • Error tracebacks: expand any red nodes to see the full Python traceback
  • Key question: "What did the LLM see as input, and what did it decide?" That explains at least 80% of agent failures
✈
Air Traffic Control Analogy

LangSmith is the ATC voice recording combined with radar playback: you can see exactly what the controller said to each pilot, what the pilot responded, and the moment-by-moment radar track of every aircraft's position. Without this, you are investigating an incident with only the final crash site as evidence.

4

Time-Travel Debug

Replay from any checkpoint with modified inputs — without side effects

LangGraph's checkpointing enables something genuinely powerful: you can pick any point in a conversation's history and replay from there with different input, without re-running the preceding steps. This is the closest thing agents have to a time machine.

Python — Time-travel debug from a specific checkpoint
# 1. Find the safe checkpoint from history (Step 2) history = list(graph.get_state_history(config)) # Pick the checkpoint at step 4 (just before the bug occurred at step 5) safe_checkpoint = history[-5] # history is newest-first, so index accordingly safe_checkpoint_id = safe_checkpoint.config["configurable"]["checkpoint_id"] # 2. Build a config that points to that checkpoint safe_config = { "configurable": { "thread_id": "your-thread-id", "checkpoint_id": safe_checkpoint_id } } # 3. Optionally: modify the state at that checkpoint before replaying graph.update_state( safe_config, {"messages": [HumanMessage(content="Modified input to test different path")]} ) # 4. Resume from that checkpoint — execution continues from where the checkpoint left off result = graph.invoke(None, safe_config) # or stream it: for event in graph.stream(None, safe_config): print(event)
What This Lets You Do
  • Test "what if the agent had received X instead of Y at step 5?" — without re-running steps 1-4
  • Inject a corrected state value and see if the agent now takes the correct path
  • Prove that a specific input causes a specific failure: repeatable, isolated reproduction
  • Save money: you only re-run from the failing point, not the whole conversation
  • Build automated regression tests: record the failing checkpoint, assert on the outcome of the replay
✈
Air Traffic Control Analogy

Time-travel debugging is like flight simulation training: take the exact situation from a real incident — the aircraft position, speed, weather, ATC instructions — and run it again in a simulator with a different controller response. You can test alternate decisions without putting real aircraft at risk. This is how aviation turns incidents into training data.

5

Isolate and Unit Test

Extract the failing node and run it in isolation with production state

By the time you reach Step 5, you know which node is causing the problem. Now you extract it and unit test it directly. This is possible because LangGraph nodes are just Python functions — they have no hidden dependencies on the graph runtime.

Python — Isolate and test a node directly
# Your node function (a plain Python function): async def my_failing_node(state: MyState, config: RunnableConfig) -> dict: # ... node logic ... return {"result": ...} # From your production failure (Step 2), you captured the exact state # that was passed to this node when it failed. Now reproduce it: problematic_state = MyState( messages=[...], # exact messages from the failing checkpoint iteration_count=3, # exact field values search_results=[...], # ... all other state fields ) # Run the node in isolation — no graph overhead, no side effects from other nodes import asyncio result = asyncio.run( my_failing_node(problematic_state, config={"configurable": {}}) ) print(result) # See exactly what this node returns with this input # Now you can write a proper unit test: # def test_my_failing_node_with_unicode_input(): # state = ... (the exact failing state) # result = asyncio.run(my_failing_node(state, config={})) # assert result["result"] is not None
Why This Works
  • Nodes are pure Python functions: they take state + config, return a state update dict. No magic.
  • You can run them without a graph, without a checkpointer, without any LangGraph infrastructure
  • This is why keeping nodes as clean, focused functions matters — it pays dividends at debug time
  • Once you can reproduce the failure in isolation, write a unit test. The test becomes your regression guard
  • Works for tool functions too: call them directly with the exact arguments from the LangSmith trace
✈
Air Traffic Control Analogy

Isolating a node for unit testing is like removing a specific instrument from an aircraft and testing it on the bench with the exact flight conditions recorded during the incident. You do not need to recreate the entire flight — just verify that this instrument, given these inputs, produces the expected output.

Common Error Patterns Decoded

These 5 errors account for the majority of LangGraph debugging sessions. Know the error, know the diagnosis.

Error 01
GraphRecursionError: Recursion limit of 25 reached
Diagnosis

Your agent looped more than 25 times. The default recursion limit (25) is a safety guard. Hitting it means the graph has a cycle that is not terminating.

Root Cause (usually one of)
  • Missing termination condition on a cycle
  • LLM keeps calling tools without concluding
  • State never reaches a value that routes to __end__
  • Conditional edge logic has a bug — the "end" branch is unreachable
Fix
  • Add iteration_count guard to State and conditional edge
  • Verify __end__ is reachable in your conditional edge router
  • Increase limit intentionally: graph.compile(recursion_limit=50) if your graph legitimately needs more depth
Check First
  • What does iteration_count show in the last checkpoint?
  • Which node is looping? Check state.metadata["source"] across checkpoints
  • Draw the graph: graph.get_graph().draw_mermaid() — is __end__ connected?
Error 02
KeyError: 'messages'  (or any state field)
Diagnosis

A node tried to access a State field that does not exist at that point in execution. The field may be missing from the State definition or never initialized.

Root Cause (usually one of)
  • State field has no default value and was never written by a previous node
  • State schema mismatch between what one team added and another expects
  • A conditional branch skipped the node that initializes a field another node depends on
Fix
  • Add defaults to all State fields: field: Optional[Type] = None
  • Use .get("field", default) in nodes instead of direct key access
  • Initialize all fields in a dedicated __start__ node or State defaults
Check First
  • Which node threw the error? What fields does it access?
  • Are those fields in your State TypedDict definition?
  • Check the execution path — did the initialization node actually run before this node?
Error 03
InvalidUpdateError / State not updating as expected
Diagnosis

A node is returning something that does not match the expected state update format, or it is returning values that conflict with the reducer annotations on State fields.

Root Cause (usually one of)
  • Node returning a full copy of State instead of only the fields it is updating
  • Type mismatch — node returns str for a field defined as list[str]
  • Mutating State in-place instead of returning a new dict with updates
Fix
  • Nodes must return a dict with only the fields they are updating — not a full copy of state
  • Never modify state in-place: treat state as immutable inside a node
  • Verify the return type of each field matches its annotation in State
Check First
  • Add print(type(state["field"]), state["field"]) at the top of the failing node
  • Print the return value of the node before it is applied to state
  • Check for any reducer annotations — a node returning a dict when the reducer expects a list will cause this
Error 04
Human-in-the-loop interrupt not firing
Diagnosis

You set interrupt_before=["my_node"] but the graph completes execution without pausing at that node. The interrupt point is not being reached.

Root Cause (usually one of)
  • The interrupt node is not in the execution path for this invocation — a conditional edge routes around it
  • You are invoking with a Command that resumes past the interrupt
  • The node name in interrupt_before does not exactly match the registered node name
Fix
  • Visualize the graph: graph.get_graph().draw_mermaid() — is the interrupt node in the execution path?
  • Use stream mode to see which nodes actually execute: for event in graph.stream(input, config): print(event)
  • Verify exact node name string matches the interrupt_before value (case-sensitive)
Check First
  • Check the conditional edge that routes to this node — is the condition met for your test input?
  • Is the graph compiled with the correct interrupt_before list? Check the compiled graph object
  • Are you testing with the right thread ID? Interrupts are per-thread
Error 05
Checkpoint not restoring correctly / Unexpected state on resume
Diagnosis

After resuming from a checkpoint, the state does not match what you expected. Fields are missing, have wrong values, or the agent starts from an unexpected node.

Root Cause (usually one of)
  • Thread ID or checkpoint ID mismatch — the config is pointing to a different thread than you think
  • A node modified state in-place instead of returning a new dict
  • The checkpointer is not persisting between calls (wrong checkpointer — see Pitfall 2)
Fix
  • Always treat state as immutable inside nodes — return new dicts, never mutate the input
  • Verify the thread_id matches exactly (case-sensitive, exact string, no trailing spaces)
  • Log the thread_id at every invoke/resume point to confirm it is consistent
Check First
  • Print config["configurable"]["thread_id"] at the point of invoke and resume — do they match?
  • Check graph.get_state_history(config) — are there checkpoints in there? If not, the thread_id is wrong
  • Confirm the checkpointer is the same instance (or connected to the same database) across calls

Debugging Tools Arsenal

Every tool you need, with its primary use case. Know these before your first production incident.

graph.get_graph().draw_mermaid()

Generate a Mermaid diagram of your graph topology as a string. Paste into mermaid.live to visualize. Essential for verifying that __end__ is reachable and conditional edges are wired correctly.

graph.get_graph().draw_mermaid_png()

Generate a PNG image of the graph visualization directly. Requires pygraphviz or Mermaid CLI. Useful for documentation and sharing in incident reports.

LangGraph Studio

Visual real-time execution environment. Run your graph in Studio to watch each node execute, see state evolve at each step, and click through the execution manually. The fastest way to understand a new graph's behavior.

LangSmith Trace

Full observability for every LLM call: exact prompts, responses, token counts, latency, and tool calls. The irreplaceable tool for answering "what did the LLM see and what did it decide?" Available automatically when LANGSMITH_API_KEY is set.

graph.get_state(config)

Current state snapshot for a thread. Use at Step 1 of every debugging session. Returns the full state dict, the next nodes to execute, and the checkpoint metadata.

graph.get_state_history(config)

Iterator of all checkpoints for a thread in reverse chronological order. Use to trace how state evolved across steps and find the exact transition where behavior deviated.

logging.DEBUG on langgraph logger

Enable verbose execution logging: import logging; logging.getLogger("langgraph").setLevel(logging.DEBUG). Shows node entry/exit, state update diffs, edge routing decisions. Verbose but comprehensive for low-level debugging.

The Debugging Mental Model

"Agents fail for 3 reasons. The 5-step framework systematically eliminates each one."

1
The LLM Made a Bad Decision
Look at LangSmith for the exact prompt that was sent and the response that came back. The LLM's decision is entirely determined by the input — find the bad input.
2
The State Is Wrong
Look at checkpoints to trace state evolution. Find the step where a field became None, wrong type, or wrong value. That is where the bug was introduced.
3
The Graph Topology Is Wrong
Look at the Mermaid diagram. Is the routing logic correct? Is __end__ reachable? Are the conditional edge conditions actually evaluated correctly?
