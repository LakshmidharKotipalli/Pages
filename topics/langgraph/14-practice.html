Your First StateGraph
TypedDict state · add_node · add_edge · compile · invoke
▾

Build a simple 2-node graph: a greeting node and a farewell node. Your state is a TypedDict with two fields: name: str and message: str. The first node sets message to "Hello, {name}!". The second node reads the current message and appends " Goodbye, {name}!" to it. Chain them with a direct edge: greeting → farewell → END.

Compile and invoke with {"name": "Alice", "message": ""}. After it works, print the final state and verify the message is exactly "Hello, Alice! Goodbye, Alice!". Then draw the graph topology on paper before running graph.get_graph().draw_mermaid() — compare your drawing to the generated output.

What You'll Learn

  • Declaring a TypedDict-based State class
  • Creating a StateGraph(State) instance
  • Defining node functions: (state: State) -> dict
  • Using add_node() and add_edge()
  • The role of compile() — what it validates and what it produces
  • Calling invoke() and reading the returned state
  • Why nodes return partial state dicts, not full state objects

Success Criteria

  • Graph executes without error end-to-end
  • Final state: message == "Hello, Alice! Goodbye, Alice!"
  • You can explain verbally what compile() does and why it is required before invoke()
  • You can explain the difference between what a node receives and what it returns
  • Given 3 nodes, you can describe what changes in the add_edge calls

Extension Challenge

Add a third node called timestamp_node that appends the current UTC timestamp (ISO format) to the message — e.g., " [2025-06-01T14:30:00Z]". Insert it between farewell and END. Verify the final message includes all three parts in order.

from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    name: str
    message: str

def greeting_node(state: State) -> dict:
    # Return only the fields you are changing
    return {"message": f"Hello, {state['name']}!"}

def farewell_node(state: State) -> dict:
    # Append to existing message
    return {"message": state["message"] + f" Goodbye, {state['name']}!"}

builder = StateGraph(State)
builder.add_node("greeting", greeting_node)
builder.add_node("farewell", farewell_node)

builder.add_edge(START, "greeting")
builder.add_edge("greeting", "farewell")
builder.add_edge("farewell", END)

graph = builder.compile()
result = graph.invoke({"name": "Alice", "message": ""})
print(result)
# Expected: {"name": "Alice", "message": "Hello, Alice! Goodbye, Alice!"}

The Persistent Counter
Conditional edges · graph loops · MemorySaver · thread_id persistence
▾

Build a graph where a single counter node increments a count field in state by 1 each time it runs. Add a conditional edge: if count < 5, loop back to the counter node; if count >= 5, route to END.

Attach a MemorySaver checkpointer. Pass config={"configurable": {"thread_id": "thread-001"}} on every invoke call. Invoke twice with the same thread_id and observe what happens on the second call — does it loop again or exit immediately? Call graph.get_state(config) after both invocations and verify the count. Explain to yourself why the second call behaves differently.

What You'll Learn

  • Writing a router function for add_conditional_edges
  • Graph cycles and how LangGraph handles loops (recursion limit)
  • MemorySaver: in-memory checkpointing for development use
  • The config dict and thread_id — what a "thread" means semantically
  • Why the second invoke starts from the persisted checkpoint, not the initial input
  • get_state(config) for reading persisted thread state

Success Criteria

  • First invoke: runs 5 iterations, final count == 5
  • Second invoke (same thread): exits immediately, count still 5
  • You can articulate why the second call did not re-run the loop
  • get_state_history(config) shows all checkpoints from the first run
  • Setting recursion_limit=3 causes a GraphRecursionError you can explain

Extension Challenge

Swap MemorySaver for SqliteSaver backed by a local counter.db file. Terminate your Python process entirely, restart it, and call invoke again with the same thread_id. The state should still be persisted. This simulates production application restarts.

from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver

class State(TypedDict):
    count: int

def counter_node(state: State) -> dict:
    return {"count": state["count"] + 1}

def should_continue(state: State) -> str:
    return "end" if state["count"] >= 5 else "continue"

builder = StateGraph(State)
builder.add_node("counter", counter_node)
builder.add_edge(START, "counter")
builder.add_conditional_edges(
    "counter",
    should_continue,
    {"continue": "counter", "end": END}
)

memory = MemorySaver()
graph = builder.compile(checkpointer=memory)
config = {"configurable": {"thread_id": "thread-001"}}

result1 = graph.invoke({"count": 0}, config=config)
print("After first invoke:", result1)  # count == 5

result2 = graph.invoke({"count": 0}, config=config)
print("After second invoke:", result2)  # still count == 5, no iterations

The Tool-Calling Agent
create_react_agent · @tool decorator · ReAct reasoning loop
▾

Build a ReAct agent using the create_react_agent prebuilt with two tools: a calculator tool that accepts a math expression string and evaluates it using Python's ast module with an operator whitelist (no arbitrary code execution), and a unit converter tool that handles meters-to-feet (m * 3.28084) and Celsius-to-Fahrenheit (c * 9/5 + 32) conversions.

Give the agent this prompt: "What is 42 multiplied by 3.14159, and if that result is a distance in meters, how many feet is it?". Let it run to completion. Afterward, iterate over state["messages"] and print each tool call and tool result to understand the full ReAct trace.

What You'll Learn

  • The @tool decorator — docstrings are the LLM's instruction manual
  • create_react_agent(model, tools) — the prebuilt ReAct graph
  • How the agent decides which tool to call and when to stop
  • Reading the messages list: HumanMessage, AIMessage, ToolMessage sequence
  • The ReAct loop: Reason → Act → Observe → Reason until done
  • Why tool docstrings must be precise and unambiguous

Success Criteria

  • Agent invokes the calculator tool first, obtains 131.94678
  • Agent then invokes the unit converter with that result
  • Final answer includes both the product and the feet equivalent (~432.9 feet)
  • You can trace the exact tool call order from state["messages"]
  • You can identify which message types appear and in what order

Extension Challenge

Add a third tool: web_search(query: str) -> str that returns a hard-coded response like "According to Wikipedia, a football field is 91.44 meters long." Craft a prompt that forces the agent to use all three tools in sequence. Observe the multi-tool trace and count how many message objects appear in the final state.

import ast, operator as op
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langgraph.prebuilt import create_react_agent

# Safe AST-based math evaluator (whitelist operators only)
_SAFE_OPS = {
    ast.Add: op.add, ast.Sub: op.sub,
    ast.Mult: op.mul, ast.Div: op.truediv,
    ast.Pow: op.pow, ast.USub: op.neg,
}

def _safe_eval_node(node):
    if isinstance(node, ast.Constant):
        return node.value
    if isinstance(node, ast.BinOp):
        return _SAFE_OPS[type(node.op)](
            _safe_eval_node(node.left),
            _safe_eval_node(node.right)
        )
    if isinstance(node, ast.UnaryOp):
        return _SAFE_OPS[type(node.op)](_safe_eval_node(node.operand))
    raise ValueError(f"Disallowed expression type: {type(node).__name__}")

@tool
def calculator(expression: str) -> str:
    """Evaluate a mathematical expression using only +, -, *, /, ** operators.
    Input must be a plain math expression like '42 * 3.14159'.
    Returns the numeric result as a string."""
    try:
        tree = ast.parse(expression.strip(), mode='eval')
        result = _safe_eval_node(tree.body)
        return str(round(result, 6))
    except Exception as e:
        return f"Error evaluating expression: {e}"

@tool
def unit_converter(value: float, from_unit: str, to_unit: str) -> str:
    """Convert between measurement units.
    Supports: meters to feet (m, ft), celsius to fahrenheit (c, f).
    Example: unit_converter(100.0, 'm', 'ft')"""
    conversions = {
        ("m", "ft"): lambda v: v * 3.28084,
        ("ft", "m"): lambda v: v / 3.28084,
        ("c", "f"):  lambda v: v * 9 / 5 + 32,
        ("f", "c"):  lambda v: (v - 32) * 5 / 9,
    }
    key = (from_unit.lower(), to_unit.lower())
    if key not in conversions:
        return f"Unsupported conversion: {from_unit} to {to_unit}"
    return f"{round(conversions[key](value), 4)} {to_unit}"

model = ChatOpenAI(model="gpt-4o-mini", temperature=0)
agent = create_react_agent(model, tools=[calculator, unit_converter])

result = agent.invoke({
    "messages": [("human",
        "What is 42 multiplied by 3.14159, and if that result "
        "is a distance in meters, how many feet is it?")]
})

print("\n--- Message Trace ---")
for msg in result["messages"]:
    mtype = type(msg).__name__
    content = getattr(msg, 'content', '')
    tool_calls = getattr(msg, 'tool_calls', [])
    name = getattr(msg, 'name', '')
    if tool_calls:
        print(f"[{mtype}] Tool calls: {[tc['name'] for tc in tool_calls]}")
    elif name:
        print(f"[{mtype}:{name}] {content}")
    else:
        print(f"[{mtype}] {str(content)[:120]}")

Human-in-the-Loop Email Sender
interrupt_before · Command(resume) · graph pausing · multi-step oversight
▾

Build a customer service assistant that drafts and sends email responses. Graph topology: classify_intent → draft_email → [INTERRUPT] → human_reviews → route to either send_email or revise_email.

State fields: customer_message: str, draft_response: str, human_decision: str, final_response: str. Use interrupt_before=["send_email"] in compile() so a human must approve every outbound email. When the graph pauses, print the draft, prompt the human for "approve" or "revise", and call graph.invoke(Command(resume={"decision": user_input}), config=config) to resume.

What You'll Learn

  • interrupt_before and interrupt_after in compile()
  • How graph execution pauses and what "paused" state looks like
  • Command(resume=value) to inject data into a resumed graph
  • Reading state from a paused graph with get_state(config)
  • Designing state schemas for multi-step human oversight workflows
  • The difference between interrupting before vs. after a node

Success Criteria

  • Graph pauses before send_email on every invocation
  • Draft is printed to console for human review
  • "approve" routes to send_email and logs "Email sent!"
  • "revise" routes to revise_email which updates the draft and loops
  • get_state(config).next shows the pending node name when paused

Extension Challenge

When the human types "revise", prompt them for specific feedback (e.g., "Make it more concise"). Store this in state as revision_notes: str. Route back to draft_email with the revision notes injected into the LLM prompt. Track revision count as revision_count: int and abort after 3 revisions with a "max revisions reached" message.

from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import Command, interrupt

class EmailState(TypedDict):
    customer_message: str
    draft_response: str
    human_decision: str
    final_response: str

def classify_intent(state: EmailState) -> dict:
    return {}  # In production: LLM classifies intent

def draft_email(state: EmailState) -> dict:
    # In production: LLM generates a draft. Mocked here.
    draft = (
        f"Dear Customer,\n\n"
        f"Thank you for contacting us regarding: '{state['customer_message']}'\n\n"
        f"We are looking into this and will resolve it within 24 hours.\n\n"
        f"Best regards,\nSupport Team"
    )
    return {"draft_response": draft}

def human_reviews(state: EmailState) -> dict:
    decision = interrupt({
        "prompt": "Review the draft below. Type 'approve' or 'revise':",
        "draft": state["draft_response"]
    })
    return {"human_decision": decision.get("decision", "approve")}

def route_after_review(state: EmailState) -> str:
    return "send_email" if state["human_decision"] == "approve" else "revise_email"

def send_email(state: EmailState) -> dict:
    print(f"\n--- EMAIL SENT ---\n{state['draft_response']}\n------------------")
    return {"final_response": state["draft_response"]}

def revise_email(state: EmailState) -> dict:
    revised = state["draft_response"] + "\n\n[P.S. Revised for clarity.]"
    return {"draft_response": revised}

builder = StateGraph(EmailState)
builder.add_node("classify_intent", classify_intent)
builder.add_node("draft_email", draft_email)
builder.add_node("human_reviews", human_reviews)
builder.add_node("send_email", send_email)
builder.add_node("revise_email", revise_email)
builder.add_edge(START, "classify_intent")
builder.add_edge("classify_intent", "draft_email")
builder.add_edge("draft_email", "human_reviews")
builder.add_conditional_edges("human_reviews", route_after_review)
builder.add_edge("send_email", END)
builder.add_edge("revise_email", "draft_email")

memory = MemorySaver()
graph = builder.compile(checkpointer=memory)
config = {"configurable": {"thread_id": "email-session-001"}}

initial = {
    "customer_message": "My order #1234 has not arrived.",
    "draft_response": "", "human_decision": "", "final_response": ""
}
# First call: runs until interrupt
graph.invoke(initial, config=config)
# Resume with approval
graph.invoke(Command(resume={"decision": "approve"}), config=config)

Parallel Research Aggregator
Send API · Annotated reducers · fan-out / fan-in pattern
▾

Build a research agent that receives a topic string and fans out to 3 parallel research branches: research_historical, research_technical, and research_recent. Each branch returns a finding string for its domain.

The critical design decision: research_results in state must use Annotated[list[str], operator.add] as its type. This tells LangGraph to merge results from all parallel branches via list concatenation rather than last-write-wins. Use the Send API in a fan_out function to dispatch all three branches simultaneously. A final synthesize node receives the merged list and produces a summary.

What You'll Learn

  • Send(node_name, state_override) for parallel dispatch
  • Annotated[list[str], operator.add] — reducer-based state merging
  • The fan-out / fan-in execution pattern and when to use it
  • Why default state (last-write-wins) is wrong for parallel execution
  • How LangGraph waits for all branches before proceeding to the join node
  • Timing parallel vs. sequential to measure actual speedup

Success Criteria

  • All 3 branches execute (add a print in each to confirm)
  • research_results contains exactly 3 strings, one per branch
  • Without operator.add reducer, only 1 result survives — demonstrate this
  • Synthesizer output references content from all 3 branches
  • You can explain why operator.add is the correct reducer here

Extension Challenge

Add import time; time.sleep(2) to the research_recent branch node. Measure total wall-clock time with time.time() before and after invoke(). Confirm total time is approximately 2 seconds (not 6), proving branches run in parallel and the slow branch does not block the others.

import operator
from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send

class ResearchState(TypedDict):
    topic: str
    research_results: Annotated[list[str], operator.add]
    summary: str

class BranchState(TypedDict):
    topic: str
    branch: str

def fan_out(state: ResearchState):
    """Dispatch 3 parallel research branches using the Send API."""
    return [
        Send("research_branch", {"topic": state["topic"], "branch": "historical"}),
        Send("research_branch", {"topic": state["topic"], "branch": "technical"}),
        Send("research_branch", {"topic": state["topic"], "branch": "recent"}),
    ]

def research_branch(state: BranchState) -> dict:
    topic, branch = state["topic"], state["branch"]
    findings = {
        "historical": f"[Historical] The origins of {topic} trace back to...",
        "technical":  f"[Technical] {topic} operates on the principle of...",
        "recent":     f"[Recent] Current developments in {topic} include...",
    }
    print(f"  Branch '{branch}' completed for topic '{topic}'")
    return {"research_results": [findings[branch]]}

def synthesize(state: ResearchState) -> dict:
    combined = "\n".join(state["research_results"])
    return {"summary": f"Research on '{state['topic']}':\n{combined}"}

builder = StateGraph(ResearchState)
builder.add_node("research_branch", research_branch)
builder.add_node("synthesize", synthesize)
builder.add_conditional_edges(START, fan_out, ["research_branch"])
builder.add_edge("research_branch", "synthesize")
builder.add_edge("synthesize", END)

graph = builder.compile()
result = graph.invoke({
    "topic": "quantum computing",
    "research_results": [],
    "summary": ""
})
print(result["summary"])

The Supervisor Agent
Supervisor pattern · specialist routing · multi-agent state design
▾

Build a supervisor with 2 specialists: a math_agent (calculator tools) and a writing_agent (text formatting tools: to_uppercase, count_words, reverse_text). The supervisor node inspects the task and outputs a routing decision: "math_agent", "writing_agent", or "FINISH".

State: task: str, current_agent: str, completed_work: Annotated[list[str], operator.add], final_answer: str. When a specialist finishes, it appends its result to completed_work and routes back to the supervisor. The supervisor decides whether to delegate again or synthesize a final_answer and stop.

What You'll Learn

  • How the supervisor's routing decision passes through state
  • State that accumulates work across multiple specialist calls
  • The collect-and-re-route pattern for multi-turn supervisor loops
  • When to use supervisor vs. parallel fan-out (sequential dependency vs. independence)
  • Building a termination condition that isn't just a loop counter

Success Criteria

  • "Calculate 15 * 7" routes to math_agent, not writing_agent
  • "Reverse the text 'hello world'" routes to writing_agent
  • Compound task requiring both agents: both called sequentially
  • completed_work accumulates results from all agents called
  • Supervisor reaches FINISH and produces a coherent final_answer

▾

▾

Success Criteria

  • Thread IDs follow the naming convention, enforced by a validator function
  • TTL archival log fires correctly for threads with stale checkpoints
  • If Postgres is unreachable, the app logs clearly and exits gracefully
  • Database tables created correctly by PostgresSaver.setup()

from langgraph.checkpoint.postgres import PostgresSaver
import psycopg
from datetime import datetime, timezone, timedelta

CONN_STRING = "postgresql://postgres:dev@localhost:5432/langgraph"

def make_thread_id(user_id: str, session_id: str) -> str:
    """Construct and validate a thread ID in the required format."""
    if not user_id or not session_id:
        raise ValueError("Both user_id and session_id must be non-empty strings")
    return f"user_{user_id}_session_{session_id}"

def archive_idle_threads(graph, known_thread_ids: list[str],
                         idle_hours: int = 1) -> list[str]:
    """Log threads that have been idle beyond the threshold."""
    now = datetime.now(timezone.utc)
    cutoff = timedelta(hours=idle_hours)
    archived = []
    for thread_id in known_thread_ids:
        config = {"configurable": {"thread_id": thread_id}}
        state = graph.get_state(config)
        if state and state.metadata and "created_at" in state.metadata:
            ts = datetime.fromisoformat(state.metadata["created_at"])
            if now - ts > cutoff:
                print(f"[ARCHIVE] thread {thread_id} idle since {ts.isoformat()}")
                archived.append(thread_id)
    return archived

Streaming Agent UI
astream_events · FastAPI SSE · token-by-token streaming · real-time UI
▾

from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import json

app = FastAPI()

async def agent_event_stream(user_message: str, thread_id: str):
    config = {"configurable": {"thread_id": thread_id}}
    async for event in graph.astream_events(
        {"messages": [("human", user_message)]},
        config=config,
        version="v2"
    ):
        event_type = event["event"]

        if event_type == "on_chain_start":
            node = event.get("name", "unknown")
            yield f"event: node_start\ndata: {json.dumps({'node': node})}\n\n"

        elif event_type == "on_chat_model_stream":
            chunk = event["data"].get("chunk")
            if chunk and chunk.content:
                yield f"event: token\ndata: {json.dumps({'token': chunk.content})}\n\n"

        elif event_type == "on_chain_end" and event.get("name") == "LangGraph":
            yield f"event: done\ndata: {json.dumps({'status': 'complete'})}\n\n"

@app.post("/chat")
async def chat_endpoint(body: dict):
    return StreamingResponse(
        agent_event_stream(
            body["message"],
            body.get("thread_id", "default-thread")
        ),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",
            "Connection": "keep-alive"
        }
    )

Production Incident Simulation
Resilience engineering · failure mode detection · recovery mechanisms · runbooks
▾

Take a working 4-node agent graph and deliberately introduce 3 failure modes. Failure 1: an external API mock that raises a ConnectionError on every 3rd call (track a call counter in a closure). Failure 2: a node that occasionally returns a string value when state expects an integer — triggered on specific input patterns.   • Structured logging that makes failures diagnosable from logs alone
  • Runbook discipline: what on-call engineers need vs. what developers assume they know

  • Failure 1: detected and recovered, total invocation succeeds on retry within 3 attempts
  • Failure 2: type error caught before state corruption, logged with field name and received vs. expected types
  • Failure 3: infinite loop terminated before recursion_limit, informative message produced

import time
import random
from functools import wraps

# --- Failure Mode 1: Flaky external API ---
_api_call_count = 0

def flaky_external_api(query: str) -> str:
    global _api_call_count
    _api_call_count += 1
    if _api_call_count % 3 == 0:
        raise ConnectionError(f"Simulated timeout on call #{_api_call_count}")
    return f"Result for: {query}"

def with_retry(max_retries: int = 3, base_delay: float = 0.5):
    """Retry decorator with exponential backoff and jitter."""
    def decorator(fn):
        @wraps(fn)
        def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return fn(*args, **kwargs)
                except ConnectionError as exc:
                    if attempt == max_retries - 1:
                        raise
                    delay = base_delay * (2 ** attempt) + random.uniform(0, 0.1)
                    print(f"[RETRY] Attempt {attempt + 1} failed ({exc}). Retrying in {delay:.2f}s")
                    time.sleep(delay)
        return wrapper
    return decorator

# --- Failure Mode 2: Type mismatch in node output ---
def validate_node_output(output: dict, expected_types: dict) -> dict:
    """Validate and safely coerce node output fields."""
    validated = dict(output)
    for field, expected_type in expected_types.items():
        if field in output:
            value = output[field]
            if not isinstance(value, expected_type):
                print(
                    f"[TYPE_MISMATCH] Field '{field}': "
                    f"expected {expected_type.__name__}, "
                    f"got {type(value).__name__} (value={value!r}). "
                    f"Coercing to {expected_type.__name__}."
                )
                try:
                    validated[field] = expected_type(value)
                except (ValueError, TypeError):
                    validated[field] = expected_type()
    return validated

# --- Failure Mode 3: Loop guard ---
def make_loop_safe_router(base_router, max_iterations: int = 10):
    """Wrap a router function to detect and break runaway loops."""
    def safe_router(state: dict) -> str:
        loop_count = state.get("_loop_count", 0)
        if loop_count >= max_iterations:
            print(
                f"[LOOP_GUARD] Loop count {loop_count} reached "
                f"max {max_iterations}. Forcing termination."
            )
            return "__end__"
        return base_router(state)
    return safe_router

How to Get the Most From These Exercises

Work through them in order. Each exercise builds vocabulary and muscle memory needed for the next. Skipping Exercise 2 (checkpointing) makes Exercise 9 (production checkpointer) confusing. Skipping Exercise 5 (parallel) makes Exercise 13 (A/B testing) opaque.

The extension challenges are not optional for senior roles. Engineers at companies running LangGraph in production will ask about exactly the patterns covered in extensions: SQLite-to-Postgres migration, multi-judge panels, cross-session memory, and resilience engineering. These are the questions that separate people who have read about LangGraph from people who have built real systems with it.

The goal is not working code. The goal is being able to explain every line of your working code.
