Debugging Kubernetes Systems

Systematic approaches to diagnosing and resolving production issues in containerized environments.

Debugging Mindset

Kubernetes adds layers of abstraction, making debugging harder. You need systematic approaches to peel back those layers and find root causes.

Information Sources

Learn what to check, in what order, and how to interpret the signals K8s gives you. Events, logs, metrics, and resource status are your clues.

Production Realities

Production debugging is different from tutorials. You need quick, safe techniques that work under pressure without causing more problems.

1. The Debugging Framework: Symptom → Diagnosis

Step 1: Identify the Symptom

What's actually broken? Be precise.

Bad: "The system is slow"

Good: "API pod takes 8 seconds to respond, normally 200ms"

Bad: "Users can't access the app"

Good: "Ingress returns 502 when accessing /api/users endpoint"

→

Step 2: Check Object Status

What does K8s tell you about the broken component?

kubectl get pods -n <namespace> kubectl get svc -n <namespace> kubectl get ingress -n <namespace> kubectl get nodes
→

Step 3: Describe the Object

Get detailed status and recent events.

kubectl describe pod/<pod-name> -n <namespace> kubectl describe svc/<service-name> -n <namespace> kubectl describe node/<node-name>
→

Step 4: Check Logs

What are the containers saying?

kubectl logs <pod-name> -n <namespace> kubectl logs <pod-name> -c <container-name> -n <namespace> kubectl logs -f <pod-name> -n <namespace> # stream kubectl logs --previous <pod-name> # crashed container
→

Step 5: Events Timeline

What happened, when?

kubectl get events -n <namespace> --sort-by=.metadata.creationTimestamp kubectl get events --all-namespaces --sort-by=.metadata.creationTimestamp
→

Step 6: Deep Dive (If Needed)

Exec into containers, check networking, resource usage

kubectl exec -it <pod-name> -- /bin/sh kubectl top pods -n <namespace> kubectl top nodes
→

Step 7: Fix and Verify

Apply fix, monitor recovery, document root cause.

2. Common Error Patterns and Solutions

CrashLoopBackOff

What It Means: Container starts, crashes, K8s restarts it, crashes again, repeats. K8s backs off between retries.

What I Check First, Second, Third

  1. First: kubectl logs <pod> - See why it's crashing
  2. Second: kubectl logs --previous <pod> - Check previous crash logs
  3. Third: kubectl describe pod <pod> - Check events, look for "Back-off restarting failed container"

Common Causes (90% of cases)

1. Application Error (40%)

Symptom: Logs show application exceptions, errors, or panic

Examples:

  • Unhandled exceptions in code
  • Missing environment variables causing app to fail startup
  • Database connection failures
  • Missing required config files

Fix: Fix application code or configuration

2. Missing Dependencies (25%)

Symptom: "command not found", "cannot open shared object file"

Examples:

  • Required binaries not in image
  • Missing shared libraries
  • Wrong base image (e.g., alpine vs debian)

Fix: Rebuild image with dependencies, use correct base image

3. Liveness Probe Failures (15%)

Symptom: Container starts but liveness probe fails repeatedly

Examples:

  • Probe endpoint doesn't exist or returns wrong status code
  • Probe timeout too short (app needs more time)
  • App runs but isn't actually ready (still initializing)

Fix: Fix probe configuration, add readiness/liveness endpoints

livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
  failureThreshold: 3
4. Configuration Errors (10%)

Symptom: App fails to parse config or validate required settings

Examples:

  • Invalid ConfigMap data
  • Missing required environment variables
  • Malformed YAML in mounted config files

Fix: Validate ConfigMaps, Secrets, and environment variable references

5. Permission Issues (5%)

Symptom: "Permission denied" errors in logs

Examples:

  • Container runs as non-root but needs root
  • Volume mount permissions incorrect
  • SecurityContext too restrictive

Fix: Adjust securityContext, runAsUser, fsGroup

securityContext:
  runAsUser: 1000
  fsGroup: 1000
  runAsNonRoot: true
6. Out of Memory (5%)

Symptom: Pod killed, events show "OOMKilled" (see OOMKilled section below)

Fix: Increase memory limits, fix memory leaks

CrashLoopBackOff Debugging Checklist

  • ✅ Checked container logs (current and previous)
  • ✅ Verified image exists and is accessible (ImagePullBackOff? check next)
  • ✅ Checked liveness/readiness probe configuration
  • ✅ Verified ConfigMaps and Secrets are mounted correctly
  • ✅ Validated environment variables are present
  • ✅ Checked resource limits (memory/CPU)
  • ✅ Tried running container locally (docker run) to isolate
  • ✅ Checked application dependencies and base image
  • ✅ Reviewed recent changes (deployment, code, config)

Production Debugging Strategy

Quick Fix: If you can't find the root cause quickly and it's blocking production, you can temporarily disable the liveness probe to keep the pod running while you investigate:

livenessProbe:
  enabled: false  # TEMPORARY ONLY!

⚠️ Warning: This is emergency-only! The app might be dead but K8s won't restart it. Document in ticket, fix properly within hours.

ImagePullBackOff / ErrImagePull

What It Means: K8s can't pull the container image from the registry.

What I Check First, Second, Third

  1. First: kubectl describe pod <pod> - Look for "Failed to pull image" message
  2. Second: kubectl get pod <pod> -o jsonpath='{.spec.containers[*].image}' - Verify image name
  3. Third: Check imagePullSecrets and registry access

Common Causes

1. Image Doesn't Exist (40%)

Symptom: "manifest not found", "not found"

Examples:

  • Wrong image name or tag
  • Typo in registry URL
  • Tag was deleted or overridden

Fix: Verify image exists: docker pull <image> locally

# Check image exists
docker pull myregistry.com/myapp:v1.2.3

# Verify tag
kubectl get pod mypod -o jsonpath='{.spec.containers[0].image}'
2. Authentication Issues (35%)

Symptom: "unauthorized", "denied", "no basic creds"

Examples:

  • Missing imagePullSecret for private registry
  • Secret has wrong credentials
  • Secret expired
  • Secret not attached to service account

Fix: Create/update imagePullSecret

# Create secret
kubectl create secret docker-registry my-registry-secret \
  --docker-server=registry.example.com \
  --docker-username=myuser \
  --docker-password=mypassword \
  --docker-email=myemail@example.com

# Attach to pod/serviceaccount
kubectl patch serviceaccount default -p '{"imagePullSecrets": [{"name": "my-registry-secret"}]}'
3. Network Issues (15%)

Symptom: "connection refused", "timeout", "no route to host"

Examples:

  • Network policies blocking registry access
  • Firewall blocking registry
  • DNS issues resolving registry hostname
  • Cluster nodes can't reach external registry

Fix: Check network policies, firewall rules, DNS

4. Registry Rate Limiting (5%)

Symptom: "rate limit exceeded", "too many requests"

Examples:

  • Docker Hub rate limits (100 pulls/6hr for anonymous)
  • Private registry limits

Fix: Authenticate (higher limits), use mirror, cache images locally

5. TLS/Certificate Issues (5%)

Symptom: "x509: certificate signed by unknown authority"

Examples:

  • Self-signed registry certificate

Fix: Add registry CA to containerd/Docker configuration on nodes

ImagePullBackOff Debugging Checklist

  • ✅ Verified image name and tag are correct
  • ✅ Can pull image manually with docker/crictl
  • ✅ Checked imagePullSecret exists and has correct credentials
  • ✅ Verified secret is attached to pod's service account
  • ✅ Checked network policies allow registry access
  • ✅ Verified nodes can reach registry (run curl/wget from node)
  • ✅ Checked for rate limiting messages
  • ✅ Tried pulling from different registry as test

OOMKilled (Out of Memory)

What It Means: Container exceeded its memory limit and was killed by the kernel.

Status: Pod shows "OOMKilled" in status, terminated with exit code 137

What I Check First, Second, Third

  1. First: kubectl describe pod <pod> - Look for "OOMKilled", "Last State: Terminated"
  2. Second: kubectl get pod <pod> -o jsonpath='{.spec.containers[*].resources}' - Check limits
  3. Third: kubectl top pod <pod> --containers - See current usage (if pod still running)

Common Causes

1. Memory Limit Too Low (50%)

Symptom: App needs more memory than allocated

Examples:

  • App legitimately needs 2GB, limit is 512MB
  • Unexpected user load or data size
  • Concurrent task processing

Fix: Increase memory limit

resources:
  requests:
    memory: "512Mi"
  limits:
    memory: "2Gi"  # Increased from 512Mi
2. Memory Leak (30%)

Symptom: Memory usage grows over time until OOM

Examples:

  • Unreleased connections or buffers
  • Cached data growing without bounds
  • Background goroutines/threads not cleaned up
  • Library bugs

Fix: Profile memory usage, find leak, fix code

# Enable memory profiling
kubectl exec -it <pod> -- go tool pprof http://localhost:6060/debug/pprof/heap

# Or use heap profiler in your language
3. No Memory Limit Set (10%)

Symptom: Pod can use unlimited memory, causes node OOM

Why It Happens: When pods have no limits, they can consume all node memory, causing kernel to kill random processes

Fix: ALWAYS set memory limits in production

resources:
  limits:
    memory: "1Gi"
  requests:
    memory: "512Mi"
4. Memory Spike (5%)

Symptom: Normal operation, then sudden spike (processing large file, etc.)

Examples:

  • Processing large file/document
  • Expensive operation loaded into memory
  • Caching large dataset

Fix: Increase limits, or process in smaller chunks

5. Java/Golang/Python Memory Issues (5%)

Language-Specific Issues:

  • Java: Heap size vs container limit mismatch. JVM sees node memory, not container limit
  • Golang: GC not releasing memory fast enough, memory fragmentation
  • Python: Object accumulation, reference cycles

Fix: Tune runtime memory settings

# Java: Set heap within container limit
env:
  - name: JAVA_OPTS
    value: "-Xms512m -Xmx1g"  # Within container limit

# Golang: Tune GOGC
env:
  - name: GOGC
    value: "100"  # GC when heap grows 100%

Production Strategy

Immediate: Increase limit to get service running, investigate leak

Short-term: Add memory monitoring and alerting at 80% of limit

Long-term: Profile application, find root cause, optimize or fix leak

OOMKilled Debugging Checklist

  • ✅ Verified memory limit is appropriate for workload
  • ✅ Checked application logs for clues before OOM
  • ✅ Reviewed recent code changes for potential leaks
  • ✅ Checked if it's gradual (leak) or sudden (spike)
  • ✅ Compared memory usage across similar pods
  • ✅ Checked for Java/Go/Python-specific memory settings
  • ✅ Verified memory requests/limits ratio (typically 1:2)
  • ✅ Considered Vertical Pod Autoscaler for automatic tuning

Pending (Pod Stuck in Pending State)

What It Means: Pod is scheduled but not running. K8s can't place it on any node.

What I Check First, Second, Third

  1. First: kubectl describe pod <pod> - Look at "Events" section (most important!)
  2. Second: kubectl get nodes - Check if nodes are Ready and have capacity
  3. Third: kubectl describe node <node> - Check node capacity, allocatable, taints

Common Causes

1. Insufficient Resources (40%)

Symptom: Events show "0/3 nodes are available: insufficient CPU/Memory"

Examples:

  • Cluster doesn't have enough CPU/Memory
  • Nodes are fully allocated
  • Pod requests too much resource

Fix: Add nodes, reduce requests, delete unused pods

# Check node capacity
kubectl top nodes
kubectl describe nodes | grep -A 5 "Allocated resources"

# Check what's consuming resources
kubectl top pods --all-namespaces --sort-by=memory
2. Node Selector / Affinity Mismatch (25%)

Symptom: Events show "0/3 nodes match pod node selector/affinity"

Examples:

  • Pod requires label "node=production" but no nodes have it
  • Anti-affinity can't schedule (all nodes already have pod)
  • Pod affinity requires specific pod that doesn't exist

Fix: Fix node labels, adjust affinity rules

# Check node labels
kubectl get nodes --show-labels

# Check pod selector
kubectl get pod mypod -o jsonpath='{.spec.nodeSelector}'
kubectl get pod mypod -o jsonpath='{.spec.affinity}'
3. Taints and Tolerations (15%)

Symptom: Events show "node(s) had taint that pod didn't tolerate"

Examples:

  • Nodes tainted with "NoSchedule" for dedicated workloads
  • Control plane nodes tainted, pod tries to schedule there
  • Pod doesn't tolerate required taint

Fix: Add toleration to pod, or remove taint from node

# Check node taints
kubectl describe nodes | grep Taints

# Add toleration to pod
tolerations:
- key: "dedicated"
  operator: "Equal"
  value: "production"
  effect: "NoSchedule"
4. Persistent Volume Issues (10%)

Symptom: Events show "pod has unbound immediate PersistentVolumeClaims"

Examples:

  • PVC not bound to any PV
  • Storage class doesn't exist
  • No available PVs match PVC requirements

Fix: Check PVC status, create PV, fix storage class

# Check PVC
kubectl get pvc
kubectl describe pvc myclaim

# Check available PVs
kubectl get pv
kubectl describe pv
5. Not Ready Nodes (5%)

Symptom: Nodes are NotReady or Unknown

Examples:

  • Node problems (disk full, kubelet down)
  • Network issues
  • Node being drained

Fix: Fix node issues

6. Pod Disruption Budget (5%)

Symptom: Can't evict pods during voluntary disruptions

Examples:

  • Node drain blocked by PDB
  • Can't achieve required availability during update

Fix: Adjust PDB, or ensure enough replicas

Pending Pod Debugging Checklist

  • ✅ Checked Events section of kubectl describe pod (CRITICAL!)
  • ✅ Verified all nodes are Ready
  • ✅ Checked cluster has sufficient CPU/memory
  • ✅ Verified node selector matches actual node labels
  • ✅ Checked affinity rules can be satisfied
  • ✅ Verified taints/tolerations match
  • ✅ Checked PVC status (if using volumes)
  • ✅ Verified pod doesn't exceed node capacity
  • ✅ Checked for scheduler errors (kubectl get events)

Network Unreachable / Connection Refused

What It Means: Can't reach pods, services, or external resources from within cluster.

What I Check First, Second, Third

  1. First: kubectl get svc - Verify service exists and has endpoints
  2. Second: kubectl get endpoints <service> - Check if service has pod endpoints
  3. Third: Test connectivity from pod: kubectl exec -it <pod> -- curl <service>

Common Causes

1. Service Has No Endpoints (30%)

Symptom: Service exists but endpoints list is empty

Examples:

  • Pod selector doesn't match pod labels
  • Pods aren't Running (they're Pending/CrashLoopBackOff)
  • Wrong port in service spec

Fix: Fix service selector or pod labels

# Check service selector
kubectl get svc myservice -o jsonpath='{.spec.selector}'

# Check pod labels
kubectl get pods --show-labels

# They must match!
2. Wrong Port (25%)

Symptom: Connection refused, timeout

Examples:

  • Service targets port 8080, container listens on 80
  • Container port name doesn't match service port name
  • Wrong protocol (TCP vs UDP)

Fix: Match service port to container port

# Service definition
ports:
- name: http
  port: 80        # Service port
  targetPort: 8080  # Container port (must match!)
3. DNS Issues (20%)

Symptom: Can't resolve service names

Examples:

  • CoreDNS not running
  • Wrong service name (case-sensitive!)
  • Wrong namespace (need FQDN for cross-namespace)
  • Network policies blocking DNS

Fix: Check DNS resolution, use correct FQDN

# Test DNS from pod
kubectl exec -it mypod -- nslookup myservice
kubectl exec -it mypod -- nslookup myservice.mynamespace.svc.cluster.local

# Check CoreDNS
kubectl get pods -n kube-system -l k8s-app=kube-dns
4. Network Policies (10%)

Symptom: Connections blocked, timeout, connection refused

Examples:

  • NetworkPolicy denies traffic between namespaces
  • Pod can't reach service due to policy rules
  • Ingress/egress rules too restrictive

Fix: Update network policies

# Check network policies
kubectl get networkpolicies --all-namespaces

# Describe to see rules
kubectl describe networkpolicy mypolicy
5. Service Not Ready (10%)

Symptom: Can reach service IP but not pods

Examples:

  • Pods not passing readiness probes
  • Service has no ready endpoints

Fix: Check readiness probes, pod status

6. External Connectivity Issues (5%)

Symptom: Pods can't reach external APIs/internet

Examples:

  • Nodes have no internet access
  • Firewall blocks egress traffic
  • Proxy configuration needed
  • VPC/routing issues

Fix: Fix network routing, configure proxy

# Test from node
curl https://google.com

# Check from pod
kubectl exec -it mypod -- curl https://google.com

# Check for proxy env vars
kubectl exec mypod -- env | grep -i proxy

Network Debugging Checklist

  • ✅ Checked service has endpoints
  • ✅ Verified service selector matches pod labels
  • ✅ Checked service port matches container port
  • ✅ Tested DNS resolution from pod
  • ✅ Checked network policies aren't blocking traffic
  • ✅ Verified pods are Ready (readiness probe passing)
  • ✅ Tested connectivity between pods directly (by IP)
  • ✅ Checked for CNI/networking plugin issues
  • ✅ Verified external connectivity (if needed)

Quick Network Tests

# Test service DNS
kubectl exec -it mypod -- curl http://myservice.mynamespace.svc.cluster.local

# Test service IP
kubectl exec -it mypod -- curl http://10.96.0.10

# Test pod IP directly
kubectl get pods -o wide  # Get pod IP
kubectl exec -it mypod -- curl http://10.244.1.5

# Check if pod can reach itself
kubectl exec -it mypod -- curl http://localhost:8080

3. Debugging by Component

Pod Issues

Quick Diagnosis Flow

Step 1: kubectl get pods - Check pod phase/status

Step 2: Based on phase, check specific issues:

  • Pending: kubectl describe pod - Check Events, scheduler
  • CrashLoopBackOff: kubectl logs - Check app errors
  • ImagePullBackOff: kubectl describe pod - Check image issues
  • Running but not working: kubectl logs + kubectl exec - Check app behavior

Step 3: kubectl describe pod - Always check Events section!

Step 4: kubectl get events --sort-by=.metadata.creationTimestamp - See recent events

Most Common Pod Issues (Frequency Order)

  1. Application crash (logs show exception) - Fix app
  2. Out of memory (OOMKilled) - Increase limit or fix leak
  3. Image pull errors - Fix image name, credentials, network
  4. Missing environment variables/config - Add env vars, ConfigMaps
  5. Readiness/liveness probe failures - Fix probes or app
  6. Can't be scheduled (Pending) - Check resources, selectors, taints
  7. Volume mount issues - Check PVC, permissions

Essential Pod Debugging Commands

# Status overview
kubectl get pods
kubectl get pods -o wide  # Shows node and IP

# Detailed info
kubectl describe pod mypod
kubectl get pod mypod -o yaml  # Full spec

# Logs
kubectl logs mypod
kubectl logs mypod --previous  # If crashed
kubectl logs mypod -c mycontainer  # Multi-container
kubectl logs -f mypod  # Follow logs

# Exec into pod
kubectl exec -it mypod -- /bin/sh
kubectl exec -it mypod -- /bin/bash

# Check resources
kubectl top pod mypod
kubectl top pod mypod --containers

# Get pod IP
kubectl get pod mypod -o jsonpath='{.status.podIP}'

# Check pod events
kubectl get events --field-selector involvedObject.name=mypod

Service Issues

Quick Diagnosis Flow

Step 1: kubectl get svc - Check service exists and type

Step 2: kubectl get endpoints <service> - CRITICAL! Check if endpoints exist

Step 3: If no endpoints:

  • Check service selector: kubectl get svc <service> -o jsonpath='{.spec.selector}'
  • Check pod labels: kubectl get pods --show-labels
  • Verify they match!

Step 4: If endpoints exist but can't connect:

  • Test pod IP directly: curl <pod-ip>:<port>
  • Test service IP: curl <service-ip>:<port>
  • Test service DNS: curl <service-name>.<namespace>.svc.cluster.local

Most Common Service Issues

  1. Service has no endpoints (selector mismatch or pods not ready)
  2. Wrong port (service port doesn't match container port)
  3. Wrong selector (label typo, wrong namespace)
  4. DNS resolution fails (wrong service name/namespace)
  5. NetworkPolicy blocking traffic
  6. LoadBalancer not provisioned (cloud provider issues)

Essential Service Debugging Commands

# List services
kubectl get svc
kubectl get svc -o wide

# Service details
kubectl describe svc myservice
kubectl get svc myservice -o yaml

# Check endpoints (CRITICAL!)
kubectl get endpoints myservice
kubectl describe endpoints myservice

# Get service IP
kubectl get svc myservice -o jsonpath='{.spec.clusterIP}'

# Get service ports
kubectl get svc myservice -o jsonpath='{.spec.ports[*]}'

# Test from pod
kubectl exec -it mypod -- curl http://myservice.mynamespace.svc.cluster.local
kubectl exec -it mypod -- curl http://<service-ip>:<port>

Ingress Issues

Quick Diagnosis Flow

Step 1: kubectl get ingress - Check ingress exists and has address

Step 2: kubectl describe ingress myingress - Check backend services

Step 3: Check ingress controller logs:

  • NGINX: kubectl logs -n ingress-nginx <controller-pod>
  • Traefik: kubectl logs -n traefik <controller-pod>

Step 4: Verify service exists and is reachable (see Service debugging above)

Step 5: Test DNS: nslookup myingress.example.com

Step 6: Test HTTP: curl -v http://myingress.example.com

Most Common Ingress Issues

  1. Ingress has no address (controller not provisioned/ready)
  2. Backend service not found (wrong service name/namespace/port)
  3. Service has no endpoints (see service debugging)
  4. TLS/SSL certificate issues
  5. Path/rule mismatch (wrong path in ingress spec)
  6. DNS not pointing to ingress IP
  7. Annotations misconfigured (especially for specific controllers)

Essential Ingress Debugging Commands

# List ingresses
kubectl get ingress
kubectl get ingress -o wide

# Ingress details
kubectl describe ingress myingress
kubectl get ingress myingress -o yaml

# Get ingress address
kubectl get ingress myingress -o jsonpath='{.status.loadBalancer.ingress[0].ip}'

# Ingress controller logs (NGINX example)
kubectl logs -n ingress-nginx -l app.kubernetes.io/name=ingress-nginx

# Test DNS
nslookup myapp.example.com
dig myapp.example.com

# Test HTTP
curl -v http://myapp.example.com
curl -v https://myapp.example.com
curl -v http://myapp.example.com/api/users

# Test with headers
curl -v -H "Host: myapp.example.com" http://<ingress-ip>

Quick Ingress Health Check

# 1. Check ingress has address
kubectl get ingress myingress

# 2. Check backend services exist
kubectl get svc
kubectl describe ingress myingress | grep -A 5 "Backend"

# 3. Check service endpoints
kubectl get endpoints <backend-service>

# 4. Check ingress controller is running
kubectl get pods -n ingress-nginx

# 5. Check controller logs for errors
kubectl logs -n ingress-nginx <controller-pod> --tail=50

DNS Issues

Quick Diagnosis Flow

Step 1: Check CoreDNS is running: kubectl get pods -n kube-system -l k8s-app=kube-dns

Step 2: Test DNS from pod: kubectl exec -it mypod -- nslookup kubernetes.default

Step 3: Test service DNS: kubectl exec -it mypod -- nslookup myservice.mynamespace.svc.cluster.local

Step 4: Check pod DNS config: kubectl get pod mypod -o jsonpath='{.spec.dnsPolicy}'

Step 5: Check for NetworkPolicy blocking DNS (port 53 UDP/TCP)

Most Common DNS Issues

  1. CoreDNS not running (crashed, pending)
  2. Wrong service name (typo, case-sensitive)
  3. Wrong namespace (need FQDN for cross-namespace)
  4. NetworkPolicy blocking DNS traffic
  5. Custom DNSPolicy misconfigured
  6. DNS latency/timeout (slow DNS servers)

Essential DNS Debugging Commands

# Check CoreDNS pods
kubectl get pods -n kube-system -l k8s-app=kube-dns

# CoreDNS logs
kubectl logs -n kube-system -l k8s-app=kube-dns --tail=50

# Test DNS resolution
kubectl exec -it mypod -- nslookup kubernetes.default
kubectl exec -it mypod -- nslookup myservice
kubectl exec -it mypod -- nslookup myservice.mynamespace.svc.cluster.local

# Test with dig
kubectl exec -it mypod -- dig myservice.mynamespace.svc.cluster.local

# Check pod DNS config
kubectl get pod mypod -o jsonpath='{.spec.dnsPolicy}'
kubectl get pod mypod -o jsonpath='{.spec.dnsConfig}'

# Check if /etc/resolv.conf is correct
kubectl exec -it mypod -- cat /etc/resolv.conf

DNS Name Formats (Must Match Exactly!)

# Same namespace
myservice

# Different namespace
myservice.mynamespace.svc.cluster.local

# Full format (always works)
<service>.<namespace>.svc.<domain>

# Examples
# - myservice.default.svc.cluster.local
# - api.production.svc.cluster.local
# - kube-dns.kube-system.svc.cluster.local

NetworkPolicy Issues

Quick Diagnosis Flow

Step 1: Check if network policies exist: kubectl get networkpolicies

Step 2: If no policies, traffic should flow (default allow)

Step 3: If policies exist, describe them: kubectl describe networkpolicy <policy>

Step 4: Check if CNI supports NetworkPolicy (Calico, Cilium, Weave do, Flannel doesn't by default)

Step 5: Check if pod labels match policy podSelector

Step 6: Test connectivity directly (by pod IP) to bypass DNS/service issues

Most Common NetworkPolicy Issues

  1. Policy denies ingress traffic (too restrictive)
  2. Policy denies egress traffic (can't reach external APIs, DNS)
  3. Pod labels don't match policy selector
  4. Wrong namespace in policy
  5. IPBlock rules incorrect
  6. CNI doesn't support NetworkPolicy

Essential NetworkPolicy Debugging Commands

# List policies
kubectl get networkpolicies --all-namespaces

# Policy details
kubectl describe networkpolicy mypolicy
kubectl get networkpolicy mypolicy -o yaml

# Check pod labels (for policy matching)
kubectl get pods --show-labels

# Check CNI
kubectl get pods -n kube-system | grep -i cni

# Test connectivity bypassing service (direct IP)
kubectl get pods -o wide
kubectl exec -it mypod -- curl http://<target-pod-ip>:<port>

# Check if CNI supports NetworkPolicy
# Common CNIs:
# - Calico: Yes
# - Cilium: Yes
# - Weave Net: Yes
# - Flannel: No (needs Calico add-on)

Quick NetworkPolicy Test

Temporarily delete the NetworkPolicy to test if it's blocking traffic:

# Backup first
kubectl get networkpolicy mypolicy -o yaml > mypolicy-backup.yaml

# Delete
kubectl delete networkpolicy mypolicy

# Test connectivity
kubectl exec -it mypod -- curl http://target-service

# If it works, policy was blocking. Restore and fix rules:
kubectl apply -f mypolicy-backup.yaml

⚠️ Use caution in production!

4. Production Debugging Strategies

Strategy 1: The Golden 5 Commands

These 5 commands diagnose 80% of issues. Memorize them.

1. Check Pod Status

kubectl get pods -n <namespace>

What It Shows: Pod health, restarts, age

Look For: CrashLoopBackOff, Pending, ImagePullBackOff, high restart counts

2. Describe Pod (Events!)

kubectl describe pod <pod-name> -n <namespace>

What It Shows: Detailed state, recent events (MOST IMPORTANT!)

Look For: Events section at bottom - this tells you what happened

3. Check Logs

kubectl logs <pod-name> -n <namespace>

What It Shows: Container stdout/stderr

Look For: Application errors, exceptions, crash messages

4. Check Events

kubectl get events -n <namespace> --sort-by=.metadata.creationTimestamp

What It Shows: All recent events in namespace

Look For: Warnings, errors related to your resources

5. Check Resources

kubectl top pods -n <namespace>

What It Shows: CPU/Memory usage

Look For: Pods hitting limits (throttling, OOM risk)

Strategy 2: Production Safety First

Debugging in production requires caution. Don't make it worse.

Debugging Safety Principles

  • Read-Only First: Use get, describe, logs commands. They don't change anything.
  • Exec Carefully: kubectl exec can impact pod performance. Don't run heavy commands.
  • Don't Delete Pods: Unless you're sure. Deleting triggers restart (downtime).
  • Test in Staging: If possible, reproduce issue in non-production environment.
  • Backup Before Changes: kubectl get <resource> -o yaml > backup.yaml
  • Document Everything: Write down what you check, what you find, what you change.
  • Communicate: Let team know you're debugging production.
  • Monitor Impact: Watch metrics while debugging. Stop if things get worse.
  • Know Rollback: Have rollback plan before applying fixes.
  • Ask for Help: Don't be hero. Escalate if stuck or time-critical.

⚠️ Dangerous Commands - Think Twice!

  • kubectl delete pod <pod> - Causes downtime, restart
  • kubectl delete deployment <deployment> - Deletes all pods!
  • kubectl delete svc <service> - Breaks networking
  • kubectl delete namespace <namespace> - DELETES EVERYTHING in namespace!
  • kubectl delete pv <pv> - Can cause data loss!
  • kubectl apply -f with wrong file - Overwrites resources
  • kubectl edit - Easy to make mistakes, use apply instead

Pro Tip: Use --dry-run=client to see what would happen without doing it:

kubectl apply -f deployment.yaml --dry-run=client

Strategy 3: Systematic Elimination

When stuck, work through layers systematically.

The OSI Model Approach to K8s Debugging

Layer 1: Is the Resource Running?
kubectl get pods
kubectl get deployments
kubectl get services

If NO: Fix why it's not starting (Pending, CrashLoopBackOff, etc.)

If YES: Go to Layer 2

Layer 2: Are Resources Healthy?
kubectl describe pod <pod>  # Check events
kubectl logs <pod>            # Check app logs
kubectl get endpoints <svc>    # Check service has endpoints

If NOT Healthy: Fix based on errors found

If Healthy: Go to Layer 3

Layer 3: Can Components Reach Each Other?
# Test pod to pod
kubectl exec -it pod1 -- curl http://<pod2-ip>:<port>

# Test pod to service
kubectl exec -it pod1 -- curl http://<service-name>

# Test DNS
kubectl exec -it pod1 -- nslookup <service-name>

If NO: Network issue (Service, NetworkPolicy, DNS)

If YES: Go to Layer 4

Layer 4: Can External Traffic Reach Ingress?
# Check ingress address
kubectl get ingress

# Check DNS
nslookup myapp.example.com

# Test HTTP
curl -v http://myapp.example.com

If NO: Ingress, DNS, or firewall issue

If YES: Issue might be application-level

Layer 5: Is the Application Working?
# Check application logs
kubectl logs -f <pod>

# Check application metrics (if exposed)
kubectl exec -it <pod> -- curl http://localhost:8080/metrics

# Check application health endpoint
kubectl exec -it <pod> -- curl http://localhost:8080/health

If NO: Application bug, configuration issue, dependency failure

Strategy 4: When to Escalate

Know when you're in over your head. Escalating early is better than breaking things further.

Escalate Immediately If:

  • ☢️ Data Loss Risk: Issue might cause data loss (PVs, databases)
  • 🔥 Production Down: Critical service down and you can't fix quickly
  • ⏱️ Time Critical: Been debugging 30+ minutes without progress
  • 🤔 Don't Understand: Issue is outside your expertise area
  • ⚠️ Risky Fix: Only fix you can think of is risky/dangerous
  • 🔄 Recurring Issue: Same problem keeps happening, root cause unclear
  • 💥 Cascading Failure: Issue spreading to other services/nodes

Before Escalating, Gather This Info:

  • Exact symptom (what's broken, when did it start)
  • What you've checked (commands run)
  • What you found (errors, events, logs)
  • Recent changes (deployments, config changes)
  • Impact (how many users affected)
  • Screenshots/output from commands

Escalation Path:

  1. Team Slack/Chat: "Hey, anyone seen this error? [details]"
  2. Senior Engineer: Direct message with context
  3. On-Call: If critical, page on-call engineer
  4. Incident Response: If SEV1/SEV2, start incident process

Strategy 5: Debugging Time Management

Don't spend hours going in circles. Time-box your debugging.

5 Minutes: Quick Wins

  • Check pod status (kubectl get pods)
  • Check logs for obvious errors
  • Check recent events
  • Verify resource exists

If not resolved: Move to 15-minute box

15 Minutes: Deep Dive

  • Describe resource (check events thoroughly)
  • Check related resources (services, endpoints, ingress)
  • Test connectivity (curl, DNS checks)
  • Compare with working similar resources

If not resolved: Move to 30-minute box

30 Minutes: Call for Help

  • Post in team chat with details
  • Ask senior engineer for pair debugging
  • Search for similar issues online
  • Consider reproducing in staging

If not resolved: Escalate formally

60+ Minutes: Escalate

  • Formal escalation to senior/staff engineer
  • Consider rollback if recent deployment
  • If critical, start incident response process
  • Document what you've tried

5. Debugging Checklist by Component

📋 Pod Debugging Checklist

Status: kubectl get pods

Details: kubectl describe pod <pod>

Logs: kubectl logs <pod>

Previous Logs: kubectl logs --previous <pod>

Events: kubectl get events --field-selector involvedObject.name=<pod>

Resources: kubectl top pod <pod>

Config: Check env vars, ConfigMaps, Secrets mounted

Probes: Check liveness/readiness probe configuration

Image: Can pull image manually

Network: kubectl exec -it <pod> -- curl localhost:<port>

📋 Service Debugging Checklist

Exists: kubectl get svc

Endpoints: kubectl get endpoints <service> ⭐ CRITICAL!

Selector: kubectl get svc <service> -o jsonpath='{.spec.selector}'

Pod Labels: kubectl get pods --show-labels

Ports: Verify service port → container port mapping

Type: Check ClusterIP, NodePort, LoadBalancer

DNS: kubectl exec -it <pod> -- nslookup <service>

Direct IP: kubectl exec -it <pod> -- curl http://<service-ip>:<port>

Pod IPs: kubectl exec -it <pod> -- curl http://<pod-ip>:<port>

NetworkPolicy: kubectl get networkpolicies

📋 Ingress Debugging Checklist

Exists: kubectl get ingress

Address: Does ingress have IP/DNS assigned?

Backends: kubectl describe ingress <ingress>

Backend Services: Do backend services exist and have endpoints?

Controller: Is ingress controller running? kubectl get pods -n ingress-nginx

Controller Logs: kubectl logs -n ingress-nginx <controller>

DNS: Does DNS point to ingress IP?

HTTP Test: curl -v http://<ingress-address>

Path: Does request path match ingress rules?

Certificates: Check TLS config if using HTTPS

📋 DNS Debugging Checklist

CoreDNS Pods: kubectl get pods -n kube-system -l k8s-app=kube-dns

CoreDNS Logs: kubectl logs -n kube-system -l k8s-app=kube-dns

Basic DNS: kubectl exec -it <pod> -- nslookup kubernetes.default

Service DNS: kubectl exec -it <pod> -- nslookup <service>

FQDN: kubectl exec -it <pod> -- nslookup <service>.<namespace>.svc.cluster.local

Pod DNS Config: kubectl exec -it <pod> -- cat /etc/resolv.conf

NetworkPolicy: Check if blocking UDP/TCP 53

Latency: kubectl exec -it <pod> -- time nslookup <service>

📋 NetworkPolicy Debugging Checklist

Policies Exist: kubectl get networkpolicies

Policy Details: kubectl describe networkpolicy <policy>

Pod Labels: kubectl get pods --show-labels

Selector Match: Do pod labels match policy podSelector?

CNI: Does CNI support NetworkPolicy? (Calico, Cilium, Weave do)

Test Direct: kubectl exec -it <pod> -- curl http://<target-pod-ip>

Egress: Can pod reach external services? DNS?

Ingress: Can other pods reach this pod?

Isolation: Are namespaces isolated by default?

📋 Storage/PVC Debugging Checklist

PVC Status: kubectl get pvc

PV Status: kubectl get pv

PVC Details: kubectl describe pvc <pvc>

PV Details: kubectl describe pv <pv>

StorageClass: kubectl get storageclass

Bound: Is PVC bound to PV?

Capacity: Enough storage available?

Access Modes: Do access modes match? (RWO, RWX, ReadOnlyMany)

Pod Mount: Is volume mounted in pod spec correctly?

Permissions: Does pod have permission to mount? (securityContext, fsGroup)

Knowledge Check

1. Pod is in CrashLoopBackOff. What's the first command to run?

Correct: kubectl logs. Logs show why container is crashing. kubectl describe is second (check events), but logs first give actual application error.

2. Pod stuck in Pending state. Where do you find the reason?

Correct: kubectl describe pod. The Events section at the bottom tells you exactly why pod isn't scheduling (insufficient resources, taints, selector mismatch, etc.).

3. Service exists but can't connect. What's the most critical check?

Correct: kubectl get endpoints. If service has no endpoints (empty list), traffic can't reach pods. 90% of service issues are selector mismatch or pods not ready.

4. Pod terminated with exit code 137. What happened?

Correct: OOMKilled. Exit code 137 = 128 + 9 (SIGKILL). Kernel killed container for exceeding memory limit. Increase limit or fix memory leak.
