Kubernetes Patterns & Anti-Patterns

Learn proven patterns that work in production and avoid common anti-patterns that have caused real incidents. These lessons come from 15+ years of production experience across hundreds of companies.

How to Think About Patterns

Patterns aren't rules: They're starting points. Every situation is different. Use these patterns as a foundation, then adapt to your context.

The shipping port analogy: Throughout this section, we'll reference the shipping port. Patterns are like "best practices for port operations"—proven methods that work most of the time.

Key principle: Understand why a pattern works. Then you'll know when to use it, when to adapt it, and when to ignore it.

Proven Production Patterns

These patterns have been battle-tested in production at companies of all sizes. They solve real problems and introduce minimal new complexity.

Pattern 1: The Sidecar Pattern

pod-design common Low Complexity

📖 What This Pattern Solves

The problem: Your application needs functionality that isn't part of the core application logic. Examples:

  • Log collection (shipping logs to centralized logging)
  • Metrics collection (exposing metrics for Prometheus)
  • Secret synchronization (fetching secrets from Vault)
  • Proxy/ambassador (handling TLS, authentication)

Why not build it into the app?

  • Mixes concerns (app logic + infrastructure)
  • Difficult to update (need to redeploy app to update logging agent)
  • Language/framework specific (each team builds their own)

Shipping port analogy: Like having a specialized worker on each crane who handles communication with the control tower. The crane operator (main container) focuses on moving cargo. The side worker (sidecar) handles reporting and coordination.

🏗️ How It Works

Concept: Run multiple containers in the same Pod. They share:

  • Network namespace: Same IP, can communicate via localhost
  • Storage: Can share volumes (emptyDir, PVC)
  • Lifecycle: Start together, stop together
Pod Structure Example
Pod (shared network & storage)
Main Container

Your application

Port: 8080

Sidecar Container

Log agent / Proxy

Port: 9090

Communicates via localhost:8080

Example: Fluentd Sidecar for Logging

Main container (app) writes logs to /var/log/app. Sidecar (Fluentd) mounts same directory, tails log files, ships to Elasticsearch.

Example: Envoy Sidecar for Service Mesh

Main container doesn't know about TLS, authentication, observability. Envoy sidecar handles all network traffic. App talks to Envoy (localhost). Envoy talks to other services.

✅ When to Use This Pattern

  • Logging agents: Fluentd, Filebeat, Fluent Bit
  • Metrics exporters: Prometheus exporters, statsd
  • Service mesh proxies: Envoy, Linkerd proxy
  • Secret rotation: Vault agent, AWS secret sync
  • Background synchronization: Database replication, cache warming
  • Debug adapters: Debug container for troubleshooting

Key indicator: If you're thinking "I wish there was a way to add this functionality without changing the app code," sidecars are often the answer.

⚠️ Trade-offs & New Problems

Benefits:

  • Separation of concerns (app vs infrastructure)
  • Language/framework agnostic (same sidecar for all apps)
  • Independent updates (update sidecar without redeploying app)
  • Simplified app code (app doesn't handle infrastructure)

Drawbacks:

  • Resource overhead: Each sidecar uses memory/CPU. With 1000 pods, that's 1000 sidecars.
  • Complexity: More containers to monitor, debug, and troubleshoot.
  • Failure surface: Sidecar crash doesn't crash pod (by default), but functionality is lost.
  • Resource contention: Sidecar and main container compete for resources on same node.

Production reality: At a company with 5,000 pods, adding Envoy sidecars (50MB each) consumed 250GB of cluster memory. That's 5 extra large nodes. Worth it for service mesh benefits, but not free.

🏭 Production Examples

Google

Pattern: Sidecar containers for logging, monitoring, debugging in Borg (K8s predecessor).

Outcome: This pattern inspired Kubernetes' multi-container pod design. Google runs billions of containers with sidecars.

Istio / Linkerd (Service Mesh)

Pattern: Envoy or Linkerd proxy as sidecar in every pod.

Outcome: Automatic mTLS, observability, traffic management without app changes. Sidecar pattern makes this possible.

Shopify

Pattern: Vault agent sidecar for secret rotation.

Outcome: Secrets automatically rotated every 24 hours. Apps read from file (sidecar writes). No app code changes.

Airbnb

Pattern: Service mesh sidecars for traffic management.

Challenge: At 30,000 pods, sidecar overhead was significant.

Solution: Optimized sidecar memory usage. Used selective meshing (not all pods need sidecars).

Pattern 2: Init Containers

pod-design intermediate Low Complexity

📖 What This Pattern Solves

The problem: Your main container needs setup before it can start. Examples:

  • Wait for database to be ready (don't crash on connection refused)
  • Generate configuration files from templates
  • Clone a git repository before running app
  • Migrate database schema before starting app server

Why not do it in the main container?

  • Startup logic mixed with application logic
  • If setup fails, container restarts and tries again (loop)
  • No way to ensure setup completes before app starts

Shipping port analogy: Before a crane can start unloading containers, it needs setup: safety checks, equipment calibration, operator briefing. Init containers handle this setup phase. Only after setup completes does the main work begin.

🏗️ How It Works

Concept: Special containers that run before main containers. They must complete successfully before main containers start.

Key characteristics:

  • Run to completion (not long-running)
  • Run sequentially (init 1, then init 2, then main)
  • Share storage with main containers
  • If any init container fails, pod restarts

Example: Database Wait Init Container

Init container runs: wait-for-it db:5432 --timeout=30

Main container starts only after database is accepting connections.

Example: Config Generation Init Container

Init container runs: Template engine that generates nginx.conf from environment variables.

Main container (nginx) starts with generated config.

Example: Git Clone Init Container

Init container runs: git clone https://github.com/config/repo.git /config

Main container mounts /config as volume, reads generated files.

✅ When to Use This Pattern

  • Dependency waiting: Wait for database, cache, API to be ready
  • Configuration generation: Templates, secrets transformation, file generation
  • Setup tasks: Database migrations, cache warming, file downloads
  • Security checks: Validate certificates, check permissions

Alternative: Readiness probes

Sometimes readiness probes are better than init containers:

  • Init: Run once at startup
  • Readiness probe: Run continuously (can handle database restarts)

Decision: Use init for one-time setup. Use readiness for ongoing health checks.

🏭 Production Example

GitLab

Challenge: GitLab needs database migrations before starting. Can't start app server if schema is old.

Solution: Init container runs db:migrate. Main container starts only after migrations complete.

Outcome: Zero-database-downtime upgrades. Migrations run before new version starts.

WordPress on K8s

Challenge: WordPress needs wp-config.php before starting.

Solution: Init container generates wp-config.php from environment variables (DB host, password, salts).

Outcome: WordPress deploys to K8s without manual config. Fully automated.

Pattern 3: Controller Pattern

advanced automation Medium Complexity

📖 What This Pattern Solves

The problem: You need to automate complex operational tasks. Examples:

  • Create a database instance automatically when someone requests it
  • Backup databases every night, clean up old backups
  • Renew TLS certificates before they expire
  • Scale applications based on custom metrics (queue length)

Why not scripts / cron jobs?

  • Scripts are one-off, hard to maintain
  • No feedback loop (scripts don't "watch" state)
  • Error handling is manual
  • Don't integrate with K8s API natively

Shipping port analogy: The control tower doesn't just give orders and hope. It continuously watches the port: Are cranes working? Are containers moving? If not, it sends new orders. Controllers are the automated loops that keep the system in desired state.

🏗️ How It Works

The reconciliation loop:

  1. Observe actual state (what exists in K8s)
  2. Compare with desired state (what you declared)
  3. Take action to make them match
  4. Repeat forever

Every K8s controller works this way:

  • Deployment controller: "I want 3 replicas" → Creates pods → Watches → If pod dies, creates new one
  • Service controller: "I want a LoadBalancer" → Creates cloud LB → Watches → Updates if endpoints change
  • Node controller: "I want all nodes healthy" → Watches node heartbeats → Marks unhealthy nodes as NotReady

Custom controllers (operators):

You write your own controller to automate your domain:

  • Database operator: "I want a Postgres database" → Operator creates StatefulSet, Service, PVC, ConfigMap
  • Cert-manager operator: "I want a TLS certificate" → Operator requests cert from Let's Encrypt, renews before expiry

✅ When to Use This Pattern

  • Application lifecycle automation: Database creation, upgrade, backup, restore
  • Resource management: Automated scaling, cleanup, optimization
  • Certificates and secrets: Automatic rotation, sync with external sources
  • Multi-resource coordination: Deploy complex apps (frontend + backend + database + cache)

Signs you need a controller:

  • You're writing scripts to manipulate K8s resources
  • You need to coordinate multiple resources
  • You want "declarative" behavior (say what you want, not how)
  • You need continuous reconciliation (react to changes)

⚠️ Trade-offs

Benefits:

  • Declarative API (say what you want, not how)
  • Self-healing (continuously fixes drift)
  • Automation (no manual intervention)
  • Reusable (share controller across teams)

Drawbacks:

  • Complexity: Writing controllers requires understanding K8s API, informers, work queues
  • Debugging: Controllers are distributed systems. Hard to trace bugs.
  • Performance: Controllers poll API server constantly. Can overwhelm if not optimized.
  • Maintenance: You're now maintaining a control plane component, not just apps.

Production reality: "I've seen companies build custom controllers that were more complex than the problems they solved. Start with existing operators (cert-manager, Prometheus operator, Postgres operator). Write custom controllers only when off-the-shelf doesn't work."

🏭 Production Examples

cert-manager

What it does: Automatically provisions and renews TLS certificates from Let's Encrypt.

Controller pattern: Watches Certificate resources → Requests from ACME → Stores as Secret → Renews before expiry.

Outcome: Zero manual certificate management. Devs declare "I want a cert for example.com". Controller handles everything.

Prometheus Operator

What it does: Automates Prometheus deployment and configuration.

Controller pattern: Watch ServiceMonitor resources → Configure Prometheus to scrape services → Reload config.

Outcome: Add new service to monitoring by creating ServiceMonitor. No manual Prometheus config changes.

TiDB Operator

What it does: Manages distributed TiDB database on K8s.

Controller pattern: Orchestrates PD, TiKV, TiDB components. Handles scaling, failover, backup, upgrade.

Complexity: 50,000+ lines of code. Manages stateful distributed system. Shows controller pattern's power.

Pattern 4: Leader Election

advanced coordination High Complexity

📖 What This Pattern Solves

The problem: You run multiple replicas of a service, but only one should do specific work. Examples:

  • Cron jobs (only one replica should run scheduled tasks)
  • Database migrations (don't run migrations from 5 pods simultaneously)
  • Expensive cleanup jobs (only one pod needed)

Why not just run 1 replica?

  • You need high availability (if leader fails, another takes over)
  • Most work is parallelizable (HTTP requests handled by all replicas)
  • Only some tasks need single-leader semantics

Shipping port analogy: Multiple control towers can see the port, but only one is "active" giving orders. If the active tower fails, another takes over instantly. No two towers give conflicting orders.

🏗️ How It Works

Concept: Use Kubernetes ConfigMap or Lease resource as a "lock". Pods race to acquire lock. Holder is "leader". Others are "standby".

Flow:

  1. All replicas start up
  2. Each tries to create a ConfigMap with same name (the "lock")
  3. First one succeeds → becomes leader
  4. Others get "AlreadyExists" error → become standby
  5. Leader renews lock periodically (heartbeat)
  6. If leader dies, lock expires. Standby pods race again.

Implementation:

  • Use leader-election library (Kubernetes client libraries include this)
  • Or use Lease resource (built for leader election)
  • Or use ConfigMap with annotations (older pattern)

Example code pattern:

if (leaderElection.Acquire()) {
    // I'm the leader
    runCronJobs()
    leaderElection.KeepAlive()  // Heartbeat
}

✅ When to Use This Pattern

  • Cron jobs: Only one replica should run scheduled tasks
  • Batch processing: Only one pod should process queue
  • Database migrations: Run schema updates from single pod
  • Expensive tasks: Cache warming, report generation

Alternative:

  • CronJob controller: K8s native. Creates one pod per schedule. No leader election needed.
  • Separate deployment: Deploy single replica for leader tasks, multi-replica for API.

When to use leader election: When you need one deployment to handle both API traffic (multi-instance) and background tasks (single instance).

⚠️ Trade-offs & Gotchas

Benefits:

  • High availability (automatic failover)
  • Simpler deployment (one deployment for all functionality)

Drawbacks:

  • Complexity: Leader election logic is subtle. Bugs cause split-brain (two leaders).
  • Network partitions: If leader can't talk to API server, it thinks it's still leader. Standby can't acquire lock.
  • Resource waste: N-1 pods are idle (just waiting for leader to fail).

Production war story: "We used leader election for cron jobs. During network blip, leader couldn't renew lock but also didn't crash. Lock expired. Standby became leader. Now we had 2 leaders running cron jobs. Database slammed by duplicate migrations. Lesson: Use proper fencing (leader ID check on every operation)."

Pattern 5: Canary Deployment

deployment progressive Medium Complexity

📖 What This Pattern Solves

The problem: You want to deploy new versions with confidence. Rolling updates replace all pods, but what if the new version has a bug?

Why not just deploy and watch metrics?

  • By the time you notice, all users are affected
  • Rollback takes time (rolling update in reverse)
  • Some bugs are subtle (10% error rate, not 100% failure)

Shipping port analogy: Before replacing all cranes with a new model, test one crane for a week. If it breaks, only one crane is affected. If it works great, then replace the rest gradually.

🏗️ How It Works

Concept: Deploy new version to small subset of users. Monitor metrics. If good, gradually increase. If bad, rollback immediately.

Steps:

  1. Deploy v2.0 alongside v1.0 (10% pods v2, 90% v1)
  2. Route 10% of traffic to v2
  3. Monitor: error rate, latency, revenue impact
  4. If metrics good → increase to 25% → 50% → 100%
  5. If metrics bad → rollback to 0% v2 instantly

Implementation approaches:

  • Two Deployments: app-v1 and app-v2. Service selects both. Adjust replica counts.
  • Service mesh: Istio/Linkerd traffic splitting (virtual services).
  • Ingress: Nginx/Traefik canary annotations.
  • Feature flags: Same deployment, feature toggles enable v2 logic.

Example: Flagger (GitOps progressive delivery)

Automated canary deployments. Metrics-driven. Auto-promotes or auto-rolls back based on SLOs.

✅ When to Use This Pattern

  • High-traffic services: Even 1% errors affect many users
  • Complex deployments: Database schema changes, API changes
  • Frequent deployments: Multiple times per day, need automation
  • Risk-averse environments: Production systems where downtime is expensive

When not to use:

  • Low traffic (canary takes too long to get statistically significant data)
  • Simple apps (rollback is fast and safe)
  • Internal tools (users are engineers, can handle bugs)

⚠️ Trade-offs

Benefits:

  • Risk reduction (only small percentage of users affected)
  • Fast rollback (instant traffic shift back)
  • Metrics-driven decisions (data, not gut feelings)
  • Confidence in deployments

Drawbacks:

  • Complexity: Need traffic splitting, metrics collection, automation
  • Testing time: Canaries take hours/days to get confidence
  • Database compatibility: v1 and v2 must both work with same database schema
  • Monitoring requirements: Need detailed metrics (error rate, latency, business metrics)

🏭 Production Examples

Facebook

Approach: Deploy to one employee first. Then 1% of users. Then 5%, 10%, 50%, 100%.

Automation: Automated rollback if metrics degrade.

Outcome: Deploy multiple times per day with confidence.

Netflix

Tool: Spinnaker (deployment platform). Automated canary analysis.

Canary analysis: Compare canary vs baseline on latency, error rate, saturation.

Outcome: Reduced deployment failures by 80%.

Weaveworks (Flagger)

Tool: Flagger for Kubernetes. Automates canary deployments on service mesh.

Process: Deploys canary, runs load tests, analyzes metrics, auto-promotes or auto-rolls back.

Outcome: Progressive delivery without manual intervention.

Anti-Patterns: What to Avoid

These patterns feel right but cause problems in production. Learn from others' mistakes so you don't repeat them.

Anti-Pattern 1: The "God Pod" Monolith

architecture CRITICAL Very Common

🪤 The Trap (Why It Feels Right)

The thinking: "We'll run our entire application in one pod. It's simpler. Everything is on localhost. No network latency. Easy to debug."

Why teams do it:

  • Lift-and-shift from VM: "Our app runs as one process, make it one container"
  • Fear of microservices overhead
  • "It's working, why change?"
  • Lack of microservices experience

Shipping port analogy: Instead of having specialized cranes for different cargo types, you build one mega-crane that tries to do everything. It can move containers, yes, but it's slow, hard to maintain, and if it breaks, everything stops.

💥 Real Production Impact

What happens:

  • Single point of failure: Pod crashes → entire application down
  • Scaling issues: Can't scale individual components (scale entire app or nothing)
  • Resource waste: One component needs 4GB CPU, everything gets 4GB
  • Slow deployments: Changing one line of code requires redeploying entire pod
  • Team conflicts: Team A wants to deploy. Team B's code isn't ready. Deadlock.
  • No isolation: Memory leak in one component crashes entire pod

Real incident:

Company ran monolithic app (web, API, worker, cron) in one pod. Memory leak in cron component caused OOMKilled. Entire app down. Had to fix leak in cron to restore web/API. No isolation meant no fault tolerance.

✅ The Right Approach

Break into microservices:

  • Web frontend (Deployment A)
  • API backend (Deployment B)
  • Background worker (Deployment C)
  • Cron scheduler (CronJob or separate Deployment)

Benefits:

  • Independent scaling (scale API more, cron less)
  • Independent deployment (Team A deploys without waiting for Team B)
  • Isolation (cron crash doesn't affect API)
  • Technology diversity (API in Go, worker in Python)

Migration strategy:

  1. Start with monolith
  2. Identify natural boundaries (API vs worker vs cron)
  3. Extract one component at a time
  4. Don't break microservices dogma (not everything needs to be separate)

When monolith is OK: Small team, simple app, low traffic. Don't over-engineer.

📖 "I've Seen This Break When..."

"Team ran Django + Celery + Nginx in one pod. 5GB memory limit. Celery job needed more memory, OOMKilled. Took down web server too. CEO's presentation failed because web was down."

Lesson: Isolate failure domains. One component shouldn't crash others.

"E-commerce site with search, recommendations, checkout in one pod. Search team deployed new version. Bug in search caused high CPU. Entire pod slow. Checkout timed out. Lost $50K in sales during sale event."

Lesson: Independent scaling and deployment prevent cascading failures.

Anti-Pattern 2: Latest Tag Hell

deployment HIGH Very Common

🪤 The Trap

The thinking: "We'll use `image: myapp:latest`. When we push new code, just `docker push myapp:latest`. K8s will pull it. Simple."

Why it feels right:

  • No version management complexity
  • "Always get the latest code" feels good
  • Easy to explain to beginners

Shipping port analogy: Like shipping containers that all say "goods" with no label identifying what's inside or what version. When you need to find a specific item, impossible. When you need to revert to previous shipment, impossible.

💥 Production Impact

What happens:

  • No rollback: Can't rollback because you don't know what version was running before
  • Unexpected updates: Someone pushes `:latest`, you didn't know. Suddenly all pods update
  • Reproducibility impossible: "Which version is running in prod?" → "I don't know"
  • Debugging nightmares: Bug in production. Which version? Can't reproduce because `:latest` has moved on

Real incident:

Deployment used `:latest`. Bug found in production. Developer asked: "What version is running?" Ops: "We're using `:latest`." Dev: "But I pushed 3 times today. Which commit is this?" No one knew. Had to check container creation time, match to git commits. Took 2 hours to figure out. By then, `:latest` had been updated again. Total confusion.

✅ The Right Approach

Use semantic versioning:

  • image: myapp:v1.2.3
  • image: myapp:git-commit-hash
  • image: myapp:20240123-143022 (timestamp)

CI/CD workflow:

  1. Build: docker build -t myapp:v1.2.3 .
  2. Push: docker push myapp:v1.2.3
  3. Tag: docker tag myapp:v1.2.3 myapp:latest
  4. Push: docker push myapp:latest
  5. Deploy K8s with v1.2.3 (not latest)

Benefits:

  • Rollback: Change image from `v1.2.3` to `v1.2.2`
  • Reproducibility: Know exactly what code is running
  • Debugging: Can match version to git commit
  • Safe testing: Dev uses `:latest`, prod uses `v1.2.3`

Exception: `:latest` is OK for development/test environments. Never for production.

📖 War Stories

"Startup deployed with `:latest`. CEO asked: 'Can we demo version from last week?' Team: 'Uhh, we'd have to find the git commit, rebuild the image, tag it...' Demo delayed 3 hours. Never again."

"New engineer deployed service X with `:latest`. Didn't know existing pods were on `:latest`. Their pull updated the image. Suddenly all service X pods restarted (K8s saw image digest changed). Unexpected outage at 2 AM. On-call engineer had no idea what changed."

Anti-Pattern 3: Ignoring Resource Limits

resources CRITICAL Common

🪤 The Trap

The thinking: "We'll let pods use whatever they need. Why limit ourselves? If the node has resources, let the app use them."

Why it feels right:

  • "Best performance" (no throttling)
  • Simple (no capacity planning)
  • "We'll scale when we need to"

Shipping port analogy: Like allowing any crane to use as much power as it wants. One crane draws 100% power, other cranes can't operate. Port efficiency drops. Everything slows down.

💥 Production Impact

What happens:

  • Noisy neighbor: One pod hogs CPU, other pods starve
  • OOMKilled cascade: One pod consumes all memory, kubelet kills ALL pods randomly to recover
  • Unschedulable pods: Scheduler can't place new pods because existing pods have no limits (looks like they'll use all resources)
  • Surprise bills: Cloud auto-scaling adds nodes unnecessarily

Real incident:

Data science team deployed ML training job without memory limits. Job had memory leak. Consumed all 64GB on node. Other pods on node (monitoring, logging, API gateway) OOMKilled. Monitoring died, so no one could see what was happening. Cluster blind for 4 hours.

✅ The Right Approach

Always set requests and limits:

resources:
  requests:
    memory: "128Mi"
    cpu: "100m"
  limits:
    memory: "256Mi"
    cpu: "200m"

Understanding requests vs limits:

  • requests: Guaranteed minimum. Scheduler uses this for placement.
  • limits: Maximum allowed. Pod throttled or killed if exceeded.

How to determine values:

  1. Run without limits in dev/test
  2. Monitor actual usage (Prometheus, kubectl top)
  3. Set request = average usage * 1.2 (buffer)
  4. Set limit = max observed usage * 1.5 (headroom)
  5. Adjust based on production metrics

Production best practice:

  • Use LimitRange to enforce resource limits by namespace
  • Use ResourceQuota to limit total namespace resources
  • Use admission webhook to reject pods without limits

The 70% rule: Keep node utilization at 70%. Leave 30% for spikes and failover.

📖 War Stories

"Node had 64GB memory. 20 pods running without limits. One pod (Go app with goroutine leak) consumed 40GB. Kubelet started OOMKilling pods randomly. Killed the database pod. Database lost data (no persistent volume). Recovery took 6 hours."

"Company spent $50K/month on K8s nodes. Added resource limits to all pods (based on actual usage). Cluster was 30% utilized. Consolidated to half the nodes. Monthly cost dropped to $30K. One day's work, $240K/year saved."

Anti-Pattern 4: Hardcoded Secrets in Images

security CRITICAL Occasional

🪤 The Trap

The thinking: "We'll bake database passwords and API keys into the container image. It's easier than managing Secrets."

Why it feels right:

  • "App just works" when deployed
  • No external dependency on Secrets
  • Quick development cycle

Shipping port analogy: Like writing the warehouse code on every container in permanent marker. Anyone who sees the container can access the warehouse. Once marked, can't change. Every container needs remarking when code changes.

💥 Production Impact

What happens:

  • Security breach: Image registry compromised → All secrets exposed
  • Git leaks: Dockerfile committed to git with secrets → Secrets in version history forever
  • No rotation: To change password, must rebuild and redeploy all images
  • Can't share images: Dev image can't be used in prod (different secrets)

Real incident:

Startup committed Dockerfile to GitHub with AWS credentials hardcoded. Image was public. Someone found credentials, used them to mine cryptocurrency. AWS bill: $80,000 in one weekend. Credentials in git history (still there even after deleting file from repo).

✅ The Right Approach

Use Kubernetes Secrets:

  1. Create Secret: kubectl create secret generic db-creds --from-literal=password=xxx
  2. Reference in Deployment: valueFrom: secretKeyRef
  3. App reads secret as environment variable or file

Even better: External secret management

  • AWS Secrets Manager: Secrets stored encrypted in AWS. App fetches at runtime.
  • HashiCorp Vault: Centralized secret management with audit logs.
  • External Secrets Operator: Syncs external secrets to K8s Secrets automatically.

Benefits:

  • Rotation: Change secret in Vault, pods reload (no rebuild)
  • Audit: Log who accessed which secrets
  • Isolation: Dev, test, prod use same image, different secrets
  • Security: Secrets encrypted at rest, IAM controls access

Never, ever:

  • Commit secrets to git (even in private repos)
  • Bake secrets into images
  • Store secrets in ConfigMaps (not encrypted)

📖 War Stories

"Dev hardcoded Stripe API key in Dockerfile. Pushed image. 6 months later, company discovered fraudulent charges. Key had been exposed in public registry. Attackers used it for months. Lost $200K."

"Team built image with prod secrets. Used same image in dev environment. Devs accidentally ran transactions in prod database from dev cluster. Data corruption. Recovery took 2 days."

Anti-Pattern 5: Running as Root

security HIGH Very Common

🪤 The Trap

The thinking: "We'll run containers as root. It's easier. No permission issues. App works as-is."

Why it feels right:

  • "Just works" without permission tuning
  • Legacy apps assume root access
  • "It's a container, isolated, what's the harm?"

Shipping port analogy: Like giving every crane operator master keys to the entire port. They can access any warehouse, any ship, any controls. If one operator is malicious or compromised, entire port is at risk.

💥 Production Impact

What happens:

  • Container breakout: Vulnerability in container runtime gives attacker host access
  • Privilege escalation: Attacker gains root on node
  • Host access: Read host filesystem, access other containers' secrets
  • Compliance failures: Security audits fail (PCI-DSS, SOC2 forbid running as root)

Real incident:

Kubernetes bug (CVE-2019-1002101) allowed attackers to escape containers and access host filesystem. If container ran as root, attacker had root access to node. If container ran as non-root, attacker had limited permissions. Running as root multiplied vulnerability impact.

✅ The Right Approach

Always run as non-root:

securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  fsGroup: 1000

Pod Security Standards:

  • Use Pod Security Policy (deprecated) or Pod Security Admission (new)
  • Enforce non-root in namespaces (baseline policy)
  • Admission webhook rejects pods running as root

Build images correctly:

FROM alpine:3.18
RUN addgroup -g 1000 appgroup && \
    adduser -D -u 1000 -G appgroup appuser
USER appuser
... (rest of Dockerfile)

Benefits:

  • Least privilege: Container has minimal permissions
  • Reduced attack surface: Container escape gives attacker limited access
  • Compliance: Meets security standards
  • Defense in depth: Even if container compromised, damage is limited

Debugging permission issues:

  • Run container as root in dev to identify required permissions
  • Add capabilities (CAP_NET_BIND_SERVICE, etc.) instead of full root
  • Use securityContext to drop all capabilities, add only what's needed

📖 War Stories

"Financial services company ran all containers as root. Security audit failed. Had to rebuild all images with non-root users. Took 3 months. Delayed product launch. Cost: $5M in lost revenue."

"Container vulnerability in runc allowed attackers to escape. Company A ran as root → attackers owned the cluster. Company B ran as non-root → attackers had user-level access, contained. Difference: complete compromise vs minor incident."

Key Takeaways: Patterns & Anti-Patterns

🎯 Patterns Are Starting Points

Not rules. Understand the problem, the pattern, and the trade-offs. Adapt to your context.

🏗️ Sidecars Separate Concerns

Infrastructure logic (logging, proxies) shouldn't be in app code. Use sidecars. But watch resource overhead.

🔄 Controllers Automate Operations

Write controllers for repetitive tasks. But they're complex. Start with existing operators before writing custom controllers.

⚠️ Anti-Patterns Have Warning Signs

"It's easier now" usually means "it'll hurt later." Hardcoding secrets, running as root, ignoring limits—all have short-term convenience, long-term pain.

🚦 Progressive Delivery Reduces Risk

Canary deployments, blue-green, feature flags. Deploy with confidence. Automate rollback when metrics degrade.

🏢 Production Experience Shows

Every pattern and anti-pattern here comes from real incidents. Learn from others' mistakes. Don't repeat them.

🎓 The Shipping Port Analogy

Throughout these patterns, the shipping port analogy helps us understand:

  • Sidecars: Specialist workers alongside crane operators
  • Controllers: Operations center continuously monitoring and adjusting
  • Leader election: One active control tower, others on standby
  • Anti-patterns: One mega-crane, unlabeled containers, operators with master keys

When in doubt, ask: "How would a well-run shipping port handle this?" The answer usually points to the right pattern.
