🚀 Essential kubectl Commands

Cluster Information

▼ 
# View cluster info
kubectl cluster-info

# Get cluster nodes
kubectl get nodes

# View node details
kubectl describe node <node-name>

# Check kubectl version
kubectl version --client

# View current context
kubectl config current-context

# View all contexts
kubectl config get-contexts

# Switch context
kubectl config use-context <context-name>

Pod Operations

▼ 
# List all pods
kubectl get pods

# List pods with more details
kubectl get pods -o wide

# List pods in all namespaces
kubectl get pods --all-namespaces

# View pod details
kubectl describe pod <pod-name>

# View pod logs
kubectl logs <pod-name>

# View logs with follow (tail -f)
kubectl logs -f <pod-name>

# View logs for a specific container
kubectl logs <pod-name> -c <container-name>

# Execute command in pod
kubectl exec -it <pod-name> -- /bin/bash

# Delete pod
kubectl delete pod <pod-name>

# Create pod from YAML
kubectl apply -f pod.yaml

▼

kubectl rollout status deployment/<name>

# Delete service

  name: nginx-deployment
  labels:
    app: nginx

  replicas: 3
  selector:
    matchLabels:

    metadata:

▼

  name: pv-volume

  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  hostPath:
    path: /mnt/data
---

metadata:
  name: pv-claim

  accessModes:
    - ReadWriteOnce
  resources:

      storage: 5Gi

- name: storage
  mountPath: /data
volumes:
- name: storage
  persistentVolumeClaim:
    claimName: pv-claim

⚡ Resource Limits Guidelines

Understanding Requests vs Limits

  • Requests: Guaranteed resources. Kubernetes schedules based on this.
  • Limits: Maximum resources. Container throttled or killed if exceeded.
  • Best Practice: Always set both. Prevents resource starvation and noisy neighbors.

Microservice Guidelines

▼
Service Size CPU Request CPU Limit Memory Request Memory Limit
Small (API, worker) 100m 500m 128Mi 256Mi
Medium (web server) 250m 1000m 256Mi 512Mi
Large (app server) 500m 2000m 512Mi 1Gi
Extra Large (batch job) 1000m 4000m 1Gi 2Gi

Note: 1000m = 1 CPU core. Adjust based on actual usage monitoring.

CPU Units Explained

▼
  • 100m = 0.1 CPU = 10% of 1 core
  • 500m = 0.5 CPU = 50% of 1 core
  • 1000m = 1 CPU = 1 full core
  • 2000m = 2 CPU = 2 full cores

Example: If you set cpu: 100m, your container is guaranteed 10% of a CPU core. If you set limits.cpu: 500m, it won't exceed 50% of a core.

Memory Units Explained

▼
  • 128Mi = 128 Mebibytes ≈ 134 MB
  • 256Mi = 256 Mebibytes ≈ 268 MB
  • 1Gi = 1 Gibibyte ≈ 1.07 GB
  • 1G = 1 Gigabyte = 1 GB

Gotcha: Kubernetes uses mebibytes (Mi) and gibibytes (Gi) by default (power of 2). Use M and G for megabytes/gigabytes (power of 10). The difference is ~5%.

📊 Pod Phases & What They Mean

✅ Pending

Pod accepted but not running. Being scheduled or pulling images.

Common causes: Cluster lacks resources, image pull slow, scheduling constraints.

🏃 Running

Pod is running. At least one container is still running or starting.

Normal state: Everything is working as expected.

✅ Succeeded

All containers terminated successfully (job completed).

Normal for jobs: Batch jobs, one-time tasks.

❌ Failed

All containers terminated, at least one failed.

Check: kubectl describe pod for exit codes and events.

❓ Unknown

State couldn't be obtained (usually communication error).

Check: Node health, network connectivity.

Container States (Inside Pod)

  • Waiting: Container starting, pulling image, or running probes
  • Running: Container executing without issues
  • Terminated: Container exited (check exit code: 0 = success, non-zero = error)

Common Event Types

▼
Event Meaning Action

Scheduled Pod assigned to a node None, this is good

Troubleshooting Flowchart

▼
1

Pod not starting?

kubectl get pods - Check if Pending or Running

If Pending: Check events with kubectl describe pod

If CrashLoopBackOff: Check logs with kubectl logs

2

Image pull errors?

Check image name and tag in YAML

Verify registry credentials: kubectl get secrets

Test locally: docker pull <image>

3

Can't reach service?

Check service exists: kubectl get svc

Check endpoints: kubectl get endpoints <service>

Verify pod labels match service selector

Check network policies blocking traffic

4

High CPU/Memory usage?

Check metrics: kubectl top pods

Review resource limits vs requests

Check if limits are too low (throttling) or too high (OOM)

5

Pod restarting repeatedly?

Check previous logs: kubectl logs <pod> --previous

Look for app crashes (unhandled exceptions)

Check liveness probe configuration

Verify resource limits (OOMKilled)

⚡ Performance Tuning Parameters

Resource Optimization

▼

CPU Optimization

  • Set appropriate requests: Base on actual usage, not guesses
  • Use CPU limits wisely: Too low = throttling, too high = noisy neighbor
  • Enable horizontal autoscaling: HPA adds pods when CPU high
  • Avoid CPU overcommit: Sum of requests ≤ node capacity

Memory Optimization

  • Set limits higher than requests: Prevents OOM, allows bursts
  • Monitor memory leaks: If memory grows constantly, fix app
  • Use vertical autoscaling: VPA adjusts based on usage
  • Profile with tools: Use heap profilers to find leaks

Probe Tuning

▼

Liveness Probe Guidelines

  • initialDelaySeconds: Give app time to start (10-30s)
  • periodSeconds: Check frequency (10-30s)
  • timeoutSeconds: Response timeout (1-5s)
  • failureThreshold: Failures before restart (3-5)
  • Gotcha: Too aggressive = restart loops. Too lenient = dead pods stay running.

Readiness Probe Guidelines

  • Faster than liveness: Smaller delays, more frequent checks
  • Check dependencies: Database, cache, external services
  • Grace period: Allow for connection establishment
  • Gotcha: If ready check fails, traffic stops but pod stays running.

Autoscaling Parameters

▼

Horizontal Pod Autoscaler (HPA)

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

Guidelines:

  • Target 70% CPU for scaling (allows headroom)
  • Set min/max replicas based on load patterns
  • Use custom metrics for autoscaling on queue length, RPS

📊 Monitoring Metrics to Watch

Cluster-Level Metrics

  • Node health: % Ready, CPU/Memory usage
  • Pod distribution: Even spread across nodes?
  • Resource utilization: Cluster CPU/Memory %
  • API server latency: Control plane responsiveness

Pod-Level Metrics

  • Restart count: Frequent restarts = crash loop
  • CPU throttling: At limits often?
  • Memory usage: Near limit = OOM risk
  • Network I/O: Traffic patterns

Application Metrics

  • Request rate: RPS per pod
  • Error rate: 5xx errors increasing?
  • Latency: P95, P99 response times
  • Queue depth: Worker queue size

Business Metrics

  • SLI/SLO: Error budget, uptime
  • Deployment frequency: How often you ship
  • Lead time: Deploy to production
  • MTTR: Mean time to recovery

⚠️ Common Gotchas & Pitfalls

"My pods can't reach each other"

Critical

The gotcha: Pods CAN reach each other by IP, but you should use Services.

Why: Pod IPs change when pods restart. Services provide stable DNS names.

Fix:

  • Create a Service for each pod you need to reach
  • Use <service-name>.<namespace>.svc.cluster.local
  • In same namespace, just <service-name> works
# Good: Use service DNS
http://my-service.default.svc.cluster.local

# Bad: Use pod IP (changes on restart)
http://10.244.1.5

"Pod stuck in Pending state"

High

The gotcha: You requested resources the cluster doesn't have.

Common causes:

  • CPU/Memory requests exceed node capacity
  • No nodes match your node selector
  • Tolerations missing for tainted nodes
  • PVC can't find matching PV

Fix:

# Check what's wrong
kubectl describe pod <pod-name>

# Common output:
# 0/3 nodes are available: 3 Insufficient cpu

# Fix: Lower your requests or add nodes

"CrashLoopBackOff - what does that mean?"

High

The gotcha: Your container is starting, then crashing, repeatedly.

Kubernetes is trying to help: It restarts the container with increasing delays (backoff).

Common causes:

  • Application error (exception on startup)
  • Misconfigured environment variables
  • Missing ConfigMaps/Secrets
  • Wrong command/entrypoint

Fix:

# Check the logs
kubectl logs <pod-name>

# If previous crashed, check previous logs
kubectl logs <pod-name> --previous

"OOMKilled - but I have enough memory!"

High

The gotcha: You set memory limit too low, or not at all.

Without limits: Container can use all node memory, then gets killed.

With low limits: Container hits limit and gets killed.

Fix:

  • Always set memory limits
  • Monitor actual usage with kubectl top pods
  • Set limits 20-30% above observed peak
  • If growing continuously, fix memory leak in app
resources:
  requests:
    memory: "128Mi"    # Guaranteed minimum
  limits:
    memory: "256Mi"    # Maximum before OOMKill

"Image pull errors in production"

Medium

The gotcha: Private registry authentication not configured.

What's happening: Pod can't pull image because it needs credentials.

Fix: Create image pull secret

# Create secret from Docker config
kubectl create secret docker-registry regcred \
  --docker-server=<registry-url> \
  --docker-username=<user> \
  --docker-password=<password>

# Use in pod spec
spec:
  imagePullSecrets:
  - name: regcred

"My pods are on the wrong nodes"

Medium

The gotcha: Scheduler places pods randomly unless you guide it.

Fix options:

  • Node selectors: Simple label matching
  • Affinity: Advanced rules (co-locate, spread)
  • Taints/tolerations: Dedicated nodes
# Node selector (simple)
spec:
  nodeSelector:
    disktype: ssd

# Affinity (advanced)
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - nginx
        topologyKey: kubernetes.io/hostname

"Rolling update broke everything"

Critical

The gotcha: New version deployed but failing, and now you're down.

Immediate fix: Rollback

# Rollback to previous version
kubectl rollout undo deployment/<name>

# Check rollout history
kubectl rollout history deployment/<name>

# Rollback to specific revision
kubectl rollout undo deployment/<name> --to-revision=2

Prevention:

  • Use readiness probes (don't route traffic to unready pods)
  • Set maxSurge and maxUnavailable carefully
  • Consider blue-green deployments for critical apps

"Config changes not applying"

Medium

The gotcha: ConfigMaps/Secrets don't auto-trigger pod restarts.

What's happening: You updated ConfigMap, but pods still have old env vars.

Fix options:

  • Restart pods: kubectl rollout restart deployment/<name>
  • Use checksum annotation to force rollout
  • Mount ConfigMap as volume (updates in real-time)
# Force restart when config changes
kind: Deployment
spec:
  template:
    metadata:
      annotations:
        checksum/config: "{{ include (print $.Template.BasePath \"/configmap.yaml\") . | sha256sum }}"

🎯 Decision Flowcharts

Which Controller to Use?

▼

Quick Decision Guide

  • Deployment: Stateful apps, web servers, APIs (95% of cases)
  • StatefulSet: Databases, distributed systems requiring stable identities
  • DaemonSet: Logging agents, monitoring on every node
  • Job: One-time tasks, batch processing
  • CronJob: Scheduled tasks, periodic backups
Use Case Controller Why
Web server, API, microservice Deployment Easy scaling, rolling updates
Database (PostgreSQL, MySQL) StatefulSet Stable network, storage identity
Logging agent (Fluentd) DaemonSet Run on every node automatically
One-time data migration Job Run to completion, track success
Daily backup job CronJob Scheduled, repeated execution
Message queue worker Deployment Scale based on queue length
Redis cluster StatefulSet Stable IPs, stable storage

Which Service Type?

▼
Service Type Use When Scope
ClusterIP Internal service-to-service communication Cluster internal only
NodePort Expose externally temporarily (testing/dev) External via <NodeIP>:<Port>
LoadBalancer Expose to internet (cloud provider LB) External via cloud LB IP
ExternalName Map to external DNS name (alias) DNS CNAME

Best Practice

Most common pattern:

  1. Use ClusterIP for internal services (default)
  2. Use Ingress for external HTTP/HTTPS traffic
  3. Use LoadBalancer for external non-HTTP traffic (rare)

Should I Use Kubernetes?

▼

✅ Yes, use Kubernetes when:

  • You're running 10+ containers across multiple servers
  • You need high availability and self-healing
  • You have complex microservices architecture
  • You're doing multi-cloud or hybrid deployments
  • You have DevOps expertise or can hire it
  • You need to scale applications automatically

❌ No, don't use Kubernetes when:

  • You have a simple monolith on one server
  • You don't have DevOps resources to maintain it
  • You're just starting with containers (learn Docker first)
  • You need something running today without complexity
  • You're okay with managed services (Lambda, App Engine)
  • Your team is small and product-focused, not infra-focused

💡 Quick Tips for Success

Start Small

Don't deploy your entire architecture on day one. Start with one service, learn the patterns, then expand.

Use Namespaces

Separate environments (dev/staging/prod) with namespaces. Prevents accidental deletions and resource conflicts.

Label Everything

Add labels to every resource. Makes querying, organizing, and debugging 10x easier.

Set Resource Limits

Always set requests and limits. Prevents noisy neighbors and OOMKilled pods.

Monitor Everything

Set up Prometheus + Grafana from day one. You can't improve what you don't measure.

Use GitOps

Store all YAML in git. Use ArgoCD or Flux for deployments. Every change is tracked and reversible.

Practice Rollbacks

Know how to rollback before you deploy. Test it in non-production first.

Learn YAML Structure

Understand the schema: apiVersion → kind → metadata → spec. Makes every manifest make sense.

🎯 Key Takeaway

This reference guide is your quick lookup companion throughout your Kubernetes journey. Bookmark it, keep it open while you work, and refer to it when you're stuck.

  • kubectl get/describe/logs - Your debugging trifecta
  • Always set resource limits - Prevents OOM and resource starvation
  • Use Services, not Pod IPs - Stable networking
  • Check events first - Tells you what Kubernetes is trying to do
  • Readiness > Liveness - Prevent traffic to unready pods
