What Each Component Actually Does

kubectl
What it does:

Your command-line interface to Kubernetes. Every interaction happens through kubectl (or API calls underneath).

🎛️ Analogy: Control panel at the shipping port operations center
Pod
What it does:

The smallest deployable unit. One or more containers that share storage/network. Always co-located and co-scheduled.

📦 Analogy: Shipping container(s) that travel together
Node
What it does:

A physical or virtual machine that runs pods. Each node has kubelet (agent), kube-proxy (network), and container runtime.

🏗️ Analogy: Crane/dock worker that moves containers
Deployment
What it does:

Manages replicated pods. Ensures specified number of pods are running. Handles rolling updates and rollbacks.

📋 Analogy: Order: "Keep 10 containers of this type running at all times"
Service
What it does:

Provides stable network endpoint (IP/DNS) for pods. Load balances traffic to pods. Pods come and go, Service stays the same.

📋 Analogy: Shipping office that directs trucks to available cranes
Namespace
What it does:

Isolates resources within a cluster. Like virtual clusters within the physical cluster.

🏢 Analogy: Different sections of the port for different companies

Common Mistakes to Avoid (with Real Production Stories)

❌ "I'll run everything in the default namespace"

MEDIUM SEVERITY

The mistake: Deploying all apps to the `default` namespace. Everything mixed together.

Real story: A new developer deployed their test app to default namespace, accidentally deleted what they thought was their pod, but it was someone else's critical service. Production outage for 2 hours.

✅ Right approach: Always use namespaces. Create one for your team, your app, your environment. kubectl create namespace myapp

❌ "I'll put secrets in my deployment YAML"

CRITICAL SEVERITY

The mistake: Adding database passwords and API keys directly in deployment manifests.

Real story: A startup committed their deployment YAML to GitHub with AWS credentials in it. Within hours, their AWS bill was $50,000 from crypto mining. The credentials were in the repo for 6 months.

✅ Right approach: Use Kubernetes Secrets. Create secrets with kubectl create secret, reference in deployments. Never commit secrets to git.

❌ "I don't need resource limits"

HIGH SEVERITY

The mistake: Deploying pods without CPU/memory limits. One pod can hog the entire node.

Real story: A data science job had a memory leak and consumed all node memory. The critical services running on that node (including monitoring) were OOMKilled. No one could see what was happening because monitoring died too.

✅ Right approach: Always set resources.requests and resources.limits. resources: {requests: {memory: "128Mi"}, limits: {memory: "256Mi"}}

❌ "I'll use latest tag for all images"

HIGH SEVERITY

The mistake: Deploying with image: myapp:latest. You have no idea what version is actually running.

Real story: Team deployed with `:latest`. During incident, they couldn't roll back because they didn't know which version was previously deployed. Was it `latest` from today? Yesterday? Last week?

✅ Right approach: Use specific version tags. image: myapp:v1.2.3. You can always update to newer versions, but you know exactly what's running.

Understanding Kubernetes' Architecture

Kubernetes consists of two parts: Control Plane (the brain) and Worker Nodes (the muscle).

Control Plane Components

API Server: Front door for all operations. Authenticates, validates, configures data.

etcd: Key-value store. The cluster's "memory" — remembers everything.

Scheduler: Decides which nodes run which pods.

Controller Manager: Runs controller loops. Notices desired vs actual state, drives to match.

Worker Node Components

kubelet: Primary node agent. Talks to API server, ensures pods are running.

kube-proxy: Network proxy. Maintains network rules on each node.

Container Runtime: Runs containers (containerd, CRI-O, etc.).

Key insight: When you run kubectl apply, you're talking to the API Server. The API Server stores your desired state in etcd. Controllers notice the change and make it happen. The Scheduler decides where pods run. Kubelet on each node actually runs them.

Why Features Exist (Design Principles)

🎯 Principle 1: Declarative API

What it is: You tell Kubernetes what you want, not how to do it.

Why it exists: Imperative approaches ("run container on node 3") don't scale. Declarative ("run 3 replicas, I don't care where") enables self-healing and automation.

In practice:

Imperative:
"Run pod on node 3"
→ What if node 3 fails? What if you need to scale? Manual work.

Declarative:
"I want 3 replicas of nginx"
→ K8s figures out where, how to handle failures, auto-scales.

🎯 Principle 2: Controller Pattern

What it is: Controllers continuously reconcile desired state (what you want) with actual state (what exists).

Why it exists: Distributed systems are constantly changing. Nodes fail, pods crash, networks glitch. Controllers ensure the system self-heals.

The loop:

1. Observe actual state (current pods: 2)
2. Compare with desired state (deployment wants: 3)
3. Take action to reconcile (start 1 new pod)
4. Repeat forever

Production tip: This is why Kubernetes "just works" — controllers are constantly fixing problems.

🎯 Principle 3: Labels and Selectors

What it is: Key-value pairs attached to objects. Selectors query for objects with specific labels.

Why it exists: Enables loose coupling. Services don't know which pods they target — they query by labels.

In practice:

Pod has labels: app=nginx, tier=frontend
Service selects: tier=frontend
→ Service finds all pods with that label, regardless of which pod they are

Production story: We added a new pod version with labels app=nginx, version=v2. Service selector app=nginx sent traffic to both v1 and v2. We tested v2, then removed v1 label. Zero downtime rollout.

🎯 Principle 4: Desired State and Reconciliation

What it is: You declare desired state; K8s continuously drives actual state to match.

Why it exists: Manual operations don't scale. Automation requires clear goals and autonomous correction.

The reality: Kubernetes is not "set it and forget it" — it's "set it and it maintains itself". You still need to monitor, but much less manual intervention.

Customizing Basics Confidently

Writing Your Own Deployment Manifests

Basic Deployment YAML

Why: Declarative configuration, version controlled, reviewable, reproducible.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "128Mi"
            cpu: "100m"
          limits:
            memory: "256Mi"
            cpu: "200m"

Breakdown: apiVersion (which API), kind (what type), metadata (name/labels), spec (what you want). The template is what each pod should look like.

Service YAML

Why: Stable network endpoint for your pods. Load balancing.

apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
  type: LoadBalancer

Key concept: Selector matches pod labels. Port is what the service listens on. TargetPort is what pods expose.

ConfigMap for Configuration

Why: Separate configuration from code. Don't rebuild images for config changes.

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  database_url: "postgres://localhost:5432/mydb"
  cache_enabled: "true"
  log_level: "info"

Usage: Reference in deployment as environment variables or mounted as files.

Secret for Sensitive Data

Why: Store passwords, API keys, certificates separately. Base64 encoded (not encrypted by default, use external secret management).

apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  password: cGFzc3dvcmQxMjM=  # base64 encoded

Security note: Secrets are only base64 encoded. Use external secret management (Vault, AWS Secrets Manager) for production encryption.

Beginning Troubleshooting Methodology

When Pods Won't Start

1
Check pod status
kubectl get pods

Look at STATUS column. Running? Pending? CrashLoopBackOff? ImagePullBackOff? Each means something different.

2
Describe the pod
kubectl describe pod <pod-name>

Shows events. This is your #1 debugging tool. Look for "Failed to pull image", "FailedScheduling", "OOMKilled".

3
Check the logs
kubectl logs <pod-name>

Shows what your application is logging. Container crashed? Logs will show why (often).

4
Get inside the pod
kubectl exec -it <pod-name> -- sh

Interactive shell. See what's actually running. Check if files exist, network works, etc.

5
Check node resources
kubectl top nodes

Out of memory or CPU? Pods won't schedule. Nodes at capacity?

Efficiency Basics and Patterns

📝 Use YAML, Not CLI

Imperative commands (kubectl run) are fine for learning. Declarative YAML is required for production. Version control, review, rollback.

kubectl apply -f deployment.yaml ← Do this
kubectl run nginx --image=nginx ← Learning only

🏷️ Label Everything

Labels make Kubernetes powerful. Query, group, select. Use consistent conventions: app=<name>, tier=<frontend/backend>, env=<dev/prod>.

📊 Set Resource Limits

Always set requests and limits. Prevents noisy neighbor problems. Enables scheduling decisions.

requests: minimum guaranteed (scheduling)
limits: maximum allowed (throttling/eviction)

🔄 Use Liveness and Readiness Probes

K8s needs to know if your pod is alive (restart if not) and ready (send traffic). Without probes, K8s can't self-heal properly.

🧹 Clean Up Old Resources

Namespaces fill up with old deployments, jobs, pods. Regular cleanup: kubectl delete all --all in dev namespaces.

📚 Use Namespaces

Organize by team, environment, project. Prevents accidental cross-team interference. Makes kubectl get pods manageable.

Reading Between the Lines: What Documentation Means vs Says

📖 Documentation says: "Kubernetes is self-healing"

Reality: Kubernetes self-heals if you configure it correctly. No probes? No resource limits? Bad YAML? It won't heal.

Production lesson: Self-healing requires proper configuration: liveness/readiness probes, resource limits, replica sets, health checks. You get out what you put in.

📖 Documentation says: "Use Deployments for stateless workloads"

Reality: Deployments work fine for many stateful workloads. But StatefulSets provide guarantees (stable network, persistent storage) that Deployments don't.

Production lesson: Understand the trade-offs. Use the right tool. Deployments for most apps, StatefulSets for databases, queues, etc.

📖 Documentation says: "kubectl is the CLI for Kubernetes"

Reality: kubectl is just an API client. Everything kubectl does, you can do with API calls. Many teams build custom tools on top of the API.

Production lesson: Learn the API, not just kubectl. The API is the foundation. kubectl is just one interface.

📖 Documentation says: "Services provide load balancing"

Reality: Services provide basic load balancing (random or round-robin). For advanced LB (session affinity, weighted routing), use Ingress or service mesh.

Production lesson: Kubernetes LB is IPVS or iptables-based. Good for basics. Production needs: external LB, ingress controller, or service mesh for advanced routing.
