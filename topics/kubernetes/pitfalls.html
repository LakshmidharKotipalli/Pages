Kubernetes Pitfalls & War Stories

These are the real production failures that have caused outages, data loss, and millions in costs. Learn from these mistakes so you don't repeat them. Every pitfall includes severity ratings, real incidents, and prevention strategies.

⚠️ How to Use These Pitfalls

For learning: Read these to understand what not to do. Each pitfall is a lesson learned the hard way by real companies.

For interviews: When asked about "challenges you've faced with Kubernetes," these pitfalls provide talking points. Show you understand production realities.

For prevention: Use the checklists to set up your cluster correctly from day one. Prevention is 100x cheaper than cure.

The shipping port analogy: These are the operational nightmares that shipping ports face and how experienced operators avoid them.

Pitfall 1: Missing or Misconfigured Health Checks

CRITICAL Very Common

🪚 The Mistake

Deploying pods without readinessProbe and livenessProbe, or configuring them incorrectly.

Common variations:

  • No health checks at all
  • Probes checking wrong endpoint (e.g., "/" instead of "/health")
  • initialDelaySeconds too short (app not ready when probes start)
  • periodSeconds too long (5+ minutes to detect failure)
  • Liveness and readiness using same endpoint

Why it happens:

  • "It works in dev, why add probes?"
  • Unclear difference between liveness and readiness
  • Copy-pasting from examples without understanding
  • Fear of "restart loops" from aggressive probes

😵 Why It Felt Right at the Time

The thinking: "Our app is simple. It starts, it works. Why add complexity?"

Reality: Kubernetes needs to know pod health to:

  • Restart dead pods (self-healing)
  • Remove broken pods from service load balancer
  • Handle rolling updates (wait for new pods to be ready before killing old ones)

Shipping port analogy: Cranes have no health monitoring. A crane breaks but keeps appearing as "available" on the dispatch board. Control tower keeps sending trucks to that crane. Trucks pile up. Operations slow down. If there were health checks ("Is crane actually working?"), broken cranes would be marked "out of service" immediately.

💥 Production Impact

Downtime: 1-4 hours typical, up to days in extreme cases

Users affected: 10-100% (depending on service criticality)

Cost impact: $10K - $1M+ (depending on business)

Specific impacts:

  • Rolling update fails: New pods not ready, old pods killed. Service down.
  • Dead pods receive traffic: Pod crashed but Service doesn't know. Users see 500 errors.
  • Restart loops: Liveness probe too aggressive. Pod crashes, restarts, crashes again. Never serves traffic.
  • Slow detection: Probe checks every 60 seconds. Pod dead for 59 seconds. High error rate.

🔧 How We Fixed It

Immediate Fix (during incident)
  • Delete broken deployment: kubectl delete deployment/app
  • Redeploy with correct probes (add to YAML, apply)
  • Or manually patch: kubectl patch deployment/app
Proper Configuration
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
  timeoutSeconds: 5
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
  timeoutSeconds: 3
  failureThreshold: 3
Key Differences
  • Liveness: "Is the app alive?" If fails → restart pod. Checks deep health (database connection, memory OK).
  • Readiness: "Can the app serve traffic?" If fails → remove from Service. Checks if app is ready (loaded config, warmed cache).
Testing Probes
  • Deploy to dev/test with aggressive probes (periodSeconds: 1)
  • Crash app, verify it restarts within expected time
  • Overload app, verify readiness probe fails before OOM
  • Check logs, verify probes aren't causing restarts

🛡️ How to Prevent It

Prevention Checklist
Automated Enforcement
  • Admission webhook: Reject pods without probes
  • Policy: OPA Gatekeeper rule requiring probes
  • CI/CD: Helm chart template requires probe values

🏭 Real Production War Stories

Major E-commerce Platform

Incident: Deployed new version with readiness probe pointing to root URL ("/"). Root endpoint returned 200 OK even if app wasn't ready (database still connecting).

What happened:

  1. Rolling update started
  2. New pods returned 200 on / (readiness probe passed)
  3. Old pods killed
  4. New pods couldn't handle requests (database not ready)
  5. 100% of requests failed
  6. Site down for 45 minutes

Fix: Changed readiness probe to "/ready" which checks database connection. Only passes when app can actually serve requests.

Cost: $2M in lost sales during Black Friday period.

SaaS Company

Incident: No liveness probe configured. App had memory leak that gradually made it unresponsive.

What happened:

  • App ran for days, memory usage slowly increased
  • Eventually app hung (99% memory, constant GC)
  • Kubernetes didn't know pod was dead (no liveness probe)
  • Pod stayed in "Running" state but served no requests
  • Users saw timeout errors
  • Took 4 hours to debug (pod looked healthy in kubectl get pods)

Fix: Added liveness probe that checks /healthz endpoint every 10 seconds. When app hangs, probe fails, pod restarts automatically.

Lesson: "Pods can be 'Running' but dead. Liveness probes detect zombie pods."

Financial Services

Incident: Liveness probe too aggressive (periodSeconds: 1, failureThreshold: 1). App occasionally took 2 seconds to respond to health check.

What happened:

  • Under load, health check response time increased to 2 seconds
  • Liveness probe timed out (timeoutSeconds: 1)
  • Pod restarted
  • New pod started, got hit with load, health check timed out again
  • Restart loop: pod up → health check fails → restart → repeat
  • Service never available

Fix: Increased timeoutSeconds to 5, failureThreshold to 3. Pod restarts only if consistently unhealthy for 15 seconds, not 1 second.

Lesson: "Probes should detect real failures, not temporary slow responses. Tune based on production metrics."

📚 Lesson Learned

Core principle: Kubernetes self-healing only works if it knows pods are unhealthy. Health checks are the communication channel between pods and K8s.

Design your health check endpoints carefully:

  • /healthz (liveness): Minimal check. App process running? Not deadlocked?
  • /ready (readiness): Can I serve traffic? Database connected? Cache warmed? Config loaded?
  • /live (optional): Deep health. All dependencies OK? (Use with caution—can cause cascading failures)

Never use the same endpoint for liveness and readiness. If app is temporarily overloaded (readiness should fail), liveness shouldn't fail (don't restart, just remove from load balancer).

Monitor probe failures in production. Alert on high probe failure rate. It's early warning of app issues before users are affected.

Pitfall 2: Forgetting Persistent Volumes for Stateful Apps

CRITICAL Common

🪚 The Mistake

Deploying a stateful application (database, message queue) without PersistentVolumes. Pod restart → all data lost.

How it happens:

  • Deploying database like a stateless app (no PVC)
  • Using emptyDir (dies with pod) instead of PVC
  • Using hostPath (works on single node, breaks when pod reschedules)
  • Forgetting to add volumeMount to container spec

Why it happens:

  • Coming from VM world (data stays on disk by default)
  • Testing locally (minikube) where data persists by accident
  • Unclear difference between emptyDir, hostPath, and PVC
  • "We'll add PVC later" (later = after data loss)

😵 Why It Felt Right at the Time

The thinking: "Pods are like VMs. I write to disk, data stays. Why do I need special storage?"

Reality: Pods are ephemeral. When a pod restarts, it's a NEW pod. Nothing persists unless explicitly mounted.

Shipping port analogy: Imagine storing goods directly on the dock instead of in warehouses. When a crane moves to a different dock, all goods left behind. Warehouses (PVC) provide permanent storage independent of cranes (pods).

💥 Production Impact

Data loss: 100% of data in pod filesystem

Downtime: Hours to days (restore from backup, replay logs)

Users affected: 100% if critical data lost

Specific impacts:

  • Database: All transactions since last backup lost
  • Message queue: All in-flight messages lost
  • File storage: All user uploads lost
  • Config: State lost, app restarts from zero

Recovery scenarios:

  • Restore from backup (hours of downtime, data loss since backup)
  • Replay transaction logs (possible for some databases, complex)
  • Manual data reconstruction (weeks, if possible at all)
  • Accept data loss (user-facing, regulatory issues)

🔧 How We Fixed It

Immediate Fix (after data loss)
  • Restore database from backup (if available)
  • Add PersistentVolumeClaim to deployment
  • Redeploy with PVC mounted
  • Restore data to new PVC
Proper Configuration
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: db-data
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Gi
  storageClassName: fast-ssd  # Use fast storage for databases

---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgresql
spec:
  template:
    spec:
      volumes:
      - name: db-data
        persistentVolumeClaim:
          claimName: db-data
      containers:
      - name: postgresql
        volumeMounts:
        - name: db-data
          mountPath: /var/lib/postgresql/data
For StatefulSets (recommended for databases)
  • Use volumeClaimTemplates (auto-creates PVC for each replica)
  • Each pod gets stable storage (postgresql-0, postgresql-1, etc.)
  • When pod reschedules, PVC follows (same data)

🛡️ How to Prevent It

Prevention Checklist
Automated Detection
  • Admission webhook: Warn if StatefulSet has no volumeClaimTemplates
  • Monitoring: Alert if PVC usage > 80% (avoid out-of-space)
  • Policy: Require PVC review for all new stateful workloads

🏭 Real Production War Stories

Healthcare Startup

Incident: Deployed PostgreSQL database using Deployment (not StatefulSet), no PVC.

What happened:

  • Database running for 3 months, storing patient records
  • Node failure scheduled for maintenance
  • Pod moved to different node
  • All data stored in pod container filesystem
  • Data stayed on old node's disk
  • New pod started with empty database
  • 3 months of patient records: gone

Recovery:

  • Old node disk retrieved from datacenter
  • Disk mounted on another machine
  • Data copied off, restored to new PVC
  • Total downtime: 18 hours

Regulatory impact: HIPAA violation investigation. Fines + mandatory audits.

Lesson: "Deployments are for stateless. StatefulSets + PVC for stateful. Period."

E-commerce Company

Incident: Redis cache using emptyDir for persistence (thinking "it's a cache, doesn't need PVC").

What happened:

  • Redis used as session store (user login tokens)
  • Node had hardware issue, pod rescheduled
  • emptyDir volume died with pod
  • All user sessions lost
  • 500,000 users logged out simultaneously
  • Login servers overloaded (everyone trying to re-login)

Fix: Migrated to Redis with PVC. Sessions persist across pod restarts.

Lesson: "Even 'caches' need persistence if they're the source of truth for anything."

Media Company

Incident: User-generated content (uploaded videos) stored in pod filesystem.

What happened:

  • Users uploaded videos to /uploads in pod
  • No PVC, videos stored in container layer
  • Deployment triggered (rolling update for new feature)
  • Old pod terminated, new pod started
  • Old pod's container deleted (all uploads lost)
  • Users who uploaded videos in last 2 hours: videos gone

Fix: Store user uploads in object storage (S3, GCS) with PVC for temporary processing only.

Lesson: "User data never goes in pod filesystem. Always external storage (S3, database with PVC)."

📚 Lesson Learned

Core principle: Pods are ephemeral by design. Data persistence is opt-in, not opt-out. If you need data to survive pod restarts, use PVC.

Decision tree:

  • Does the app write data? No → no PVC needed
  • Is the data reconstructible? Yes (cache) → PVC optional
  • Is the data irreplaceable? Yes (database, user data) → PVC required

Storage classes matter:

  • Use fast SSD for databases (low latency)
  • Use standard HDD for logs/archives (cheaper)
  • Consider network storage (NFS, EFS) for shared access

PVC ≠ Backup: PVC protects against pod/node failures. It doesn't protect against:

  • Accidental deletion (kubectl delete pvc)
  • Corruption (database writes bad data)
  • Disaster (datacenter destruction)

You still need backups: regular snapshots, point-in-time recovery, cross-region replication.

Pitfall 3: Namespace Confusion & Resource Collisions

HIGH Very Common

🪚 The Mistake

Deploying everything to the default namespace, causing naming conflicts, accidental deletions, and security issues.

Common variations:

  • All apps in default namespace
  • Dev, test, prod in same namespace
  • Multiple teams sharing one namespace
  • No resource quotas, namespaces compete for resources

Why it happens:

  • Default namespace exists, why create others?
  • Simplest path (kubectl apply without -n namespace)
  • Small team, "we don't need isolation"
  • Unclear when to use namespaces

😵 Why It Felt Right at the Time

The thinking: "We have 5 apps. Why make 5 namespaces? Complicates things."

Reality: Namespaces provide isolation for resources, security, and organization. As you scale, lack of namespaces becomes chaos.

Shipping port analogy: Like having all companies' containers in one big pile. No separation. Company A accidentally picks up Company B's container. Crane operator doesn't know which container belongs to whom. Disorganized operations, accidental mix-ups, security risks.

💥 Production Impact

Common incidents:

  • Naming conflicts: Two teams deploy "database" Deployment. Second overwrites first.
  • Accidental deletions: kubectl delete all deletes everything, including other teams' apps.
  • Noisy neighbor: Team A's job uses all CPU, Team B's apps starve.
  • Security issues: Team A can see Team B's pods, logs, secrets.
  • Resource exhaustion: No quota limits, one job can fill cluster disk.

Downtime: 1-8 hours typical

Users affected: 10-100% (depending on conflict)

🔧 How We Fixed It

Migrate to Namespaced Structure
# Create namespaces
kubectl create namespace dev
kubectl create namespace staging
kubectl create namespace production
kubectl create namespace team-a
kubectl create namespace team-b

# Set resource quotas
kubectl apply -f quota.yaml -n production

# Deploy to namespace
kubectl apply -f app.yaml -n production
# or set default namespace in context
kubectl config set-context --current --namespace=production
Resource Quota Example
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
  namespace: production
spec:
  hard:
    requests.cpu: "100"
    requests.memory: 200Gi
    limits.cpu: "200"
    limits.memory: 400Gi
    persistentvolumeclaims: "10"
Network Policy (isolation)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-from-other-namespaces
  namespace: production
spec:
  podSelector:
    matchLabels: {}
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector: {}

🛡️ How to Prevent It

Prevention Checklist
Best Practices
  • One namespace per environment: dev, staging, production
  • One namespace per team: team-a, team-b, platform
  • Resource quotas: Every namespace has CPU/memory limits
  • Network policies: Isolate production from dev/staging
  • RBAC: Teams can only access their namespaces

🏭 Real Production War Stories

Startup

Incident: New developer joined, followed runbook: kubectl delete pod -l app=database to restart database.

What happened:

  • Developer was in default namespace
  • Ran command without -n flag
  • Deleted ALL pods with label app=database in ALL namespaces
  • Dev, staging, production databases all deleted simultaneously
  • Production down for 2 hours

Fix: Separate namespaces. Developer only has access to dev namespace. Can't affect production.

Lesson: "Namespaces are security boundaries. RBAC + namespaces = least privilege."

Mid-size Company

Incident: Team A deployed job named "processor". Team B had job with same name.

What happened:

  • Both teams using default namespace
  • Team A's job overwritten by Team B's job
  • Team A's critical data processing stopped
  • Team A didn't notice for 12 hours (job didn't fail, just replaced)
  • Data pipeline broken, analytics wrong

Fix: Separate namespaces per team. Resource names don't conflict.

Lesson: "Namespaces prevent naming conflicts. They're not just organization, they're isolation."

📚 Lesson Learned

Core principle: Namespaces are the primary isolation mechanism in Kubernetes. Use them from day one, not after problems start.

When to create namespaces:

  • Per environment: dev, staging, prod
  • Per team: team-a, team-b, shared-services
  • Per application: For large, multi-component apps
  • Per security boundary: Isolate sensitive workloads

Namespaces provide:

  • Resource isolation: Quotas prevent noisy neighbor
  • Security isolation: RBAC restricts access
  • Network isolation: NetworkPolicy controls traffic
  • Naming isolation: Same names in different namespaces don't conflict

Rule of thumb: If you have more than 5 developers or more than 3 applications, you need namespaces.

Pitfall 4: Image Pull Errors & Registry Issues

HIGH Common

🪚 The Mistake

Pods stuck in ImagePullBackOff or ErrImagePull state. Can't start because image can't be pulled.

Common causes:

  • Wrong image name or tag (typo, non-existent tag)
  • Private registry without imagePullSecrets
  • Registry authentication expired
  • Registry down or unreachable
  • Image too large, pull timeout
  • Network policies block registry access

💥 Production Impact

Downtime: 30 minutes to several hours (depending on fix)

Users affected: 100% of service users

Typical scenario:

  • Deploy new version with updated image tag
  • Pods fail to start (ImagePullBackOff)
  • Old pods already killed (rolling update)
  • Zero pods running → service completely down
  • Debug: kubectl describe pod shows image pull error
  • Fix: Correct image tag, redeploy
  • Recovery time: 30-90 minutes of downtime

🔧 How We Fixed It

Immediate Debugging
# Check pod status
kubectl get pods

# Describe pod to see error
kubectl describe pod 
# Look for: Events -> Failed to pull image

# Common errors:
# - "Failed to pull image": image doesn't exist
# - "denied": authentication required
# - "timeout": registry unreachable
Solutions by Error Type
  • Wrong tag: Fix image name, redeploy
  • Auth required: Create imagePullSecret, add to deployment
  • Registry unreachable: Check network, DNS, firewall
  • Timeout: Pre-pull images, use local registry cache
imagePullSecrets Setup
# Create secret from registry credentials
kubectl create secret docker-registry regcred \
  --docker-server=registry.example.com \
  --docker-username=user \
  --docker-password=pass \
  --docker-email=user@example.com

# Add to deployment
spec:
  template:
    spec:
      imagePullSecrets:
      - name: regcred

🛡️ How to Prevent It

Prevention Checklist
Best Practices
  • Validate in CI: docker pull before kubectl apply
  • Use image tag policy: No :latest in production
  • Private registry: Use auth, rotate secrets regularly
  • Registry reliability: Use managed registries (ECR, GCR) with SLA
  • Rollback strategy: Keep previous images in registry

🏭 Real Production War Stories

Startup

Incident: Deployed with image tag :latest. CI pushed new image with same tag. Existing pods suddenly pulled new image.

What happened:

  • Deployed v1.0 with tag :latest
  • CI pushed v1.1 with tag :latest (overwrote)
  • Kubernetes saw image digest changed
  • Pods restarted with new image unexpectedly
  • v1.1 had bug. Production broke at 2 AM.
  • On-call confused: "I didn't deploy anything"

Fix: Use immutable tags (v1.0.0, v1.0.1). Never :latest in prod.

Enterprise

Incident: Image registry authentication rotated. Deployments started failing.

What happened:

  • Security team rotated registry credentials
  • imagePullSecrets still had old credentials
  • Deployments failed with "authentication required"
  • No one knew secret needed updating
  • Downtime: 4 hours (debugging + updating secrets)

Fix: Automate secret rotation. Monitor image pull errors.

Key Takeaways: Production Pitfalls

🎯 Health Checks Are Non-Negotiable

Every deployment needs liveness and readiness probes. Kubernetes can't self-heal if it doesn't know pods are unhealthy.

💾 Stateful Workloads Need Persistent Volumes

Pods are ephemeral. Data in pod filesystem dies with the pod. Use PVC for any data you care about.

🏢 Namespaces Provide Isolation

Don't run everything in default. Namespaces prevent naming conflicts, enable resource quotas, and provide security boundaries.

🐳 Image Management Matters

Use specific version tags. Validate images before deploying. Set up imagePullSecrets for private registries.

📊 Prevention Is Cheaper Than Cure

Every pitfall here has prevention strategies. Implement them before incidents occur. Reactive fixes cost 100x more.

🚢 The Shipping Port Analogy Holds

Health checks, persistent storage, organized sections—these are port operations basics. Kubernetes is no different.

⚠️ The Most Critical Pitfalls

Based on production incidents, these are the pitfalls that cause the most damage:

  1. Missing health checks → Complete outages, rolling update failures
  2. No persistent volumes for databases → Catastrophic data loss
  3. Running as root → Security breaches
  4. Secrets in images/git → Credential leaks, cryptocurrency mining
  5. No resource limits → Noisy neighbor, cascading failures

Priority: Address these first. They have the highest severity and frequency.
