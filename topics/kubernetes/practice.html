Exercise 1: Pod Lifecycle

Scenario

You deploy a pod with the following configuration:

apiVersion: v1
kind: Pod
metadata:
  name: test-pod
spec:
  containers:
  - name: app
    image: nginx:latest
    livenessProbe:
      httpGet:
        path: /health
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 10
      periodSeconds: 5

The nginx image listens on port 80, not 8080. The /health and /ready endpoints don't exist.

Questions

  1. What phase will the pod be in initially? (Pending, Running, Failed, Succeeded)
  2. What will happen to the pod after ~30 seconds? Why?
  3. What will the pod's restart count be after 2 minutes? Why?
  4. How would you fix this deployment?

Solution

Show Solution
  1. Phase: Running - The container starts successfully (nginx image exists), so pod phase is Running.
  2. After 30 seconds: Pod will be in CrashLoopBackOff. The livenessProbe will fail repeatedly because:
    • Wrong port (80 vs 8080)
    • /health endpoint doesn't exist
    • After 3 failed probes (15 seconds), K8s restarts container
    • Container crashes again, repeats with backoff
  3. Restart count: ~2-3 restarts after 2 minutes
    • First restart after ~15 seconds (3 livenessProbe failures)
    • Second restart after another ~15 seconds
    • Third restart might be in backoff period
  4. Fix: Change probe configuration: 
    livenessProbe:
  httpGet:
    path: /  # nginx default page
    port: 80  # correct port
readinessProbe:
  httpGet:
    path: /
    port: 80

    Or disable liveness/readiness probes if nginx default endpoints are acceptable.

Hands-On Try

  1. Create the broken YAML above: kubectl apply -f broken-pod.yaml
  2. Watch the pod: kubectl get pods -w
  3. Describe the pod: kubectl describe pod test-pod - look at Events
  4. Check logs: kubectl logs test-pod and kubectl logs test-pod --previous
  5. Fix the YAML, apply: kubectl apply -f fixed-pod.yaml
  6. Verify pod is Running and ready: kubectl get pods

Exercise 2: Service Networking

Scenario

You have a deployment with 3 replicas of a web application:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
        tier: frontend
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80

You create a service:

apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80

Questions

  1. How can you check if the service has endpoints?
  2. If you run kubectl get endpoints web-service and see <none>, what are 3 possible reasons?
  3. How do you test connectivity from a pod to the service?
  4. What's the difference between accessing web-service vs web-service.default.svc.cluster.local?

Solution

Show Solution
  1. Check endpoints: 
    kubectl get endpoints web-service
kubectl describe svc web-service  # Shows "Endpoints:" section
  2. Why no endpoints (<none>):
    1. Label mismatch: Pod labels don't match service selector
      • Pod has: app: web, tier: frontend
      • Service selects: app: web ✓ This should match
    2. Pods not Running: All pods are Pending/CrashLoopBackOff 
      kubectl get pods -l app=web
    3. Pods not Ready: Pods are Running but readiness probe failing 
      kubectl get pods -l app=web  # Check READY column
    4. Wrong namespace: Looking in wrong namespace for pods 
      kubectl get pods -A -l app=web
  3. Test connectivity: 
    # From a pod
kubectl run -it --rm debug --image=busybox --restart=Never -- wget -O- http://web-service

# Or with curl
kubectl run -it --rm debug --image=curlimages/curl --restart=Never -- curl http://web-service

# Test DNS resolution
kubectl run -it --rm debug --image=busybox --restart=Never -- nslookup web-service
  4. Service name formats:
    • web-service - Works only within same namespace
    • web-service.default.svc.cluster.local - Fully Qualified Domain Name (FQDN), works from any namespace
    • Format: <service>.<namespace>.svc.<domain>

Hands-On Try

  1. Deploy the resources: kubectl apply -f web-deployment.yaml and kubectl apply -f web-service.yaml
  2. Check endpoints: kubectl get endpoints web-service
  3. Test from debug pod: kubectl run -it --rm debug --image=busybox -- wget -O- http://web-service
  4. Break it: Change service selector to app: wrong-label
  5. Check endpoints again: kubectl get endpoints web-service - should be empty
  6. Fix selector back to app: web, verify endpoints return

Exercise 3: Resource Limits

Scenario

You have a node with 2 CPUs and 4GB RAM. You deploy these pods:

# Pod A
resources:
  requests:
    cpu: "1"
    memory: "1Gi"
  limits:
    cpu: "2"
    memory: "2Gi"

# Pod B
resources:
  requests:
    cpu: "1"
    memory: "1Gi"
  limits:
    cpu: "2"
    memory: "2Gi"

# Pod C
resources:
  requests:
    cpu: "500m"
    memory: "500Mi"
  limits:
    cpu: "1"
    memory: "1Gi"

All pods are Running. Now you deploy Pod D:

# Pod D
resources:
  requests:
    cpu: "1"
    memory: "1Gi"
  limits:
    cpu: "2"
    memory: "2Gi"

Questions

  1. What state will Pod D be in? Why?
  2. Which resource (CPU or memory) is the bottleneck?
  3. If Pod A starts using 1.5 CPUs (within its limit), what happens to Pod B?
  4. How would you verify which resource is preventing scheduling?

Solution

Show Solution
  1. Pod D state: Pending
    • Reason: Insufficient CPU requests on the node
    • Pod A, B, C request: 1000m + 1000m + 500m = 2500m CPU
    • Node has: 2000m CPU
    • Pod D needs: 1000m CPU
    • Total requested would exceed node capacity
  2. Bottleneck: CPU (not memory)
    • CPU requests: 2500m requested, 2000m available → 500m short
    • Memory requests: 2.5GB requested, 4GB available → 1.5GB free
    • K8s scheduler uses requests (not limits) for scheduling decisions
  3. If Pod A uses 1.5 CPUs:
    • Nothing happens to Pod B immediately
    • Pods can burst up to their limits (2 CPU for Pod A)
    • Pod A is within its limit, so no throttling or termination
    • However: If Pod A exceeds 2 CPU, it gets throttled (CPU limited, not killed)
    • If Pod A exceeds 2GB memory, it gets OOMKilled
  4. Verify bottleneck: 
    kubectl describe pod pod-d  # Look at Events section
# Output: "0/1 nodes are available: 1 Insufficient cpu"

kubectl describe node  # Look at "Allocated resources"
# Shows CPU Requests vs Capacity

Hands-On Try

  1. Check your node capacity: kubectl describe node | grep -A 5 "Capacity"
  2. Create Pod A: kubectl apply -f pod-a.yaml, check status: kubectl get pods
  3. Create Pod B: kubectl apply -f pod-b.yaml, check status
  4. Create Pod C: kubectl apply -f pod-c.yaml, check status
  5. Create Pod D: kubectl apply -f pod-d.yaml, check status (should be Pending)
  6. Run: kubectl describe pod pod-d - look at Events for "Insufficient cpu"
  7. Delete Pod A: kubectl delete pod pod-a
  8. Check Pod D status: kubectl get pods - should be scheduled now

Exercise 4: Deployment Strategy Design

Scenario

You're architecting the deployment strategy for a critical payment processing service with these requirements:

  • Must maintain 99.99% uptime (4.5 minutes downtime/month max)
  • Deployments happen 2-3 times per week
  • New versions must be tested with 1% of real traffic before full rollout
  • If issues detected, must rollback within 30 seconds
  • Database schema changes happen occasionally (backward incompatible)
  • Current setup: 20 replicas across 3 availability zones

Questions

  1. Would you use RollingUpdate or Recreate strategy? Explain the trade-offs.
  2. How would you implement the 1% traffic testing requirement?
  3. How would you ensure rollback within 30 seconds?
  4. How do you handle deployments that include database schema changes?
  5. What K8s features would you use and how would you configure them?

Solution

Show Solution
  1. Deployment Strategy: RollingUpdate
    • Why: Zero downtime is critical. Recreate would cause 30+ second gaps.
    • Configuration: 
      strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 25%        # Can create 5 extra pods during rollout
    maxUnavailable: 0    # Never take down existing pods until new ones ready
  revisionHistoryLimit: 10  # Keep 10 revisions for rollback
    • Why maxUnavailable: 0: Ensures capacity maintained during rollout
    • Challenge: Database schema changes still special handling (see Q4)
  2. 1% Traffic Testing: Use service mesh or custom routing
    • Option A (Service Mesh - Recommended): Istio/Linkerd traffic splitting 
      # Istio VirtualService
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: payment-service
spec:
  http:
  - match:
    - headers:
        x-canary:
          exact: "true"
    route:
    - destination:
        host: payment-service
        subset: v2  # 1% traffic to new version
      weight: 1
  - route:
    - destination:
        host: payment-service
        subset: v1  # 99% to old version
      weight: 99
    • Option B (Two Deployments): Run both versions, use separate services with different selectors, adjust service selectors gradually
    • Option C (Ingress Controller): Nginx/GCE ingress can do canary splits
    • Monitoring: Watch error rates, latency, payment success metrics during canary
  3. 30-Second Rollback:
    • Preparation: Use readiness probes to prevent traffic to unready pods 
      readinessProbe:
  httpGet:
    path: /healthz/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 2
  successThreshold: 2
    • Automated Rollback: Configure deployment to auto-rollback on failure 
      # Not native to K8s, use CI/CD or ArgoCD
# In CI/CD:
# 1. Deploy new version
# 2. Run smoke tests
# 3. If tests fail: kubectl rollout undo deployment/payment-service
    • Manual Rollback Command: 
      kubectl rollout undo deployment/payment-service

# Or rollback to specific revision
kubectl rollout undo deployment/payment-service --to-revision=5

# Rollback time: ~10-20 seconds (pod termination + new pod startup)
    • Practice: Run rollback drills quarterly to ensure team can execute in under 30 seconds
  4. Database Schema Changes: Requires careful orchestration
    • Backward Compatible Changes (Easy): Add columns, add tables → RollingUpdate works fine
    • Backward Incompatible Changes (Hard): Drop columns, rename fields, change types
    • Strategy - Blue-Green Deployment:
      1. Deploy v2 pods but don't send traffic (scale to 0 replicas initially)
      2. Run database migration script (v2 compatible)
      3. Start v2 pods, verify with smoke tests
      4. Switch traffic to v2 (service selector or ingress change)
      5. Monitor for issues, rollback if needed
      6. After success, scale down v1 deployment
      # Step 1: Deploy v2 (no traffic)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service-v2
spec:
  replicas: 0  # Start scaled down

# Step 2: Run DB migration
kubectl apply -f migration-v2.yaml
kubectl exec -it migration-job -- ./migrate.sh

# Step 3: Scale up v2
kubectl scale deployment payment-service-v2 --replicas=10

# Step 4: Switch traffic
kubectl patch svc payment-service -p '{"spec":{"selector":{"version":"v2"}}}'
    • Alternative: Expand-Contract Pattern (safer but slower)
      1. Deploy v2 that supports both old and new schema
      2. Migrate database to new schema (but keep old columns)
      3. Rollout v2 with RollingUpdate
      4. After v2 fully rolled out, drop old schema columns
  5. Complete Configuration: 
    apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
  labels:
    app: payment
    version: v1
spec:
  replicas: 20
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 5        # 25% of 20
      maxUnavailable: 0
  revisionHistoryLimit: 10
  minReadySeconds: 10   # Pod must be ready 10s before continues
  progressDeadlineSeconds: 600  # Fail rollout if takes >10 min
  selector:
    matchLabels:
      app: payment
  template:
    metadata:
      labels:
        app: payment
        version: v1
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - payment
              topologyKey: topology.kubernetes.io/zone
      containers:
      - name: payment
        image: payment-service:v1.2.3
        ports:
        - containerPort: 8080
        env:
        - name: VERSION
          value: "v1.2.3"
        resources:
          requests:
            cpu: "500m"
            memory: "512Mi"
          limits:
            cpu: "1"
            memory: "1Gi"
        livenessProbe:
          httpGet:
            path: /healthz/live
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
          failureThreshold: 3
        readinessProbe:
          httpGet:
            path: /healthz/ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 2
          successThreshold: 2
          failureThreshold: 2

Exercise 5: High Availability Architecture

Scenario

Your company is moving a monolithic application to Kubernetes. The application architecture:

  • Web frontend (stateless)
  • API gateway (stateless)
  • Business logic service (stateless, connects to cache and database)
  • Redis cache (stateful)
  • PostgreSQL database (stateful)

Requirements:

  • Survive loss of 1 availability zone
  • 99.9% uptime SLA
  • Handle 10x traffic spikes during peak hours
  • Data durability (RPO=0 for database)
  • Rolling upgrades without downtime

Questions

  1. How do you design the deployment to survive AZ loss?
  2. How do you handle 10x traffic spikes?
  3. How do you ensure data durability for PostgreSQL?
  4. What K8s resources would you use for each component?
  5. What's your monitoring and alerting strategy?

Solution

Show Solution
  1. Survive AZ Loss: Multi-AZ deployment with pod anti-affinity
    • Topology Spread Constraints: Distribute pods across AZs 
      apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-gateway
spec:
  replicas: 9  # 3 per AZ minimum
  template:
    spec:
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: ScheduleAnyway
        labelSelector:
          matchLabels:
            app: api-gateway
    • Why 9 replicas: If one AZ fails (3 replicas lost), still have 6 replicas remaining (2 per AZ)
    • Stateless components (Web, API, Business Logic): Deployments with multi-AZ spread
    • Stateful components (Redis, PostgreSQL):
      • Redis: Redis Cluster or Redis Sentinel with replicas across AZs
      • PostgreSQL: Patroni operator for HA with 3 nodes (1 per AZ)
  2. Handle 10x Traffic Spikes: Autoscaling + overprovisioning
    • Horizontal Pod Autoscaler (HPA): 
      apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-gateway-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-gateway
  minReplicas: 9
  maxReplicas: 100
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Pods
    pods:
      metric:
        name: requests_per_second
      target:
        type: AverageValue
        averageValue: "1000"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
      - type: Percent
        value: 100
        periodSeconds: 30
      - type: Pods
        value: 10
        periodSeconds: 30
      selectPolicy: Max
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60
    • Cluster Autoscaler: Automatically add nodes when pods can't be scheduled 
      # Cloud provider specific, e.g., AWS EKS
# Enable in node group settings
# Min nodes: 6 (2 per AZ)
# Max nodes: 50
    • Circuit Breakers: Protect downstream services from overload
      • Use service mesh (Istio) or application-level circuit breakers
      • Queue-based load leveling (Kafka/RabbitMQ) for async workloads
    • Pre-warming: If predictable spikes (daily 2pm peak), scale up proactively using KEDA or scheduled scaling
  3. Data Durability for PostgreSQL: Patroni + replication + backups
    • Patroni Operator: Manages PostgreSQL HA with automated failover 
      apiVersion: postgresql.k8s.enterprisedb.io/v1
kind: Cluster
metadata:
  name: postgres-cluster
spec:
  instances: 3
  primaryUpdateStrategy: unsupervised
  postgresql:
    parameters:
      max_connections: "200"
      shared_buffers: "256MB"
      synchronous_commit: "on"
  postgresqlConfiguration:
    synchronous_commit: "on"
    synchronous_standby_names: "ANY 1"
  bootstrap:
    initdb:
      database: appdb
      owner: appuser
  storage:
    size: 100Gi
    storageClass: gp3-encrypted
  walStorage:
    size: 50Gi
  affinity:
    podAntiAffinityType: required
    topologyKey: topology.kubernetes.io/zone
  monitoring:
    enabled: true
    • Replication: Synchronous replication to 2 standbys (1 in each other AZ)
    • Backups: Continuous archiving + daily full backups
      • WAL archiving to S3/GCS (continuous backup)
      • Daily full backups to separate region (disaster recovery)
      • Backup retention: 30 days
    • Recovery Testing: Quarterly restore drills to verify backups
  4. K8s Resources by Component:
    Component K8s Resource Configuration
    Web Frontend Deployment Replicas: 9, HPA enabled
    API Gateway Deployment Replicas: 9, HPA enabled
    Business Logic Service Deployment Replicas: 15, HPA enabled
    Redis StatefulSet 3 replicas (1 per AZ)
    PostgreSQL StatefulSet (via Patroni) 3 replicas (1 per AZ)
    All Services Service ClusterIP, topology-aware hints
    External Access Ingress Multiple ingress controllers for HA
  5. Monitoring and Alerting:
    • Metrics Collection: Prometheus + Grafana
      • Cluster metrics (node/pod resource usage)
      • Application metrics (request rate, error rate, latency)
      • Database metrics (connections, replication lag)
    • Logging: ELK Stack or Loki
      • Aggregate logs from all pods
      • Centralized query and alerting on logs
    • Alerts (PagerDuty/Slack): 
      # Critical alerts (page immediately)
- PodCrashLooping (5+ restarts in 10 min)
- DeploymentRolloutFailed
- DatabaseReplicationLag > 30s
- ErrorRate > 1%
- P99Latency > 1s
- APIDown (endpoint returning 5xx)

# Warning alerts (Slack, investigate next business hour)
- HighCPU (>80% for 5min)
- HighMemory (>85% for 5min)
- DiskSpaceLow (<20% free)
- ReplicaSetMismatch (desired != ready)
    • Dashboards:
      • Cluster overview (nodes, pods, resource usage)
      • Application performance (requests, errors, latency)
      • Database health (connections, replication, queries)
      • Business metrics (orders processed, payment success rate)

Exercise 6: ConfigMap vs Environment Variable Trade-offs

Scenario

Your team is deploying a microservice that needs configuration. The configuration includes:

  • Database connection string
  • API keys for external services
  • Feature flags (true/false values)
  • JSON-formatted rate limiting rules (100+ lines)
  • Logging configuration (log levels, output formats)

Some developers propose using environment variables for everything. Others suggest ConfigMaps for non-sensitive data.

Questions

  1. What are the trade-offs between ConfigMaps and environment variables?
  2. For each configuration item above, which approach would you use and why?
  3. How do you handle sensitive data (API keys, passwords)?
  4. How do you manage configuration differences between environments (dev, staging, prod)?
  5. What's a strategy that balances simplicity and best practices?

Solution

Show Solution
  1. Trade-offs: ConfigMaps vs Environment Variables
    Aspect Environment Variables ConfigMaps
    Simplicity ✅ Simple, works everywhere ❌ Extra K8s resource to manage
    Dynamic Updates ❌ Requires pod restart ✅ Can update without restart (if mounted as volume)
    Large Config ❌ Length limits, hard to read ✅ Handles multi-line configs
    Structured Data ❌ JSON in env vars is ugly ✅ Native support for structured data
    Portability ✅ Works outside K8s (Docker, etc.) ❌ K8s-specific
    Visibility ✅ Easy to see with kubectl describe ✅ Easy to see with kubectl describe
    Versioning ❌ In deployment manifests ✅ ConfigMaps are versionable resources
  2. Recommendation by Configuration Type:
    • Database Connection String: Environment Variable
      • Small, simple string
      • Often needs to be secret (see Q3)
      • Portable across different deployment targets
      • HOWEVER: If connection string has many parameters, consider ConfigMap for non-sensitive parts (host, port) + Secret for password
    • API Keys: Secret (not ConfigMap or env var directly)
      • NEVER commit secrets to Git
      • Use K8s Secret or external secret store (Vault, AWS Secrets Manager)
      • Mount as env var or file - both work, but env var is simpler
    • Feature Flags: Environment Variables
      • Simple true/false values
      • Easy to override per deployment
      • Quick to see what flags are set
      • Exception: If you have 50+ feature flags, consider ConfigMap to avoid cluttering env var list
    • Rate Limiting Rules (JSON, 100+ lines): ConfigMap (mounted as file)
      • Too large for env var (length limits, readability)
      • JSON is structured data - better as file
      • Can update rules without pod restart (if app watches file)
      • Easy to validate JSON in ConfigMap
    • Logging Configuration: ConfigMap (mounted as file)
      • Logging configs are often multi-line (log levels per package, output formats)
      • Structured format (YAML/JSON) works better as file
      • Can update log level without restart (if app supports hot reload)
  3. Sensitive Data Handling:
    • Option 1: K8s Secrets (baseline) 
      apiVersion: v1
kind: Secret
metadata:
  name: api-secrets
type: Opaque
stringData:
  api-key: sk_live_abc123...
  database-password: P@ssw0rd!

# Mount as env var in pod
env:
- name: API_KEY
  valueFrom:
    secretKeyRef:
      name: api-secrets
      key: api-key
    • Option 2: External Secret Store (recommended for production)
      • AWS Secrets Manager, Google Secret Manager, Azure Key Vault, HashiCorp Vault
      • Better audit logging, rotation, integration with cloud IAM
      • Use External Secrets Operator to sync into K8s: 
        apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: api-secrets
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: SecretStore
  target:
    name: api-secrets
    creationPolicy: Owner
  data:
  - secretKey: api-key
    remoteRef:
      key: prod/payment-service/api-key
    • Best Practices:
      • Enable encryption at rest for etcd (K8s secrets are base64 encoded, not encrypted by default)
      • Rotate secrets regularly
      • Use RBAC to restrict who can view secrets
      • Never commit secrets to Git
  4. Multi-Environment Configuration:
    • Strategy 1: Separate Namespaces 
      # Namespace per environment
namespaces:
  - payment-service-dev
  - payment-service-staging
  - payment-service-prod

# Different ConfigMaps per namespace
configmap-dev.yaml:     LOG_LEVEL=debug
configmap-staging.yaml:  LOG_LEVEL=info
configmap-prod.yaml:    LOG_LEVEL=warn
    • Strategy 2: Kustomize Overlays 
      # base/configmap.yaml
apiVersion: v1
kind: ConfigMap
data:
  log_level: info

# overlays/dev/kustomization.yaml
bases:
- ../../base
patchesStrategicMerge:
- configmap-dev.yaml

# overlays/dev/configmap-dev.yaml
apiVersion: v1
kind: ConfigMap
data:
  log_level: debug  # Override
    • Strategy 3: Helm Values 
      # values.yaml
config:
  logLevel: info

# values-dev.yaml
config:
  logLevel: debug

# values-prod.yaml
config:
  logLevel: warn
  5. Balanced Strategy:
    1. Small, simple configs → Environment Variables
      • Feature flags, simple settings
      • Portable, easy to understand
    2. Large, structured configs → ConfigMaps (mounted as files)
      • JSON/YAML configs, rate limiting rules
      • Supports hot reload without restart
    3. Sensitive data → Secrets or External Secret Store
      • API keys, passwords, certificates
      • Use K8s Secrets for dev/staging
      • Use external secret stores (Vault, AWS SM) for production
      • Environment differences → Helm or Kustomize
        • Same manifests, different values/overlays
        • GitOps for declarative configuration

Exercise 7: Troubleshooting Scenario - Production Incident

Scenario

Incident Timeline:

  • 10:00 AM: Deployment of new API version (v2.3.0) using RollingUpdate
  • 10:02 AM: Monitoring alerts: Error rate spike from 0.1% to 15%
  • 10:03 AM: Customer complaints: "API returning 500 errors"
  • 10:05 AM: You're paged to investigate

What you find:

  • Deployment has 15 replicas (old version v2.2.0)
  • New version v2.3.0 is rolling out
  • Rollout is stuck at 5 new pods (v2.3.0), 10 old pods (v2.2.0)
  • New pods keep crashing (CrashLoopBackOff)
  • Old pods are healthy but overloaded (handling 2x traffic)

Questions

  1. What's your immediate priority - fix the new version or restore capacity?
  2. What commands do you run to diagnose the issue?
  3. The pod logs show: Error: Database connection failed: invalid username. What happened?
  4. What's your rollback plan?
  5. After incident, how do you prevent this from happening again?

Solution

Show Solution
  1. Immediate Priority: Restore Capacity
    • Why: Old pods are healthy but only 10 replicas handling full load. New pods can't start. System at risk.
    • Action: Rollback immediately to restore capacity: 
      kubectl rollout undo deployment/api-service

# Verify rollback
kubectl rollout status deployment/api-service
kubectl get pods -l app=api-service
    • Alternative (if rollback fails): Scale up old version manually: 
      kubectl scale deployment/api-service --replicas=20
    • Don't waste time debugging first: Customers are affected. Restore service, then investigate root cause.
  2. Diagnostic Commands (After service restored): 
    # 1. Check deployment status
kubectl get deployments api-service
kubectl describe deployment api-service

# 2. Check pod status
kubectl get pods -l app=api-service
kubectl describe pod 

# 3. Check pod logs
kubectl logs 
kubectl logs  --previous  # If already crashed

# 4. Check events
kubectl get events --sort-by=.metadata.creationTimestamp

# 5. Check recent changes
kubectl rollout history deployment/api-service
kubectl diff deploy api-service  # If using GitOps

# 6. Check configuration
kubectl get configmaps -l app=api-service -o yaml
kubectl get secrets -l app=api-service -o yaml
  3. Root Cause - Database Username Error:
    • What happened: New version (v2.3.0) expects different database username or Secret wasn't updated
    • Possible causes:
      1. Deployment manifest wrong - references wrong Secret key
      2. Secret not created for new environment
      3. Code change broke environment variable loading
      4. Configuration drift - staging and prod Secrets different
    • Verification: 
      # Check what secrets are mounted
kubectl describe pod  | grep -A 10 "Mounts:"

# Check secret contents
kubectl get secret api-secrets -o yaml

# Check env vars in pod
kubectl exec  -- env | grep -i db
  4. Rollback Plan:
    1. Immediate: Rollback deployment 
      kubectl rollout undo deployment/api-service
    2. Verify: Check pods are healthy and error rate drops 
      kubectl rollout status deployment/api-service
kubectl get pods -l app=api-service

# Check error rate in monitoring dashboard
    3. If rollback fails: Emergency scaling 
      # Delete stuck deployment
kubectl delete deployment api-service

# Recreate old version from backup
kubectl apply -f api-service-v2.2.0-backup.yaml

# Or use saved rollout revision
kubectl rollout undo deployment/api-service --to-revision=5
    4. Communicate:
      • Update incident status channel
      • Notify customers (if major)
      • Document timeline for postmortem
  5. Prevention (Post-Incident Actions):
    • Pre-deployment Validation:
      1. Deploy to staging first, run smoke tests 
        kubectl apply -f api-service-v2.3.0.yaml -n staging
kubectl run smoke-test --image=tester -- ./test-api.sh
      2. Validate ConfigMaps and Secrets exist before deploying 
        # In CI/CD pipeline
kubectl get configmap api-config -n prod
kubectl get secret api-secrets -n prod

# Deploy only if resources exist
kubectl apply -f api-service.yaml
      3. Use dry-run to validate manifests 
        kubectl apply -f api-service.yaml --dry-run=client
    • Better Rollout Strategy:
      1. Reduce maxSurge to prevent too many new pods at once 
        rollingUpdate:
  maxSurge: 2  # Only 2 new pods at a time
  maxUnavailable: 0
      2. Add health check validation before continuing rollout 
        readinessProbe:
  httpGet:
    path: /healthz/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
  successThreshold: 3  # Must pass 3 times before ready
      3. Use progressive delivery (canary deployments)
        • Deploy 1 replica, test, then continue rollout
        • Use Argo Rollouts or Flagger for automated canary
    • Testing:
      1. Unit tests: Mock database connection, test config loading
      2. Integration tests: Test with real database in staging
      3. Contract tests: Verify API backward compatibility
    • Configuration Management:
      1. Use GitOps (ArgoCD, Flux) - all configuration in Git
      2. Validate ConfigMaps and Secrets in CI/CD before merge
      3. Use Kustomize or Helm to ensure consistency across environments
      4. Implement ConfigMaps/Secrets as part of deployment (always deployed together)
    • Monitoring Improvements:
      1. Add alert for deployment failures 
        # Alert: If rollout takes >15 minutes or fails
- DeploymentRolloutFailed
- PodCrashLooping (immediate page during rollout)
      2. Add deployment dashboards showing:
        • Rollout progress
        • Error rate by version
        • Pod health by version
    • Process:
      • Conduct postmortem within 48 hours
      • Create action items, assign owners, set deadlines
      • Follow up on action items in next sprint
      • Update runbooks with lessons learned

Exercise 8: NetworkPolicy Design

Scenario

You're securing a microservices application with these components:

  • Frontend (web) → API Gateway → Business Service → Database
  • Business Service → External API (payment processor)
  • Admin Service → Business Service (internal operations)
  • Monitoring (Prometheus) → All services (scraping metrics)

Security Requirements:

  • Frontend should only talk to API Gateway
  • API Gateway should only talk to Business Service
  • Only Business Service can talk to Database
  • Only Business Service can call external APIs
  • Admin Service can only talk to Business Service
  • Monitoring can scrape all services
  • Default deny: If not explicitly allowed, deny traffic

Questions

  1. How do you implement default deny for the namespace?
  2. What NetworkPolicies do you create for each service?
  3. How do you handle monitoring (Prometheus) that needs to talk to all services?
  4. How do you test that NetworkPolicies are working correctly?
  5. What's the maintenance burden of this approach?

Solution

Show Solution
  1. Default Deny: 
    apiVersion: networking.k8s.io/v1
kind:NetworkPolicy
metadata:
  name: default-deny
  namespace: production
spec:
  podSelector: {}  # Empty selector = all pods
  policyTypes:
  - Ingress
  - Egress

    What this does: Blocks all ingress and egress traffic to all pods in the namespace unless explicitly allowed by other policies.

  2. NetworkPolicies by Service: 
    # Frontend: Can only receive traffic from Ingress, can only talk to API Gateway
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: frontend-policy
spec:
  podSelector:
    matchLabels:
      app: frontend
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: ingress-nginx  # Allow from ingress controller
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: api-gateway
    ports:
    - protocol: TCP
      port: 8080

---

# API Gateway: Can receive from Frontend, can talk to Business Service
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-gateway-policy
spec:
  podSelector:
    matchLabels:
      app: api-gateway
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: business-service
    ports:
    - protocol: TCP
      port: 8080

---

# Business Service: Can receive from API Gateway and Admin Service
# Can talk to Database and external API
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: business-service-policy
spec:
  podSelector:
    matchLabels:
      app: business-service
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api-gateway
    - podSelector:
        matchLabels:
          app: admin-service
    ports:
    - protocol: TCP
      port: 8080
  egress:
  # Allow DNS
  - to:
    - namespaceSelector:
        matchLabels:
          name: kube-system
    ports:
    - protocol: UDP
      port: 53
  # Allow to Database
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 5432
  # Allow to external API (payment processor)
  - to:
    - namespaceSelector: {}  # Any namespace
    ports:
    - protocol: TCP
      port: 443

---

# Database: Can only receive from Business Service
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: database-policy
spec:
  podSelector:
    matchLabels:
      app: database
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: business-service
    ports:
    - protocol: TCP
      port: 5432

---

# Admin Service: Can receive from admin network (VPN), can talk to Business Service
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: admin-service-policy
spec:
  podSelector:
    matchLabels:
      app: admin-service
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - ipBlock:
        cidr: 10.0.0.0/8  # VPN network
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: business-service
    ports:
    - protocol: TCP
      port: 8080
  3. Monitoring (Prometheus) Access:
    • Option A: Add Prometheus podSelector to each policy
      • Add to ingress of each service policy
      • Flexible but verbose
    • Option B: Separate policy for monitoring (recommended) 
      # Allow Prometheus to scrape all services
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-monitoring
spec:
  podSelector: {}  # All pods
  policyTypes:
  - Ingress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: monitoring
    - podSelector:
        matchLabels:
          app: prometheus
    ports:
    - protocol: TCP
      port: 9090  # Metrics endpoint
    • Don't forget DNS: All services need to resolve DNS 
      # Allow DNS for all pods (usually in kube-system namespace)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          name: kube-system
    ports:
    - protocol: UDP
      port: 53
  4. Testing NetworkPolicies:
    • Test 1: Verify allowed traffic works 
      # From frontend pod, should reach API Gateway
kubectl exec -it frontend-xxx -- curl http://api-gateway:8080/healthz

# From API Gateway, should reach Business Service
kubectl exec -it api-gateway-xxx -- curl http://business-service:8080/healthz
    • Test 2: Verify denied traffic is blocked 
      # From frontend pod, should NOT reach Database
kubectl exec -it frontend-xxx -- curl http://database:5432
# Should timeout or get connection refused
    • Test 3: Use network policy testing tools
      • kubectl-netcheck - Tests connectivity between pods
      • NetworkPolicy Calculator - Visualizes policies
      • ci/dragon - Network policy audit tool
    • Test 4: Monitor and logs 
      # Check if policy is being applied
kubectl get networkpolicies

# Check pod labels match policy selectors
kubectl get pods --show-labels
  5. Maintenance Burden:
    • High initially, lower after setup:
      • Creating all policies takes time (1-2 days for complex app)
      • Testing and debugging is tedious
      • Once working, day-to-day maintenance is minimal
    • Ongoing maintenance tasks:
      • Adding new services: Must create/update policies (30 min per service)
      • Changing communication patterns: If service A now needs to talk to service C, update policies
      • Audit: Quarterly review of policies to remove stale rules
      • Documentation: Keep policy documentation in sync with reality
    • Ways to reduce burden:
      • Use CNI with better policy UI: Calico, Cilium have better visualization
      • Policy as Code: Generate policies from service definitions
      • Automated testing: Include policy testing in CI/CD
      • Start loose, tighten over time: Begin with broader policies, refine over months
    • Is it worth it?
      • Yes: For regulated industries (PCI, HIPAA), security compliance, production environments
      • Maybe: For internal tools, dev/test environments (depends on threat model)
      • No: For local development, simple prototypes (overkill)

Exercise 10: Design a Multi-Region Kubernetes Platform

Scenario

A global SaaS company wants to build a Kubernetes platform that serves customers in US, Europe, and Asia-Pacific with these requirements:

  • Latency < 100ms for 99% of requests
  • Data sovereignty: Customer data must remain in their region
  • Global consistency: User account changes propagate within 5 seconds
  • Disaster recovery: Survive loss of an entire region
  • Compliance: GDPR (Europe), CCPA (California), data localization laws
  • Cost optimization: Don't overprovision low-traffic regions
  • Operational simplicity: Small platform team (3 engineers)

Questions

  1. How many K8s clusters, how are they architected?
  2. How do you handle cross-region data replication and consistency?
  3. How do you achieve global load balancing with regional failover?
  4. How do you manage deployments across multiple regions?
  5. What's your observability and incident response strategy?
  6. Write an Architecture Decision Record (ADR) explaining your approach.

Solution Approach

Expert-level exercise with multiple valid approaches. Consider:

  • 3-6 clusters (one per region or per availability zone grouping)
  • Multi-master vs single-master per region
  • Global load balancer (Route53, Cloudflare, GSLB)
  • Database replication strategy (active-active vs active-passive)
  • GitOps with multi-cluster controllers (ArgoCD App of Apps)
  • Service mesh across regions (Istio multi-cluster)

In an interview, discuss trade-offs and justify your choices.
