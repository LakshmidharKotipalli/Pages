Configuration Files 6 locations
~/.claude/CLAUDE.md

Scope: Global — applies to all projects on this machine.

Contains: Universal preferences that apply regardless of project. Personal style preferences, default tool behaviors, global prohibited patterns, preferences about how Claude communicates.

Loaded: At session start, before project CLAUDE.md. First 200 lines are loaded eagerly. Remainder loads based on available token budget.

Best practice: Keep under 80 lines. These are your non-negotiables across all work.

./CLAUDE.md (project root)

Scope: Project — applies to all sessions in this project directory.

Contains: Project conventions, architecture notes, prohibited dependencies, required patterns, team standards, pointer to .claude/rules/ for domain-specific rules.

Loaded: After global CLAUDE.md. Git-versioned — changes require team review. This is the file that should be treated like an API contract.

Best practice: Under 100 lines. Move domain-specific rules to .claude/rules/. Version with pull requests.

.claude/rules/*.md

Scope: Module — applies when activated by context or invoked explicitly.

Contains: Domain-specific rules that are too detailed for the root CLAUDE.md. One file per domain: frontend.md, backend.md, infra.md, testing.md.

Loaded: After project CLAUDE.md, in alphabetical order by filename. Later files can override earlier files for the same rule.

Best practice: Use these to keep CLAUDE.md short. Separate by concern, not by length. Each file should be independently coherent.

.claude/settings.json

Scope: Project — technical configuration for Claude Code runtime.

Contains: MCP server definitions (name, command, arguments, environment variables), permission mode setting, tool access restrictions, model routing preferences.

Loaded: At session start, before CLAUDE.md. Changes require restarting the session to take effect.

Best practice: Version with git. Never commit API keys directly — use environment variable references. Review before every new team member onboards.

.claude/skills/*/SKILL.md

Scope: Project skills — custom skills defined for this project specifically.

Contains: YAML frontmatter (name, description, invocation trigger, required tools) followed by instruction markdown that Claude follows when the skill is invoked.

Loaded: Lazily — only when the skill is invoked. Not loaded at session start. Consumes context only when active.

Best practice: One skill per procedure. Name files to match the skill name. Add "defer to CLAUDE.md" clause to avoid convention conflicts with project rules.

~/.claude/projects/<project>/memory/

Scope: Project-specific persistent memory across sessions.

Contains: Auto-generated session notes from Claude, plus your manually written notes via /memory. Architecture decisions, key constraints, discovered patterns, session handoff notes.

Loaded: At session start, after CLAUDE.md. Content is summarized and injected into the context window automatically.

Best practice: Review periodically and remove stale entries. Use /memory before ending long sessions to write explicit handoff notes for the next session.

Permission Modes 3 modes
ask

Default Mode

Claude asks for explicit confirmation before every tool use: file reads, file writes, shell commands, MCP calls. Maximum human oversight.

Use for: Production environments, regulated industries, unfamiliar codebases, any situation where you need to review each action before it happens.

Trade-off: Slower due to confirmation prompts. Best for learning or high-stakes work.

auto-edit

File Autonomy

Claude can read and write files freely without asking. It still asks before running shell commands, MCP writes, or any operation with external side effects.

Use for: Most daily development work. The right balance for trusted codebases where you want speed on file operations but still review commands.

Trade-off: Files can change without per-file confirmation. Git diff and review before committing is more important here.

full-auto (YOLO)

Full Autonomy

Claude executes all operations without asking. File writes, shell commands, MCP operations — all run autonomously. Maximum throughput.

Use for: Trusted, isolated environments. CI/CD pipelines. Agent teams where the team lead coordinates rather than humans. Worktree-isolated work where mistakes can be discarded.

Caution: Never use in production systems. Never use in shared repositories. A mistake in full-auto requires a git reset rather than a simple "no" at the prompt.

Model Selection Guide 3 models
Model Capability Profile Ideal Tasks Avoid For Relative Cost
Opus Highest reasoning depth. Best at multi-step planning, nuanced trade-off analysis, novel problem framing. Architecture design, complex debugging with unclear root cause, system design decisions, code that requires deep contextual understanding. Routine code generation, simple lookups, sub-agent tasks with clear specifications, anything that Sonnet handles reliably. Highest — 3-5x Sonnet
Sonnet Strong reasoning with faster response time. Handles most software engineering tasks reliably. Good default for daily work. Feature implementation, code review, refactoring with moderate complexity, debugging with known symptom patterns, writing documentation. Simple one-line lookups where Haiku is sufficient. Highly complex architectural decisions where Opus is worth the cost. Balanced — good default
Haiku Fast and cost-effective. Best for well-defined tasks with clear specifications and low ambiguity tolerance. Sub-agent tasks with precise specifications, codebase search and exploration, simple file transformations, quick lookups, cost-sensitive bulk operations. Complex reasoning, ambiguous tasks, anything requiring multi-step inference or nuanced judgment. Haiku can miss subtle issues that Sonnet catches. Lowest — 10-20x cheaper than Opus
Routing pattern for agent teams: Main agent (Sonnet or Opus for complex work), Explore sub-agents (Haiku — they do targeted lookups with clear specs), Plan sub-agents (Sonnet — planning requires real reasoning), Bash sub-agents (Haiku — command execution is mechanical). This pattern reduces cost by 40-60% compared to running everything on Sonnet.

SKILL.md Structure Required format
SKILL.md is the open standard for agent skills. It works across Claude Code, OpenAI Codex, Cursor, Gemini CLI, and other agentic tools. A skill written once can be distributed to any compatible runtime.
YAML Frontmatter Fields (between --- markers)
Field Required Description
name Yes Human-readable skill name. Used in /help listings.
description Yes What the skill does. Used by the runtime to match natural language invocations.
invocation Yes The trigger pattern. Can be a slash command name, a natural language phrase, or both.
tools Recommended List of tools the skill needs access to. Runtime can verify permissions before invoking.
model Optional Preferred model for this skill. Overrides session default for skill execution.
version Optional Semantic version of the skill. Useful for skills distributed via marketplace or plugins.
Markdown Body

Everything after the closing --- is instruction markdown that Claude reads and follows when the skill is invoked. This is standard markdown — use headers, lists, and emphasis to structure the procedure clearly.

Recommended structure for the body:

1. One-paragraph description of what the skill accomplishes and when it should be used.

2. Prerequisite conditions — what must be true before the skill starts.

3. Step-by-step procedure — numbered list of actions. Be specific about what to check, read, write, or verify at each step.

4. Success criteria — how to know the skill completed correctly.

5. Failure handling — what to do if a step fails or produces unexpected results.

6. "Defer to CLAUDE.md for all project conventions" — prevents skill instructions from overriding project standards.

Hook Events Reference 5 events

Hooks are shell commands that run at specific session lifecycle events. They receive JSON context about the event and can take action, log, or modify the environment.

Event When It Fires JSON Context Available Common Uses
session.start When a new Claude Code session begins Session ID, project path, timestamp, user identity Usage tracking, environment validation, dependency checks, notification to team that a session started
session.resume When a paused or detached session resumes Session ID, previous end time, project path Logging session gap duration, refreshing environment variables, checking for upstream changes since last session
tool.pre Before any tool call executes (Bash, Read, Write, Edit, MCP, etc.) Tool name, tool parameters, session context Security scanning before writes, secret detection before commits, compliance checks, rate limiting, cost budgeting
tool.post After a tool call completes Tool name, parameters, result summary, duration Usage metering, audit logging, cost tracking, performance profiling
config.change When CLAUDE.md or settings.json changes Changed file path, change type (add/modify/delete) Notification when project configuration changes, triggering validation of the new configuration, team alerts for significant CLAUDE.md updates
Hook configuration location: Hooks are defined in .claude/settings.json under the hooks key. Each hook entry specifies the event, the shell command to run, and whether to block (await the hook result before continuing) or non-block (fire and continue).
Context Management Cheat Sheet Token budgets

Use these estimates to reason about context budget before starting work. Actual token counts vary by content, but these ballpark figures hold for typical code files and conversations.

Operation Approximate Token Cost Notes
Fresh session start 5,000 – 10,000 CLAUDE.md + system prompt + any auto memory loaded
Reading a small file (100 lines) +2,000 – 5,000 Dense code costs more than sparse documentation
Reading a large file (1,000 lines) +20,000 – 50,000 A single large file can consume 5-10% of a 1M token window
Sub-agent delegation +0 to main context Sub-agents have isolated windows. Only the result summary returns to main context.
Sub-agent result summary +1,000 – 5,000 The finding returned to main context, not the files the sub-agent read
/compact operation Reduces to ~20-30% of current usage With a good focus string, key context is preserved at high fidelity
Auto-compaction threshold Triggers at ~80% window utilization Blunt summarization — less precise than manual /compact with focus string
MCP tool descriptions (without lazy loading) +2,000 – 8,000 per server Each MCP server's tool manifest loaded at session start
MCP lazy loading (Tool Search) ~95% reduction vs. eager loading Only the tools actually used are loaded, not the entire server manifest
Performance Tuning Checklist Optimization guide

A team that applies these optimizations consistently can reduce Claude Code costs by 40-70% and reduce average session duration without reducing output quality.

Context Optimizations (highest impact):

Keep CLAUDE.md under 100 lines. Every line of CLAUDE.md occupies permanent space in every session. Bloated CLAUDE.md files are the single most common cause of unnecessary token consumption. Extract to .claude/rules/ modules for domain-specific content.

Use sub-agents for all research operations. Reading files to understand the codebase is exploratory work that belongs in a sub-agent's isolated context, not the main context. If you need to understand 20 files to plan a feature, spawn an Explore agent — do not read 20 files inline.

Run /compact proactively at task phase boundaries. Do not wait for auto-compaction. Manual compaction with a targeted focus string preserves significantly more useful context than automatic compaction at the 80% threshold.

Use /clear between unrelated tasks. Carrying context from a morning bug fix into an afternoon feature implementation adds noise and costs tokens. A fresh start costs 10 minutes of re-orientation and saves hours of context-related debugging.

Model Routing Optimizations (second highest impact):

Haiku for sub-agents with precise specifications. Explore sub-agents with clear tasks (find all functions matching pattern X) need fast lookup capability, not deep reasoning. Haiku is 10-20x cheaper than Opus for this workload with comparable output quality on well-specified tasks.

Sonnet for main agent default. Opus is warranted for genuinely complex architectural decisions. Most daily development work sits comfortably in Sonnet's capability profile. Reserve Opus for the 10-15% of tasks that genuinely require it.

MCP Optimizations (server connection management):

Enable lazy loading (Tool Search) for all MCP servers. This reduces per-session context overhead by ~95% for MCP tool descriptions. The tools load only when actually needed rather than pre-loading every server's full tool manifest at session start.

Limit active MCP connections to 5-7 servers. Each connected server adds overhead even with lazy loading. Disconnect servers not needed for the current project context. Re-connect when the project scope requires them.

Parallel Work Optimizations:

Use worktrees for parallel agent work. Sequential agent work is inherently serial and often avoidable. Two agents in two worktrees working on independent features complete in half the time of sequential execution.

Use agent teams for work with genuine parallelism. Do not apply agent teams to tasks that are fundamentally sequential. The coordination overhead (~20-30% of total time) is only justified when the parallel speedup exceeds that cost — typically tasks where at least 2 streams can run simultaneously for 60%+ of the total duration.
