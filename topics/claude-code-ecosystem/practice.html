Practice Exercises

Knowing the architecture is necessary but not sufficient. Fluency comes from being forced to apply the concepts under pressure — to design, diagnose, and decide without the answer in front of you.

Fifteen questions across four difficulty levels. Each one tests a different layer of the Claude Code mental model. Click the header to reveal the answer only after you have committed to your own response.

How to Use These Exercises

Read each question. Take 2-3 minutes to formulate your own answer before clicking to reveal. The goal is not to memorize the model answers — it is to develop the habit of structured reasoning about Claude Code problems. The model answers represent one strong response; your answer may be better in some dimensions. Compare and note where your reasoning diverged.

Film production analogy: these exercises are like table reads. The script is the question; your answer is the performance. Only after you have performed do you look at how the director would have staged it.

Warm-Up Questions

Orientation level — these verify that the fundamentals are solid. If any of these trip you up, revisit the Foundation and Context sections before moving forward.

Q1: You open a fresh Claude Code session. What are the first 3 things you check before starting work? Warm-Up 1

Model Answer:

1. Run /context to see what is loaded. A fresh session is not a blank slate — it loads CLAUDE.md, the system prompt, and potentially auto memory from previous sessions. Knowing the starting token budget and what is already present tells you how much room you have to work with before you start reading files or expanding context.

2. Check CLAUDE.md is current and relevant. Verify the file reflects your current intent for this session. If you are starting a new feature, does CLAUDE.md include the conventions for that feature area? If you recently changed your project's architecture, does CLAUDE.md reflect it? Stale instructions compound silently.

3. Verify MCP servers are connected for the tools you will need. If you plan to query a database, push to GitHub, or interact with external services through MCP, confirm those connections are live before you start. Discovering mid-task that your GitHub MCP server needs re-authentication is an unnecessary interruption.

Why this matters: These three checks take under two minutes and prevent the three most common mid-session disruptions: running out of context unexpectedly, discovering that CLAUDE.md does not cover your current task, and hitting a broken tool connection when you need it most.
Q2: Your colleague says Claude keeps forgetting their coding standards mid-session. What is your first diagnosis? Warm-Up 2

Model Answer:

First diagnosis: CLAUDE.md length and structure. The most common cause of mid-session standard drift is a CLAUDE.md that has grown organically and exceeded the reliable loading range. Check the line count — if it is over 100 lines, check whether the standards being "forgotten" are defined in the latter half of the file.

Second check: /context utilization. If the session runs for a long time or reads many large files, context utilization rises. Auto-compaction summarizes the conversation, including CLAUDE.md context, with less fidelity than the original. The standards are not forgotten — they are compressed into a summary that loses the specific wording.

Resolution: Extract to .claude/rules/ modules. Move service-specific or domain-specific rules into .claude/rules/frontend.md, .claude/rules/backend.md, etc. These load modularly and are less likely to be trimmed by compaction. Keep CLAUDE.md itself under 80 lines of high-priority, universal conventions.

The anti-pattern to name: The colleague likely treats CLAUDE.md as an append-only log of "things I told Claude." Every time Claude made a mistake, they added a rule to prevent it. After six months, the file is 400 lines of historical corrections that Claude cannot reliably load in full.
Q3: What is the difference between /compact and /clear? Warm-Up 3

Model Answer:

/compact is an editorial operation. It summarizes the conversation history into a condensed form, reducing token consumption by 70-80% while preserving the key decisions, context, and task state. The session continues — you do not lose your working context, just the verbosity. Think of it as editing the screenplay down to a shooting script. The intent is preserved; the exploration is compressed.

/clear is a reset operation. It wipes the entire conversation history. CLAUDE.md reloads from disk. Auto memory loads from the previous session notes. But the conversation is gone — every decision discussed, every file read in context, every intermediate result. You are starting a new session in an existing terminal window. Think of it as wrapping the current day of shooting and starting fresh tomorrow.

When to use each:

/compact: You are mid-task, context is getting heavy, but you want to continue the same thread of work with the key context preserved. Use it proactively before you hit the auto-compaction threshold.

/clear: The current session has gone off the rails, you are switching to a completely different task, or you want a clean slate without the accumulated drift of a long session. Also use it when starting work on a completely different area of the codebase.

Core Questions

Practitioner level — these test whether you can apply the architecture to real design decisions. A senior engineer should be able to answer these fluently.

Q4: Design a CLAUDE.md structure for a team of 12 working on a monorepo with 3 services. Core 1

Model Answer:

Root CLAUDE.md (target: under 80 lines): Universal conventions that apply to all three services. Commit message format and PR process. Naming conventions for variables, files, and branches. Prohibited dependencies or patterns that apply project-wide. A pointer to the .claude/rules/ directory for service-specific overrides. This file is the team constitution — it should change rarely and require team consensus to change.

.claude/rules/frontend.md: React component patterns, CSS methodology (Tailwind/styled-components), state management approach, accessibility requirements, bundle size budgets, and test patterns for UI components. Only applies when working in the frontend service directory.

.claude/rules/backend.md: API design conventions (REST patterns, error response format), database access patterns, authentication middleware requirements, logging standards, and performance thresholds. Only applies to backend service work.

.claude/rules/infra.md: Terraform patterns, naming conventions for cloud resources, required tags, cost optimization guidelines, and security group defaults. Only applies to infrastructure changes.

Optional extension: Each service directory can have its own CLAUDE.md for highly local overrides. The load order is global → root project → .claude/rules/ → local service directory CLAUDE.md. Later rules override earlier ones.
Governance note: Version these files with git. Require PR review for changes. When a rule changes, add a comment explaining why — future engineers (and Claude) need that context.
Q5: You need Claude to review a 50-file PR. How do you approach this without exhausting context? Core 2

Model Answer:

Do not read all 50 files into the main context. That approach will exhaust the window before the review is complete and produce degraded analysis for the later files as context pressure mounts.

Decompose by service or module boundary: Identify natural groupings in the 50 files. A 50-file PR touching three services likely has clear subsystem boundaries. Spawn one Explore or code-reviewer sub-agent per subsystem. Each agent receives: the files for its subsystem, the PR description, and a specific review mandate (security issues, convention violations, test coverage gaps, or performance regressions).

Each sub-agent runs in isolation with its own context window, reads its assigned files, and returns a structured findings summary. The main agent synthesizes the findings into a unified review document — noting cross-cutting concerns that appeared in multiple sub-agent reports.

If changes need to be tested: Use worktree isolation. Checkout the PR branch in a worktree, point a Bash agent at it to run the test suite, and return results to the main agent for inclusion in the review.

The result: A thorough review of all 50 files with zero context pressure on the main agent, because all the file reading happened in isolated sub-agent windows. The main agent only ever sees the structured summaries — a fraction of the token cost of reading the files directly.
Q6: When should you use a skill versus an MCP server versus a plugin? Core 3

Model Answer:

Use a Skill when: You have a repeatable procedure Claude should follow — a sequence of steps with decision points, not just a single action. A deployment checklist, a code review protocol, an onboarding workflow. Skills are Claude's procedures: they encode "how to do X" in a form Claude executes. Skills do not connect Claude to external data sources — they guide its behavior.

Use an MCP Server when: Claude needs to read from or write to an external data source or service that is not the local filesystem. Database queries, GitHub API calls, Jira ticket management, Slack messaging. MCP servers are Claude's senses and hands in the external world. Without MCP, Claude can only work with files on the local machine.

Use a Plugin when: You want to package skills, MCP server configurations, hooks, and commands together for distribution across a team or the community. A plugin is a deployable bundle — install it once and everything it provides is available. Plugins are the distribution mechanism for extensibility, not the extensibility itself.

The mental model: Skill = what to do. MCP = what to connect to. Plugin = how to distribute both. A mature workflow often involves all three: an MCP server that connects to your CI/CD system, a skill that orchestrates the deployment procedure, and a plugin that bundles both for team distribution.
Q7: An agent team has 3 teammates working on a feature. One discovers the database schema needs to change. What happens? Core 4

Model Answer:

The discovering teammate messages the team lead about the schema change requirement. This is the peer-to-peer communication mechanism of the TeammateTool — the discovering agent cannot unilaterally pause the other teammates or modify the shared task list. Only the team lead has that authority.

The team lead evaluates impact: Which of the other two teammates' current tasks are affected by the schema change? If Teammate 2 is updating the API layer that queries the affected tables, its current work will be based on the old schema and may need to be redone after the change. The team lead can message Teammate 2 to pause its current worktree until the schema change merges.

The schema change gets prioritized: The discovering teammate gets a revised task to implement the schema migration. The team lead updates the shared task list to reflect the new dependency ordering: schema migration must complete and merge before the API layer work resumes.

After the schema migration merges: Teammate 2's worktree gets rebased or the branch is updated to include the schema change. Work resumes with the correct foundation.

Why clear task boundaries matter: If the initial task definitions had included explicit schema dependencies ("this task assumes schema version X"), the team lead would have caught this conflict at task definition time rather than mid-execution. Pre-flight dependency analysis is cheaper than mid-flight coordination overhead.
Q8: Your company wants to track Claude Code usage for cost optimization. What hooks and metrics would you set up? Core 5

Model Answer:

Hooks to instrument:

session.start / session.end: Capture session duration (wall clock and active time), the developer identifier, and the project being worked on. This gives you per-developer per-project usage patterns and total session time trends.

tool.pre / tool.post: Capture which tools are invoked, which model is being used for each invocation, and approximate token counts if available in the JSON context. This reveals which tools dominate token consumption and whether model routing is being used appropriately.

config.change: Log whenever CLAUDE.md or settings.json changes. Useful for correlating configuration changes with cost spikes or drops.

Metrics to track: Tokens per session by model (Opus / Sonnet / Haiku ratio), cost per developer per week, sub-agent spawn rate (high rates may indicate task decomposition problems), context utilization at session end (consistently near 100% suggests sessions are too long or tasks too broad), and skill invocation frequency (reveals which procedures are most commonly automated).

Dashboard insight: The most actionable metric is the Opus-to-Haiku ratio. A team that runs all sub-agents on Opus and all lookups on Opus instead of Haiku can reduce costs by 40-60% through model routing alone, with no reduction in output quality.

Advanced Questions

Senior level — these require synthesizing multiple concepts and making architectural trade-off decisions. There is no single right answer, but strong answers name specific trade-offs.

Q9: Design an agent team configuration for migrating a monolith to microservices. Advanced 1

Model Answer:

Team Lead: Architect. Responsible for defining service boundaries based on domain analysis. Maintains the shared task list with explicit dependency ordering. Reviews and approves merge proposals from teammates. Does not write code directly — coordinates and reviews.

Teammate 1: API Extractor. Worktree: feature/extract-api-contracts. Task: Identify all REST endpoints in the monolith, document their contracts (request/response shape, auth requirements, error codes), and produce a service assignment map. Prerequisite: none. Blocker for: Teammates 2, 3, and 4 who all need to know the API contracts before proceeding.

Teammate 2: Data Modeler. Worktree: feature/data-models. Task: For each identified service boundary, define the schema that service owns. Identify data that crosses service boundaries (candidates for eventual consistency via events). Prerequisite: Teammate 1's service assignment map. Blocker for: Teammate 4's infrastructure work.

Teammate 3: Test Migrator. Worktree: feature/test-migration. Task: For each monolith test, classify it as unit/integration/e2e, map it to the service that will own it after extraction, and flag tests that implicitly cross service boundaries (candidates for new integration tests). Prerequisite: Teammate 1's service assignment map.

Teammate 4: Infrastructure. Worktree: feature/infra-services. Task: Create Terraform modules and Docker configurations for each new service. Set up service discovery, health checks, and basic observability. Prerequisite: Teammate 2's data models (to size databases correctly).

Coordination design: Teammate 1's output is the critical path. The team lead should not spawn Teammates 2, 3, and 4 until Teammate 1 has delivered a reviewed and approved service assignment map. Starting parallel work on contested boundaries wastes effort that gets discarded when the boundaries are revised.
Q10: Your CLAUDE.md has a rule that conflicts with a skill's instructions. What happens and how do you resolve it? Advanced 2

Model Answer:

What happens: CLAUDE.md loads at session start and is part of the base context. Skills load dynamically when invoked and inject their own instruction context alongside the existing context. If both are present and conflict, behavior is undefined — Claude will attempt to satisfy both and may follow either, depending on which instruction is more recent in its context window and how the compaction state has affected relative salience. Behavior will be inconsistent and may change between sessions.

Resolution Strategy 1: Make skills explicitly defer to project rules. Add a line to the skill's SKILL.md instructions: "For all style, architecture, and convention decisions, defer to the project's CLAUDE.md and .claude/rules/ files. This skill's instructions govern only the procedure defined here, not project-wide standards." This creates a precedence hierarchy in the instructions themselves.

Resolution Strategy 2: Remove the conflict at source. If a skill's instruction is substantively different from CLAUDE.md, one of them is wrong. Either update CLAUDE.md to include the skill's best practice project-wide (if the skill is right), or update the skill to match your project's standard (if CLAUDE.md is right).

The deeper problem: Conflicts between CLAUDE.md and skills usually indicate that skills were installed from the community without being reviewed for project compatibility. Every installed skill should be audited for conflicts before being added to a team project. The open skills standard makes cross-tool compatibility easy, but it does not make project-specific compatibility automatic.
Q11: How would you set up Claude Code for a regulated industry (healthcare/finance) with compliance requirements? Advanced 3

Model Answer:

Permission mode: always ask. Never auto-edit or full-auto in a regulated environment. Every file write, every command execution, every external API call requires explicit human review. The compliance requirement for human oversight of automated systems is not compatible with autonomous agent modes.

Hooks for compliance: pre-commit hook that runs secret scanning (no credentials, PII, or PHI in committed code), compliance template enforcement (required file headers, required audit log entries, required test coverage minimums). These hooks run before Claude's changes can be committed, creating a mandatory compliance gate.

CLAUDE.md: explicit prohibited and required patterns. Prohibited: logging of PHI/PII, storing credentials in code, direct production database writes. Required: all data access through audit-logged repositories, all external API calls through the approved integration layer, all user-facing changes with accessibility compliance notes.

MCP: read-only access to production data. Never configure write access to production databases or production systems through MCP in a regulated environment. Claude can query to understand the current state; all writes go through the same reviewed deployment pipeline as human-authored code.

Auto memory: controlled or disabled. Auto memory stores session notes that may contain sensitive architectural details. Either disable it or implement audit logging of all auto memory entries. Any persistent storage of session content falls under data retention policies.

Agent teams: defer until security review completes. The experimental nature of agent teams and the peer-to-peer communication patterns create audit gaps that most regulated industries cannot accept without a formal security review and documented risk acceptance.
Q12: Compare the context management strategies for a 30-minute bug fix versus a 4-hour feature implementation. Advanced 4

Model Answer:

Bug fix (30 minutes): Minimal context management needed. /clear to start with a clean slate. State the bug: the symptom, the file it manifests in, and the last known working state if applicable. Read only the relevant files — resist the temptation to read the full codebase for context. Fix, test, done. The short duration means auto-compaction will not trigger, and context pressure will not be a factor unless the bug spans many files.

Feature implementation (4 hours): Context management is an active practice, not a background concern. The session has a lifecycle:

Phase 1 — Research (45-60 minutes): Spawn Explore sub-agents to read relevant subsystems. Protect the main context from file reading overhead. At the end of research, run /compact "focus on architecture decisions and key constraints for [feature name]" to lock in the relevant findings before implementation begins.

Phase 2 — Implementation (2-2.5 hours): Monitor context with /context every 30 minutes. Delegate file-heavy tasks to sub-agents. At the end of phase 2, run /compact "focus on remaining tasks and decisions made so far" before entering the testing phase.

Phase 3 — Testing (45-60 minutes): Run /context to check remaining room. If utilization is above 70%, consider using a Bash sub-agent to run the test suite rather than running tests inline. Close the session with a /memory entry summarizing what was built and what was deferred.

The underlying principle: Context management scales with task duration and complexity. A 30-minute task runs in a single breath. A 4-hour task requires deliberate pacing, like a long film shoot with scheduled breaks and director reviews between scenes.

Expert Questions

Architect level — these ask you to critique the current architecture and think about its limits. Strong answers acknowledge what the current design does well before identifying its constraints.

Q13: If you were designing Claude Code v3.0, what architectural changes would you make to context management? Expert 1

Model Answer:

The fundamental problem with the current model: The context window is a single flat container. Everything goes in the same space — CLAUDE.md, conversation history, file contents, tool results. When it fills up, everything compresses together with no differentiation between what must be preserved (architectural decisions, CLAUDE.md rules) and what can be compressed (file contents that were read for a single lookup).

Proposed: Structured context tiers. Pinned (never compacted): CLAUDE.md, active task state, key architectural decisions. This tier persists at full fidelity for the entire session. Warm (compacted lazily): Recent conversation, recent file reads, current sub-agent results. Standard compaction applies here. Cold (evicted to indexed storage): Earlier conversation, files read early in the session, completed sub-task results. Retrievable on demand via context lookup, but not consuming active window space.

Proposed: Per-topic compaction. Instead of compacting the entire conversation history when the window fills, compact only the topics that are no longer active. "We resolved the database schema question two hours ago — compact that thread." Preserve the threads that are still in active use at full fidelity.

Proposed: Skill-declared context contracts. Skills declare their minimum context requirements in YAML frontmatter. The runtime guarantees those requirements are met before invoking the skill. A skill that needs the full CLAUDE.md declares it; a skill that only needs the current task state declares that. This prevents skills from silently degrading when context is tight.

What the current design does right: The simple flat window is predictable and easy to reason about. Developers can run /context and immediately understand their situation. The proposed tiered model adds power but also adds cognitive complexity. The design challenge is preserving that transparency while adding the structural differentiation that makes longer sessions more reliable.
Q14: How do agent teams fundamentally change the software development lifecycle compared to sub-agents? Expert 2

Model Answer:

Sub-agents: human-in-the-loop with AI assistance. The human directs every delegation. The human decides which sub-agent to spawn, what to ask, and what to do with the result. The AI executes bounded tasks; the human maintains strategic coherence. The SDLC is: human identifies task, human delegates to AI, AI executes, human reviews, human decides next step. The developer remains the active decision-maker throughout.

Agent teams: AI-in-the-loop with human oversight. The team lead agent decides which teammates to spawn, how to decompose the task, and how to sequence the work. The human defines the goal and reviews the result, but is not in the decision loop for intermediate steps. The SDLC becomes: human defines goal, team lead decomposes into work, teammates execute in parallel, team lead synthesizes, human reviews and ships.

The shift in developer role: With sub-agents, you are an active director making constant decisions. With agent teams, you are an executive producer — you define the vision, set the constraints, and review the output, but the director (team lead agent) and crew (teammates) do the implementation work. The skills required change: less "how do I implement this" and more "how do I define this clearly enough that an AI team can implement it without me."

The SDLC transformation: Define (human) → Decompose (team lead agent) → Implement in parallel (teammates) → Synthesize (team lead agent) → Review (human) → Ship (human or agent). The implementation phases become substantially automated, shifting human time from writing code to defining requirements and reviewing output.

The unanswered question: As agent teams become more capable, where does the human's technical judgment get exercised? If agents make the implementation decisions, the human's critical path becomes requirements definition and output review. This is a different skill profile than traditional software development — closer to systems design and quality assurance than to implementation.
Q15: What are the limits of the current Claude Code architecture and what problems can it not solve well? Expert 3

Model Answer:

Limit 1: Context window ceiling for truly large codebases. Even a 1M token context window is insufficient for analyzing a mature enterprise codebase with tens of millions of lines of code. Sub-agents help by distributing the reading, but synthesis across massively decomposed analyses loses fidelity. The solution today (multiple analysis passes) is expensive and imperfect.

Limit 2: No persistent state beyond text files. Auto memory is text. CLAUDE.md is text. Between sessions, Claude's "memory" is whatever was written to these files. There is no persistent model of the codebase, no evolving understanding of project history, no memory of "we tried this pattern six months ago and it failed." Every session, Claude starts from the same base model with only the text notes as continuity.

Limit 3: Agent teams coordination overhead can exceed benefit. For genuinely parallel, well-separated tasks, agent teams provide clear value. For tasks with emergent dependencies (which are most real-world features), the coordination overhead of a team can exceed the benefit. The feature that seems decomposable often is not, and discovering mid-execution that Teammate 2's work blocks Teammate 3's work requires expensive re-coordination.

Limit 4: No visual understanding (without browser integration). Claude Code cannot look at a running application and debug a visual rendering issue. It can read the code that produces the UI but cannot observe the result. Claude in Chrome addresses this for web interfaces, but the core CLI is blind to visual output.

Limit 5: Cost scales linearly with usage. There is no caching of common patterns across sessions. Every session that reads the same architectural documentation or the same library files pays the same token cost. The more complex the project, the higher the fixed session cost, with no amortization across sessions or team members.

What it does exceptionally well: Bounded, well-defined tasks with clear success criteria. Code generation from clear specifications. Codebase exploration and synthesis. Procedural task execution (builds, deploys, refactors with defined scope). The limits appear at the edges — extremely large codebases, long-running projects, highly visual work, and tasks where the definition of done is inherently ambiguous.
