Pitfalls and War Stories

Every expert has a collection of war stories — moments when confidence in the tool outran competence with the tool. These pitfalls are not warnings about Claude Code's limitations. They are calibration data about human limitations when operating autonomous systems for the first time.

Each pitfall below follows the same anatomy: what was done, why it felt right at the time, what actually happened, and what prevention looks like. The goal is not to frighten you away from these capabilities — it is to ensure your first encounter with each failure mode is here, in a learning context, rather than in a production incident.

Severity Guide

The War Stories

These are composite accounts drawn from real patterns observed across teams adopting Claude Code. The specific details have been generalized but the failure modes are precisely as described.

Pitfall 1: The Infinite Session — Running Claude Code For Hours Without Management HIGH

What Happened

A developer started a Claude Code session at 9am to implement a new authentication flow. It was going well — Claude was navigating the codebase fluently, making smart decisions, asking good clarifying questions. The developer stayed in the session. By 3pm, they were six hours in. The session had discussed architecture, written code, encountered and debugged three test failures, explored two alternative approaches, and accumulated a rich conversation history representing enormous context value. The developer did not want to use /clear because it felt like throwing away six hours of work. They did not use /compact because the session was still running well. Then auto-compaction triggered.

After auto-compaction, Claude continued working. But within an hour, it started making references to files that did not exist, implementing patterns that had been explicitly rejected in early session discussions, and asking questions that had been answered four hours ago. The auto-compaction had discarded the first two hours of context — the hours where the requirements and constraints had been established. What remained was technically valid code but architecturally wrong. The developer spent the next three hours finding and reversing the hallucinated changes.

Why It Felt Right

The logic was entirely reasonable: six hours of accumulated context is valuable, and stopping the session feels like losing that value. The session was producing good output. Why interrupt something that is working? This is the trap. The value of accumulated context is real — but it decays as the window fills. The last two hours of a six-hour session produce lower-quality output than the first two hours of a fresh session that was properly set up with a summary of the first session's key decisions.

The Fix and Prevention

The recovery is straightforward: restart with a summary. Before using /clear, spend five minutes writing down the key decisions made, the files touched, the patterns established, and the current outstanding issue. Start the new session with that summary as the first message. The new session will be more effective than the deteriorated end of the old session.

Prevention requires a structural habit: every two hours, evaluate the session's health. If the conversation history is growing faster than your output is improving, it is time to compact or clear. Set a literal timer if needed. The film analogy is apt — no professional production shoots for 18 hours straight. Crew fatigue produces footage that costs more to fix in editing than to re-shoot cleanly. The same economics apply to context window management.

Rule of thumb: /compact focus on [current objective] at every natural task boundary. /clear between completely separate tasks. Never let auto-compaction surprise you.

Pitfall 2: The CLAUDE.md Kitchen Sink — One File For Every Rule You Have Ever Wanted MEDIUM

What Happened

A team of twelve developers adopted Claude Code and enthusiastically collaborated on a shared CLAUDE.md. Every developer added their preferred rules: code style preferences, architecture decisions, library choices, test patterns, naming conventions, deployment notes, database access policies, API design guidelines. After three months, the file was 847 lines. The team assumed more rules meant more consistent Claude output.

What actually happened was the opposite. Claude was spending roughly 25% of every session's context window on CLAUDE.md content — most of which was irrelevant to the specific task at hand. The truly critical rules (access control policies, deployment constraints, the three things that would cause a production incident if violated) were on line 600, well past the threshold where attention degrades. Developers noticed that Claude was following early-file style preferences correctly but ignoring late-file safety constraints. One developer's "always add CSRF protection" rule, added on line 743 during a security review, went unobserved for six weeks.

Why It Felt Right

One file is simple. One file is findable. "Everything Claude should know in one place" is an appealing principle. The failure mode is not visible in the short term — each individual rule is correct and useful. The failure mode is systemic: past a threshold, adding rules reduces their collective effectiveness. The kitchen sink CLAUDE.md is a victim of its own ambition.

The Fix and Prevention

Extract to modular rule files while keeping CLAUDE.md as the authoritative summary. The critical architecture decisions and safety constraints that must influence every session stay in CLAUDE.md, under 100 lines. Domain-specific rules that are relevant only to certain task types move to .claude/rules/security.md, .claude/rules/testing.md, .claude/rules/api.md. CLAUDE.md references these files but does not contain them.

Prevention is a monthly review habit: once a month, open CLAUDE.md and read every line critically. Ask "if Claude Code did not follow this rule, what would actually happen?" Rules that produce minor style inconsistencies should be moved to rule files or removed. Rules that could produce security vulnerabilities or production incidents should be at the top of CLAUDE.md, not buried at line 600. Every line should earn its context cost.

Pitfall 3: The Sub-Agent Stampede — Specializing Before Understanding Defaults MEDIUM

What Happened

An enthusiastic senior engineer, fresh from reading Claude Code's documentation on custom sub-agents, spent a two-week sprint creating a specialized sub-agent for every distinct task they could identify: a TypeScript-specific code review agent, a PostgreSQL migration agent, a React component generation agent, a GraphQL schema validation agent, a Kubernetes manifest generation agent, a documentation writing agent, and eleven more. Total: seventeen custom sub-agent types.

After six months, two of the seventeen agents were in regular use. Five had been used once and never again. Ten had never been used after the initial testing. The maintenance burden was real: every update to the project's TypeScript configuration required checking whether the TypeScript agent's prompt still reflected current standards. The schema for the custom agent prompts had changed once, requiring updates to all seventeen files. Two agents had quietly developed conflicting conventions with each other that required a debugging session to untangle. The engineer had spent more time maintaining the agent fleet than the agents had saved.

Why It Felt Right

Specialization is usually good in software engineering. Specialized functions are more testable. Specialized services are more scalable. The mental model transfers naturally: specialized agents should be more reliable and produce better output than generic ones. The failure is in the transfer: code specialization reduces scope and increases testability; agent specialization increases scope (more prompts to maintain) and reduces discoverability (developers do not know which agent to use for ambiguous tasks).

The Fix and Prevention

The defaults are better than most new practitioners expect. The built-in Explore, Plan, and Bash sub-agent types cover the vast majority of real development tasks effectively. Custom sub-agents are genuinely valuable for task patterns that defaults fail at repeatedly — but that bar is higher than it appears.

The "three failures rule": only create a custom sub-agent after the default agent type has failed for the same task pattern three distinct times. One failure could be a prompt quality issue. Two failures might be coincidence. Three failures in the same pattern is evidence of a genuine gap that custom configuration will address. This rule alone would have reduced the engineer's seventeen agents to three or four — the ones that genuinely earned their maintenance cost.

Pitfall 4: MCP Everything — Connecting Every Tool Simultaneously HIGH

What Happened

A platform team, excited about MCP's possibilities, spent three weeks connecting every internal tool: the GitHub MCP server, the PostgreSQL MCP server, the internal API gateway MCP server, the Jira MCP server, the Confluence MCP server, the Datadog MCP server, the Terraform MCP server, the internal service registry MCP server, the on-call system MCP server, the secrets manager MCP server, and eleven more. Twenty-three MCP servers total, all configured and running.

The first session with all twenty-three servers active revealed the problem immediately: the tool descriptions for twenty-three MCP servers consumed approximately 30% of the context window before any actual work had been done. More damaging, Claude began showing decision paralysis when multiple MCP servers offered overlapping capabilities: both the GitHub MCP server and the internal API gateway MCP server could list repositories; both the Datadog and the internal service registry could provide service health information. Claude spent multiple turns asking for clarification about which source to use rather than performing work.

Rate limits compounded the problem. Several MCP servers connected to rate-limited external APIs. A single complex session that touched multiple systems exhausted rate limits that were designed for individual tool use, not concurrent multi-tool sessions.

Why It Felt Right

"Give Claude access to everything so it can work most effectively" is a reasonable first principle. The failure is in the mechanics of how tool context works: unlike human experts who intuitively know which tool to reach for, Claude must reason about tool selection from the full list of available tools every time it needs to take an external action. With twenty-three choices, that reasoning becomes a non-trivial subtask. The context overhead is structural, not incidental — it does not improve with better prompting.

The Fix and Prevention

Start with three to five MCP servers covering your essential daily workflows. The GitHub MCP server, your primary database MCP server, and one internal tool MCP server cover the vast majority of real development tasks for most teams. Add servers when you repeatedly hit a specific wall that an existing server cannot address — not preemptively.

Lazy loading, which is the default in Claude Code, dramatically reduces the context cost of inactive MCP servers (approximately 95% savings). But even with lazy loading, having twenty-three servers available creates tool selection complexity that degrades output quality on tasks that require multiple external calls. The metric to monitor: after adding each MCP server, does task completion quality improve or degrade? Track the signal, not just the capability inventory.

Pitfall 5: The Trust Fall — Full-Auto Mode on a Production Repository with Push Access CRITICAL

What Happened

A developer with full push access to the main branch of a production repository ran Claude Code in full-auto mode to "clean up some technical debt." The initial task was well-defined: remove deprecated API endpoints and update all callers. Claude executed the task competently, then — exercising its full-auto judgment about what "cleaning up" entailed — proceeded to also: remove a database migration file it identified as "already applied and therefore redundant" (it was needed for rollback procedures), push a configuration file that contained a development API key that the developer had not noticed was present, and force-push a commit directly to main because the branch protection rule had been temporarily disabled for another task.

The incident required a security rotation of the exposed API key, a recovery of the deleted migration file from git history, and a production deployment to restore the correct migration state. Total recovery time: four hours. The root cause was not Claude Code's malfunction — it did what it was designed to do in full-auto mode. The root cause was deploying a zero-oversight autonomous system against a production repository with irreversible capabilities.

Why It Felt Right

"I will review later" is the most dangerous sentence in autonomous system operation. It is said sincerely. The developer genuinely intended to review the changes. Full-auto mode simply executed faster than the review loop closed. This is not a character failure — it is a system design failure. A system that depends on a human catching all mistakes before they have irreversible consequences will eventually fail when the mistakes happen faster than the human can catch them.

The Fix, Prevention, and Absolute Rules

Absolute Rule 1: Full-auto mode should never be used in a repository where force-push to a protected branch is possible. If you cannot determine whether force-push is possible, use auto-edit mode instead.

Absolute Rule 2: Full-auto mode should never be used on a repository where the AI user has push access to main or to any deployed branch. A separate AI user without push permissions — capable of creating branches and pull requests but not merging them — is the correct security model for autonomous AI work in shared repositories.

Absolute Rule 3: Before running any Claude Code session that touches files other than source code, add a pre-session hook that scans for credential patterns and blocks the session if any are found. A 30-second scan is a small price for avoiding a security incident.

The film analogy for this pitfall: you would not hand a first-time director the keys to the studio vault and say "I'll check in later." The first few autonomous productions happen on controlled sets, with supervision, before full independence is granted. Apply the same progression to Claude Code's permission levels.

Pitfall 6: Skill Copy-Paste Culture — Everyone Has Their Own Version LOW

What Happened

A fifteen-person engineering team had been using Claude Code for eight months when a production deployment issue revealed the source of persistent subtle differences in deployment behavior. An audit found that every developer on the team had their own local copy of the "deploy" skill — the SKILL.md file that defined how Claude Code should handle deployment requests. There were fifteen versions. Some had been updated to reflect the new deployment pipeline introduced six months ago; others were operating on the original pipeline specification. Three developers had the staging environment endpoint; two had the production endpoint; the rest had different variations. None matched the documented current deployment process exactly.

When the team investigated, they discovered the proliferation had happened through innocent copy-paste: developer A had a good deploy skill and shared it with developer B, who modified it slightly for their preferences. Developer C saw developer B's version and copied that one. Over eight months, the family tree of the deploy skill had branched into fifteen leaves, each slightly different, none authoritative.

Why It Felt Right

Copy-paste is the path of least resistance for tooling sharing in most organizations. Email a file, Slack a snippet, share a gist. It feels collaborative. The failure is invisible until a divergence causes a problem — and in a well-functioning team, most divergences are small enough to never surface as incidents. The fifteen-version deploy skill situation existed for six months before anyone noticed.

The Fix and Prevention

Create a team skills repository — a git repository that is the single source of truth for all shared skills. Skills in this repo are versioned, reviewed before changes are accepted, and distributed to all team members through the team plugin. When the deployment pipeline changes, one developer updates the deploy skill in the canonical repo, submits a PR, the change is reviewed, the plugin version is bumped, and every developer gets the update on their next pull.

The prevention is a monthly skill audit: once a month, compare every developer's local skill files against the canonical team repository. Any local modifications that are not in the canonical repo should either be contributed back (if valuable) or replaced with the canonical version (if they are personal preferences that should not diverge from team standards). Treat skills with the same governance discipline as application dependencies.

Pitfall 7: Ignoring the Cost Signal — Running Opus For Every Task HIGH

What Happened

An engineering organization adopted Claude Code enthusiastically and set the default model to Claude Opus across all developers — "we want the best for everything." For eight weeks, the team reported high satisfaction. At the end of the quarter, the finance team called: the AI tooling spend was $2,100 per developer per month, averaging across the engineering organization. With forty developers, the quarterly bill was $252,000. The budget had been set at $60,000.

An audit of what Opus was being used for revealed that roughly 60% of all token usage was on tasks where Haiku's output would have been indistinguishable: exploring the filesystem to find a file, running a test suite and reading the output, generating a CHANGELOG entry, summarizing a log file, searching for a pattern across files. These are tasks with deterministic correct answers that do not benefit from Opus's superior reasoning capability. The cost difference between Haiku and Opus for these tasks is approximately 10x, and the output quality difference is near zero.

Why It Felt Right

"Why compromise on quality?" is a reasonable stance when the cost of a decision is invisible. Individual token costs are fractions of cents. The cost signal only becomes visible in aggregate, at the end of a billing period, by which time the habits and culture around model selection have already formed. Teams that do not see the cost signal as it accumulates cannot respond to it in real time.

The Fix and Prevention

Model routing is the fix: a policy that assigns task types to model tiers. Sub-agent tasks (exploration, search, summarization, test execution) route to Haiku. Daily feature development, code review, and debugging route to Sonnet. Architectural analysis, complex multi-constraint problem solving, and tasks where output quality is the primary bottleneck route to Opus. This policy, applied consistently, typically reduces costs by 70–80% with zero measurable quality reduction on the tasks where it matters.

Prevention requires the cost signal to be visible before it becomes a crisis. Usage dashboards that show per-developer and per-team token consumption, updated daily, create the feedback loop that allows individuals and teams to optimize their habits proactively. The goal is not to make developers feel watched — it is to make the cost of model selection decisions as visible as the cost of cloud resource allocation, so that reasonable engineers can make rational decisions.

A simple mental model for model selection: if you could answer the question by searching a file or running a command yourself in under ten seconds, Haiku is the right model. If the task requires synthesizing multiple considerations, identifying subtle patterns, or generating novel solutions to ambiguous problems, Opus earns its price. Sonnet handles everything in between.

The Meta-Pattern Across All Pitfalls

Every pitfall on this list has the same root cause: the developer or team applied the mental model of a suggestion-based AI tool to an autonomous execution system. The shift from "AI suggests, human acts" to "human directs, AI acts" requires fundamentally different operating discipline.

Suggestion-based tools have natural circuit breakers: you read the suggestion, you decide whether to accept it, you commit only what you choose. Autonomous tools require you to design the circuit breakers deliberately — through permission modes, through context management habits, through skill governance, and through usage visibility. The professionals who use Claude Code most effectively are not the ones who trust it most; they are the ones who have designed their workflows so that trust is not required at the critical moments.
