🟢 Warm-up Exercises (Beginner)

Q1: What are the 5 stages of the RAG pipeline? ▼

Answer:

  1. Query Preprocessing: Clean, rewrite, expand queries
  2. Retrieval: Search vector database for relevant documents
  3. Reranking: Reorder results for precision
  4. Prompt Assembly: Combine context with instructions and query
  5. Generation: LLM produces grounded response
Q2: What's the difference between vector search and keyword search? ▼

Answer:

Vector search: Uses embeddings to find semantically similar content. Great for concepts and meaning. "Why am I tired?" finds relevant documents about fatigue, sleep, energy.

Keyword search (BM25): Finds exact term matches. Great for specific names, codes, technical terms. "Find document about API-1234" works better with keyword search.

Hybrid search: Combines both for best results.

Q3: Why do we chunk documents before embedding? ▼

Answer:

  • Context window limits: LLMs can't process entire documents at once
  • Precision: Smaller chunks allow more precise retrieval
  • Cost: Embedding smaller chunks is cheaper
  • Flexibility: Can retrieve multiple relevant sections from one document
Q4: What is cosine similarity? ▼

Answer:

Cosine similarity measures how similar two vectors are by calculating the cosine of the angle between them.

Range: -1 to 1

  • 1: Identical (same direction)
  • 0: Orthogonal (unrelated)
  • -1: Opposite (different meaning)

Used in RAG to: Find document chunks most similar to the query embedding.

Q5: What's the main benefit of RAG over fine-tuning? ▼

Answer:

RAG advantages:

  • Timeliness: Can access current information (update vector DB in minutes)
  • Privacy: Keep data in your infrastructure
  • Attribution: Can cite sources
  • Cost: Much cheaper than retraining
  • Flexibility: Update knowledge without model changes

Fine-tuning is better for: Format, style, domain-specific language, behavior.

🟡 Core Exercises (Intermediate)

Q1: When would you use Adaptive RAG vs naive RAG? ▼

Answer:

Use naive RAG when:

  • All queries are similar complexity
  • Cost and latency aren't major concerns
  • Simple use case or prototype

Use Adaptive RAG when:

  • Query complexity varies widely
  • Cost optimization is important
  • Need to optimize both simple and complex queries
  • Production system at scale

Example: Simple factual questions get direct LLM response. Moderate questions get single-step retrieval. Complex questions get multi-step retrieval with reranking.

Q2: What are the trade-offs between smaller and larger chunk sizes? ▼

Answer:

Smaller chunks (100-300 tokens):

  • ✅ More precise retrieval
  • ✅ Better for specific information
  • ❌ May miss broader context
  • ❌ More chunks to store and search
  • ❌ More embedding API calls

Larger chunks (500-1000 tokens):

  • ✅ More context per chunk
  • ✅ Fewer chunks to manage
  • ✅ Fewer API calls
  • ❌ Less precise retrieval
  • ❌ May include irrelevant information

Sweet spot: 500 tokens with 50-100 token overlap for many use cases.

Q3: How does parent-child retrieval work and when is it useful? ▼

Answer:

How it works:

  1. Chunk documents into small pieces (children)
  2. Link each chunk to its parent (larger section or document)
  3. Search on small chunks for precision
  4. Return parent chunks for generation (more context)

When to use:

  • Documents have complex structure
  • Context matters for answer quality
  • Need precise retrieval but surrounding information
  • Legal documents, contracts, academic papers

Benefit: Best of both worlds - precise retrieval with full context.

Q4: What are the key metrics for evaluating RAG systems? ▼

Answer:

Retrieval metrics:

  • Precision@K: How many retrieved docs are relevant?
  • Recall@K: How many relevant docs did we retrieve?
  • MRR: Mean Reciprocal Rank - is the first result relevant?

Generation metrics:

  • Faithfulness: Does answer match retrieved context?
  • Answer Relevance: Does it answer the question?
  • Citation Accuracy: Are sources correctly cited?

System metrics: Latency (P50, P95, P99), throughput, cost per query

Business metrics: User satisfaction, adoption, task completion

Q5: How does caching work in RAG and what are the trade-offs? ▼

Answer:

How it works: Store query-response pairs. On subsequent similar queries, return cached response instead of processing through full pipeline.

Benefits:

  • 60-80% reduction in latency for cached queries
  • 50-70% reduction in costs (fewer API calls)
  • Better user experience for common queries

Trade-offs:

  • Stale data risk: Cached responses may be outdated
  • Cache invalidation: Need strategy for when to update
  • Memory/storage: Cache storage costs
  • Complexity: Added infrastructure (Redis, etc.)

Best practice: Shorter TTLs for time-sensitive data, longer for static content.

🟠 Advanced Exercises (Advanced)

Q1: Design a RAG system for a legal document analysis use case ▼

Key considerations:

  • Accuracy is critical: Use best available components (GPT-4, reranking)
  • Chunking: Semantic chunking to preserve legal clauses, parent-child retrieval
  • Citations: Mandatory citation requirements for verification
  • Evaluation: Comprehensive evaluation with legal experts
  • Fallback: "I don't know" for low-confidence queries
  • Security: On-prem deployment, strict access controls

Architecture:

  1. Semantic chunking (clause-level)
  2. Parent-child retrieval (small chunks, large context)
  3. Hybrid search (vector + keyword for case names, citations)
  4. Reranking (domain-specific if available)
  5. GPT-4 with mandatory citations
  6. Faithfulness checks before returning
Q2: How would you optimize RAG for cost at scale (1M+ queries/day)? ▼

Cost optimization strategies:

  1. Aggressive caching: 60% hit rate saves 60% of costs
  2. Adaptive routing: Use GPT-3.5 for 70% of queries, GPT-4 for 30%
  3. Self-host embeddings: Eliminate embedding API costs
  4. Self-host LLMs: Use Llama for simple queries
  5. Reduced retrieval: Optimize chunk size and k values
  6. Skip reranking: For low-stakes queries where precision less critical
  7. Queue and batch: Process non-real-time queries in batches

Expected savings: 70-80% cost reduction while maintaining 90%+ quality.

Q3: How would you debug a RAG system that's producing hallucinations? ▼

Systematic debugging approach:

  1. Check retrieval: Are retrieved documents relevant? If not → fix retrieval
  2. Check context: Does retrieved context contain the answer? If not → improve retrieval
  3. Check prompt: Does prompt emphasize using retrieved context? If not → strengthen prompt
  4. Check temperature: Is it too high? Lower to 0-0.3 for factual
  5. Check model: Try different LLM (some are better at grounding)
  6. Add citations: Require source references for all claims
  7. Implement "I don't know": When confidence is low

Most common cause: Retrieved documents are irrelevant or don't contain the answer. Fix retrieval first.

Q4: Compare and contrast Graph RAG vs traditional RAG ▼

Traditional RAG:

  • Documents → chunks → vector embeddings
  • Retrieval based on semantic similarity
  • Great for straightforward Q&A
  • Limited for complex relationships

Graph RAG:

  • Documents → knowledge graph (entities and relationships)
  • Retrieval based on graph traversal
  • Excellent for complex, multi-hop reasoning
  • Captures relationships between entities

When to use Graph RAG:

  • Questions requiring multiple connected pieces of information
  • Relationship-heavy domains (supply chains, org charts, networks)
  • "How does X relate to Y?" type questions

Trade-off: More complex to build and maintain. Use when relationships matter.

🔴 Expert Exercises (Expert)

Q1: Design a multi-tenant RAG system where each tenant has isolated data but shared infrastructure ▼

Key challenges:

  • Data isolation: Tenant A can't access Tenant B's documents
  • Performance: Can't sacrifice latency for isolation
  • Scalability: Support thousands of tenants
  • Cost: Shared infrastructure to remain economical

Architecture:

  1. Vector DB: Tenant metadata filtering, separate collections per tenant
  2. Application: Tenant-aware routing, per-tenant caching
  3. Access control: Enforce at application layer (user permissions)
  4. Evaluation: Per-tenant metrics, fair resource allocation
  5. Billing: Per-tenant cost tracking

Best practice: Tenant ID in all queries, enforced at database level for security.

Q2: How would you design a RAG system that handles real-time data (news, finance)? ▼

Key challenges:

  • Freshness: Data becomes stale in minutes
  • Volume: Continuous stream of new documents
  • Caching: Can't cache for long periods

Architecture:

  1. Streaming ingestion: Real-time document processing pipeline
  2. Smart invalidation: Cache invalidation triggered by data updates
  3. Time-based filtering: Metadata filtering for recency
  4. Short TTLs: Cache TTL of 1-5 minutes for time-sensitive data
  5. Hybrid approach: Combine real-time RAG with periodic snapshots

Trade-off: Accept higher costs for freshness. Optimize through aggressive caching of stable data.

Q3: Design an evaluation strategy for a RAG system where ground truth doesn't exist ▼

When no ground truth exists:

  1. Synthetic test set: Generate questions from documents, have LLM answer, use as baseline
  2. Human evaluation: Sample queries, domain experts rate answers
  3. LLM-as-judge: Use GPT-4 to evaluate faithfulness and relevance
  4. Consistency checks: Same query should give consistent answers
  5. Citation analysis: Are citations present and accurate?
  6. User feedback: Thumbs up/down, satisfaction ratings
  7. A/B testing: Compare changes against baseline

Best approach: Combine multiple methods. LLM-as-judge for continuous evaluation, human evaluation for periodic validation, user feedback for real-world assessment.

🎯 Scenario-Based Challenges

Real-World Scenarios

Apply your knowledge to realistic situations. Think through these before revealing the answers.

Scenario 1: Your RAG system worked great in testing but fails in production. What do you do? ▼

Immediate actions:

  1. Check logs for error patterns or anomalies
  2. Review sample failed queries - what's different?
  3. Compare test data vs production data - any differences?
  4. Check resource utilization - are you hitting limits?
  5. Verify external dependencies (APIs, DBs) - are they failing?

Common causes:

  • Production queries are more complex/different than test queries
  • Volume is higher, exposing bottlenecks
  • Real data is messier than synthetic test data
  • External dependencies failing under load

Fix: Improve testing to match production conditions. Add load testing. Test on real data samples.

Scenario 2: Stakeholders want to cut RAG costs by 50% without sacrificing quality. What's your plan? ▼

Cost optimization plan (in priority order):

  1. Implement aggressive caching: Target 60% hit rate (saves ~50% of costs)
  2. Adaptive routing: GPT-3.5 for 70%, GPT-4 for 30% (saves ~60% of LLM costs)
  3. Self-host embeddings: Use open-source models (saves ~20% of total)
  4. Reduce retrieval: Optimize k values, skip reranking for some queries (saves ~10%)
  5. Negotiate pricing: Volume discounts with providers

Expected outcome: 70-80% cost reduction while maintaining 90%+ quality. Monitor quality metrics throughout.

Scenario 3: Your team can't agree on chunking strategy. How do you decide? ▼

Decision framework:

  1. Define success criteria: What metrics matter? (Accuracy, latency, cost)
  2. Run A/B tests: Test multiple chunking strategies on real data
  3. Measure impact: Evaluate retrieval precision, answer quality, latency
  4. Analyze by query type: Different strategies may work for different queries
  5. Make data-driven decision: Choose based on actual performance

Likely outcome: No single strategy is best. Consider adaptive chunking - different strategies for different document types or query types.
