Patterns & Anti-Patterns

Proven Patterns and Common Mistakes in RAG Implementation

← Previous Next: Trade-offs →

The Value of Patterns

💡 Why Patterns Matter

Patterns: Reusable solutions to common problems. They're not rules, but starting points that have worked for others.

Anti-patterns: Approaches that seem right but lead to problems. Learning these helps you avoid common traps.

Remember: Context matters. A pattern that's perfect in one situation may be wrong in another. Always adapt to your specific needs.

Proven RAG Patterns

🎯 Pattern 1: Hybrid Search ▼

What

Combine vector similarity search with keyword search (BM25), then merge and rerank results.

Why It Works

  • Vector search: Great for semantic similarity ("why do I feel tired?")
  • Keyword search: Great for exact matches ("acetyl-L-carnitine")
  • Together: Best of both worlds, covers more query types

When to Use

  • Domain with specific terminology (medical, legal, technical)
  • Users search by product codes, names, or exact phrases
  • You're not sure which approach will work better

Implementation

# Pseudo-code for hybrid search
vector_results = vector_db.search(query, k=10)
keyword_results = keyword_db.search(query, k=10)
combined = merge_and_dedupe(vector_results, keyword_results)
reranked = reranker.rank(query, combined)
final_results = reranked[:5]

Real Example

"A medical Q&A system improved retrieval precision from 68% to 84% by adding BM25 to vector search."

🎯 Pattern 2: Query Transformation ▼

What

Transform user queries before retrieval to improve relevance. Techniques include HyDE, query expansion, and rewriting.

Common Transformations

  • HyDE: Generate hypothetical answer, search for similar documents
  • Query expansion: Add synonyms and related terms
  • Step-back: Abstract to higher-level concepts
  • Multi-query: Generate multiple query variations

When to Use

  • User queries are vague or underspecified
  • Domain has multiple ways to express concepts
  • Users aren't expert searchers

Real Example

"E-commerce site saw 35% improvement in search relevance by expanding 'phone charger' to include 'power adapter', 'charging cable', 'USB charger'."

🎯 Pattern 3: Parent-Child Retrieval ▼

What

Store and search small chunks for precision, but retrieve and use larger parent chunks for context.

How It Works

  1. Chunk documents into small pieces (100-300 tokens)
  2. Link each chunk to its parent (larger section or whole document)
  3. Search on small chunks (more precise)
  4. Return parent chunks (more context)

When to Use

  • Documents have complex structure
  • Context matters for answer quality
  • You need precision but also surrounding information

Real Example

"Legal document analysis system improved by 40% when they switched from fixed-size chunks to parent-child retrieval."

🎯 Pattern 4: Metadata Filtering ▼

What

Use document metadata (date, author, type, etc.) to filter search results before or after vector search.

Why It Matters

  • Vector similarity doesn't capture temporal relevance
  • Some queries need specific document types
  • Users often care about recency or source

When to Use

  • Time-sensitive queries ("latest", "recent")
  • Source-specific queries ("from the finance team")
  • Type-specific queries ("policies", "guidelines")

Implementation Tips

  • Extract metadata during ingestion
  • Store metadata in vector DB
  • Apply filters before vector search (pre-filter) or after (post-filter)
  • Make metadata queryable from user input
🎯 Pattern 5: Recursive Retrieval ▼

What

Iteratively retrieve and refine, using results from previous retrieval to guide the next.

How It Works

  1. Initial retrieval based on user query
  2. Analyze results: Are they relevant? Is more information needed?
  3. If needed, refine query and retrieve again
  4. Repeat until satisfied or max iterations reached

When to Use

  • Complex, multi-hop questions
  • Initial retrieval is uncertain
  • Questions that require multiple pieces of information

Real Example

"Research assistant improved by 50% on complex queries by implementing recursive retrieval that gathered information from multiple sources."

🎯 Pattern 6: Decomposition ▼

What

Break complex queries into simpler sub-questions, retrieve for each, then combine answers.

How It Works

  1. Analyze complex query
  2. Decompose into 2-5 sub-questions
  3. Retrieve and answer each sub-question
  4. Synthesize answers into comprehensive response

When to Use

  • User asks multi-part questions
  • Questions require comparing multiple aspects
  • Single retrieval can't capture all needed information

Real Example

"Customer support bot improved satisfaction by 35% by decomposing 'How does pricing work for nonprofits with annual contracts?' into separate pricing and nonprofit queries."

🎯 Pattern 7: Citation-First Generation ▼

What

Design the RAG system to require citations from the start. Every claim must reference a retrieved document.

How It Works

  • Prompt explicitly requires citations
  • LLM includes source references
  • Post-processing validates citations
  • Failed citations trigger retry or fallback

When to Use

  • Always for factual content
  • Especially for medical, legal, financial
  • When trust and verifiability matter

Why It's Critical

"Citations reduce hallucinations by 60% and make errors immediately traceable. They're essential for enterprise adoption."

Common Anti-Patterns

⚠️ What Are Anti-Patterns?

These are approaches that seem reasonable but lead to problems. They're traps that engineers commonly fall into.

Learn these so you can avoid repeating others' mistakes.

🚫 Anti-Pattern 1: Framework Over-Reliance ▼

The Trap

Using LangChain/LlamaIndex for everything without understanding what they're doing. Copying tutorials without customization.

Why It Seems Right

  • "It's faster to use existing components"
  • "The framework handles edge cases"
  • "Everyone uses LangChain"

The Problems

  • Don't understand failures when they occur
  • Can't optimize for specific needs
  • Framework abstractions hide important details
  • Debugging becomes much harder

Real Example

"Team spent 3 weeks debugging a RAG issue that was caused by a LangChain abstraction. They rebuilt without the framework in 2 days and fixed the problem."

Better Approach

  • Build from scratch first to learn fundamentals
  • Use frameworks for speed, not as crutches
  • Understand what framework components actually do
  • Be ready to replace framework code when needed
🚫 Anti-Pattern 2: Premature Optimization ▼

The Trap

Optimizing for latency, cost, or complexity before achieving baseline functionality and quality.

Why It Seems Right

  • "We need to be production-ready from day one"
  • "Optimization is part of good engineering"
  • "Let's get it right the first time"

The Problems

  • Waste time optimizing the wrong things
  • Add complexity before understanding needs
  • Delay actual value delivery
  • Can't measure if optimizations help

Real Example

"Team spent 6 weeks implementing custom caching and parallelization. Launch was delayed 2 months. Post-launch analysis showed 60% of their optimization was unnecessary."

Better Approach

  • Get it working first, optimize later
  • Measure before optimizing
  • Optimize based on actual bottlenecks, not guesses
  • Simplicity beats cleverness every time
🚫 Anti-Pattern 3: Evaluation Neglect ▼

The Trap

Building and deploying RAG systems without implementing evaluation metrics. Relying on "it seems to work."

Why It Seems Right

  • "Evaluation is complex and time-consuming"
  • "We can tell if it works by using it"
  • "Let's focus on building first"

The Problems

  • Can't detect regressions
  • Don't know if changes help or hurt
  • Can't communicate quality objectively
  • Small problems compound unnoticed

Real Example

"Team deployed RAG system without evaluation. Three months later, they realized retrieval precision had dropped from 75% to 58% due to data drift. No one had noticed."

Better Approach

  • Implement evaluation from day one (RAGAS, DeepEval)
  • Track metrics continuously
  • Set up alerts for metric degradation
  • Make evaluation part of the development process
🚫 Anti-Pattern 4: Over-Engineering ▼

The Trap

Implementing complex features (Agentic RAG, Graph RAG, etc.) when simple solutions would suffice.

Why It Seems Right

  • "We need the best solution"
  • "Complex problems need complex solutions"
  • "Let's future-proof this"

The Problems

  • More complex = more failure points
  • Harder to debug and maintain
  • Slower and more expensive
  • Often unnecessary for actual use case

Real Example

"Startup built Graph RAG system for FAQ bot. 6 months of work. A simple vector search would have achieved 95% of the benefit with 10% of the effort."

Better Approach

  • Start with naive RAG
  • Add complexity only when needed
  • Measure if complex additions actually help
  • Simpler is usually better
🚫 Anti-Pattern 5: Ignoring Data Quality ▼

The Trap

Focusing all effort on model and retrieval optimization while neglecting the quality and preparation of source data.

Why It Seems Right

  • "The model is the smart part"
  • "Retrieval is where we optimize"
  • "Data prep is boring grunt work"

The Problems

  • Garbage in, garbage out
  • Can't fix bad data with better models
  • Data issues compound through pipeline
  • Users notice poor data quality immediately

Real Example

"Company spent months tuning their RAG system. Then someone finally cleaned the source documents and accuracy jumped 30% overnight."

Better Approach

  • Clean and normalize data before ingestion
  • Remove duplicates and low-quality content
  • Structure data for retrieval
  • Quality check source documents
  • Data quality > Model sophistication
🚫 Anti-Pattern 6: One-Size-Fits-All Chunking ▼

The Trap

Using the same chunking strategy for all document types and use cases.

Why It Seems Right

  • "Keep it simple"
  • "Consistency is good"
  • "We don't have time to optimize per document type"

The Problems

  • Different documents have different structures
  • Fixed-size chunks break semantic boundaries
  • Some use cases need larger context
  • Suboptimal retrieval across the board

Real Example

"Legal system used 500-token chunks for everything. Contracts worked okay, but court opinions (which are long and structured) failed completely."

Better Approach

  • Adapt chunking to document structure
  • Use semantic chunking when possible
  • Different strategies for different document types
  • Test and optimize per type

Pattern Selection Guide

🎯 How to Choose the Right Pattern

Not all patterns apply to every situation. Use this guide to decide which patterns to implement for your use case.

Decision Framework

If Your Use Case Is... Start With These Patterns Consider Adding...
Simple Q&A
(FAQ, basic questions)		 Naive RAG, Hybrid search Citations, Caching
Domain-Specific
(Medical, legal, technical)		 Hybrid search, Domain embeddings Metadata filtering, Query transformation
Complex Questions
(Multi-hop, analytical)		 Decomposition, Recursive retrieval Agentic RAG, Graph RAG
Real-Time Data
(News, finance)		 Metadata filtering (time), Short cache TTL Smart invalidation, Streaming ingestion
Large Documents
(Contracts, reports)		 Parent-child retrieval, Semantic chunking Decomposition, Recursive retrieval
High Volume
(Consumer-facing)		 Aggressive caching, Adaptive RAG Model routing, Query classification

Progressive Pattern Adoption

Start Simple, Add Complexity Thoughtfully

Week 1-2: Naive RAG with basic vector search

Week 3-4: Add hybrid search and basic evaluation

Week 5-8: Add reranking and caching

Week 9-12: Add query transformations or other advanced patterns based on actual needs

Key principle: Only add complexity when you've identified a specific problem it will solve.

Key Takeaways

✓ Pattern Principles

  • Patterns are starting points, not rules: Adapt to your specific context
  • Hybrid search is usually right: Vector + keyword covers more cases
  • Evaluation isn't optional: Implement from day one
  • Simple > Complex: Don't over-engineer
  • Data quality matters: Garbage in, garbage out
  • Match patterns to use cases: One size doesn't fit all
  • Progressive adoption: Start simple, add complexity thoughtfully
