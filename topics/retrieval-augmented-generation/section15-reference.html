Chunking Strategies

Strategy When to Use Pros Cons
Fixed-size (500 tokens, 50 overlap) Starting out, unknown data Simple, predictable, fast May break semantic boundaries
Recursive (small → large) Documents with structure Preserves some structure Complex, variable sizes
Semantic (by meaning) When precision critical Best semantic coherence Slowest, most complex
Parent-child (small chunks, large context) When context matters Precise retrieval + full context More storage, complex

Common Gotchas & Solutions

🐛 Gotcha: Retrieval Returns Irrelevant Docs ▼

Symptoms: Retrieved documents don't match query intent, answers are generic or wrong

Causes:

  • Poor chunking (breaking semantic boundaries)
  • Wrong embedding model for domain
  • Query preprocessing issues
  • Vector database not tuned

Solutions:

  • Try semantic chunking instead of fixed-size
  • Fine-tune embeddings for your domain
  • Add query preprocessing (HyDE, expansion)
  • Implement hybrid search (vector + BM25)
  • Add reranking
🐛 Gotcha: High Latency ▼

Symptoms: Responses take >2 seconds, users abandon queries

Causes:

  • Vector database queries slow
  • Too many LLM calls
  • No caching
  • Network latency

Solutions:

  • Implement caching (60% hit rate achievable)
  • Use faster LLM for simple queries (adaptive)
  • Optimize vector DB (indexing, connection pooling)
  • Parallelize independent operations
  • Use smaller context models when possible
🐛 Gotcha: High Costs ▼

Symptoms: API costs spiraling, budget overruns

Causes:

  • Using expensive models for all queries
  • High retrieval volume (large k)
  • No caching
  • Redundant retrieval

Solutions:

  • Implement adaptive model selection
  • Aggressive caching (TTL tuning)
  • Self-host when cost-sensitive
  • Optimize chunk size (smaller = more embeddings)
  • Batch requests when possible
🐛 Gotcha: Hallucinations ▼

Symptoms: System invents facts, confident but wrong answers

Causes:

  • Retrieved context is insufficient
  • LLM ignores retrieved context
  • Prompt doesn't emphasize grounding
  • No citation requirements

Solutions:

  • Improve retrieval precision (reranking, hybrid)
  • Better prompt engineering ("use only the following context")
  • Require citations in response
  • Implement faithfulness checks
  • Add "I don't know" handling
🐛 Gotcha: Context Window Overflow ▼

Symptoms: Errors about token limits, truncated context

Causes:

  • Retrieving too many chunks
  • Chunks are too large
  • LLM context window exceeded

Solutions:

  • Reduce retrieval k (retrieve fewer documents)
  • Reduce chunk size
  • Use models with larger context windows
  • Implement context compression before generation
  • Use parent-child retrieval (retrieve small, use large)

Troubleshooting Checklist

When RAG Goes Wrong: Step-by-Step Debugging

Step 1: Check Retrieval

  • Are retrieved documents actually relevant?
  • Is the search finding documents that should exist?
  • Try increasing k to see if relevant docs are further down the list
  • Test with known queries and documents

Step 2: Check Reranking

  • Is reranking reordering results correctly?
  • Compare results before and after reranking
  • Check if reranking is making things worse
  • Try different rerankers or parameters

Step 3: Check Prompt Assembly

  • Is context being added to the prompt correctly?
  • Is the context within the model's context window?
  • Does the system prompt clearly instruct the LLM?
  • Are citations being included properly?

Step 4: Check Generation

  • Is the LLM using the retrieved context?
  • Is the LLM hallucinating despite good context?
  • Try different LLMs (some are better at RAG than others)
  • Adjust temperature (lower = more faithful to context)

Step 5: Check End-to-End

  • Run evaluation metrics (RAGAS, DeepEval)
  • Review specific failure cases manually
  • Check for patterns in failures
  • Test with synthetic queries covering edge cases

Performance Tuning Parameters

Parameter Range Effect of Increasing When to Adjust
chunk_size 100-1000 tokens More context per chunk, less precise Adjust based on document structure
chunk_overlap 0-200 tokens More context preservation, more redundancy Increase if breaking semantic boundaries
retrieval_k 3-20 Higher recall, lower precision, slower Increase if missing relevant docs
rerank_top_k 3-10 More context for LLM, slower Increase if answers lack detail
temperature 0.0-1.0 More creative, less faithful Keep low (0-0.3) for RAG
cache_ttl 300-7200 seconds Higher hit rate, but stale data Increase for static data

Key Commands (Python)

# Basic RAG pipeline (without frameworks)

# 1. Create embeddings
from openai import OpenAI
client = OpenAI()
embedding = client.embeddings.create(
input="Your text here",
model="text-embedding-3-small"
)
vector = embedding.data[0].embedding

# 2. Store in vector DB (ChromaDB example)
import chromadb
client = chromadb.Client()
collection = client.create_collection("docs")
collection.add(
documents=["text1", "text2"],
embeddings=[vec1, vec2],
ids=["doc1", "doc2"]
)

# 3. Retrieve
results = collection.query(
query_embeddings=[query_vector],
n_results=5
)

# 4. Generate
response = client.chat.completions.create(
model="gpt-3.5-turbo",
messages=[
{"role": "system", "content": "Answer using the context"},
{"role": "user", "content": f"Context: {context}\n\nQuestion: {query}"}
]
)

Essential Monitoring Setup

Minimum Viable Monitoring

Track these metrics from day one:

  • Per-query latency: P50, P95, P99
  • Per-query cost: Break down by component
  • Cache hit rate: Monitor effectiveness
  • Error rate: Track failures
  • User satisfaction: Thumbs up/down, ratings

Advanced monitoring (production):

  • Retrieval precision (automated evaluation)
  • Answer faithfulness (automated evaluation)
  • System resource utilization
  • Query patterns and trends
  • A/B test results for improvements
