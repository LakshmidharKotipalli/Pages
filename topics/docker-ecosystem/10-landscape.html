Section 10 · Landscape

Technology Landscape

Docker is one product in a rich ecosystem. Understanding what else exists — and when to use each tool — is the mark of a container-mature practitioner.

The Container Technology Stack

Where Each Tool Fits in the Stack

containerd — The Real Workhorse

📦
containerd
CNCF Graduated K8s Default OCI Compliant

The industry's most widely deployed container runtime — used underneath both Docker and directly by Kubernetes. containerd handles image pulls, storage, and container lifecycle. It's the runtime that actually runs your containers; Docker is the user-friendly experience layer on top.

Why it matters: Understanding containerd means understanding what actually runs in production Kubernetes. When "docker ps" doesn't work on a K8s node, crictl ps using containerd's CRI interface is the equivalent.

Key feature: Snapshotter plugins (overlayfs, stargz, nydus) — different ways to manage image layer storage. stargz enables lazy pulling (start container before image fully downloaded), crucial for large ML model containers.

Podman — The Rootless Alternative

🦭
Podman (Pod Manager)
Red Hat / RHEL Rootless Daemonless

Podman runs containers without a privileged background daemon. Each podman run is a direct process — no daemon intercepting. This means better security (no root daemon), easier systemd integration (containers as systemd units), and no single-point-of-failure daemon.

Pods: Podman supports pod groups (multiple containers sharing network/IPC namespace) matching Kubernetes pod semantics. Generate Kubernetes YAML directly from a running pod: podman generate kube mypod.

When to choose Podman: RHEL/Fedora/CentOS environments (it's the default). CI runners where you want rootless builds. Organizations migrating away from Docker Desktop licensing. Systems where systemd integration matters.

Kata Containers — Strong Isolation

🔒
Kata Containers
Higher Security OCI Compatible Multi-tenant

OCI-compatible containers that run inside lightweight VMs. From the outside, they look like Docker containers. On the inside, each container (or pod) runs in its own microVM with its own kernel. Combines container convenience with VM-level isolation.

When it matters: Multi-tenant Kubernetes clusters serving untrusted workloads (public cloud functions, user-submitted code, academic computing). Financial services requiring hardware-level isolation between tenants. Any environment where kernel-level isolation between containers is a compliance requirement.

Cost: 100-200ms higher startup time vs runc. Small memory overhead per VM. Worth it when isolation requirements demand it.

Container Registries

🌐
Docker Hub
The original public registry. Home to 8M+ public images. Rate limited for anonymous pulls. Free tier limited. Best for OSS image distribution; not recommended as primary private registry.
🟠
AWS ECR
Tight AWS IAM integration. Built-in vulnerability scanning. Cross-region replication. Pay per GB stored. Best choice if your workloads run on AWS.
⚓
Harbor
Open-source CNCF project. Self-hosted. Vulnerability scanning, replication, robot accounts, webhook support. Best for on-premises or air-gapped environments.
🦊
GitLab Container Registry
Built into GitLab. Images automatically linked to their CI pipeline. Great for integrated dev workflow without separate registry management.
