Section 03B · Interview

Interview Conversations

7 interview question trees — each with a strong opener, follow-up paths, production stories, and the nuanced answer that shows real-world experience. Click any question to expand.

Interview Strategy
Don't recite definitions. Start with the problem the feature solves, then explain the mechanism, then give a production example. Interviewers at senior levels want to hear "we ran into X, which taught us Y."
1. "Explain how Docker containers differ from Virtual Machines. Why would you choose one over the other?" ▼
Strong Opener

"The fundamental difference is what gets virtualized. A VM virtualizes the hardware — you get a simulated CPU, memory, and disk, with a full guest OS kernel on top. A Docker container virtualizes only the process environment — it shares the host kernel but gets isolated namespaces and resource limits. This makes containers start in milliseconds instead of minutes and use megabytes instead of gigabytes."

The Technical Depth

"Docker achieves isolation through two Linux kernel mechanisms: namespaces (PID, network, mount, UTS, IPC, user) give each container its own view of system resources, and cgroups enforce resource limits on CPU, memory, and I/O. There's no hypervisor, no guest OS — the container process runs directly on the host kernel."

The Trade-off Nuance

"But this kernel sharing is also the security boundary difference. With a VM, a compromised guest needs a hypervisor escape to reach the host. With containers, a kernel exploit in the shared kernel can affect all containers. For multi-tenant environments serving untrusted code — think public cloud workloads — VMs or microVMs like Firecracker are still the right choice."

When to choose which
  • Choose containers: fast iteration, microservices, consistent dev/prod environments, cost efficiency at scale
  • Choose VMs: strong security isolation, different OS kernels required (Windows on Linux host), compliance requirements, untrusted workloads
  • Choose both: Kubernetes nodes are VMs, but pods are containers — you get VM-level node isolation + container-level workload density
Likely Follow-ups

→ "What if I need to run a Windows container on a Linux host?" (Docker Desktop does this via a hidden Linux VM) → "How does container density compare to VMs?" (10-100x more containers per host) → "What's Kata Containers?" (OCI-compatible containers backed by lightweight VMs)

2. "Walk me through how you'd build a production-ready Docker image for a Node.js application." ▼
Strong Opener

"A production-ready image needs to be minimal, reproducible, secure, and cache-efficient. I'd use a multi-stage build: a full Node.js builder stage that installs all dependencies and runs the build, followed by a minimal runtime stage (Node Alpine) that only copies the production artifact. This can take a 1.5GB image to under 100MB."

Layer Caching Strategy

"The critical caching insight: copy package.json and package-lock.json first, run npm ci, then copy the rest of the source. Since package.json rarely changes, the dependency installation layer gets cached on 90%+ of builds. Without this ordering, every source code change triggers a full npm install."

Security Hardening

"I'd create a non-root user (node user already exists in the official Node image), use --chown when copying files, set USER node before the CMD, and use a read-only root filesystem where possible. I'd also run npm ci --only=production to exclude dev dependencies from the runtime image, and pin the Node version (not latest)."

What I'd include / exclude
  • Include: HEALTHCHECK with curl or wget, ENTRYPOINT for signal handling (or use tini), explicit EXPOSE
  • Exclude: npm, git, build tools, test files, .env files, .git directory (via .dockerignore)
  • Secrets: never in ENV instructions or RUN commands — use BuildKit secret mounts or runtime secrets
Production Story

"At a previous job, we had a CI pipeline where every deployment took 8 minutes just to build the Docker image. After restructuring the Dockerfile to copy package files first and using BuildKit's registry cache, we got it to 45 seconds for unchanged dependencies. This unblocked the team from doing multiple daily deployments."

Follow-ups

→ "How do you handle environment-specific config?" (Env vars at runtime, not build time) → "What about base image vulnerabilities?" (Automated scanning in CI, Dependabot for base images) → "Distroless images?" (No shell, package manager — smallest possible attack surface)

3. "How does Docker networking work? How do containers communicate with each other?" ▼
Strong Opener

"Docker creates virtual networks using Linux bridges and veth (virtual Ethernet) pairs. Each container gets a network namespace with a virtual interface connected to a Docker bridge network. Containers on the same bridge network can communicate using service names — Docker's embedded DNS resolver maps service names to container IPs automatically."

The Four Network Drivers
  • bridge (default): Private virtual network on the host. Containers communicate by IP or service name. Traffic to the outside world is NAT'd through the host IP.
  • host: Container uses the host's network namespace directly. No network isolation. Used for performance-critical applications or when you need to expose many ports. Dangerous in multi-tenant environments.
  • none: No networking. Completely isolated. Useful for batch jobs that process files without any network.
  • overlay: Multi-host networking for Docker Swarm. Uses VXLAN tunnels to span containers across multiple Docker hosts. Kubernetes uses a similar concept with CNI plugins (Flannel, Calico, Cilium).
DNS Resolution

"When you run docker-compose up, each service gets a DNS entry matching its service name. So your app can connect to postgres:5432 instead of an IP address. This is done by Docker's internal DNS server (127.0.0.11) injected into each container's /etc/resolv.conf."

Port Mapping Mechanics

"Port mapping (-p 8080:80) creates an iptables rule that NATs traffic from host port 8080 to the container's internal port 80. The container process binds to 0.0.0.0:80 inside its network namespace; external traffic hits the host's 8080 and gets forwarded."

Security Consideration

"By default, containers can reach any other container on the same bridge network. For microservices, it's best practice to create separate user-defined bridge networks per service tier and explicitly connect containers that need to communicate. This prevents a compromised container from reaching unrelated services."

4. "What's the difference between Docker volumes, bind mounts, and tmpfs? When do you use each?" ▼
Strong Opener

"This is fundamentally about where data lives and who controls it. Volumes are managed by Docker and stored in Docker's data directory — portable, easy to back up, Docker controls the path. Bind mounts reference a specific host path — the host controls it, and any change on either side is immediately visible. tmpfs is ephemeral in-memory storage that never touches disk."

When to use each
  • Volumes for production database data, any persistent state in production. Docker manages them, they work across platforms, they're the right abstraction when you don't care about the exact host path.
  • Bind mounts for development — mounting your source code into the container so code changes are reflected immediately without rebuilding. Also for injecting config files from the host. Never use in production if avoidable (host path dependency, platform-specific issues).
  • tmpfs for secrets that must not hit disk (payment tokens, private keys during computation), caches that don't need persistence, or improving performance of write-heavy processes.
The Permissions Gotcha

"The most common bind mount pain point is file permissions. Your host user is UID 1000, the container user is UID 999 — suddenly you can't write to mounted files. Solutions: match UIDs, use ACLs, or use --user flag. In production, volumes sidestep this because Docker manages the path."

Follow-ups

→ "How do you backup Docker volumes?" (docker run --volumes-from, or direct copy from /var/lib/docker/volumes) → "Volume drivers?" (NFS, AWS EBS, Azure File — for persistence that follows the container across nodes)

5. "How would you debug a container that's crashing on startup?" ▼
The Systematic Approach

"I follow a consistent sequence: logs → exit code → interactive shell → image inspection."

Step-by-step
  • Step 1 — Logs: docker logs <container> (or --follow for streaming, --tail 100 for last 100 lines). 80% of crashes are diagnosed here — application error, missing env var, port conflict.
  • Step 2 — Exit Code: docker inspect <container> --format '{{.State.ExitCode}}'. Exit 1 = application error. Exit 137 = OOMKilled (memory limit exceeded). Exit 139 = segfault. Exit 143 = SIGTERM received (graceful shutdown). Exit 125, 126, 127 = container runtime errors (bad CMD, permission denied).
  • Step 3 — Override entrypoint: docker run --entrypoint /bin/sh -it <image> to get a shell and poke around. Check if files exist, env vars are set, the binary is executable.
  • Step 4 — Inspect the image: docker inspect <image> to see configured CMD, ENTRYPOINT, ENV, working directory. docker history <image> to see all layers.
  • Step 5 — Exec into running container: If the container runs briefly, docker exec -it <container> /bin/sh while it's alive. Or use a debug sidecar.
Common Causes
  • Missing environment variables (DB_HOST, API_KEY not set)
  • Database not ready yet (startup race — use wait-for-it.sh or healthcheck-based depends_on)
  • Port already in use on host
  • Insufficient memory limits → OOMKilled
  • Wrong architecture (ARM image on x86 or vice versa)
  • File doesn't exist or isn't executable (wrong COPY path in Dockerfile)
6. "What Docker security practices do you follow in production?" ▼
Strong Opener

"Container security is defense in depth. No single measure is sufficient. I think in four layers: image security, runtime security, host security, and network security."

Image Security
  • Pin base image versions with digest (FROM node:18-alpine@sha256:...), not just tags
  • Run vulnerability scanners (Trivy, Docker Scout, Snyk) in CI — block on critical/high CVEs
  • Use minimal base images: Alpine, distroless, scratch where possible — less software = less attack surface
  • Multi-stage builds to exclude build tools from runtime images
  • Never embed secrets in images (not in ENV, RUN, or COPY)
Runtime Security
  • --user nonroot or create a dedicated user in Dockerfile — avoid running as UID 0
  • --read-only root filesystem with explicit tmpfs mounts for writable directories
  • --cap-drop ALL --cap-add <specific caps> — drop all capabilities, add only what's needed
  • Custom seccomp profiles to restrict syscalls (Docker's default is already restrictive, but you can tighten further)
  • --no-new-privileges — prevent privilege escalation via setuid binaries
Production Story

"We had a container escape near-miss where a misconfigured container had --privileged — someone had added it to debug a storage driver issue and it never got removed. We added an OPA policy to the admission controller that blocked privileged containers unless explicitly annotated with an approval. This kind of guardrail is essential at scale."

7. "If Kubernetes deprecated Docker, does that mean we can't use Docker images anymore?" ▼
Strong Opener

"This is one of the most misunderstood announcements in the container world. Kubernetes deprecated dockershim — the compatibility shim that let Kubernetes talk to Docker's API. It did NOT deprecate Docker images. All Docker images follow the OCI image spec and work perfectly with containerd, CRI-O, and any other OCI-compliant runtime."

Why it happened

"Kubernetes communicates with container runtimes through the Container Runtime Interface (CRI). Docker's runtime didn't speak CRI natively, so Kubernetes maintained a translation layer (dockershim). This was extra complexity with no benefit — containerd (which Docker uses internally) does speak CRI. Removing dockershim meant Kubernetes could talk directly to containerd, skipping the Docker layer entirely."

What actually changed
  • Dev workflow: zero change — you still use docker build, docker push, Docker Desktop
  • CI/CD: zero change — your pipeline still builds Docker images that Kubernetes pulls
  • Cluster nodes: nodes no longer have Docker installed — they use containerd or CRI-O directly
  • Debugging: you can no longer SSH into a K8s node and run docker ps to see pods (use crictl ps instead)
The Insight That Impresses

"This change actually exposed something important: Docker was always a developer experience layer on top of lower-level infrastructure. Containerd is the real runtime — Docker was always using it internally. The deprecation clarified the architecture: OCI images are the standard, and runtimes are interchangeable commodities."
