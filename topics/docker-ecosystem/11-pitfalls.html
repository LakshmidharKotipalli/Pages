Section 11 · Pitfalls

Pitfalls & War Stories

These incidents happened. Some cost millions. All taught something. The pattern: each mistake felt reasonable at the time, and the damage only revealed itself under production conditions.

💥 The "Works on My Machine" Backwards
P1 Incident ● HIGH
The Mistake

Team containerized their Python ML service. Dockerfile used FROM python:3.9 without pinning the minor version. CI/CD built fresh. Three months in, upstream pushed Python 3.9.18, which included a security patch for the cryptography library. The new base image pulled a slightly different NumPy version. The ML model's prediction output changed subtly — not an error, just different floating-point results due to a computation order change.

Why It Felt Right

"Python 3.9 will always be Python 3.9." The team was confident the semantics would be stable. They weren't tracking that :3.9 is a mutable tag that moves with patch releases.

Production Impact

The fraud detection model started flagging 23% more transactions as fraudulent. Customer support was flooded. Revenue processing was halted for 6 hours while the team identified the cause. Initial assumption was a data pipeline issue, which sent them in the wrong direction for 3 hours.

Fix

Pin base images by SHA256 digest: FROM python:3.9.17-slim@sha256:abc123.... Any change to the base image requires a deliberate, reviewed Dockerfile update. Use Dependabot or Renovate for automated, PR-based base image updates with full test runs.

Lesson Learned

Mutable image tags in Dockerfiles are time bombs. Every tag except SHA digest can change underneath you. The reproducibility guarantee of containers is only as strong as the immutability of your base image reference.

💾 The Disk Full Incident — Docker Layer Accumulation
P2 Incident ● HIGH
The Mistake

A CI server ran docker builds all day. Each build pulled images, created build containers, and left dangling intermediate images. Nobody ran docker system prune. Over 6 weeks, /var/lib/docker grew to 800GB on a 1TB disk. When the disk hit 100%, the Docker daemon crashed, taking down all CI runners simultaneously. Mid-week at 2 PM.

Why It Felt Right

"Docker cleans up after itself." Developers assumed Docker managed its own storage. Docker does not automatically remove old images, stopped containers, or dangling build cache. It accumulates indefinitely without intervention.

Production Impact

All CI pipelines failed simultaneously. Team couldn't deploy during a critical release window. Manual cleanup required SSH access and careful docker image prune to free space without deleting needed images. 4-hour recovery.

Fix

Schedule docker system prune -f --volumes nightly on CI servers (with exclusions for needed images). Set Docker daemon's --max-concurrent-uploads and configure log rotation. Monitor /var/lib/docker disk usage with alerts at 60% and 80%. Use BuildKit's garbage collection configuration to auto-prune build cache. On Kubernetes nodes, configure image garbage collection (imageGCHighThresholdPercent) in kubelet config.

Lesson Learned

Docker's storage is not self-maintaining. Build up disk monitoring for Docker hosts before you need it. Treat Docker disk usage like database disk usage — alert well before it's critical.

🔑 The Secret Baked Into 10,000 Pulled Images
Security Incident ● CRITICAL
The Mistake

A developer needed a private npm registry token to install packages during docker build. They added ARG NPM_TOKEN and ENV NPM_TOKEN=$NPM_TOKEN so it was available during the RUN npm install step. The token was in the image's environment variable layer. The image was pushed to Docker Hub as a public image.

The Discovery

A security researcher running automated scans on Docker Hub found the token in the image metadata. They reported it responsibly, but not before the image had been pulled 10,000 times by users of the open-source project. It was impossible to know if any of those pulls had extracted the token.

Fix

Rotate the token immediately. Rebuild the image with the token never touching ENV. Use BuildKit secret mounts: RUN --mount=type=secret,id=npmrc cat /run/secrets/npmrc — the secret is only available during that RUN command and never written to any layer. Audit all public images for secrets using tools like Truffelhog or GitLeaks. Add pre-commit hooks to catch secrets before they're committed to Dockerfiles.

Lesson Learned

Image layers are permanent records. Anything ever written to any layer — even if later deleted — is potentially extractable. Build-time secrets must use BuildKit secret mounts or multi-stage builds where the secret is used in stage 1 and stage 2 starts clean.

💀 OOMKilled in Silence
P2 Incident ● MEDIUM
The Mistake

Memory limits were set "conservatively" at 512MB for a Java service. During peak traffic, the JVM's garbage collector needed more heap. The container hit the memory limit, received SIGKILL from the kernel (OOMKill), and was restarted. This happened 50 times per hour during peak load. Users saw intermittent failures but no clear pattern. The JVM doesn't emit a clean error message on OOMKill — it just dies.

Why It Was Hard to Detect

The container restarted fast enough that it wasn't considered "down." Kubernetes liveness probes showed the service as healthy. Error rate metrics showed 2% failures, which was within SLA. But those failures were clustered in 5-minute windows that correlated with traffic spikes — which nobody noticed because they were looking at 1-hour averages.

Fix

Alert on container_oom_events_total > 0. Add OOMKill to your standard container monitoring. For JVM services: set -XX:MaxRAMPercentage=75 so JVM limits itself to 75% of the container memory limit, leaving headroom. Profile actual memory usage under load before setting limits — use docker stats or container memory metrics at 95th percentile traffic.

Lesson Learned

Exit code 137 is OOMKill. Always alert on it. Memory limits need to be set based on profiled peak usage plus headroom, not arbitrary "conservative" numbers. JVM services are especially sensitive — MaxRAMPercentage is your friend.

🌐 The Container That Could SSH
Security Finding ● HIGH
The Mistake

An ops engineer, frustrated with having to rebuild images to debug production issues, added an SSH server to the application Docker image. "Just for emergencies." This image ran in production. The SSH key was the same as the team's dev SSH key. The container ran as root.

The Risk

Any attacker who could reach the container's network (which was exposed via a misconfigured security group for "testing") could SSH in as root. From root in a container, with the right kernel version, container escape is possible. This was a critical infrastructure risk for 8 months before a security audit found it.

Fix

Remove SSH from all application images — no exceptions. For debugging: use kubectl exec / docker exec which requires host access. Use ephemeral debug containers (kubectl debug) that are separate from the application image. Use distroless images that make adding SSH physically impossible. Institute image scanning in CI that flags SSH-related packages.

Lesson Learned

Operational convenience and security are often at odds. Pick security. "Just for emergencies" tools that require privileged access are vulnerabilities, not emergency exits. The container should be debuggable via exec from the orchestrator — that already requires host-level authentication.
