Section 01 · Foundation

Foundation & Context

Docker is a process isolation system that packages an application and all its dependencies into a portable, self-contained unit — solving the "works on my machine" problem by making the machine part of the artifact.

The One-Sentence Truth
Docker doesn't virtualize hardware. It isolates processes using Linux kernel primitives — namespaces for isolation and cgroups for resource control — and wraps a layered filesystem around them so you can ship that isolated environment as a single artifact.

Why Docker Exists — First Principles

Before Docker, deploying software meant one of two things: either you installed it directly on the host (dependency hell) or you spun up a full Virtual Machine (slow, heavy, wasteful). The gap between development and production was enormous.

The problem had two axes. First, dependency isolation: App A needs Python 3.8, App B needs Python 3.11, and the OS can only have one. Second, environment parity: the developer laptop has different libraries, kernel patches, and config than the production server. Docker solved both by making the environment a first-class artifact that travels with the code.

The key insight was using existing Linux kernel features that had existed since 2008 (namespaces) and 2007 (cgroups) — Docker didn't invent new technology, it made existing technology accessible and composable.

The Linux Kernel Primitives

🏠
Namespaces
Isolate what a process sees. Each container gets its own PID space, network interfaces, filesystem mounts, hostname, user IDs, and IPC. The process thinks it's alone on the machine.
🎛️
cgroups (Control Groups)
Limit what a process can use. CPU shares, memory limits, I/O bandwidth, network priority. Without cgroups, one container could starve all others.
📚
OverlayFS / UnionFS
Stack read-only image layers and add a thin writable layer on top. 20 containers can share the same 500MB base image while each has its own 1MB write layer — massive disk savings.
🔒
seccomp + Capabilities
Restrict which syscalls a container process can make and drop dangerous Linux capabilities (like CAP_NET_ADMIN). Defense in depth for container security.

Key Concepts & Abbreviations

Image vs Container ▼

Image = a read-only, layered blueprint. Like a class in OOP. It's a filesystem snapshot plus metadata (what command to run, what port to expose). You can't run an image directly — it's a template.

Container = a running instance of an image. Like an object instantiated from a class. Docker adds a thin writable layer on top of the image layers, starts the specified process in an isolated namespace. Kill the container, the writable layer is gone (unless you have a volume).

Dockerfile ▼

A recipe for building an image. Each instruction (FROM, RUN, COPY, CMD) creates a new immutable layer. Layers are cached by their content hash — if nothing changed, Docker reuses the cached layer. This is why order matters: put rarely changing instructions first (OS packages) and frequently changing ones last (app code).

Registry ▼

A server that stores and distributes Docker images. Docker Hub is the public default. ECR (AWS Elastic Container Registry), GCR (Google Container Registry), ACR (Azure Container Registry), and Harbor are common private options. Images are referenced as registry/repo:tag — e.g., docker.io/nginx:1.25-alpine.

Docker Compose ▼

A tool for defining and running multi-container applications using a YAML file. Instead of running three docker commands manually (start postgres, start redis, start app), you define all services, networks, and volumes in one docker-compose.yml and run docker compose up. Essential for local dev. Often replaced by Kubernetes in production.

Docker Daemon (dockerd) ▼

The background service that does the actual work — managing containers, images, networks, and volumes. The Docker CLI communicates with it via a REST API over a Unix socket (/var/run/docker.sock). Since Docker 1.11, dockerd delegates container lifecycle to containerd, which delegates low-level operations to runc (the OCI runtime). This decoupling allowed Kubernetes to adopt containerd directly, bypassing Docker entirely.

Volume vs Bind Mount vs tmpfs ▼

Volume: Docker-managed persistent storage at /var/lib/docker/volumes/. Survives container restarts and removal. Best for databases, persistent data. Docker handles backups and migrations.

Bind Mount: Mounts a specific host directory into the container. Your local ./src mounted at /app/src — changes on either side reflect immediately. Great for development, dangerous in production (host path dependency).

tmpfs: In-memory only, never written to disk. Use for sensitive data like secrets that shouldn't persist, or for speed with ephemeral caches.

The OCI Standard

Open Container Initiative

In 2015, Docker, CoreOS, and others formed the OCI to standardize container formats and runtimes. This was critical: it meant Kubernetes could run containers without requiring Docker at all.

OCI defines: Image Spec (how images are structured and stored), Runtime Spec (what a container runtime must do), and Distribution Spec (how images are pushed/pulled). Any OCI-compliant tool can build, ship, and run containers interchangeably.

Historical Context

2008–2012
The Pre-Docker World
Linux containers (LXC) existed but required deep kernel expertise. Puppet/Chef managed VM configuration. Heroku pioneered "buildpacks" for app packaging. The problem was understood; the solution was painful.
2013
Docker 0.1 — dotCloud's Internal Tool Goes Public
Solomon Hykes demos Docker at PyCon. Initial release wraps LXC with a simple CLI and layered images. The shipping container metaphor immediately resonates with developers.
2014–2015
Ecosystem Explosion
Docker Compose (formerly Fig), Docker Swarm, Docker Machine. Every cloud provider adds container support. Mesosphere, CoreOS, and Kubernetes enter the orchestration space. Docker ships its own runtime (libcontainer, replacing LXC).
2016–2017
The Orchestration Wars
Docker Swarm vs Kubernetes vs Mesos. Kubernetes, backed by Google's production experience with Borg, gradually wins. Docker Inc. eventually bundles Kubernetes alongside Swarm.
2019–2022
Kubernetes Deprecates Docker
Kubernetes removes dockershim in v1.24. This doesn't mean containers go away — it means containerd (the runtime Docker uses internally) becomes the direct interface. Docker images still work fine.
2023–Present
Maturity & Specialization
Docker Desktop remains the dominant developer experience. Podman gains traction for daemonless, rootless containers. BuildKit becomes the default builder. Container security (Sigstore, SLSA) becomes priority. WASM containers emerge as a new frontier.
