Section 04 · System Design

System Design Integration

Docker isn't just a deployment tool — it touches every pillar of system design. Understanding how containerization decisions ripple through reliability, cost, security, and observability is what separates architects from practitioners.

How Docker Connects to System Design Pillars

⚡
Reliability
Immutable images eliminate configuration drift. Health checks enable self-healing. But shared kernel means a kernel panic takes all containers on that host — VM-level blast radius for node failures.
📡
Availability
Containers start in seconds enabling fast horizontal scaling. Zero-downtime deploys via rolling updates. But stateful containers are hard — databases need careful storage and connection management.
🏎️
Latency
Container startup adds ~100ms overhead (vs ~1ms for a process). Container networking adds ~5-10μs per hop on bridge networks. DNS resolution in service meshes adds per-request overhead. Cold starts matter for serverless-style workloads.
🚀
Throughput
Packing more containers per host (10-100x vs VMs) dramatically increases workload density. CPU and memory isolation via cgroups prevents noisy-neighbor problems that reduce effective throughput.
🔧
Maintainability
Dockerfiles are infrastructure as code. Rolling back = redeploying a previous image tag. Immutable deployments mean no "snowflake servers." But Dockerfile complexity and base image maintenance add their own burden.
💰
Cost
Higher server density reduces infrastructure cost. Faster build cycles reduce CI compute. But registry storage, egress for image pulls, and operational tooling add new cost lines often overlooked.
🔒
Security
Process isolation improves over bare metal. Minimal images reduce attack surface. But container escape vulnerabilities, privileged containers, and the Docker socket as a root equivalent introduce new attack vectors.
📊
Observability
Ephemeral containers make log collection critical — logs must be shipped before container dies. Container IDs constantly change — correlation requires labels, not hostnames. cgroups metrics enable per-container resource monitoring.

System Design Scenario: CI/CD Pipeline with Docker

Let's walk through how Docker fits into a complete CI/CD system design, and the decisions that matter.

Docker in a CI/CD Pipeline — End to End

Key System Design Decisions in This Pipeline

Build Strategy Decisions

Cache source: Where does the build cache come from? Local cache is fastest but CI runners are ephemeral. Registry-based cache (BuildKit --cache-from) works across runners but adds ~30s for cache pull. S3-based cache is another option. Most teams settle on registry cache for balance.

Build context size: Sending a large build context (huge node_modules, large assets) to the Docker daemon is a major CI bottleneck. A thorough .dockerignore is as important as Dockerfile optimization.

Parallel builds: Multi-stage builds can run stages in parallel with BuildKit. For a monorepo building 20 services, build matrix in CI with registry cache reduces build time from 40min to 8min.

Image tagging: Tag with git commit SHA (immutable + traceable) + branch name (mutable reference). Never use only latest in production — you lose the ability to audit what's deployed.

Registry Architecture Decisions

Public vs Private: Docker Hub for OSS projects and public base images. Private registry (ECR, GCR, Harbor) for proprietary applications — keeps images internal, avoids rate limits (Docker Hub: 100 pulls/6hr for anonymous, 200 for authenticated free accounts).

Registry geography: If you deploy to multiple regions, push to regional registries to reduce pull latency. AWS ECR replication copies images across regions automatically. Cold pulls across continents can add 2-5 minutes to deployment time.

Retention policy: Define when to delete old images — every untagged image consumes storage and costs money. Common policy: keep the last 10 tags per repo, keep images < 90 days, never delete tags matching semantic versioning patterns.

Image scanning: ECR, GCR, and Harbor all offer integrated scanning. Shift scanning left: block CI if critical CVEs are found in the built image. Some teams also scan at pull time on K8s nodes.

Deployment Strategy

Rolling update: Replace containers one by one. Zero downtime if health checks are configured. Risk: brief period where old and new versions serve traffic simultaneously — backward-compatible API changes required.

Blue-green: Maintain two identical production environments. Deploy to green, test, switch traffic. Immediate rollback by flipping back. Costs 2x infrastructure during transition. Best for risky releases.

Canary: Route small percentage (1-5%) of traffic to new version. Monitor error rate, latency, business metrics. Gradually increase percentage. Requires traffic splitting (service mesh or weighted load balancer).

Image pull policy: Always pull on deploy to get the latest security patches even if the tag hasn't changed? Or pull only when tag changes? IfNotPresent is faster but can miss security updates. Always is safer but slower and puts load on the registry.

Rollback Strategy

The Docker advantage: Rolling back is just redeploying the previous image tag. If you tagged with git commit SHA, you can deploy any commit's image instantly — no rebuilding needed.

What rollbacks don't fix: Database schema migrations. If the new version ran a migration that removed a column, rolling back the app to the version that expects that column will break. This is why backward-compatible migrations (add column before deploying, remove column after old version is gone) are essential.

Keep N previous images: Ensure your registry keeps at least the last 3-5 deployable images. If a critical bug shows up 2 hours later and you've already garbage-collected old images, you're rebuilding from source under pressure.

Docker in Multi-Tier Architecture

The Standard Three-Tier Containerized App

Tier 1 — Load Balancer (nginx/HAProxy container): Routes requests, terminates SSL. Typically 2 replicas for HA. Stateless — no persistent data needed.

Tier 2 — Application containers: Stateless services. Scale horizontally. Multiple replicas behind load balancer. Health check on /health endpoint. No local state — session data in Redis.

Tier 3 — Database (PostgreSQL/MySQL): Stateful — requires persistent volume. Usually single primary in simple setups. Separate Docker network from app tier. Critical: resource limits set conservatively to avoid OOMKill during query spikes.

When NOT to Containerize

Docker Doesn't Solve Everything
  • GPU workloads: Containerizing GPU-accelerated jobs requires NVIDIA Container Toolkit and careful driver management. Often simpler to use bare metal or dedicated GPU VMs.
  • High-performance networking: Container network stack adds overhead. DPDK, RDMA, and SR-IOV applications may need host networking or bypass containers entirely.
  • Legacy applications with kernel dependencies: Apps that depend on specific kernel modules, kernel version, or low-level OS features may not containerize cleanly.
  • Very small teams, simple apps: A 3-person startup running a Rails monolith may find Docker and Kubernetes overhead higher than the benefit. Platform-as-a-Service (Heroku, Render) may be better.
