Section 06 · Evolution

The Evolution Story

2013 to 2028: How Docker changed software delivery, sparked an ecosystem, lost the orchestration war, and became indispensable infrastructure. The story of how we got here — and where we're going.

2013 — Year 0: The Demo Heard Round the Dev World
Docker 0.1 Launches at PyCon

Solomon Hykes' 5-minute demo: a Python application running in a container, built from a Dockerfile, and shipped as an image. The audience reaction was immediate — this solved a problem everyone had but couldn't articulate.

State of the industry: Puppet, Chef, and Ansible were the state of the art for configuration management. Vagrant gave developers VMs. Heroku had pioneered 12-factor apps. AWS was dominant but deployments were still heavily manual.

What Docker changed: It packaged the application AND its environment. Not "install our app on your environment" but "here is the environment, your app included."

Technical foundation: Wrapped LXC (Linux Containers) with a simpler API and layered Union Filesystems. Introduced the concept of a Dockerfile as a reproducible build recipe. Open-sourced on GitHub immediately.

2014–2015 — The Cambrian Explosion
Docker Inc. Raises $40M, Ecosystem Explodes

Every major cloud provider and tool vendor announced Docker support within 18 months. Amazon Container Service, Google Container Engine, Microsoft Azure Containers all launched. Docker Hub went from 0 to millions of images.

Docker abandons LXC: Docker 0.9 shipped libcontainer, a native Go library for container runtime, removing the LXC dependency. This was crucial for portability and gave Docker full control of the runtime.

The orchestration problem emerges: Running one container is easy. Running 100 containers across 10 servers — deciding where each runs, restarting failures, networking them together — required new tooling. Docker Swarm, Google Kubernetes (June 2014), Mesosphere Marathon, and CoreOS Fleet all launched to compete.

Fig becomes Docker Compose: Docker acquires Fig, the multi-container compose tool. Local development with multiple services becomes standardized.

Security concerns emerge early: Docker's privileged mode, the Docker socket as root equivalent, and running containers as root-by-default drew scrutiny from security researchers. The NIST published early container security guidance.

2016–2017 — The Orchestration Wars
Docker vs Kubernetes vs Mesos — and the Winner Emerges

Three serious contenders: Docker Swarm (simplest, Docker-native), Kubernetes (backed by Google, most powerful), Apache Mesos with Marathon (best for mixed workloads, popular at Twitter/Airbnb). Companies placed bets.

Kubernetes' secret weapon: It was based on Google's decade of experience running Borg — their internal cluster manager. Every design decision had been tested at billion-container scale. The design was complex but battle-proven.

The CNCF formation (2016): Cloud Native Computing Foundation formed with Kubernetes as the first project. Google, IBM, Red Hat, and others donated massive engineering resources. Docker contributed containerd.

Docker's strategic mistake: Docker Inc. tried to compete with Kubernetes rather than integrate it. The result: Kubernetes got all the enterprise attention. Docker eventually bundled Kubernetes in Docker Desktop but the perception had shifted.

OCI standardization (2015-2017): The Open Container Initiative standardized image format and runtime spec. This was momentous: it decoupled the image format from any single vendor. Docker, rkt, Mesos could now run the same images.

2018–2019 — Kubernetes Wins, Runtime Decouples
Containerd Graduates, BuildKit Arrives, Docker Inc. Struggles

Kubernetes became the de-facto standard. Every major cloud offered managed Kubernetes. Docker Swarm adoption stalled. Docker Inc. laid off engineers and pivoted focus to developer tooling.

containerd becomes independent: Docker donated containerd to CNCF. It became the reference container runtime for Kubernetes — the real runtime underneath, now independent of Docker. This was the beginning of Docker's decoupling from the runtime layer.

BuildKit ships (Docker 18.09): A completely rebuilt build engine with parallel execution, improved caching, secret mounts, and SSH forwarding. Dramatically faster builds. Changed the Dockerfile development experience.

Docker Desktop growth: Even as Docker Inc. struggled commercially, Docker Desktop became the dominant developer tool. VSCode's Remote Containers extension (2019) made developing-in-containers mainstream.

The Mirantis acquisition (2019): Mirantis acquired Docker's Enterprise product line. Docker Inc. refocused on Docker Desktop and Docker Hub as developer tools.

2020–2022 — Kubernetes Removes Docker, Podman Rises
Dockershim Deprecated — The "Docker is Dead" Non-Event

Kubernetes 1.20 (December 2020) announced dockershim deprecation. The industry briefly panicked. Most infrastructure engineers realized: nothing in their workflow actually changed for end-users.

What actually changed: Kubernetes nodes no longer ran Docker daemon. They ran containerd (or CRI-O) directly. You couldn't SSH into a node and run docker ps to see pods. You needed crictl instead. CI pipelines didn't change.

Podman emerges seriously: Red Hat's daemonless, rootless container tool gained enterprise adoption. Podman is Docker-compatible (same CLI, same Dockerfiles) but doesn't require a running daemon. Better security story for servers.

Docker Desktop licensing change (2022): Docker Desktop became paid for larger companies. This accelerated Podman, Lima, and Rancher Desktop adoption as alternatives on developer machines.

Supply chain security spotlight: Log4Shell (December 2021) showed how transitive dependencies in container images could compromise thousands of services. SBOM (Software Bill of Materials) generation became a compliance requirement. Sigstore and cosign for image signing went mainstream.

2023–2025 — Maturity, Security, and New Frontiers
BuildKit as Default, Docker Scout, and WASM Containers

The container ecosystem reached maturity. The tools stopped changing quarterly and stabilized into supported, versioned products with clear support windows.

Docker Scout (2023): Integrated vulnerability and software composition analysis. Developers see CVEs at build time in Docker Desktop. The shift-left security vision finally became practical.

WASM containers (2023-2024): Docker announced WebAssembly container support. WASM modules can run as containers via the WasmEdge runtime, enabling near-native performance, language portability, and a security sandbox stronger than Linux namespaces. Still experimental but directionally important.

AI/ML containerization: Every ML framework (PyTorch, TensorFlow, JAX) ships official Docker images. GPU-accelerated containers became standard. Large model serving (LLMs) drove requirements for specialized container runtimes with GPU passthrough.

eBPF changes the game: Cilium (eBPF-based CNI) replaces iptables for Kubernetes networking, offering dramatically better performance and observability. Container networking is being reimplemented using eBPF programs loaded directly into the kernel.

2026–2028 — Where We're Heading
Platform Engineering, Function-Scale Containers, AI-Assisted Dockerfiles

Platform Engineering: Large organizations are building Internal Developer Platforms (IDPs) that abstract Docker/Kubernetes entirely. Developers write code; the platform produces OCI images, deploys them, and manages the lifecycle. Backstage, Port.io, and similar tools define the developer interface.

Function-granularity containers: Firecracker microVMs, WASM, and optimized cold-start runtimes are enabling container-per-request workloads. AWS Lambda runs your code in a microVM that starts in <100ms. The line between containers and functions is blurring.

AI-generated Dockerfiles: LLMs generate surprisingly good Dockerfiles. The Dockerfile may eventually be an implementation detail — you describe your application, the platform produces the container artifact. The important knowledge becomes understanding what a good container looks like, not memorizing syntax.

Confidential containers: Intel TDX, AMD SEV-SNP enabling encrypted containers where even the cloud provider can't read the workload memory. Critical for healthcare, finance, and AI model protection.
