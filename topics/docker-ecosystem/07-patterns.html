Section 07 · Patterns

Patterns & Anti-Patterns

The difference between a good and bad container strategy is often not which tools you use, but how you use them. Here are the patterns that survive production — and the traps that look reasonable until they blow up.

✅ Proven Patterns

1. Multi-Stage Build
Separate your build environment from your runtime environment
Essential

The pattern: Use multiple FROM statements in a Dockerfile. Stage 1 contains all build dependencies (compilers, test frameworks, dev libraries). Stage 2 starts fresh from a minimal base and only COPY the built artifact from Stage 1.

Why it works: Build tools can be 10-100x larger than the actual application binary. A Go application binary is typically 10-20MB; the Go compiler toolchain is 800MB+. A React app's compiled output is 500KB; node_modules for building it is 300MB+.

Real impact: Team at a fintech startup reduced their Docker image from 1.8GB to 47MB for a Node.js API. Deploy times fell from 8 minutes to under 90 seconds. The attack surface dropped by 97% (fewer installed packages = fewer vulnerabilities).

When to use: Almost always for compiled languages (Go, Rust, Java, .NET). For interpreted languages (Python, Node), use it to separate build-time package installation from runtime, and to exclude test/dev dependencies.

2. Sidecar Container Pattern
Extend a container's capabilities without modifying it
Architecture

The pattern: Run a secondary container alongside your main container that handles a cross-cutting concern: log shipping (Fluentbit), metrics collection (Prometheus exporter), configuration reload (Consul template), or network proxying (Envoy).

Why it works: The sidecar handles infrastructure concerns so the application container stays focused on business logic. The application doesn't need to be modified to gain observability, security, or network capabilities.

Classic example: Your legacy monolith can't push metrics in Prometheus format. Add a StatsD sidecar that accepts StatsD metrics and translates them to Prometheus. The monolith is unchanged. Service meshes (Istio, Linkerd) use this pattern for every pod — inject Envoy sidecar, gain mutual TLS and traffic management for free.

Docker Compose example: App container + Fluentbit sidecar. Both containers share a Docker volume where the app writes log files and Fluentbit reads and ships them to Elasticsearch.

3. Init Container Pattern
Run setup tasks before your main container starts
Architecture

The pattern: Run a temporary initialization container before the main application container. The init container performs setup tasks — database migrations, schema validation, secret fetching, config generation — and must succeed before the main container starts.

Why it works: Separates setup concerns from application concerns. The app container starts only when the environment is ready. Failed init = deployment stops before corrupting production state.

In Docker Compose: Use depends_on with a health check — the migration container must exit 0 before the app starts. In Kubernetes, init containers are first-class: they run sequentially to completion before app containers start.

Real example: Django application with PostgreSQL. Init container runs python manage.py migrate. If the migration fails (schema conflict, DB unreachable), the app never starts. No partial migrations in production.

4. One Process Per Container
Containers should have a single, well-defined responsibility
Principle

The pattern: Each container runs exactly one process (or one cohesive process tree). Don't run nginx + application + cron + log shipper in a single container. Give each its own container.

Why it works: Process isolation, independent scaling, independent failure/recovery, independent logging. If your web server crashes, it doesn't take down your cron jobs. You can scale web workers without scaling cron workers.

The nuance: "One process" doesn't mean "one binary." A Node.js process with worker threads, or a Go binary with goroutines — that's fine. The principle is: one functional unit per container, not one OS process per container.

5. Immutable Infrastructure
Never modify running containers — replace them
Principle

The pattern: To "update" an application, you don't exec into the container and change files. You build a new image, push it to the registry, and redeploy. The old container is replaced entirely.

Why it works: Eliminates configuration drift (the "it worked differently in prod than in staging" problem). Every deployment is a clean slate from a known-good image. Rollback is just deploying the previous image. Audit trail is the image tag history.

The corollary: No persistent data in containers. State must live in external volumes, databases, or object storage. The container itself is disposable.

6. Environment-Specific Config via Environment Variables
Build once, configure everywhere — the 12-factor way
12-Factor

The pattern: Build ONE image. Configure it for dev, staging, and production via environment variables at runtime, not at build time. Database URLs, API keys, feature flags — all via env vars injected by the orchestrator or Compose file.

Why it works: The same image that was tested in staging goes to production. Nothing changes except configuration. Build artifacts become true environment-portable units.

Secrets management: Env vars are visible in docker inspect output. For sensitive secrets, use Docker Secrets (Swarm), Kubernetes Secrets, HashiCorp Vault, or AWS Secrets Manager — fetched at runtime and injected as env vars or files.

❌ Dangerous Anti-Patterns

Anti-Pattern 1: Running as Root
The default that's a security disaster waiting to happen
HIGH SEVERITY

What happens: By default, most Dockerfiles don't specify a USER — the process runs as root inside the container. If the container process is compromised, the attacker has root access inside the container, which maps to real capabilities on the host depending on other security controls.

Why it feels right: It's the default. Root has no permission errors. Files owned by root don't cause issues. Easy.

The risk: In combination with a mounted Docker socket or a volume that maps to a sensitive host path, root in a container is effectively root on the host. Even without those, container escape CVEs often require privileged process access to escalate.

Fix: Add RUN addgroup --system app && adduser --system --ingroup app app and USER app at the end of your Dockerfile. Most official base images (node, python, nginx) already include a non-root user.

Anti-Pattern 2: Secrets in Dockerfiles / Images
Layer history never forgets what you added and deleted
HIGH SEVERITY

What happens: Developer adds ENV API_KEY=secret123 to a Dockerfile, or runs RUN wget --header="Authorization: Bearer secret123". Even if they delete it in a later RUN command, it's preserved in the image layer history forever. docker history or pulling the layer directly reveals it.

The scope: Secrets pushed to Docker Hub are public. Private registries with misconfigured access controls expose them. Image scanning tools (Trivy, Semgrep) will find them in CI — but only if you're scanning.

Fix: Use BuildKit --secret mounts for build-time secrets (SSH keys, private npm tokens). Use runtime environment variables for operational secrets. Use Vault or AWS Secrets Manager for secrets that shouldn't be in env vars either.

Anti-Pattern 3: Using :latest Tag in Production
Mutable references in immutable infrastructure
HIGH SEVERITY

What happens: You deploy myapp:latest. It works. Two weeks later, a new image is pushed to :latest. When node 3 of your cluster restarts and pulls the image, it gets the new version while nodes 1 and 2 still run the old one. Your cluster is now running mixed versions silently.

Why it feels right: :latest sounds like "the newest stable version," which sounds like what you want in production. It's also less typing.

Fix: Tag with git commit SHA: myapp:abc1234. Use semantic versioning for stable releases: myapp:2.4.1. The tag must be immutable in production. Pin base images with SHA digest: FROM node:18-alpine@sha256:abc....

Anti-Pattern 4: Giant Monolithic Images
The "install everything just in case" trap
MEDIUM SEVERITY

What happens: The base image is Ubuntu Full. Then you install curl, vim, git, build-essential, debugging tools "just in case you need to debug in prod." The image is 2GB. Every deployment pulls 2GB across every node. CI takes 8 minutes just on the push step.

Why it feels right: Having tools available feels safe. "What if I need to debug?" is a legitimate concern when you don't trust your observability tooling.

The real cost: Larger attack surface (more packages = more CVEs), slower deploys, higher registry storage costs, longer cold starts. The "I might need to debug" problem is better solved with ephemeral debug containers (kubectl debug) that share the pod's process namespace.

Fix: Use Alpine or distroless. Multi-stage builds. Remove package caches after installing (apt-get clean && rm -rf /var/lib/apt/lists/* in the same RUN command).

Anti-Pattern 5: Storing Persistent Data in Containers
Data that dies when the container dies
HIGH SEVERITY

What happens: Application writes logs to /var/log/app inside the container. Or stores uploaded files to /uploads. Container restarts (OOMKill, node failure, update) — all that data is gone. The container filesystem is ephemeral.

Why it feels right: It just "works" during development. The container has a filesystem; why not use it?

Fix: Logs → stdout/stderr, collected by a log driver. File uploads → object storage (S3, GCS). Database files → Docker volume. Session state → Redis. The rule: containers are stateless, state lives outside.

Anti-Pattern 6: Mounting the Docker Socket in Containers
Giving a container the keys to the kingdom
HIGH SEVERITY

What happens: You mount -v /var/run/docker.sock:/var/run/docker.sock so your container can talk to the Docker daemon. The container can now start privileged containers, read any other container's filesystem, or escape to the host entirely. This is effectively root access to the host.

Common legitimate uses: CI runners (Jenkins, GitLab Runner), monitoring tools (Portainer), and auto-configuration tools (Traefik, nginx-proxy). All legitimate — but must be used with full awareness of the security implications.

Alternatives: Docker-in-Docker (DinD) with a separate Docker daemon (still risky but isolated). Kaniko for building Docker images without Docker (runs as non-root, doesn't need the socket). Buildah for builds. Restrict socket access with socket proxies (Tecnativa's docker-socket-proxy).

Anti-Pattern 7: Ignoring the .dockerignore File
Sending your entire repo to the build daemon
MEDIUM SEVERITY

What happens: No .dockerignore means the entire build context (current directory) is sent to the Docker daemon before building. In a Node project, this includes node_modules (hundreds of MB). In a git repo, it includes .git history. Slow. Potentially leaks sensitive files.

Fix: Always create a .dockerignore. Minimum contents: .git, node_modules, .env, *.log, dist, coverage, *.md (if not needed in image), .DS_Store, Dockerfile*, docker-compose*.

Anti-Pattern 8: Fat docker-compose Files Without Structure
One file to define all environments — but differently
MEDIUM SEVERITY

What happens: A single docker-compose.yml tries to serve development, testing, and staging with conditional environment variables and commented blocks. It becomes unmaintainable. Environment-specific differences get missed. "We're on latest for dev but 1.4.2 for staging — wait, no, that was changed last Tuesday."

Fix: Base docker-compose.yml with common configuration + docker-compose.override.yml for local dev (auto-merged) + docker-compose.test.yml for CI + explicit docker-compose.staging.yml. Compose merges files with -f flag. Config is clear, environment-specific, and version-controlled.
