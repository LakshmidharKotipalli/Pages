Section 12 · Debugging

Debugging Playbook

Systematic approaches to every common Docker failure mode. Start from symptoms, follow the diagnostic steps, find the root cause. Ordered by what to check first.

Container Won't Start (Crash on Startup)

1
Check the logs first
docker logs <container_id> docker logs --tail 50 <container_id>

80% of crashes have their cause right here. Look for: missing environment variables, database connection errors, port in use, missing files, import/dependency errors.

2
Check the exit code
docker inspect <container> --format '{{.State.ExitCode}}'

Exit 0 = clean exit (CMD finished normally — wrong if you expected a server)
Exit 1 = application error
Exit 125 = Docker daemon error (bad docker run flag)
Exit 126 = Command found but not executable
Exit 127 = Command not found (wrong CMD path, binary not installed)
Exit 137 = SIGKILL / OOMKilled (out of memory)
Exit 139 = Segmentation fault
Exit 143 = SIGTERM received (graceful shutdown initiated)

3
Override entrypoint and shell in
docker run --entrypoint /bin/sh -it <image> docker run --entrypoint /bin/bash -it <image>

If the image has no shell (distroless), use a debug image:

docker run -it --pid=container:<running_container> --net=container:<running_container> busybox sh

Once inside: check if binary exists (which app), check permissions (ls -la), check env vars (env), test connectivity (wget -qO- http://db:5432).

4
Inspect the image metadata
docker inspect <image> docker history <image>

Check: correct CMD/ENTRYPOINT, working directory, expected ENV vars, exposed ports. Common issue: ENTRYPOINT set to a shell script that doesn't have execute permissions.

Container Starts but Service is Unreachable

1
Verify the process is actually listening
docker exec <container> ss -tlnp docker exec <container> netstat -tlnp

Is the server bound to 0.0.0.0:8080 or 127.0.0.1:8080? If it's bound to localhost, it won't accept connections from outside the container. The app must bind to 0.0.0.0 inside the container for port mapping to work.

2
Verify port mapping
docker port <container> docker ps --format "table {{.Ports}}"

Is the port mapped? Is it mapping to the right host interface? 0.0.0.0:8080->8080/tcp vs 127.0.0.1:8080->8080/tcp (latter only accessible from localhost).

3
Test from inside the container
docker exec <container> wget -qO- http://localhost:8080/health docker exec <container> curl -s http://localhost:8080/health

If it works from inside but not from the host, the issue is port mapping or host firewall. If it doesn't work from inside, the service isn't running properly.

Containers Can't Communicate with Each Other

1
Verify they're on the same network
docker network inspect <network_name> docker inspect <container> --format '{{json .NetworkSettings.Networks}}'

Both containers must be on the same user-defined bridge network. Containers on the default bridge network (created with bare docker run) can't use service name DNS — only custom networks support that.

2
Test DNS resolution
docker exec <container> nslookup <other_service_name> docker exec <container> ping <other_service_name>

Service names only resolve if containers use Docker's embedded DNS (127.0.0.11). Check /etc/resolv.conf inside the container: docker exec <container> cat /etc/resolv.conf

Symptom → Diagnosis Quick Reference

Container keeps restarting
  • Application crashes on startup (check logs)
  • OOMKilled (check exit code 137)
  • Health check failing (check healthcheck config)
  • Dependency not ready (DB, cache, API)
  • Resource limit too low
Image builds are very slow
  • No .dockerignore → large build context
  • Package install before source copy → no cache
  • BuildKit not enabled
  • No registry cache in CI
  • Base image not cached locally
Disk space full on Docker host
  • Dangling images accumulating
  • Stopped containers not removed
  • Build cache growing
  • Container logs not rotated
  • Large volumes not cleaned up
Permission denied errors in container
  • Running as non-root, files owned by root
  • Bind mount: host UID ≠ container UID
  • Volume not readable by container user
  • Binary not executable (chmod +x missing)
  • Read-only filesystem, trying to write
Container can't reach the internet
  • No network specified (isolated)
  • iptables NAT rules missing/corrupt
  • Host firewall blocking container traffic
  • IP forwarding disabled on host
  • DNS misconfiguration
Docker build fails on RUN step
  • Package not found (wrong OS/arch)
  • No internet access in build network
  • Authentication required (private registry/npm)
  • Command not found in base image
  • Previous layer cached, force rebuild: --no-cache

Essential Debug Commands Reference

# Container inspection docker inspect <container> # Full JSON: config, state, network, mounts docker stats <container> # Live CPU/memory/network/IO metrics docker top <container> # Processes running inside container docker diff <container> # Files changed in writable layer # Log inspection docker logs -f --tail 100 <container> # Follow last 100 log lines docker logs --since 5m <container> # Logs from last 5 minutes docker logs --timestamps <container> # Include timestamps # Network debugging docker network ls # List all networks docker network inspect <network> # Show connected containers, IPs docker exec <container> ip addr show # Container's network interfaces # Image debugging docker history --no-trunc <image> # All layers and their commands docker run --rm -it <image> ls -la / # Quick filesystem peek # Cleanup commands docker system prune -f # Remove stopped containers, dangling images docker system prune -f --volumes # Also remove unused volumes (CAREFUL) docker system df # Disk usage by Docker docker image prune -a --filter "until=168h" # Images older than 7 days
