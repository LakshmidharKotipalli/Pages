Section 08 · Trade-offs

Trade-offs Analysis

The hardest Docker questions don't have single right answers — they depend on context, team maturity, security requirements, and workload characteristics. Here's how to think through the most important decisions.

1. Containers vs Virtual Machines

🐳 Containers
  • ✓ Start in milliseconds
  • ✓ ~5-100x higher density per host
  • ✓ Shared kernel = less overhead
  • ✓ Image sizes in MB, not GB
  • ✓ Perfect dev/prod parity
  • ✗ Shared kernel = security boundary is weaker
  • ✗ Kernel exploits affect all containers
  • ✗ Can't run different OS kernels
  • ✗ More complex networking
VS
🖥️ Virtual Machines
  • ✓ Strong hardware isolation boundary
  • ✓ Can run any OS kernel
  • ✓ Hypervisor escape harder to achieve
  • ✓ Kernel-level isolation for untrusted code
  • ✗ Boot in minutes, not milliseconds
  • ✗ Full OS overhead per workload
  • ✗ GB-scale images
  • ✗ Complex configuration management
  • ✗ No easy dev/prod parity
⚖️ It Depends On...
  • Security requirements: Running untrusted multi-tenant code (public cloud functions)? Use VMs or microVMs (Firecracker). Running your own trusted services? Containers are fine.
  • OS diversity: Need Windows and Linux on the same host? VMs. All Linux? Containers.
  • Density requirements: Trying to maximize hosts? Containers win by a wide margin.
  • Real answer: Most organizations use both — VMs as the cloud infrastructure layer, containers as the workload packaging layer. Kubernetes nodes are VMs; pods are containers.

2. Docker vs Podman

🐳 Docker
  • ✓ Industry-standard developer experience
  • ✓ Docker Desktop: outstanding Mac/Win support
  • ✓ Largest ecosystem and documentation
  • ✓ BuildKit: best-in-class image builder
  • ✓ Docker Compose well-established
  • ✗ Requires running daemon (dockerd)
  • ✗ Daemon runs as root by default
  • ✗ Docker Desktop paid for orgs >250 employees
  • ✗ Docker socket is a security concern
VS
🦭 Podman
  • ✓ Daemonless — no background service required
  • ✓ Rootless by default — better security posture
  • ✓ OCI-compliant — Docker images work fine
  • ✓ alias docker=podman for most workflows
  • ✓ Pods: group containers like K8s pods
  • ✗ Less polished desktop experience
  • ✗ Fewer third-party integrations assume Podman
  • ✗ Compose support via podman-compose is less mature
  • ✗ Smaller community, less documentation
⚖️ It Depends On...

Developer machines: Docker Desktop is the better experience, especially on Mac. CI/CD servers: Podman or kaniko are often better — no daemon to manage, rootless by default. RHEL/Fedora/CentOS environments: Podman is the default and Red Hat's preferred tool. Large enterprises concerned about Docker Desktop licensing: Podman or Rancher Desktop as alternatives.

3. Docker Compose vs Kubernetes

🐙 Docker Compose
  • ✓ Simple YAML that most devs understand in an hour
  • ✓ Perfect for local development
  • ✓ Small team production deployments viable
  • ✓ No control plane to manage
  • ✓ Fast iteration cycle
  • ✗ No automatic rescheduling on node failure
  • ✗ No built-in horizontal autoscaling
  • ✗ Single-host limitation (Swarm for multi-host)
  • ✗ Limited to Docker's networking model
VS
☸️ Kubernetes
  • ✓ Automatic rescheduling on node failures
  • ✓ Horizontal pod autoscaling
  • ✓ Rich ecosystem (service mesh, operators, GitOps)
  • ✓ Battle-tested at massive scale
  • ✓ Declarative, self-healing
  • ✗ Steep learning curve (weeks to months)
  • ✗ Control plane overhead (3 masters minimum for HA)
  • ✗ YAML verbosity (Deployment + Service + Ingress for one service)
  • ✗ Operational complexity at small scales
⚖️ The Decision Framework
  • Team size ≤ 5, ≤ 10 services: Docker Compose on a single server or simple VPS. Kubernetes overhead exceeds benefit.
  • Team size 5-20, multiple services, starting to hit scaling needs: Managed Kubernetes (EKS, GKE, AKS). Let the cloud manage the control plane.
  • Large org, complex requirements, compliance: Kubernetes with full platform engineering investment — GitOps, service mesh, custom operators.
  • The false dichotomy: Compose for local dev, Kubernetes for production is the most common setup. They're not mutually exclusive.

4. Base Image Trade-offs

Base Image Size Packages CVE Surface Shell Best For
Ubuntu/Debian full ~100MB ~1000+ High bash Development, debugging
Alpine Linux ~5MB ~50 Low sh Production (glibc gotchas!)
Distroless ~2MB ~10 Very Low None Minimal attack surface production
scratch 0MB 0 Minimal None Statically compiled binaries (Go, Rust)
Official language images Varies Varies Medium bash/sh Quick start, well-maintained
Alpine glibc Warning
Alpine uses musl libc instead of glibc. Most C software is compiled against glibc. Some packages (cryptographic libraries, some Python packages with C extensions, some Node native modules) may behave differently or fail on Alpine. Test thoroughly before using Alpine in production. When in doubt, use the slim variant of official images (node:18-slim, python:3.11-slim) which are Debian-based but minimal.

5. Image Build Strategy

Traditional Dockerfile

Best for: Full control, custom requirements, all languages, security-specific needs.

Requires: Team expertise to write good Dockerfiles. Ongoing maintenance as base images update.

Risk: Bad Dockerfiles are common — layer ordering mistakes, secrets baked in, oversized images.

Cloud Native Buildpacks (CNB)

Best for: Heroku-like experience. Automatic language detection. No Dockerfile required.

Used by: Heroku, Google Cloud Run, Paketo Buildpacks (Spring, Node, Go, Python, Ruby).

Trade-off: Less control over the final image. Buildpack must support your specific requirements. Harder to customize for unusual configurations.

Language-Specific Builders (Jib for Java, ko for Go)

Best for: Teams that want optimal images without Dockerfile expertise.

Jib: Builds Docker images directly from Maven/Gradle without Docker daemon. Produces layered images that cache extremely well (dependencies layer is rarely rebuilt). Part of Google Cloud ecosystem.

ko: Builds OCI images from Go binaries in a single command. No Dockerfile. Produces minimal images from scratch. Excellent for K8s operators and CLI tools.

Trade-off: Framework-specific. Not appropriate for non-Java/Go projects. Less flexibility than Dockerfiles.
