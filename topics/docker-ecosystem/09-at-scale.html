Section 09 · At Scale

Docker at Scale

What works for 10 containers on one host stops working at 1,000 containers across 50 hosts. Here's what breaks, what companies like Netflix and Uber learned, and what you need to monitor when containers go large.

The Scale Threshold Truth
At small scale, Docker's rough edges are ignorable. At medium scale, they become operational pain. At large scale, they become incidents. Understanding which problems appear at which scale helps you invest in solutions at the right time.

What Breaks First (By Scale)

10 Containers — The Happy Path

At this scale, Docker Compose on a single host or a few VMs works well. Problems are manageable.

What you encounter:

  • Layer cache invalidation causing slow rebuilds (fix with proper Dockerfile ordering)
  • Port conflicts between services on shared hosts
  • Logs scrolling past in terminal — need to add a log aggregator
  • Forgetting to set resource limits, getting occasional OOMKills
  • Volume permissions between host user and container user

Monitoring needs: Docker stats, Portainer for UI, basic log tailing. Nothing fancy required.

100 Containers — Operational Pain Begins

Single-host Compose starts straining. You're now on a small cluster, likely Kubernetes or Swarm.

What breaks:

  • Registry bottleneck: 100 containers pulling images from Docker Hub → rate limiting. Need a private registry with mirroring. A 500MB image × 100 nodes on redeploy = 50GB of registry egress simultaneously.
  • DNS amplification: Each container DNS queries Docker's internal resolver heavily. On container-dense hosts, the DNS resolver becomes a hotspot.
  • iptables rules explosion: Docker creates iptables rules for each container port mapping. At 100+ containers, iptables performance degrades. This is why eBPF-based networking (Cilium) matters at scale.
  • Log volume: 100 containers generating logs → overwhelming central logging if not filtered/sampled. Structured logging becomes mandatory.
  • Secrets sprawl: Environment variables in docker-compose files, hard-coded in run scripts, different between environments. Need a secrets manager (Vault, AWS Secrets Manager).

Monitoring needs: Per-container CPU/memory metrics, centralized logging (ELK stack, Loki), distributed tracing (Jaeger, Tempo), proper alerting.

1,000+ Containers — Engineering Required

This is enterprise/large startup scale. Kubernetes is mandatory. The problems are complex distributed systems problems.

What breaks at this scale:

  • Image pull storms: A deployment rolling update across 50 nodes all pulling the same 800MB image simultaneously brings the registry to its knees. Solutions: pre-pull images, P2P image distribution (Dragonfly, Kraken), registry caching at each node (stargz snapshotter for lazy pulling).
  • Layer deduplication at scale: OverlayFS layer deduplication works per-node, not cluster-wide. Node A and Node B each store the same 500MB base image layer separately. Solutions: node image cache pre-warming, shared NFS/CephFS volumes.
  • Noisy neighbor CPU contention: cgroups CPU shares only enforce limits under contention. A node with 40 containers is fine at normal load; during a traffic spike, CPU-intensive containers starve others. Use CPU pinning or guaranteed QoS class in Kubernetes.
  • Metadata database bottleneck: The Docker daemon's (and containerd's) metadata database (bbolt) can become a write bottleneck during rapid container churn. Seen in large CI environments spinning up hundreds of short-lived containers per minute.
  • Network connection table exhaustion: Each container has its own network namespace. High-traffic services create many connections. Host connection tables (conntrack) and ephemeral port ranges can be exhausted. Requires kernel tuning.

Monitoring needs: Full observability stack — metrics (Prometheus/Thanos), logs (Loki), traces (Tempo/Jaeger), eBPF-based network monitoring, container image inventory and vulnerability reporting, cost attribution per team/service.

How Major Companies Solved Scale Problems

🎬
Netflix — Image Distribution
At 100,000+ containers, image pull latency during deployments was a critical problem. Netflix open-sourced Dynomite (Redis cluster) and contributed to P2P image distribution research. The key insight: get images to nodes before you need them, not when you need them.
🚗
Uber — Container Density
Uber runs thousands of microservices. Their key optimization: grouping services by resource profile (CPU-intensive vs memory-intensive), running them on different node types, and careful cgroup CPU shares to prevent noisy-neighbor problems during peak demand.
🔍
Google — containerd Performance
Google's open-sourcing of containerd improvements for their GKE customers included stargz/eStargz (seekable compressed image format enabling lazy layer pulling) — you can start a container before the full image is downloaded. Particularly valuable for large ML model containers.
🏦
Financial Services — Security at Scale
Large banks containerizing workloads add: mandatory vulnerability scanning in CI with CVE SLAs, OPA/Gatekeeper admission control to block non-compliant images, mandatory image signing (cosign + Sigstore), and FIPS-compliant base images for regulated workloads.

Operational Complexity Growth Curve

The Non-Linear Complexity Curve

Managing 10 containers is not 10x more complex than managing 1. It's roughly the same. But managing 100 containers is not 10x more complex than 10 — it's 50x. And managing 1,000 is not 10x more than 100 — it's potentially 200x. This non-linearity is why organizations eventually need platform engineering teams dedicated to making containers manageable for application developers.

The solution isn't to accept this complexity — it's to abstract it. This is why Kubernetes Operators, Helm charts, Kustomize, and Internal Developer Platforms exist: to hide orchestration complexity behind simpler interfaces.
