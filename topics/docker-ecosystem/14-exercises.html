Practice Exercises

15 exercises ordered by difficulty — from warm-up conceptual questions to expert-level design challenges. The goal isn't to memorize answers but to build durable mental models. Click to reveal answers.

🌱 Warm-Up (Conceptual Foundations)

Exercise 1
Explain the difference between an image and a container to a developer who has never used Docker.
Warm-up
Strong Answer

An image is like a class in OOP — it's a read-only template that defines what a container should look like: which files are included, what process to run, what ports to expose, and what environment variables are expected. You build an image from a Dockerfile; you can't "run" an image directly.

A container is an instance of that class — a running process with its own isolated filesystem, network, and process space, created from the image. Like instantiating an object from a class, you can create many containers from one image. When the container stops, its writable layer (any changes it made) is discarded unless you used a volume.

Concrete analogy: A Docker image is like a class definition. A Docker container is like a running instance of that class. The constructor is docker run.

Bonus Points For

Mentioning that images are made of layers (each Dockerfile instruction creates a layer) and that multiple containers from the same image share the read-only layers, making storage efficient.

Exercise 2
What Linux kernel features make Docker containers possible? Why does Docker work on Mac/Windows if it needs Linux?
Warm-up
Strong Answer

Docker requires three Linux kernel features: namespaces (isolate what a process can see — PID space, network interfaces, filesystem mounts, hostname), cgroups (control what resources a process can use — CPU, memory, I/O), and Union filesystems like OverlayFS (stack read-only image layers with a writable layer on top).

On Mac and Windows, Docker Desktop runs a lightweight Linux VM (HyperKit on older Mac, Apple Hypervisor on Apple Silicon, WSL2 on Windows). Your Docker containers run inside that VM — not directly on your Mac/Windows host. This is why Docker Desktop on Mac is slightly slower than on Linux: there's a VM hypervisor layer in the middle.

Exercise 3
You have a Node.js app that prints "Hello World" and exits. What's wrong with this Dockerfile?
Warm-up
The Broken Dockerfile
FROM ubuntu:latest RUN apt-get update && apt-get install -y nodejs COPY . /app RUN npm install CMD node /app/index.js
Problems Found
  • FROM ubuntu:latest: Mutable tag, will change under you. Use a specific version. Also, ubuntu is 100MB+ — use FROM node:18-alpine instead (5MB base + only what you need)
  • apt-get without cleanup: apt cache stays in the layer, bloating the image. Need && rm -rf /var/lib/apt/lists/*
  • COPY . before npm install: Cache-busting! Every source code change causes a full npm install. Should COPY package*.json first, RUN npm install, then COPY the rest
  • No .dockerignore: node_modules from the host gets copied into the build context
  • No USER specification: Runs as root — security risk
  • CMD format: Shell form CMD node /app doesn't receive signals properly. Use exec form: CMD ["node", "/app/index.js"]

⚙️ Core (Building Real Understanding)

Exercise 4
Design a multi-stage Dockerfile for a Go REST API that produces the smallest possible production image.
Core
Ideal Approach
# Stage 1: Build FROM golang:1.21-alpine AS builder WORKDIR /app # Copy dependency manifests first (cache layer) COPY go.mod go.sum ./ RUN go mod download # Copy source and build COPY . . RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o server . # Stage 2: Minimal runtime FROM scratch # CA certificates for HTTPS calls COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/ COPY --from=builder /app/server /server EXPOSE 8080 ENTRYPOINT ["/server"]

Why scratch? Go compiles to a self-contained binary with no runtime dependencies. FROM scratch means literally zero OS — just the binary. Final image: ~8-15MB vs ~800MB for the build stage.

-ldflags="-s -w": Strip debug symbols and DWARF info, reducing binary size ~30%.

CGO_ENABLED=0: Static binary. CGO would require glibc in the runtime stage.

Exercise 5
Design a docker-compose.yml for a web application with a Node.js API, PostgreSQL database, and Redis cache. The API should not start until both DB and cache are ready.
Core
Key Considerations

depends_on with condition: service_healthy ensures the dependency is not just started, but passing its health check. Without this, "depends_on" only waits for the container to start — not for Postgres to be ready to accept connections (which takes longer).

Named volume for postgres data persists across container restarts. tmpfs for Redis could be used if caching is purely ephemeral.

healthcheck in postgres: pg_isready checks actual TCP connection acceptance, not just process start.

Environment variables from .env file: Never hardcode passwords in docker-compose.yml — use env_file or variable substitution from shell environment.

Exercise 6
A container is running but users report it's unresponsive. Walk through your complete diagnosis process.
Core
Complete Diagnosis Flow
  1. Check if the process is still running: docker top <container> — is the app process there?
  2. Check resource consumption: docker stats <container> — is it CPU-bound (100% CPU), memory-bound (near limit), or fine on both?
  3. Check recent logs: docker logs --tail 100 --since 5m <container> — any errors, panic messages, connection pool exhaustion?
  4. Test from inside: docker exec <container> wget -qO- http://localhost:8080/health — does it respond from inside?
  5. Check the health check status: docker inspect --format '{{.State.Health.Status}}' <container>
  6. Check open connections: docker exec <container> ss -s — is the connection table full?
  7. If CPU is 100%: likely an infinite loop, runaway GC, or CPU-intensive operation. Profile with language-specific tools.
  8. If memory is near limit: potential OOMKill imminent, or memory leak. Check GC metrics for managed languages.

🔧 Advanced (Production Thinking)

Exercise 7
Your team is deploying a new container and receiving exit code 137. What does this mean and how do you fix it?
Advanced
Answer

Exit code 137 = SIGKILL (signal 9). In Docker context this almost always means OOMKill — the container exceeded its memory limit and the Linux kernel OOM killer killed the process.

Diagnosis: Run docker inspect <container> --format '{{.State.OOMKilled}}' — if true, confirmed OOMKill. Also check kernel messages: dmesg | grep -i oom on the host.

Fix options (in order of preference):

  1. Profile real memory usage under load (docker stats at peak traffic), then set memory limit to 1.5x the measured p99
  2. For JVM: add -XX:MaxRAMPercentage=75 so JVM limits itself to 75% of container memory limit
  3. Fix memory leaks in the application
  4. If legitimately needs more memory, increase the limit

Never just increase memory limits without understanding why the container is using more memory than expected — it may be a memory leak that will grow indefinitely.

Exercise 8
How would you securely handle a private npm registry token needed during docker build?
Advanced
Wrong Approach (Never Do This)
# WRONG: Secret in layer history forever ARG NPM_TOKEN ENV NPM_TOKEN=$NPM_TOKEN RUN echo "//registry.npmjs.org/:_authToken=${NPM_TOKEN}" > .npmrc && npm install RUN rm .npmrc # too late — the layer with .npmrc still exists!
Correct Approach: BuildKit Secret Mounts
# Correct: Secret available ONLY during this RUN, never in layers FROM node:18-alpine WORKDIR /app COPY package*.json ./ RUN --mount=type=secret,id=npmrc,target=/root/.npmrc npm ci COPY . . CMD ["node", "server.js"]

Build with: DOCKER_BUILDKIT=1 docker build --secret id=npmrc,src=$HOME/.npmrc .

The secret file is available at /root/.npmrc only during that RUN instruction. It never appears in any image layer. docker history will not show it.

🏛️ Expert (Architect-Level)

Exercise 9
Design an image build and distribution strategy for a company with 150 microservices deployed across 3 AWS regions.
Expert
Key Decisions to Cover

Registry Architecture: One ECR registry per region. Source region (us-east-1) is the primary; ECR replication automatically copies new image tags to us-west-2 and eu-west-1. Deployment in each region pulls from local regional registry — no cross-region egress costs, no latency.

Golden Base Images: Create a team that maintains approved base images (company-base-node:18, company-base-java:17, company-base-python:3.11). These are security-scanned, hardened versions that all teams must use. When a CVE hits OpenSSL, one base image update flows to all 150 services via Renovate bot PRs.

Build Caching Strategy: Each ECR repo has a cache layer. CI uses --cache-from ECR/service:cache and --cache-to type=registry,ref=ECR/service:cache. Different services don't share cache (too complex), but CI runners in the same AWS account pull cache from ECR cheaply.

Image Tagging Convention: ECR/service:commit-sha (immutable, deployed) + ECR/service:latest (updated on main branch merge, for non-production convenience). Never deploy using :latest.

Security Gates: ECR scanning enabled. CI pipeline blocks on critical CVEs (Trivy in CI + ECR basic scanning). Cosign signs images after push; K8s admission controller (Connaisseur or Ratify) verifies signature before allowing deployment.

Retention Policy: Keep all tagged releases (semver), delete untagged after 7 days, delete all non-release tags older than 90 days. Expected storage steady state: 500GB across 3 regions.
