Section 12: Debugging Process

Systematic Debugging

LLM applications are fundamentally different to debug than traditional software. Outputs are non-deterministic, errors are often semantic rather than syntactic, and the "stack trace" spans multiple AI model calls. Master these techniques to tame the chaos.

Step 1: Enable Verbose Logging

The first step in any LangChain debugging session is seeing what's happening inside the chain. LangChain offers three levels of visibility:

Logging Levels

LevelWhat It ShowsWhen to Use
VerboseIntermediate steps, tool calls, outputsDuring development
DebugFull prompts, raw responses, metadataDeep investigation
LangSmithVisual trace trees, cost, latencyProduction monitoring

Cautions

  • Never enable verbose/debug in production (performance + security)
  • Verbose output may contain sensitive data (prompts, user inputs)
  • Use LangSmith tracing instead for production debugging
  • Debug mode is extremely chatty — use sparingly

Step 2: Use LangSmith Tracing

LangSmith is the purpose-built observability platform for LangChain. It provides visual traces of every chain execution, making it the single most powerful debugging tool available.

What LangSmith Shows You

  • Visual trace tree: See every step, nested call, and handoff
  • Token usage: Exact token count per LLM call
  • Latency breakdown: Time spent in each component
  • Input/Output: Full prompts and responses at every step
  • Error highlighting: Pinpoint exactly which step failed
  • Cost calculation: Dollar cost per trace
  • Feedback collection: Attach human ratings to traces

LangSmith Debugging Workflow

  1. Reproduce the issue with tracing enabled
  2. Find the trace in LangSmith dashboard
  3. Expand the trace tree to find the failing step
  4. Inspect the input prompt — is context correct?
  5. Check the LLM output — did parsing fail?
  6. Look at token count — was context truncated?
  7. Compare with a working trace — what's different?

Step 3: Test Components in Isolation

When a chain fails, break it apart. Test each component independently to isolate the failure point.

Component Isolation Checklist

  1. Test Retriever: Query it directly — are the retrieved docs actually relevant? Check scores, sources, and content previews.
  2. Test LLM: Send a simple query directly — is the model responding? Check for timeout, auth, or rate limit errors.
  3. Test Prompt Template: Format with sample data — does the assembled prompt look correct? Check for missing variables or malformed context.
  4. Test Output Parser: Feed it raw LLM output — does it parse correctly? Check for format mismatches between expected and actual output.
  5. Test Memory: Save and load context — is history being stored? Verify the memory_key matches the prompt variable name.

What to Check at Each Stage

ComponentCheckCommon Issue
RetrieverDoc count, relevance scoresEmpty results, wrong docs
LLMResponse content, tokens usedTimeout, auth failure
PromptFormatted output correctnessMissing variables

MemoryStored messages, variable namesWrong memory_key

Step 4: Use Debug Callbacks

Callbacks let you hook into every LangChain operation for custom logging, metrics, and error handling. They act as middleware that intercepts each step of execution.

Callback Events You Can Intercept

EventWhat It Captures
on_llm_startModel name, prompt content, prompt length
on_llm_endResponse content, token usage, latency
on_llm_errorError type, error message, failed prompt
on_chain_startChain name, input keys and values
on_tool_startTool name, tool input
on_retriever_endNumber of documents, source metadata

Building Your Debug Callback

Create a custom callback handler that logs every operation with timestamps, token counts, and latency. This handler intercepts:

  • LLM calls: Log model name, prompt preview, output preview, and exact token usage
  • Chain operations: Log chain name, input keys, and intermediate outputs
  • Tool invocations: Log which tool was called, with what input, and what it returned
  • Retriever results: Log document count, sources, and content snippets
  • Errors: Log full error context with the input that caused the failure

Step 5: Common Issues & Fixes

Empty Retriever Output

Symptoms: Chain returns generic answers, retriever returns 0 docs

  • Verify vector store has data (check total document count)
  • Test similarity search directly with a known query
  • Ensure embedding model matches: same model for ingest AND query
  • Lower similarity threshold or increase number of results (k)
  • Verify documents were properly chunked and embedded

Malformed LLM Output

Symptoms: Parser exceptions, JSON parse errors

  • Use an auto-fixing parser with retry logic
  • Add explicit format instructions in the prompt
  • Use Pydantic parser for validated structured output
  • Enable parse error handling (don't crash, recover gracefully)
  • Try a more capable model (GPT-4 follows format instructions better)

Timeout Errors

Symptoms: ReadTimeout, ConnectTimeout

  • Increase request timeout to 120 seconds
  • Add automatic retry with exponential backoff (3 retries)
  • Reduce prompt size to speed up LLM response time
  • Use streaming for long responses to avoid idle timeout

Token Limit Exceeded

Symptoms: "Max tokens exceeded" error

  • Count tokens before sending (use a tokenizer)
  • Retrieve fewer documents or use smaller chunks
  • Use token-limited memory to cap conversation history
  • Switch to a model with larger context window
  • Use map-reduce pattern for large document sets

Memory Not Working

Symptoms: Bot forgets previous messages

  • Check memory is passed to chain constructor correctly
  • Verify memory_key matches prompt template variable name
  • Ensure return_messages=True for chat models
  • Check session ID for Redis/database-backed memory
  • Inspect memory contents directly to see stored state

Agent Picks Wrong Tool

Symptoms: Agent uses search instead of calculator

  • Improve tool descriptions — be very specific about when to use each
  • Add examples to tool descriptions
  • Reduce number of tools (5-7 max per agent)
  • Mark final-answer tools with return_direct=True
  • Use OpenAI Functions Agent (better at tool selection)

Debugging Decision Tree

Chain returns wrong answer?

  1. Check retriever: Are relevant docs being found?
  2. Check prompt: Is context formatted correctly?
  3. Check LLM: Is the model following instructions?
  4. Check parser: Is output being parsed correctly?

Chain returns no answer?

  1. Enable verbose mode to see where it stops
  2. Test each component in isolation
  3. Check for silently caught exceptions
  4. Verify API keys and model availability

Chain is too slow?

  1. Profile with LangSmith to find bottleneck
  2. Enable streaming for perceived speed
  3. Add caching for repeated queries
  4. Use async operations for parallel execution
  5. Reduce prompt/context size

Chain costs too much?

  1. Track token usage per chain with callbacks
  2. Implement model routing (cheap model first)
  3. Add semantic caching layer
  4. Reduce retrieved document count
  5. Use smaller models where quality permits

Testing LangChain Applications

Unit Testing Strategy

Use mock LLMs that return predetermined responses. This removes non-determinism and API cost from your test suite. Test that each chain step processes inputs correctly and passes outputs to the next step. Run unit tests on every commit in CI.

Evaluation Testing Strategy

Create gold-standard datasets in LangSmith with known correct answers. Run your chain against these datasets and measure: QA correctness, helpfulness, relevance, and toxicity scores. Compare scores across chain versions to catch quality regressions before production deployment.

Golden Rules of LLM Debugging

  • Don't guess, measure: Always use tracing (LangSmith) before making changes
  • Isolate components: Test retriever, LLM, parser, memory separately
  • Compare traces: Find a working case and diff against the failing one
  • Check the prompt first: 80% of LLM issues are prompt issues
  • Use deterministic settings: Set temperature=0 when debugging for reproducibility
  • Log everything: Token counts, latencies, costs — you can't optimize what you don't measure
