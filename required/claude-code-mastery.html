Section 1

Core Concepts & Setup

Claude Code is an agentic AI coding assistant that reads your codebase, edits files, runs commands, and integrates with your development tools — available in terminal, IDE, desktop app, and browser.

Full Codebase Awareness
Claude reads your entire project, understands architecture, tracks dependencies, and makes cross-file changes intelligently.
Agentic Execution
Claude plans multi-step tasks, runs tests, fixes failures, creates PRs, and iterates until the job is done — autonomously.
Permission-Safe
Granular permission system with pre-approval rules, hook-based guards, and multiple safety modes to keep you in control.

Installation

bash
# macOS, Linux, or WSL
curl -fsSL https://claude.ai/install.sh | bash

# After install, start in your project
cd your-project
claude
Auto-updates
Native installs automatically update in the background. You always get the latest features and security fixes without any manual action.
bash
brew install --cask claude-code

# Manual update needed periodically
brew upgrade claude-code
powershell
# PowerShell
irm https://claude.ai/install.ps1 | iex

# WinGet
winget install Anthropic.ClaudeCode

# CMD
curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd
bash
npm install -g @anthropic-ai/claude-code

First Steps

1
Authenticate
Run claude in your project directory. You'll be prompted to log in with your Claude.ai account or Anthropic Console API key. Authentication is handled once and persists across sessions.
2
Explore Your Codebase
Ask Claude to explore your project: "What does this codebase do? What's the main architecture?" — Claude will read files, trace dependencies, and give you a comprehensive overview.
3
Initialize Project Memory
Run /init to create a CLAUDE.md file bootstrapped from your codebase. This becomes Claude's persistent memory about your project.
4
Start Working
Describe what you want in plain language. Claude will plan, code, test, and iterate. Use Shift+Tab to enter Plan Mode first if you want to review the approach before execution.
Power Move: Pipe Input to Claude
Claude follows Unix philosophy — you can pipe data directly to it for powerful one-liners:
bash
# Monitor logs and get alerted about anomalies
tail -f app.log | claude -p "Alert me if you see errors"

# Review security issues in changed files
git diff main --name-only | claude -p "Review these files for security issues"

# Bulk translate strings and open PR
claude -p "translate new strings to French and raise a PR"

Permission Modes — Quick Switch

Default
Claude asks for permission before each action. Safest for exploration.
Plan Mode
Claude describes what it will do and waits. Great for reviewing before committing.
Accept Edits
File edits auto-accepted. Other actions still require approval.
Bypass Perms
Skip all permission checks. Use for CI/CD automation only.
Quick Mode Toggle
Press Shift+Tab during any session to cycle between Default, Plan Mode, and Auto-Accept Mode — without breaking your current conversation.

Section 3

Memory System

Claude Code's memory system lets you give Claude persistent, project-aware context that survives across sessions. Master this and Claude always knows exactly how your project works.

CLAUDE.md Files — You Write
Markdown files you maintain with explicit instructions, coding standards, architecture decisions, and conventions. Claude reads these at session start and follows them throughout.
Auto Memory — Claude Writes
Claude automatically records patterns, debugging insights, and architectural notes it discovers while working. Stored per-project and loaded at session start.

Memory Hierarchy

TypeLocationScopeShared With
Managed Policy /Library/Application Support/ClaudeCode/CLAUDE.md Org-wide rules by IT All users in org
Project Memory ./CLAUDE.md or ./.claude/CLAUDE.md Team-shared project rules Team via git
Project Rules ./.claude/rules/*.md Modular topic-specific rules Team via git
User Memory ~/.claude/CLAUDE.md Personal preferences, all projects Just you
Local Override ./CLAUDE.local.md Personal project-specific (gitignored) Just you
Auto Memory ~/.claude/projects/<project>/memory/ Claude's auto-generated notes Just you
Loading Priority
More specific instructions take precedence over broader ones. CLAUDE.md files above the working directory load in full at launch. Files in child directories load on-demand when Claude reads files in those directories.

What to Put in CLAUDE.md

CLAUDE.md example
# Project: My SaaS App

## Tech Stack
- TypeScript + Next.js 14 (App Router)
- PostgreSQL via Drizzle ORM
- Tailwind CSS v4
- Bun as package manager (never use npm/yarn)

## Commands
- Build: `bun run build`
- Test: `bun test --watch`
- Lint: `bun run lint`
- DB migrations: `bun run db:migrate`

## Architecture
- API routes in `src/app/api/`
- Server actions in `src/actions/`
- DB schema in `src/db/schema.ts`
- All env vars must be validated in `src/env.ts`

## Coding Standards
- Use functional components only, no class components
- Always add JSDoc comments for public functions
- Error boundaries required for all async boundaries
- Use Zod for all external data validation
- Prefer `const` over `let`, never use `var`

## Review Checklist
- [ ] No hardcoded secrets or API keys
- [ ] Input validation on all user-facing routes
- [ ] Tests for all new business logic
- [ ] Logging added for error paths

CLAUDE.md Imports

You can import other files into CLAUDE.md using @path/to/file syntax — great for keeping different concerns in separate files:

CLAUDE.md
# See project overview
See @README for project overview.
See @package.json for available scripts.

# Import specific instruction files
- Git workflow: @docs/git-instructions.md
- API conventions: @docs/api-conventions.md

# Import personal preferences from home dir (works across worktrees)
- Personal prefs: @~/.claude/my-project-preferences.md

Modular Rules with .claude/rules/

For larger projects, split instructions into focused files using the .claude/rules/ directory. Rules can even be path-specific using YAML frontmatter:

.claude/rules/api-rules.md
---
paths:
  - "src/api/**/*.ts"
  - "src/routes/**/*.ts"
---

# API Development Rules
- All endpoints must include Zod input validation
- Use standard error response format: `{ error: string, code: string }`
- Include OpenAPI documentation comments
- Rate limiting required on all public endpoints
Auto Memory — How It Works
  • Claude automatically saves project patterns, build commands, and key architecture notes
  • Stored in ~/.claude/projects/<project>/memory/MEMORY.md
  • First 200 lines loaded at every session start
  • Claude creates topic files (e.g., debugging.md, api-conventions.md) for detailed notes
  • Enable with: export CLAUDE_CODE_DISABLE_AUTO_MEMORY=0
Best Practices
  • Be specific: "Use 2-space indentation" beats "format code properly"
  • Include commands: Build, test, lint — Claude won't search for them
  • Document architecture: Key files, module relationships, important abstractions
  • Add review checklists: Claude will follow them for every PR
  • Review periodically: Update as your project evolves
  • Use /init to bootstrap from existing codebase
Pro Tip
Tell Claude to save things directly: "Remember that we use pnpm, not npm" or "Save to memory that the API tests need a local Redis instance." Claude will update MEMORY.md immediately and reference it in future sessions.

Hook Configuration Format

.claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",        // regex matching tool name
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/lint.sh",
            "timeout": 30,
            "async": false         // set true to run in background
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/validate-cmd.sh"
          }
        ]
      }
    ]
  }
}

Practical Hook Examples

Block rm -rf and other destructive commands before they execute. The script exits with code 2 to deny the action:

.claude/hooks/block-rm.sh
#!/bin/bash
COMMAND=$(cat | jq -r '.tool_input.command')

if echo "$COMMAND" | grep -q 'rm -rf'; then
  echo "Blocked: rm -rf is not allowed" >&2
  exit 2   # exit 2 = blocking error
fi

exit 0   # exit 0 = allow
.claude/hooks/lint-check.sh
#!/bin/bash
INPUT=$(cat)
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

if [[ "$FILE" == *.ts || "$FILE" == *.tsx ]]; then
  if ! npx eslint "$FILE" 2>&1; then
    # Return feedback to Claude as JSON
    echo '{"decision":"block","reason":"ESLint errors found. Please fix them."}'
    exit 0
  fi
fi
exit 0
.claude/hooks/session-context.sh
#!/bin/bash
# Inject environment variables for the session
if [ -n "$CLAUDE_ENV_FILE" ]; then
  echo 'export NODE_ENV=development' >> "$CLAUDE_ENV_FILE"
  echo 'export API_URL=http://localhost:3000' >> "$CLAUDE_ENV_FILE"
fi

# Add context Claude can see and act on
echo '{
  "hookSpecificOutput": {
    "hookEventName": "SessionStart",
    "additionalContext": "Current branch: '"$(git branch --show-current)"'\nOpen issues: '"$(gh issue list --limit 5 --json number,title | jq -r '.[] | "#\(.number): \(.title)"')"'"
  }
}'

Use a fast Claude model to evaluate whether an action should be allowed — no shell script needed:

settings.json — prompt hook
{
  "hooks": {
    "Stop": [{
      "hooks": [{
        "type": "prompt",
        "prompt": "Evaluate if all tasks are complete: $ARGUMENTS. Check conversation history. Return {\"ok\": true} to allow stopping, or {\"ok\": false, \"reason\": \"tasks remaining\"} to continue.",
        "timeout": 30
      }]
    }]
  }
}

Exit Code Reference

exit 0 — Success
Claude Code parses stdout for JSON output. If no JSON, action proceeds normally. Stdout shown in verbose mode (Ctrl+O).
exit 2 — Block
Blocking error. JSON stdout is ignored. Stderr text is fed back to Claude as an error message. Blocks the action (for events that support blocking).
Other Exit Codes
Non-blocking error. Stderr shown in verbose mode. Execution continues. Use for warnings or when hook itself failed but shouldn't stop Claude.
MCP Tool Matchers
MCP tools follow the pattern mcp__<server>__<tool>. Use regex patterns like mcp__memory__.* to match all tools from a server, or mcp__.*__write.* to match any write-like tool across all servers.

Managing Hooks

Type /hooks to open the interactive hooks manager — add, view, and delete hooks without editing JSON files directly. Hooks are labeled by source: [User] [Project] [Local] [Plugin]

Section 5

MCP Servers

Model Context Protocol (MCP) is an open standard for connecting Claude Code to external tools, databases, APIs, and services. Connect GitHub, Slack, Jira, Postgres, Sentry, and hundreds more.

GitHub
Review PRs, create issues, manage branches, search code, and automate GitHub workflows directly from Claude.
Databases
Query PostgreSQL, MySQL, SQLite naturally. Ask "What's our revenue this month?" and Claude writes and runs the query.
Observability
Pull Sentry errors, analyze Datadog metrics, check Statsig flags. Debug production issues without leaving Claude.

Adding MCP Servers

bash — HTTP transport (recommended)
# Basic HTTP server
claude mcp add --transport http github https://api.githubcopilot.com/mcp/

# With Bearer token auth
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp \
  --header "Authorization: Bearer your-token"

# Notion, Jira, Slack examples
claude mcp add --transport http notion https://mcp.notion.com/mcp
claude mcp add --transport http slack https://mcp.slack.com/mcp
claude mcp add --transport http linear https://mcp.linear.app/mcp

# Then authenticate with OAuth
claude  # start Claude
> /mcp   # opens auth manager
bash — local stdio transport
# PostgreSQL database
claude mcp add --transport stdio db \
  -- npx -y @bytebase/dbhub \
  --dsn "postgresql://user:pass@host:5432/db"

# Filesystem access
claude mcp add --transport stdio filesystem \
  --env TARGET_DIR=/workspace \
  -- npx -y @modelcontextprotocol/server-filesystem

# Airtable with API key
claude mcp add --transport stdio --env AIRTABLE_API_KEY=YOUR_KEY airtable \
  -- npx -y airtable-mcp-server

# Windows: wrap with cmd /c
claude mcp add --transport stdio my-server \
  -- cmd /c npx -y @some/package
bash — managing servers
claude mcp list            # List all configured servers
claude mcp get github      # Details for a specific server
claude mcp remove github   # Remove a server

# Import from Claude Desktop (macOS/WSL)
claude mcp add-from-claude-desktop

# Add from JSON config
claude mcp add-json weather '{"type":"http","url":"https://weather.api/mcp"}'

# Inside Claude session
> /mcp   # view status, authenticate, manage

Section 6

Sub-Agents & Task Tool

Sub-agents are specialized AI assistants that handle specific types of tasks in their own isolated context window. Spawn parallel researchers, enforce tool restrictions, and reuse specialized configurations across projects.

Explore (Built-in)
Fast, read-only agent using Haiku model. Optimized for searching and analyzing codebases without making changes. Keeps exploration results out of main context.
Plan (Built-in)
Research agent used in plan mode. Gathers context from your codebase before presenting a plan. Read-only tools only — can't make changes.
General-Purpose (Built-in)
Full-capability agent for complex multi-step tasks requiring both exploration and file modification. Inherits all tools from main conversation.

Creating Custom Sub-Agents

Sub-agents are Markdown files with YAML frontmatter. Store them in ~/.claude/agents/ (personal) or .claude/agents/ (project-shared):

.claude/agents/code-reviewer.md
---
name: code-reviewer
description: Expert code reviewer. Use proactively after code changes. Reviews for quality, security, and maintainability.
tools: Read, Grep, Glob, Bash
model: sonnet        # sonnet | opus | haiku | inherit
permissionMode: plan # default | acceptEdits | dontAsk | bypassPermissions | plan
maxTurns: 20
memory: user          # persistent memory: user | project | local
---

You are a senior code reviewer. When invoked:
1. Run `git diff` to see recent changes
2. Focus on modified files only
3. Check for: security issues, error handling, test coverage, naming

Format feedback by priority:
- Critical (must fix): security, data loss, crashes
- Warning (should fix): code quality, performance
- Suggestion (consider): style, readability

Sub-Agent Configuration Fields

FieldRequiredDescription
nameYesUnique identifier (lowercase, hyphens). Used for Task(agent-name) invocation
descriptionYesWhen Claude should delegate to this agent. Write this carefully — Claude uses it to decide when to use the agent
toolsNoComma-separated tool allowlist. If omitted, inherits all tools
disallowedToolsNoTools to explicitly deny even if inherited
modelNosonnet, opus, haiku, or inherit (default)
permissionModeNoPermission mode for this agent's actions
maxTurnsNoMaximum agentic turns before agent stops
skillsNoSkills to preload into agent context at startup
mcpServersNoMCP servers available to this agent only
hooksNoLifecycle hooks scoped to this agent
memoryNoPersistent memory: user, project, or local
isolationNoworktree — run in isolated git worktree
backgroundNotrue — always run as background task

Powerful Sub-Agent Patterns

Spawn multiple agents to research different areas simultaneously — they run in parallel, then Claude synthesizes results:

prompt
Research the auth, database, and API modules using separate subagents in parallel.
Each should map out: key files, main functions, dependencies, and potential issues.

Chain agents for sequential workflows where each result feeds the next:

prompt
1. Use the code-reviewer subagent to find all performance issues
2. Pass those findings to the optimizer subagent to fix them
3. Use the test-writer subagent to add regression tests for each fix
.claude/agents/db-reader.md
---
name: db-reader
description: Execute read-only database queries for analysis and reporting
tools: Bash
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-readonly.sh"
---

You are a database analyst with read-only access.
Execute SELECT queries only. If asked to write data, explain you cannot.

bash — define agents at launch
claude --agents '{
  "code-reviewer": {
    "description": "Expert code reviewer. Use proactively after code changes.",
    "prompt": "You are a senior code reviewer. Focus on code quality, security, best practices.",
    "tools": ["Read", "Grep", "Glob", "Bash"],
    "model": "sonnet"
  },

Foreground vs Background Sub-Agents
Press Ctrl+B during a running task to move it to the background while you continue working. Background agents auto-deny permission prompts they weren't pre-approved for — Claude asks upfront for all needed permissions before backgrounding.

  "includeCoAuthoredBy": false,

  // Cleanup sessions older than N days
  "cleanupPeriodDays": 30
}

Permission Rule Syntax

Rules follow the format Tool or Tool(specifier). Evaluated in order: deny wins, then ask, then allow. First match wins:

permission rule examples
Bash                   # All bash commands
Bash(npm run *)        # Commands starting with 'npm run'
Bash(git diff *)       # Any git diff variant
Read(./.env)           # Reading the .env file specifically
Read(./secrets/**)     # Anything in secrets/ (recursive)
Edit(/path/to/file)    # Write to specific file
WebFetch(domain:example.com)  # Fetches to example.com
MCP(tool-name)         # Specific MCP tool access
Task(Explore)          # Spawning the Explore subagent
Skill(deploy *)        # Running any deploy skill

Custom Skills (Slash Commands)

Skills are your own custom slash commands. Create a SKILL.md file and Claude adds it to its toolkit. Invoke with /skill-name or let Claude trigger them automatically.

.claude/skills/review-pr/SKILL.md
---
name: review-pr
description: Comprehensive PR review covering security, quality, and performance
argument-hint: [pr-number]
disable-model-invocation: true  # only you can trigger this
allowed-tools: Bash(gh *), Read, Grep
---

Review PR $ARGUMENTS comprehensively:

## Live PR Context
- PR diff: !`gh pr diff $ARGUMENTS`
- Comments: !`gh pr view $ARGUMENTS --comments`
- Changed files: !`gh pr diff $ARGUMENTS --name-only`

## Review Checklist
1. Security: input validation, auth checks, secrets exposure
2. Logic: edge cases, error handling, race conditions
3. Performance: N+1 queries, unnecessary computation
4. Tests: coverage for new logic
5. Docs: JSDoc/docstrings for public APIs

Format feedback by severity: Critical > Warning > Suggestion

Skill Locations & Scopes

LocationPathApplies To
EnterpriseManaged settingsAll users in organization
Personal~/.claude/skills/<name>/SKILL.mdAll your projects
Project.claude/skills/<name>/SKILL.mdCurrent project (shareable)
Plugin<plugin>/skills/<name>/SKILL.mdWhere plugin is enabled

Skill Frontmatter Options

SKILL.md frontmatter reference
---
name: deploy-staging              # becomes /deploy-staging command
description: "Deploy to staging"    # Claude uses this to decide when to invoke
argument-hint: [service-name]     # shown in autocomplete
disable-model-invocation: true    # only manual /slash-command invocation
user-invocable: false             # hidden from menu, Claude-only
allowed-tools: Bash(kubectl *), Bash(helm *)  # tools auto-approved
model: sonnet                      # override model for this skill
context: fork                      # run in isolated subagent context
agent: Explore                     # which subagent type (with context: fork)
---

Dynamic Context Injection with !

The !`command` syntax in skills runs shell commands before Claude sees anything. The output replaces the placeholder — Claude gets real data, not the command itself:

skill with dynamic context
---
name: debug-session
description: Start a debugging session with full environment context
---

Debug $ARGUMENTS with this context:

Current branch: !`git branch --show-current`
Recent commits: !`git log --oneline -5`
Test failures: !`bun test 2>&1 | grep FAIL | head -20`
Open issues: !`gh issue list --limit 5 --json number,title`
Env vars: !`cat .env.example`

Context Window Management

Auto-Compaction

Claude automatically compacts the context when it nears capacity (~95%). Use /compact manually with optional focus instructions, e.g., /compact focus on auth changes.

Context Visibility

Run /context to visualize context usage as a colored grid. Set CLAUDE_CODE_AUTOCOMPACT_PCT_OVERRIDE=50 to compact at 50% capacity for large projects.

Best Practices for Power Users

  • Write a comprehensive CLAUDE.md with build commands, architecture, and coding standards
  • Use Plan Mode (/plan or Shift+Tab) before large refactors to review Claude's approach first
  • Create custom skills for repeated workflows: /review-pr, /deploy-staging, /commit-msg
  • Set up PostToolUse hooks for auto-linting and test runs after file edits
  • Use PreToolUse hooks to block dangerous commands and enforce security policies
  • Pre-approve common commands in settings.json to reduce permission interruptions
  • Name sessions with /rename for easy resumption later: /rename auth-refactor-2026
  • Use -w (worktree) flag to run parallel Claude sessions without interference
  • Pipe data to Claude: git diff main | claude -p "review for security"
  • Use MCP servers to connect Claude to your actual project tools (GitHub, Jira, DB)
  • Create specialized sub-agents for code review, testing, and documentation
  • Use --max-turns and --max-budget-usd in scripts to control costs
