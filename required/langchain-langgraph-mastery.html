Prerequisites & LLM Primer

Before we dive into LangChain, let's make sure you understand the foundation — what LLMs actually are and how they work.

💡
The Core Idea

An LLM (Large Language Model) is a program that has read billions of pages of text and learned to predict "what word comes next". That's it. Everything else — answering questions, writing code, reasoning — emerges from this one ability at massive scale.

🧠

What is an LLM?

A neural network with billions of parameters, trained to predict the next token in a sequence. Models like GPT-4, Claude, and Gemini are all LLMs. They don't "know" things — they've learned statistical patterns from text.

🔤

What is a Token?

Text is broken into chunks called tokens. "Hello world" = 2 tokens. Code can be 3–4x more tokens than plain English. Models charge by token and have a max context window (how much they can read at once).

🎲

How LLMs Generate Text

They predict one token at a time, sampling from a probability distribution. Temperature controls randomness: 0 = deterministic, 1 = creative. This is called autoregressive generation.

💬
Chat Models vs Base LLMs — A base LLM just completes text. A chat model has been fine-tuned with RLHF (Reinforcement Learning from Human Feedback) to follow instructions and be helpful. LangChain works with both, but you'll almost always use chat models (like ChatOpenAI, ChatAnthropic).
⚠️
LLM Limitations to Keep in Mind: LLMs have a knowledge cutoff, can hallucinate facts, have limited context windows (8K–2M tokens), and are stateless by default — they don't remember previous conversations unless you explicitly pass history. LangChain and LangGraph exist specifically to work around these limitations.

LangChain Fundamentals

LangChain is the most popular framework for building LLM-powered applications. Let's understand why it was created before learning how to use it.

The Problem LangChain Solves

😩

Without LangChain

You write raw API calls. You manually format prompts. You write custom retry logic. You hand-roll memory systems. You build your own document loaders, text splitters, vector store integrations. Every app starts from scratch.

😎

With LangChain

You snap together pre-built components. Prompts, models, parsers, memory, retrievers — all composable via a unified interface. 100+ integrations out of the box. 6 lines of code instead of 60.

🧱
The LEGO Analogy

Think of LangChain as LEGO bricks for AI apps. Each component (prompt, model, parser, retriever) is a standardized brick. They all have the same interface, so you can combine any of them freely. LangChain provides both the bricks AND a language for snapping them together (LCEL).

🟢 Beginner · Section 3

LCEL & Core Primitives

LangChain Expression Language (LCEL) is the modern way to compose LangChain components. It uses a pipe operator (|) to chain components together — just like Unix pipes.

🔧
The Unix Pipe Analogy

You know how Unix pipes work: cat file.txt | grep "error" | wc -l. Each command takes input and passes output to the next. LCEL does exactly this for AI components: prompt | model | parser. The output of each piece becomes the input of the next.

Chat Models — The Brain

Chat models take a list of messages (not just a string) and return a message. LangChain wraps every LLM provider with a standardized ChatModel interface so you can swap providers without changing code.

Python
from langchain_openai import ChatOpenAI
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import HumanMessage, SystemMessage

# All models share the same interface
model = ChatOpenAI(model="gpt-4o", temperature=0.7)
# model = ChatAnthropic(model="claude-3-5-sonnet-20241022")

# Call with messages
response = model.invoke([
    SystemMessage(content="You are a helpful assistant."),
    HumanMessage(content="What is LangChain?")
])

print(response.content)       # The text response
print(response.usage_metadata) # Token counts
💡
Key insight: model.invoke() for single calls, model.stream() for streaming, model.batch() for parallel calls. All chat models support these three modes.

Prompt Templates — Dynamic Instructions

Prompt templates let you create reusable, parameterized prompts. Instead of hardcoding text, you define a template with placeholders and fill them in at runtime.

Python
from langchain_core.prompts import (
    ChatPromptTemplate,
    MessagesPlaceholder,
    HumanMessagePromptTemplate
)

# Simple prompt with variables
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are an expert in {domain}. Be concise."),
    ("human", "{question}")
])

# Format with variables
messages = prompt.invoke({
    "domain": "machine learning",
    "question": "Explain backpropagation"
})

# MessagesPlaceholder — for injecting conversation history
chat_prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    MessagesPlaceholder(variable_name="history"),  # ← inject history here
    ("human", "{input}")
])

Output Parsers — Structure the Response

LLMs always return text. Output parsers transform that text into structured Python objects — strings, lists, JSON, Pydantic models — so you can actually use the response in code.

Python
from langchain_core.output_parsers import StrOutputParser, JsonOutputParser
from langchain_core.pydantic_v1 import BaseModel, Field

# 1. StrOutputParser — just extract the string content
parser = StrOutputParser()

# 2. JsonOutputParser — parse JSON from LLM response
json_parser = JsonOutputParser()

# 3. PydanticOutputParser — validate + parse into a typed object
class MovieReview(BaseModel):
    title: str = Field(description="Movie title")
    score: int = Field(description="Rating from 1-10")
    summary: str = Field(description="One sentence summary")

pydantic_parser = JsonOutputParser(pydantic_object=MovieReview)

# The parser adds format instructions to your prompt automatically:
print(pydantic_parser.get_format_instructions())

Runnables — The LCEL Building Blocks

Every LCEL component is a Runnable — it has .invoke(), .stream(), and .batch(). The pipe operator combines Runnables into chains. Special Runnables give you superpowers:

Python
from langchain_core.runnables import (
    RunnablePassthrough,   # pass input unchanged
    RunnableParallel,      # run multiple branches simultaneously
    RunnableLambda,        # wrap any function as a Runnable
    RunnableWithMessageHistory  # add memory to any chain
)

# RunnablePassthrough — pass input through unchanged
chain = RunnablePassthrough() | model | StrOutputParser()

# RunnableParallel — run branches in parallel, merge results
parallel = RunnableParallel({
    "question": RunnablePassthrough(),
    "context": retriever  # fetch docs in parallel with question
})

# RunnableLambda — turn any function into a composable piece
uppercase = RunnableLambda(lambda x: x.upper())
chain = prompt | model | StrOutputParser() | uppercase

Building an LCEL Chain

Now let's put it all together. The | pipe operator creates a chain where data flows left to right. This is the heart of modern LangChain development.

Python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# Define components
prompt = ChatPromptTemplate.from_template(
    "Explain {topic} in simple terms a 15-year-old would understand."
)
model = ChatOpenAI(model="gpt-4o")
parser = StrOutputParser()

# Compose with | operator — this is LCEL!
chain = prompt | model | parser

# Invoke the chain
result = chain.invoke({"topic": "neural networks"})
print(result)

# Stream token by token
for chunk in chain.stream({"topic": "backpropagation"}):
    print(chunk, end="", flush=True)

# Batch multiple inputs in parallel
results = chain.batch([
    {"topic": "attention mechanisms"},
    {"topic": "transformers"},
    {"topic": "tokenization"}
])

🟡 Intermediate · Section 4

Memory Systems

LLMs are stateless — they forget everything after each call. Memory systems give your chatbot the ability to remember past conversations.

📓
The Notepad Analogy

Think of memory as a notepad the AI reads before every response. Buffer memory = copies every message verbatim. Summary memory = writes a condensed summary instead. Vector memory = looks up only the most relevant past messages from a database.

ConversationBufferMemory

Stores every single message in full. Simple and accurate but grows forever — will eventually overflow the context window.

U
Hi! My name is Alex.
AI
Hello Alex! Nice to meet you. How can I help?
U
What's the capital of France?
AI
The capital of France is Paris.
U
What's my name?
💾 Memory contains ALL 4 previous messages verbatim → AI can answer "Your name is Alex" ✓

ConversationSummaryMemory

After each turn, an LLM summarizes the conversation so far. Stores the summary instead of raw messages. Great for long conversations — uses far fewer tokens.

U
Hi! My name is Alex. I'm building a startup.
AI
Great to meet you Alex! Startups are exciting...
U
We're working on an AI coding assistant.
💾 Summary memory stores: "User named Alex is building a startup focused on an AI coding assistant." — Much shorter than raw messages, but still captures the key info.

VectorStoreRetrieverMemory

Stores messages as vector embeddings in a vector database. At each turn, semantically retrieves only the most relevant past messages. Best for long sessions, worst for recent context.

U
My favorite food is sushi.
AI
Great taste! Japanese cuisine is wonderful.
— 50 messages later —
U
What should I have for dinner?
🔍 Vector search finds "favorite food is sushi" as most relevant → AI suggests sushi-related options ✓ (even after 50 messages!)

Modern Memory with RunnableWithMessageHistory

Python
from langchain_core.chat_history import BaseChatMessageHistory
from langchain_community.chat_message_histories import ChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

# In-memory store keyed by session_id
store = {}

def get_session_history(session_id: str) -> BaseChatMessageHistory:
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

# Wrap any chain with memory
with_memory = RunnableWithMessageHistory(
    chain,                                      # your LCEL chain
    get_session_history,                        # function to get/create history
    input_messages_key="input",              # where user message goes
    history_messages_key="history"            # where to inject history
)

# Invoke with session tracking
config = {"configurable": {"session_id": "user-123"}}
with_memory.invoke({"input": "My name is Alex"}, config=config)
with_memory.invoke({"input": "What is my name?"}, config=config) # Remembers!

Buffer Memory

Best for: Short conversations, <5k tokens

Limit: Grows unboundedly, will overflow context window

Summary Memory

Best for: Long conversations needing key facts retained

Limit: Can lose nuance; costs extra LLM calls to summarize

Vector Memory

Best for: Very long sessions, semantic retrieval of past context

Limit: Can miss recent context; needs vector DB setup

🟡 Intermediate · Section 5

Tools & Tool Calling

Tools let LLMs interact with the real world — search the web, run code, query databases, call APIs. Tool calling is how LLMs act, not just think.

🔨
The Contractor Analogy

An LLM without tools is like a brilliant contractor who can only talk about building houses. Add tools and now they can actually pick up a hammer, use a calculator, and look up building codes. The LLM decides which tool to use based on the task — just like a contractor picks the right tool for each job.

How Tool Calling Works — Step by Step

Tool Calling Flow: "What's the weather in Paris?"

1
User sends query
2
LLM picks tool
3
Tool executes
4
Result returned
5
Final answer
USER INPUT

"What's the weather in Paris right now?"

The user's message is sent to the LLM along with a list of available tool schemas — JSON descriptions of what each tool does and what parameters it accepts.

LLM DECISION

LLM selects: get_weather(city="Paris")

The LLM reads the tool schemas and decides it needs the get_weather tool. Instead of generating a text response, it outputs a structured tool call — a JSON object with the tool name and arguments. It does NOT make the call itself.

EXECUTION

Your code runs: get_weather("Paris")

LangChain sees the tool call, finds the function registered as get_weather, and executes it. The function calls a real weather API and returns actual data. The LLM has no control over this step — it's just Python code.

TOOL RESULT

Result sent back: {"temp": 18, "condition": "Cloudy"}

The tool result is appended to the conversation as a ToolMessage and sent back to the LLM. The LLM now has the real data it needed.

FINAL ANSWER

"It's currently 18°C and cloudy in Paris."

The LLM synthesizes the tool result into a natural language response. The user gets an accurate, up-to-date answer — no hallucination because the data came from a real API call.

Creating Tools with LangChain

Python
from langchain_core.tools import tool
from langchain_core.pydantic_v1 import BaseModel, Field

# Method 1: @tool decorator (simplest)
@tool
def search_web(query: str) -> str:
    """Search the web for current information. Use for facts, news, prices."""
    # Your actual search implementation here
    return f"Results for: {query}"

# Method 2: StructuredTool with Pydantic schema (for complex args)
class DatabaseQueryInput(BaseModel):
    table: str = Field(description="Database table to query")
    filter_col: str = Field(description="Column to filter on")
    value: str = Field(description="Value to filter by")

@tool(args_schema=DatabaseQueryInput)
def query_database(table: str, filter_col: str, value: str) -> str:
    """Query a SQL database. Returns matching rows as JSON."""
    return f"SELECT * FROM {table} WHERE {filter_col} = '{value}'"

# Bind tools to the model
tools = [search_web, query_database]
model_with_tools = model.bind_tools(tools)

# The model now knows about these tools and can call them
response = model_with_tools.invoke("What's the latest news on AI?")
print(response.tool_calls)  # LLM decided to call search_web!

RAG — Retrieval Augmented Generation

RAG solves the LLM's biggest limitation: it only knows what it was trained on. RAG lets you give the LLM access to your own documents at query time — in real time, no retraining needed.

📚
The Open-Book Exam Analogy

A standard LLM is like a closed-book exam — it can only answer from what it memorized during training. RAG gives the LLM an open book — before answering, it looks up relevant passages from your documents and uses them as context. It's not memorizing; it's reading and quoting.

RAG Code: From Scratch to Working Pipeline

Python — Full RAG Pipeline
from langchain_community.document_loaders import PyPDFLoader, WebBaseLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

# ── STEP 1: LOAD DOCUMENTS ──────────────────────────
loader = PyPDFLoader("company_handbook.pdf")
documents = loader.load()  # List of Document objects

# ── STEP 2: SPLIT INTO CHUNKS ───────────────────────
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,      # characters per chunk
    chunk_overlap=200,    # overlap between chunks (prevents losing context)
    separators=["\n\n", "\n", ".", " "]
)
chunks = splitter.split_documents(documents)
# Each chunk is a Document with page_content + metadata

# ── STEP 3: EMBED + STORE ───────────────────────────
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.from_documents(chunks, embeddings)
# vectorstore.save_local("my_index")  # save to disk

# ── STEP 4: CREATE RETRIEVER ────────────────────────
retriever = vectorstore.as_retriever(
    search_type="similarity",  # or "mmr" for diversity
    search_kwargs={"k": 4}     # return top 4 chunks
)

# ── STEP 5: BUILD THE RAG CHAIN ─────────────────────
prompt = ChatPromptTemplate.from_template("""
Answer the question based ONLY on the following context.
If the answer isn't in the context, say "I don't know."

Context: {context}

Question: {question}
""")

def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | ChatOpenAI(model="gpt-4o")
    | StrOutputParser()
)

# ── USE IT! ─────────────────────────────────────────
answer = rag_chain.invoke("What is the vacation policy?")
print(answer)

🟣 Intermediate · Section 7

LangGraph — Why It Exists

LangChain chains are great for linear workflows. But real-world AI applications need loops, branches, and the ability to pause and wait. That's what LangGraph was built for.

⚠️
The Problem with Simple Chains: LCEL chains execute exactly once, left to right. You can't loop back, branch conditionally based on LLM output, pause for human approval, or run agents that retry on failure. These are dealbreakers for real agents.
🗺️
The Road Network Analogy

LCEL chains are a straight highway — fast, efficient, but you can only go from A to B in a fixed path. LangGraph is a full road network with roundabouts, intersections, and dead ends. Your data can take different routes based on conditions, loop back when needed, and merge from multiple paths.

Core Concepts: State, Nodes, Edges

📋

State

A shared Python dictionary (TypedDict) that all nodes can read and update. Think of it as the whiteboard everyone in the office shares. The state is the single source of truth for what's happening in your workflow.

⚙️

Nodes

Python functions that take the current state and return an update to it. Each node does one thing: call an LLM, run a tool, query a database, make a decision. Nodes are where the actual work happens.

→

Edges

Connections between nodes that define the flow. Regular edges always go from A to B. Conditional edges look at the current state and decide which node to go to next — this is what enables branching and routing.

StateGraph vs MessageGraph

StateGraph Most Common

You define a custom TypedDict state with any fields you want. State can hold messages, tool results, metadata, flags, counters — anything. Most flexible, use for most use cases.

Python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, add_messages

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    user_name: str
    steps_taken: int
    is_done: bool

MessageGraph Simpler

A pre-built state where the state IS just a list of messages. Great for simple chatbots where you only need conversation history. Limited flexibility compared to StateGraph.

Python
from langgraph.graph import MessageGraph

# State is automatically just a list of messages
# No custom TypedDict needed
graph = MessageGraph()
graph.add_node("chat", chat_node)
graph.set_entry_point("chat")
graph.set_finish_point("chat")

Building Your First LangGraph Agent

Python — Complete StateGraph Example
from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, START, END, add_messages
from langchain_openai import ChatOpenAI
from langchain_core.messages import BaseMessage

# ── 1. DEFINE STATE ────────────────────────────────
class AgentState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]
    # add_messages is a "reducer" — it appends new messages
    # instead of overwriting the whole list

# ── 2. DEFINE NODES ────────────────────────────────
model = ChatOpenAI(model="gpt-4o")

def call_model(state: AgentState) -> AgentState:
    """Node: call the LLM with current messages"""
    response = model.invoke(state["messages"])
    return {"messages": [response]}  # add_messages reducer appends this

def should_continue(state: AgentState) -> Literal["tools", END]:
    """Conditional edge: check if the LLM wants to call a tool"""
    last_msg = state["messages"][-1]
    if hasattr(last_msg, "tool_calls") and last_msg.tool_calls:
        return "tools"  # → go to tool node
    return END          # → we're done

def call_tools(state: AgentState) -> AgentState:
    """Node: execute any tool calls the LLM requested"""
    # tool execution logic here
    tool_results = execute_tools(state["messages"][-1].tool_calls)
    return {"messages": tool_results}

# ── 3. BUILD THE GRAPH ─────────────────────────────
graph_builder = StateGraph(AgentState)
graph_builder.add_node("agent", call_model)
graph_builder.add_node("tools", call_tools)

graph_builder.add_edge(START, "agent")  # always start at agent
graph_builder.add_conditional_edges(
    "agent",        # from this node...
    should_continue, # ...call this function to decide...
    {"tools": "tools", END: END}  # ...and map to these targets
)
graph_builder.add_edge("tools", "agent")  # tools always go back to agent

# ── 4. COMPILE AND RUN ─────────────────────────────
graph = graph_builder.compile()
result = graph.invoke({"messages": [("user", "What's 42 * 17?")]})

LangGraph Advanced Features

Checkpointing, human-in-the-loop, streaming, and subgraphs — the features that make LangGraph production-ready.

Checkpointing — The Save-Game Feature

LangGraph saves the full state at every node execution. This enables resuming workflows, human approval flows, and long-running tasks that survive server restarts.

🎮
Save-Game Analogy

Checkpointing is exactly like a video game save point. If something goes wrong (error, user needs to review, server crashes), you resume from the last checkpoint — not from the beginning.

Python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.checkpoint.sqlite import SqliteSaver

# In-memory checkpointer (dev/testing)
memory = MemorySaver()

# SQLite checkpointer (persists to disk — survives restarts!)
with SqliteSaver.from_conn_string(":memory:") as checkpointer:
    graph = graph_builder.compile(checkpointer=checkpointer)

# Each run needs a thread_id to track state
config = {"configurable": {"thread_id": "session-42"}}

result1 = graph.invoke({"messages": ["What is Python?"]}, config)
result2 = graph.invoke({"messages": ["What about Java?"]}, config)
# The graph REMEMBERS the first question when answering the second!

# Inspect the saved state at any point
state = graph.get_state(config)
history = list(graph.get_state_history(config))

Human-in-the-Loop

Pause the graph at critical points — a human reviews and approves before execution continues. Essential for high-stakes actions (sending emails, database writes, API calls).

Python
from langgraph.graph import StateGraph, interrupt

def dangerous_action_node(state):
    """This node pauses for human approval before acting."""
    action_plan = state["plan"]
    
    # PAUSE and send plan to human for review
    human_decision = interrupt({
        "message": "Please review this action plan",
        "plan": action_plan
    })  # Execution STOPS here until resumed
    
    if human_decision["approved"]:
        execute_plan(action_plan)
        return {"status": "executed"}
    else:
        return {"status": "cancelled"}

# Compile with checkpointer (required for HITL)
graph = builder.compile(checkpointer=MemorySaver())

# Resume with human's decision (can be hours later!)
graph.invoke(
    Command(resume={"approved": True}),
    config={"configurable": {"thread_id": "session-42"}}
)

Streaming — Real-Time Visibility

LangGraph can stream three types of events: state updates between nodes, LLM tokens as they generate, and node transition events. Choose based on what your UI needs.

Python
# Stream mode 1: "values" — full state after each node
for event in graph.stream(inputs, mode="values"):
    print("State:", event)

# Stream mode 2: "updates" — only what changed (more efficient)
for node_name, state_delta in graph.stream(inputs, mode="updates"):
    print(f"Node {node_name} updated: {state_delta}")

# Stream mode 3: "messages" — get LLM tokens in real-time!
for msg, metadata in graph.stream(inputs, stream_mode="messages"):
    if isinstance(msg, AIMessageChunk):
        print(msg.content, end="", flush=True)  # Real-time tokens!

# Stream mode 4: "events" — granular everything
async for event in graph.astream_events(inputs, version="v2"):
    if event["event"] == "on_chat_model_stream":
        print(event["data"]["chunk"].content, end="")

Subgraphs — Graphs within Graphs

A subgraph is a complete LangGraph graph used as a single node inside a parent graph. This is the key architectural pattern for multi-agent systems — each agent is a subgraph.

Python
# Define a subgraph (a complete mini-agent)
research_builder = StateGraph(ResearchState)
research_builder.add_node("search", search_node)
research_builder.add_node("summarize", summarize_node)
research_builder.add_edge("search", "summarize")
research_builder.set_entry_point("search")
research_builder.set_finish_point("summarize")
research_agent = research_builder.compile()  # ← compiled subgraph

# Use the subgraph AS A NODE in the parent graph
parent_builder = StateGraph(MainState)
parent_builder.add_node("researcher", research_agent)   # ← subgraph as node!
parent_builder.add_node("writer", writer_agent)
parent_builder.add_edge("researcher", "writer")

🟠 Advanced · Section 9

AI Agents & the ReAct Pattern

An AI agent is a system where an LLM autonomously decides what actions to take, executes them, observes the results, and loops until the task is complete. ReAct is the foundational pattern that makes this work.

🤖

What Makes Something an "Agent"?

An agent has four things: (1) Perception — it receives input. (2) Reasoning — an LLM thinks about what to do. (3) Action — it uses tools to affect the world. (4) Memory — it remembers what it's done. Remove any one of these and it's just a chain.

🔄

ReAct = Reason + Act

ReAct (Reasoning + Acting) is a prompting strategy where the LLM alternates between thinking out loud (Thought) and taking action (Action), then observing the result (Observation) — repeating until it has an answer. Like a detective's inner monologue.

🕵️
The Detective Analogy

A detective doesn't guess the answer immediately. They reason: "The suspect had no alibi. Let me check the CCTV footage." Then they act: review the footage. Then they observe: "The suspect is on the footage." Then they reason again. ReAct agents do exactly this — think, act, observe, repeat.

ReAct Trace — Step Through a Real Example

Task: "What was Tesla's revenue in 2023 and how does it compare to 2022?"

2

3

4

THOUGHT

Agent reasons about what it needs

The user wants Tesla's 2023 revenue AND a comparison with 2022. I don't have real-time financial data in my training. I need to search for both numbers. Let me start with 2023 revenue.

ACTION

search_web("Tesla annual revenue 2023")

The agent calls the search_web tool with a specific query. The LLM doesn't guess — it knows it needs real data. This is the "Act" part of ReAct.

OBSERVATION

Tool returns: "Tesla 2023 annual revenue: $96.77 billion"

The search result is returned as an observation. The agent now has the 2023 number. But it still needs 2022 to make the comparison the user asked for.

Agent realizes it needs more data

I have 2023 revenue: $96.77B. I still need 2022 revenue to calculate the year-over-year comparison. Let me search for that too.

ACTION

search_web("Tesla annual revenue 2022")

A second tool call — same tool, different query. Notice how the agent adapts based on what it learned from the first observation. This iterative refinement is the power of ReAct.

OBSERVATION

Tool returns: "Tesla 2022 annual revenue: $81.46 billion"

Now the agent has both data points. It has all the information needed to answer the original question. No more tool calls needed — time to synthesize.

FINAL ANSWER

"Tesla's 2023 revenue was $96.77B, up 18.8% from $81.46B in 2022"

The agent synthesizes both search results, does the percentage calculation, and delivers a grounded, accurate answer. No hallucination — every number came from real search results.

Agent Types Comparison

🔧 Tool-Calling Agent Most Common

▼

Uses the LLM's native function/tool-calling capability. The LLM is given tool schemas and outputs structured tool calls. Modern and reliable. LangGraph's create_react_agent creates this type.

Best for: Most use cases. Fast, reliable, supported by all major LLMs (GPT-4, Claude, Gemini).

Limitation: Doesn't work well with models that lack native tool calling support.

💭 ReAct Agent Classic

▼

Uses a prompt engineering trick to make the LLM output thoughts and actions as text (e.g., Action: search[query]). The framework then parses this text to know what tool to call. Works with any LLM, even those without native tool support.

Best for: Open-source models, complex reasoning chains where you want to see the thinking.

Limitation: Text parsing can be fragile. Slower than native tool calling.

📋 Plan-and-Execute Agent Structured

▼

Two-phase approach: (1) A planner LLM creates a full plan upfront listing every step. (2) An executor carries out each step in order. More predictable than pure ReAct but less flexible — the executor can't deviate from the plan if something unexpected happens.

Best for: Well-defined tasks with clear steps (data analysis pipelines, code generation tasks).

Limitation: The planner must predict all steps in advance — fails on complex tasks with unknown unknowns.

Multi-Agent Systems

When one agent isn't enough. Multi-agent systems decompose complex tasks across specialized agents that work together — each agent is an expert at one thing.

🏢
The Company Analogy

A solo contractor (single agent) can do everything but is limited by one person's capacity. A company (multi-agent) has specialists — researchers, writers, editors, reviewers — each expert in their role. A manager (supervisor) delegates tasks and a CEO (orchestrator) sets the direction. Same principle applies to multi-agent AI systems.

Three Architectural Patterns — Click to Explore

👔 Supervisor Pattern
One supervisor agent receives the task and delegates to specialized worker agents. The supervisor synthesizes results. Clear hierarchy, easy to debug.
Best for: Tasks with clear subtasks that can be assigned
Limitation: Supervisor is a bottleneck
🐝 Swarm Pattern
No central controller. Agents collaborate peer-to-peer, passing control to each other based on their expertise. Highly parallel and resilient.
Best for: Autonomous systems, exploratory tasks
Limitation: Hard to debug, non-deterministic
🌲 Hierarchical Pattern
Multiple levels: an orchestrator manages supervisors, who each manage worker agents. Scales to very complex tasks. LangGraph subgraphs make this elegant.
Best for: Enterprise workflows, complex research
Limitation: Complex to build and maintain

Agent Memory Systems

🗒️ Scratchpad (In-context)

The agent's working memory — the conversation history and tool results within the current context window. Temporary, cleared after each run. This is what ReAct uses.

📖 Episodic Memory

Records of past experiences and interactions. "Last time I tried X, it failed because Y." Stored externally (vector DB or SQL), retrieved at start of new tasks.

🧠 Semantic Memory

Factual knowledge about the domain — like a knowledge base the agent can query. "User's timezone is EST." "Company policy prohibits X." Stored in a vector store.

⚙️ Procedural Memory

Knowledge about HOW to do things — stored as system prompts or few-shot examples. "Here's how to format API requests." Updated rarely, changes agent behavior fundamentally.

🔵 Advanced · Section 11

Advanced RAG Patterns

Naive RAG is just the beginning. As applications scale, you need smarter retrieval, better ranking, and eventually agents that decide how to retrieve.

📄 Level 1: Naive RAG Baseline

▼

What it is: The basic RAG you just learned. Load → Split → Embed → Store → Retrieve top-k → Generate. Simple and works for many cases.

Query
→
Embed
→
Top-K Search
→
LLM Answer

Limitations: Fixed chunking misses context. Embedding similarity ≠ relevance. No query rewriting. Can retrieve irrelevant chunks confidently.

🚀 Level 2: Advanced RAG Production Standard

▼

What it adds: Multiple techniques to improve retrieval quality:

  • Query Rewriting — LLM rewrites the user query to be more specific before searching
  • HyDE (Hypothetical Document Embeddings) — Generate a hypothetical answer first, embed that, search with it (counterintuitively works great)
  • Multi-Query Retrieval — Generate 3-5 variations of the query, retrieve for each, merge results
  • Re-ranking — After initial retrieval, use a cross-encoder model to re-rank by true relevance
  • Hybrid Search — Combine dense (embedding) search with sparse (BM25 keyword) search for better coverage
  • Parent-Child Chunking — Retrieve small chunks but return surrounding parent context for coherence
Python — Multi-Query Retriever
from langchain.retrievers.multi_query import MultiQueryRetriever

# This retriever uses an LLM to generate multiple query variations
multi_retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(),
    llm=ChatOpenAI(temperature=0)
)
# For "tell me about Python" it generates:
# 1. "Python programming language features"
# 2. "Python overview and capabilities"
# 3. "What is Python used for?"
# Retrieves for all 3, deduplicates, returns merged results

🧩 Level 3: Modular RAG Flexible Architecture

▼

What it is: RAG as a collection of swappable, independently deployable modules. You don't have a fixed pipeline — you have a toolkit of modules and a routing layer that assembles the right pipeline per query.

  • Query Router — Routes queries to different retrievers based on query type
  • Retrieval Modules — Dense, sparse, graph, SQL — pick the right one
  • Reranking Module — Optional post-retrieval ranking
  • Fusion Module — Merge results from multiple sources (e.g., RAG Fusion)
  • Generation Module — Can choose different LLMs for different tasks

Why it matters: Easier to A/B test modules, easier to maintain, easier to upgrade one component without breaking others.

🤖 Level 4: Agentic RAG State of the Art

▼

The big idea: Instead of always retrieving, an agent first decides whether it needs to retrieve at all. If yes, it decides how to retrieve. It can retrieve multiple times, from different sources, and refine based on what it finds.

Adaptive RAG

Route queries by complexity: simple factual questions → direct LLM answer (no retrieval). Complex questions → RAG. Very complex → multi-step agent.

Self-RAG

The LLM critiques its own retrieved documents and generated answer. If answer is unsupported or low quality, it retrieves again and regenerates.

Corrective RAG (CRAG)

After retrieval, a grader LLM evaluates chunk relevance. If chunks are irrelevant, it falls back to web search. Only uses good sources.

RAG with Planning

An agent creates a research plan, then retrieves from multiple sources in parallel, synthesizes results, and self-verifies before responding.

When you have multiple items to process, don't do them one by one. LangChain/LangGraph provide patterns for parallel execution that dramatically speed up batch operations.

    "summary": summarize_chain,      # These 3 run at

map_chain = RunnableLambda(lambda docs:
    summarize_chain.batch(docs, config={"max_concurrency": 5})
)
# STEP 2: Reduce — combine summaries into final answer
reduce_chain = combine_summaries_chain

Python — LangGraph Map-Reduce (fan-out)
from langgraph.constants import Send

def fan_out_node(state) -> list[Send]:
    """Spawn a parallel subgraph for each item"""
    return [
        Send("process_item", {"item": item})
        for item in state["items"]
    ]  # All process_item nodes run in PARALLEL

def collect_results(state) -> dict:
    """Gather all parallel results"""
    return {"final": aggregate(state["results"])}

Cost Optimization Strategies

LLM API calls are expensive at scale. Here are proven strategies to reduce costs without sacrificing quality.

💾

Caching

Cache LLM responses for identical inputs. LangChain has built-in InMemoryCache and SQLiteCache. Especially powerful for embedding calls — embeddings for the same text should never be computed twice.

Python
from langchain.cache import SQLiteCache
import langchain
langchain.llm_cache = SQLiteCache(".langchain.db")
🎯

Model Routing

Use cheap models (GPT-3.5, Haiku) for simple tasks and expensive models (GPT-4o, Sonnet) only when needed. A routing classifier can decide which model tier to use based on query complexity.

✂️

Prompt Compression

Long prompts cost more. Techniques: use concise system prompts, compress conversation history with summarization, filter retrieved chunks before inserting into context, use tiktoken to count tokens before sending.

📦

Batching

Use .batch() with max_concurrency to process multiple items in parallel without exceeding rate limits. Much faster and more efficient than sequential calls in a loop.

Example 1: Chatbot with Memory

A full conversational chatbot using LCEL + RunnableWithMessageHistory. The bot remembers the entire conversation history per session.

Python — chatbot_with_memory.py
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.output_parsers import StrOutputParser
from langchain_community.chat_message_histories import ChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory

# ── 1. DEFINE THE CHAIN ────────────────────────────
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful AI assistant. Be concise and friendly."),
    MessagesPlaceholder(variable_name="chat_history"),
    ("human", "{input}")
])
model = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)
chain = prompt | model | StrOutputParser()

# ── 2. ADD MEMORY ──────────────────────────────────
store = {}  # In production, use Redis or a database

def get_session_history(session_id: str) -> ChatMessageHistory:
    if session_id not in store:
        store[session_id] = ChatMessageHistory()
    return store[session_id]

chatbot = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="input",
    history_messages_key="chat_history"
)

# ── 3. USE IT ───────────────────────────────────────
session_config = {"configurable": {"session_id": "alice-session-1"}}

def chat(message: str) -> str:
    return chatbot.invoke({"input": message}, config=session_config)

# Conversation that remembers context
print(chat("Hi! My name is Alice and I love Python."))
print(chat("What's a great Python project for beginners?"))
print(chat("And what's my name again?"))  # It remembers: Alice!

Example 2: Complete RAG Pipeline

Load a PDF, chunk it, embed it, store in FAISS, then query it. A production-ready RAG pipeline with source citations.

Python — rag_pipeline.py
from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain.chains import create_retrieval_chain
from langchain.chains.combine_documents import create_stuff_documents_chain
from langchain_core.prompts import ChatPromptTemplate

# ── 1. INGEST ───────────────────────────────────────
loader = PyPDFLoader("your_document.pdf")
docs = loader.load()

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800, chunk_overlap=150
)
chunks = splitter.split_documents(docs)
print(f"Created {len(chunks)} chunks")

# ── 2. EMBED & STORE ────────────────────────────────
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.from_documents(chunks, embeddings)
vectorstore.save_local("faiss_index")  # Persist to disk!

# To load later: vectorstore = FAISS.load_local(...)

# ── 3. BUILD RAG CHAIN ──────────────────────────────
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})
llm = ChatOpenAI(model="gpt-4o", temperature=0)

system_prompt = ("""You are a helpful assistant answering questions about documents.
Use ONLY the provided context. If unsure, say you don't know.
Always cite the page numbers from the source documents.

Context: {context}""")

prompt = ChatPromptTemplate.from_messages([
    ("system", system_prompt),
    ("human", "{input}")
])

qa_chain = create_stuff_documents_chain(llm, prompt)
rag_chain = create_retrieval_chain(retriever, qa_chain)

# ── 4. QUERY ────────────────────────────────────────
result = rag_chain.invoke({"input": "What is the main conclusion?"})
print("Answer:", result["answer"])
print("Sources:", [d.metadata["page"] for d in result["context"]])

Example 3: ReAct Agent with Tools

A full LangGraph ReAct agent with multiple tools and built-in retry logic. The agent can search the web, do math, and check the current date.

from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langgraph.prebuilt import create_react_agent
from langgraph.checkpoint.memory import MemorySaver
from datetime import datetime
import math

# ── 1. DEFINE TOOLS ─────────────────────────────────
@tool
def search_web(query: str) -> str:
    """Search the internet for current information."""
    # Replace with real search (e.g., Tavily, SerpAPI)
    return f"[Search results for: {query}]"

@tool
def calculate(expression: str) -> str:
    """Evaluate a mathematical expression. Input: Python math expression."""
    try:
        result = eval(expression, {"__builtins__": {}}, vars(math))
        return str(result)
    except Exception as e:
        return f"Error: {e}"

@tool
def get_current_date() -> str:
    """Get the current date and time."""
    return datetime.now().strftime("%Y-%m-%d %H:%M:%S")

)

config = {"configurable": {"thread_id": "research-1"}}

Example 4: Multi-Agent Research System

A supervisor pattern with a Researcher agent and a Writer agent. The supervisor delegates and synthesizes. Built with LangGraph subgraphs.

Python — multi_agent_research.py
from typing import TypedDict, Annotated, Literal
from langgraph.graph import StateGraph, START, END, add_messages
from langgraph.prebuilt import create_react_agent
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, BaseMessage

llm = ChatOpenAI(model="gpt-4o")

# ── SPECIALIZED AGENTS ──────────────────────────────
researcher = create_react_agent(llm, [search_web], 
    state_modifier="You are a research specialist. Gather facts and data.")

writer = create_react_agent(llm, [],
    state_modifier="You are a writing specialist. Create clear, engaging content.")

# ── SUPERVISOR SETUP ────────────────────────────────
class SupervisorState(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]
    next_agent: str

supervisor_prompt = """You are a supervisor. Given the conversation, decide who to call next.
Options: researcher, writer, FINISH
Respond with ONLY one word."""

def supervisor_node(state: SupervisorState):
    response = llm.invoke([
        ("system", supervisor_prompt),
        *state["messages"]
    ])
    next_agent = response.content.strip()
    return {"next_agent": next_agent, "messages": [response]}

def route_to_agent(state) -> Literal["researcher", "writer", "__end__"]:
    n = state["next_agent"]
    if n == "FINISH": return "__end__"
    return n  # "researcher" or "writer"

# ── BUILD THE GRAPH ─────────────────────────────────
builder = StateGraph(SupervisorState)
builder.add_node("supervisor", supervisor_node)
builder.add_node("researcher", researcher)   # subgraph as node!
builder.add_node("writer", writer)            # subgraph as node!

builder.add_edge(START, "supervisor")
builder.add_conditional_edges("supervisor", route_to_agent)
builder.add_edge("researcher", "supervisor")  # always back to supervisor
builder.add_edge("writer", "supervisor")

graph = builder.compile()

# ── RUN ─────────────────────────────────────────────
result = graph.invoke({
    "messages": [HumanMessage(content="Research and write a blog post on LangGraph")]
})

Example 5: Human-in-the-Loop Approval Workflow

An agent that pauses before taking a dangerous action (sending an email) and waits for human approval. Uses LangGraph's interrupt mechanism.

Python — hitl_workflow.py
from langgraph.graph import StateGraph, START, END, interrupt
from langgraph.checkpoint.memory import MemorySaver
from langgraph.types import Command
from typing import TypedDict

class EmailState(TypedDict):
    task: str
    draft_email: str
    approved: bool
    status: str

# ── NODES ───────────────────────────────────────────
def draft_email_node(state: EmailState) -> EmailState:
    """Generate email draft (no approval needed yet)"""
    draft = llm.invoke(f"Write a professional email about: {state['task']}")
    return {"draft_email": draft.content, "status": "draft_ready"}

def approval_node(state: EmailState) -> EmailState:
    """PAUSE here and wait for human review"""
    decision = interrupt({
        "message": "Please review this email draft before sending:",
        "draft": state["draft_email"],
        "options": ["approve", "reject"]
    })  # ← EXECUTION STOPS HERE until human responds
    
    approved = decision.get("choice") == "approve"
    return {"approved": approved, "status": "reviewed"}

def send_email_node(state: EmailState) -> EmailState:
    if state["approved"]:
        send_email(state["draft_email"])  # Actually send!
        return {"status": "sent"}
    else:
        return {"status": "cancelled"}

# ── BUILD + COMPILE ─────────────────────────────────
builder = StateGraph(EmailState)
builder.add_node("draft", draft_email_node)
builder.add_node("approve", approval_node)
builder.add_node("send", send_email_node)
builder.add_edge(START, "draft")
builder.add_edge("draft", "approve")
builder.add_edge("approve", "send")

checkpointer = MemorySaver()
graph = builder.compile(checkpointer=checkpointer)
config = {"configurable": {"thread_id": "email-task-1"}}

# Start the workflow
graph.invoke({"task": "project deadline extension"}, config)
# → runs draft_node, then PAUSES at approval_node

# ... human reviews the draft ...
# Resume with decision (can be seconds or hours later)
graph.invoke(Command(resume={"choice": "approve"}), config)
# → resumes from approval_node, continues to send_node

Interview Cheat Sheet

25 must-know concepts, definitions, and gotchas for LangChain/LangGraph interviews. Search, filter by topic.

🔍

LCEL (LangChain Expression Language)

LC

A declarative syntax using the pipe operator (|) to compose Runnables. Every component is a Runnable with .invoke()/.stream()/.batch(). Enables streaming, parallelism, and async out of the box.

chain = prompt | model | parser

Runnable Interface

LC

The common interface every LangChain component implements. Key methods: invoke(input), stream(input), batch([inputs]), ainvoke() (async). All composable via |.

model.invoke() / .stream() / .batch()

RunnableWithMessageHistory

LC

The modern way to add memory to any LCEL chain. Wraps a chain and automatically injects/saves conversation history per session_id. Works with any chat history backend (Redis, SQL, in-memory).

RunnableWithMessageHistory(chain, get_history)

RecursiveCharacterTextSplitter

RAG

The most common text splitter. Splits on a hierarchy of separators (\n\n → \n → . → space) trying to keep semantic units together. Key params: chunk_size and chunk_overlap.

RecursiveCharacterTextSplitter(chunk_size=1000)

Vector Store vs Retriever

RAG

A vector store is the database that stores embeddings. A retriever is an interface that takes a query and returns relevant documents. You convert a vector store to a retriever with .as_retriever().

retriever = vectorstore.as_retriever(search_kwargs={"k": 4})

MMR (Maximal Marginal Relevance)

RAG

A retrieval strategy that balances relevance AND diversity. Avoids returning 4 nearly identical chunks. Use search_type="mmr". Better than pure similarity when chunks are repetitive.

retriever = vs.as_retriever(search_type="mmr")

StateGraph vs MessageGraph

LG

StateGraph: custom TypedDict state — any fields you want. Most flexible, use for most cases. MessageGraph: state is automatically just a list of messages. Simpler but less flexible. Prefer StateGraph.

StateGraph(MyStateTypedDict)

State Reducers (add_messages)

LG

Reducers define how state fields are updated. Default: overwrite. add_messages reducer appends to the list instead of overwriting. Critical for message history fields. Use Annotated[list, add_messages].

messages: Annotated[list, add_messages]

Checkpointer

LG

Saves the full graph state after every node. Required for: human-in-the-loop, multi-turn conversations, long-running workflows, resuming after failures. MemorySaver for dev, SqliteSaver or Postgres for production.

graph = builder.compile(checkpointer=MemorySaver())

interrupt() function

LG

Calling interrupt(value) inside a node PAUSES graph execution and returns the value to the caller. Execution resumes when Command(resume=...) is passed. Requires a checkpointer.

result = interrupt({"question": "Approve?"})

Conditional Edges

LG

Edges where the next node is decided at runtime based on the current state. The routing function receives state and returns a node name (or END). This is what enables branching logic in graphs.

builder.add_conditional_edges("node", router_fn)

Send() — Fan-Out Pattern

LG

Return a list of Send(node_name, state) from a node to spawn multiple parallel subgraph executions. Results are collected and merged when all finish. The LangGraph map-reduce primitive.

return [Send("worker", {"item": x}) for x in items]

ReAct Pattern

Agents

Reason + Act. The agent alternates between thinking (Thought) and doing (Action) in a loop. Each Action produces an Observation that informs the next Thought. Continues until the agent decides it has the final answer.

create_react_agent(llm, tools)

@tool Decorator

Agents

Turns a Python function into a LangChain Tool. The function's docstring becomes the tool description the LLM uses to decide when to call it — write great docstrings! The type hints define the schema.

@tool\ndef my_tool(arg: str) -> str: ...

bind_tools()

Agents

Attaches tool schemas to a model, enabling tool calling. The model then knows what tools are available and can output structured tool calls. Required step before running an agent.

model_with_tools = model.bind_tools(tools)

Supervisor Pattern

Multi-Agent

A central supervisor agent receives the task, decides which specialist agent to call, receives results, and loops until done. The supervisor uses an LLM to make routing decisions. Clear, debuggable hierarchy.

add_conditional_edges("supervisor", route_to_agent)

Swarm Pattern

Multi-Agent

Agents pass control to each other peer-to-peer using Command(goto="agent_name"). No central supervisor. Agents self-organize. Highly parallel. Harder to debug. LangGraph supports this natively.

return Command(goto="other_agent", update={...})

HyDE (Hypothetical Document Embeddings)

RAG

Generate a hypothetical perfect answer to the query, embed THAT, and use it to search the vector store. Counterintuitively, a fake perfect answer embeds closer to real relevant docs than the raw question.

HypotheticalDocumentEmbedder

Corrective RAG (CRAG)

RAG

After retrieval, an LLM grader evaluates each chunk's relevance. If all chunks are irrelevant → fall back to web search. If some are relevant → use those. Never trust retrieval blindly.

grade = llm.invoke(f"Is this relevant? {chunk}")

LangSmith Tracing

Observability

Set LANGCHAIN_TRACING_V2=true and every LLM/chain/agent call is automatically logged with full inputs, outputs, latency, token counts, and cost. Zero code changes required.

export LANGCHAIN_TRACING_V2=true

LLM Caching

Production

Cache identical LLM requests to avoid redundant API calls. InMemoryCache for dev, SQLiteCache for local persistence. Especially valuable for embeddings — never embed the same text twice.

langchain.llm_cache = SQLiteCache()

RunnableParallel

LC

Runs multiple Runnables in parallel on the same input, merging results into a dict. Use when you need multiple independent computations from the same input (e.g., retrieve + embed at the same time).

RunnableParallel({"a": chain1, "b": chain2})

Thread ID

LG

A string that identifies a specific conversation/workflow run. The checkpointer uses it to save and load state. Different thread IDs = separate sessions. Essential for multi-user applications.

{"configurable": {"thread_id": "user-123"}}

max_iterations Guard

Agents

Always set a maximum iteration limit on agents to prevent infinite loops and runaway costs. In LangGraph, implement in the conditional edge: if steps > max_steps → return END. In AgentExecutor: max_iterations=10.

if state["steps"] > 10: return END

When to use LangChain vs LangGraph

Architecture

LangChain LCEL: Linear pipelines, single-pass RAG, simple chatbots. LangGraph: Anything with loops, branching, human approval, multi-agent coordination, long-running stateful workflows. When in doubt, start with LCEL and migrate to LangGraph when you need loops.

Question 1 of 10 · LangChain Basics
What does the pipe operator (|) do in LCEL (LangChain Expression Language)?
Correct! The | pipe operator is borrowed from Unix. In LCEL, prompt | model | parser means: run the prompt, pass the result to the model, then pass the model's output to the parser. Data flows left-to-right, and each component must accept the output type of the previous one.
Question 2 of 10 · Memory
You have a customer support chatbot with very long conversations (100+ messages). Which memory type is BEST suited to avoid context window overflow while keeping key information?
Correct! ConversationBufferMemory stores EVERY message and will eventually overflow the context window in long conversations. ConversationSummaryMemory uses an LLM to write a running summary of the conversation, which stays compact regardless of conversation length — perfect for 100+ message sessions.
Question 3 of 10 · RAG
Why is chunk_overlap important when splitting documents for RAG?
Correct! Without overlap, a sentence at the end of chunk 1 and the beginning of chunk 2 gets split in half — neither chunk has the full context. Overlap (typically 10-20% of chunk_size) copies some text from the end of each chunk to the beginning of the next, ensuring no sentence is cut off without context on either side.
Question 4 of 10 · LangGraph
In LangGraph, what is the purpose of Annotated[list[BaseMessage], add_messages] in the state definition?
Correct! In LangGraph, state fields are overwritten by default — if you return {"messages": [new_msg]}, it replaces ALL previous messages. The add_messages reducer changes this behavior: it APPENDS the new messages to the existing list. Without this, every node would erase conversation history!
Question 5 of 10 · LangGraph
What is required in LangGraph to implement human-in-the-loop (interrupt) patterns?
Correct! When interrupt() is called, LangGraph saves the current state to the checkpointer and pauses execution. When Command(resume=...) is called later (potentially hours later), LangGraph loads the saved state and continues from where it stopped. Without a checkpointer, there's nowhere to save the state, so interrupt() cannot work.
Question 6 of 10 · Agents
In the ReAct pattern, the agent produces a "Thought" before each action. What is the PRIMARY purpose of the Thought step?
Correct! The Thought step is the "Chain of Thought" part of ReAct. By thinking step-by-step before acting, the LLM produces significantly better decisions. It's like asking someone to explain their reasoning before acting — it forces more deliberate, considered choices. Research shows this dramatically improves agent accuracy compared to just outputting actions directly.
Question 7 of 10 · Multi-Agent
Which multi-agent pattern has NO central supervisor and allows agents to hand off control to each other peer-to-peer?
Correct! The Swarm pattern has no central controller. Agents use Command(goto="other_agent") to pass control to whichever agent is best suited for the current task. Like a real swarm, coordination emerges from local decisions rather than central direction. This makes it highly parallel and resilient, but harder to debug than the Supervisor pattern.
Question 8 of 10 · Advanced RAG
What is HyDE (Hypothetical Document Embeddings) and why does it work?
Correct! This is counterintuitive but true: a question like "What is backpropagation?" has a very different embedding than an answer about backpropagation. HyDE generates a fake ideal answer first, then uses THAT answer's embedding to search the vector store. The fake answer's embedding is much closer to real relevant documents than the raw question's embedding was.
Question 9 of 10 · Production
Your LangGraph agent is entering an infinite loop — it keeps calling the same tool repeatedly. What is the BEST architectural fix?
Correct! The standard defense against infinite agent loops is a step counter in the state. Each node increments it. A conditional edge checks: if steps > MAX_STEPS → return END (with a "max iterations reached" message). This pattern prevents runaway costs and infinite loops while being easy to implement. The max limit should be tuned based on your task complexity.
Question 10 of 10 · Architecture
When should you choose LangGraph over a simple LCEL chain?
Correct! LCEL is perfect for linear pipelines — it's simpler, faster to write, and easier to understand. Only reach for LangGraph when you genuinely need: (1) loops/cycles — the agent can revisit nodes, (2) conditional branching based on runtime state, (3) human-in-the-loop approval, or (4) coordinating multiple agents. Adding LangGraph complexity unnecessarily makes simple things harder.
