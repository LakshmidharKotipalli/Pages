Section 01

APIs — Application Programming Interfaces

The universal language that lets software systems talk to each other. Every app you use depends on APIs under the hood.

What is an API?

🍽️
Think of a restaurant. You (the client) sit at a table. You don't walk into the kitchen — instead, a waiter (the API) takes your order, carries it to the kitchen (the server), and brings back your food (the response). You never need to know how the kitchen works. That waiter is the API — a defined, standardized way to request something and get a result.

An API (Application Programming Interface) is a set of rules and contracts that define how two software components communicate. It specifies what requests you can make, what format to use, and what responses to expect.

📋
Contract
An API is a contract. It promises: "Send me X in this format, and I'll return Y in that format." The implementation behind it can change — as long as the contract holds.
🔌
Abstraction
APIs hide complexity. You call stripe.charge() without knowing Stripe's internal database structure. The API is the public face of a private system.
🌐
Web APIs
Most modern APIs are Web APIs — they work over HTTP/HTTPS. Your mobile app talks to a backend server via a Web API. The server talks to other services via APIs too.

REST vs GraphQL vs gRPC

🍔
REST is a fixed menu — you order "the burger" and get exactly what they serve. GraphQL is "build your own burger" — you pick exactly which toppings (fields) you want. gRPC is a walkie-talkie — fast, compact, binary signals between services, not meant for browsers.

REST — Representational State Transfer

REST is not a protocol — it's an architectural style with 6 key constraints. When an API follows these constraints, it's called "RESTful."

6 REST Constraints
  • 1. Client-Server — UI and data are separate
  • 2. Stateless — each request is self-contained
  • 3. Cacheable — responses can be cached
  • 4. Uniform Interface — consistent URLs and methods
  • 5. Layered System — can add proxies, CDNs
  • 6. Code on Demand — optional: server can send code
REST Resource Design
GET /users — list all users
GET /users/42 — get user #42
POST /users — create a user
PUT /users/42 — replace user #42
PATCH /users/42 — update part of #42
DELETE /users/42 — delete user #42
✅ Best for:
Public APIs, CRUD apps, browser clients, when you need caching and simplicity. REST is the most widely understood API style in the industry.

GraphQL — Query Language for APIs

Created by Facebook in 2012. Instead of many fixed endpoints, GraphQL gives you one endpoint where clients ask for exactly what they need — no more, no less.

REST — Over-fetching problem
REST Response
/* You just wanted user's name, but you got ALL this: */
{
  "id": 42, "name": "Alice", "email": "alice@mail.com",
  "age": 28, "address": "...", "phone": "...",
  "createdAt": "...", "lastLogin": "...",  // unnecessary!
  "preferences": {...}  // also unnecessary!
}
GraphQL — Ask for exactly what you need
GraphQL Query
query {
  user(id: 42) {
    name         # only this!
    posts {
      title        # and this!
    }
  }
}
Key GraphQL Concepts:
Schema — typed contract for all data; Query — fetch data (like GET); Mutation — change data (like POST/PUT/DELETE); Subscription — real-time updates over WebSocket; Resolver — function that fetches data for a field.
✅ Best for:
Mobile apps (save bandwidth), complex UIs with many data shapes, rapid frontend development, when multiple clients need different data views from the same API.

gRPC — Google Remote Procedure Call

Created by Google. Uses Protocol Buffers (binary format, not JSON) over HTTP/2. Up to 10x faster than REST for internal service-to-service communication.

Define service in .proto file
user.proto
syntax = "proto3";

message UserRequest {
  int32 id = 1;
}

message UserResponse {
  int32  id   = 1;
  string name = 2;
}

service UserService {
  rpc GetUser(UserRequest) returns (UserResponse);
}
Why gRPC is fast
  • ⚡ Binary Protocol Buffers — 3-10x smaller than JSON
  • 🔀 HTTP/2 multiplexing — many requests on one connection
  • 📡 Streaming — server/client/bidirectional stream support
  • 🔐 Strongly typed — contract enforced at compile time
  • 🌍 Polyglot — generate client code in 10+ languages
⚠️ Limitation:
Not natively supported by browsers (needs gRPC-Web proxy). Best for backend microservice communication.

HTTP Methods

HTTP methods tell the server what action you want to perform on a resource. Click any row to expand details.

GET
Retrieve Data
Fetch a resource — read-only, no side effects
Idempotent:Yes — calling it 100 times returns same result
Has body:No (not recommended)
Safe:Yes — doesn't modify server state
Cache:Yes — browsers cache GET responses
Example:GET /api/products/42
▼
POST
Create Data
Submit data to create a new resource
Idempotent:No — calling twice creates two resources!
Has body:Yes — contains the resource data
Safe:No — modifies server state
Response:201 Created with Location header
Example:POST /api/orders {"product":42,"qty":2}
▼
PUT
Replace Data
Replace an entire resource with new data
Idempotent:Yes — same result every time
Has body:Yes — full resource representation
Difference from PATCH:PUT replaces ALL fields; PATCH only updates specified fields
Example:PUT /api/users/42 {"name":"Alice","age":28}
▼
PATCH
Partial Update
Update specific fields of an existing resource
Idempotent:Usually yes, but not guaranteed by spec
Has body:Yes — only the fields to update
Use case:Update a user's email without touching other fields
Example:PATCH /api/users/42 {"email":"new@mail.com"}
▼
DELETE
Remove Data
Delete a resource from the server
Idempotent:Yes — deleting something twice is same as deleting once
Has body:Rarely (technically allowed)
Response:204 No Content or 200 OK
Example:DELETE /api/posts/17

HTTP Status Codes

Status codes are 3-digit numbers that tell you what happened. Hover each one for details.

● 2xx — Success
200
200 OK — Request succeeded. Standard success response.
201
201 Created — Resource successfully created (after POST).
204
204 No Content — Success but no body (after DELETE).
206
206 Partial Content — Partial range response (video streaming).
● 3xx — Redirection
301
301 Moved Permanently — URL changed forever. Update bookmarks.
302
302 Found — Temporary redirect. Original URL may come back.
304
304 Not Modified — Cached version is still valid. Don't re-download.
● 4xx — Client Errors
400
400 Bad Request — Your request is malformed or invalid.
401
401 Unauthorized — Not authenticated. Log in first.
403
403 Forbidden — Authenticated but no permission for this.
404

405

409

422

429

● 5xx — Server Errors
500

502

503

504
504 Gateway Timeout — Upstream server didn't respond in time.

API Authentication

API Keys — Simple Token Authentication

🏷️
An API key is like a library card. You show it at the entrance, and the library knows who you are. Anyone who gets your card can use it — so keep it secret!
API Key Usage
# In URL (less secure — shows in logs!)
GET /weather?api_key=abc123xyz

# In Header (preferred)
Authorization: Api-Key abc123xyz
# or custom header
X-API-Key: abc123xyz
✅ Pros
Simple to implement, good for server-to-server, easy rate limiting per key
❌ Cons
No expiry by default, no user identity, if leaked must be rotated immediately

OAuth 2.0 — Delegated Authorization

🔑
OAuth is like a hotel key card. Instead of giving a guest your master key, the hotel issues a limited-access card. The guest can open their room but not the safe. OAuth lets users authorize apps to act on their behalf without sharing passwords.
OAuth 2.0 Grant Types:
Authorization Code — web apps (most secure, uses PKCE); Client Credentials — machine-to-machine (no user); Implicit — deprecated, was used for SPAs; Password — deprecated, user gives password directly.

JWT — JSON Web Tokens

🎫
JWT is a tamper-proof visitor badge. The badge contains your name, clearance level, and expiry. Security doesn't need to call HQ to check — they read the badge. But the badge has a security seal: if anyone modifies it, the seal breaks. That seal is the cryptographic signature.

Click each part of the JWT to learn what it contains:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiI0MiIsIm5hbWUiOiJBbGljZSIsInJvbGUiOiJhZG1pbiIsImV4cCI6MTcwMDAwMDAwMH0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
Header — Algorithm and token type.
Decoded: {"alg": "HS256", "typ": "JWT"}
HS256 = HMAC-SHA256 (symmetric). RS256 = RSA (asymmetric, preferred for distributed systems).
Payload (Claims) — The actual data.
Decoded: {"sub": "42", "name": "Alice", "role": "admin", "exp": 1700000000}
Standard claims: sub (subject/user ID), exp (expiry), iat (issued at), iss (issuer). Add custom claims too.
Signature — Tamper detection seal.
Created by: HMAC-SHA256(base64(header) + "." + base64(payload), secret_key)
⚠️ Anyone can READ a JWT (it's just Base64) — never put sensitive data in the payload! The signature only ensures it hasn't been MODIFIED.
Section 02

FastAPI — Modern Python Web Framework

The fastest Python framework for building APIs — with automatic validation, documentation, and async support out of the box.

What is FastAPI and Why Use It?

⚡
FastAPI is like a sports car with automatic GPS and airbags built in. Other frameworks (like Flask) are like a basic car — you add GPS and safety features yourself. FastAPI gives you high speed (async), navigation (auto-docs), and safety (type validation) out of the box.
🚀
High Performance
Based on Starlette (ASGI) and Pydantic. Comparable speed to Node.js and Go. One of the fastest Python frameworks available.
🤖
Auto Docs
Write code, get interactive Swagger UI and ReDoc automatically. No manual documentation writing — the types you define become the docs.
✅
Type Safety
Uses Python type hints + Pydantic v2 for automatic request validation. Wrong data type? FastAPI returns a 422 error with a clear message before your code even runs.

Relative Performance (requests/sec, higher is better)

FastAPI
~58,000 req/s
Flask
~13,000
Django
~8,000

Note: Benchmarks vary by use case. ASGI + async gives FastAPI its performance edge.

Creating Your First FastAPI App + Routes, Path & Query Params

main.py — Complete FastAPI Example
from fastapi import FastAPI, HTTPException, Query
from typing import Optional

app = FastAPI(
    title="My API",
    description="A great API",
    version="1.0.0"
)

# ─── Path Parameter ───────────────────────────────────────────
# {item_id} in the URL is captured as a typed Python argument
@app.get("/items/{item_id}")          # route decorator
async def get_item(item_id: int):      # type hint = validation
    if item_id > 1000:
        raise HTTPException(status_code=404, detail="Item not found")
    return {"item_id": item_id, "name": "Widget"}

# ─── Query Parameters ─────────────────────────────────────────
# /users?skip=0&limit=10&search=alice
@app.get("/users")
async def list_users(
    skip: int = 0,                    # default = 0
    limit: int = Query(10, le=100),    # max=100, validation!
    search: Optional[str] = None       # optional param
):
    users = get_users_from_db(skip, limit, search)
    return {"users": users, "total": len(users)}
Path Parameters
Part of the URL path itself. Defined with {curly_braces} in the route and as function arguments with type hints. FastAPI validates the type automatically — if you pass a string where an int is expected, it returns 422.
GET /items/42 → item_id = 42
GET /items/abc → 422 Error!
Query Parameters
Added after ? in the URL. Optional or required, with defaults. Use Query() for extra validation like min/max length, regex patterns, etc.
/users?skip=20&limit=5&search="alice"

Request Body with Pydantic Models

Pydantic models define the shape and validation rules for data. FastAPI automatically deserializes, validates, and documents them.

📋
Think of Pydantic as a smart form with built-in validation. The form knows an email must have an "@", age must be a positive number, and the name can't be empty — and it tells you exactly what's wrong if you fill it incorrectly.
Pydantic Models + Request Body
from fastapi import FastAPI
from pydantic import BaseModel, EmailStr, Field, field_validator
from typing import Optional
from datetime import datetime

class UserCreate(BaseModel):
    name: str = Field(..., min_length=2, max_length=50)   # required, validated
    email: EmailStr                                        # auto email validation
    age: int = Field(..., ge=0, le=150)                    # 0 ≤ age ≤ 150
    bio: Optional[str] = None                               # optional

    @field_validator('name')
    @classmethod
    def name_must_not_contain_numbers(cls, v: str) -> str:
        if any(c.isdigit() for c in v):
            raise ValueError('Name must not contain numbers')
        return v.strip()

class UserResponse(BaseModel):              # separate response model
    id: int
    name: str
    email: EmailStr
    created_at: datetime

app = FastAPI()

@app.post("/users", response_model=UserResponse, status_code=201)
async def create_user(user: UserCreate):    # body auto-parsed & validated
    # user.name, user.email, user.age are all guaranteed correct types
    db_user = save_to_db(user)
    return db_user                            # auto-serialized to JSON
Pro tip — Separate Input/Output Models:
Use different Pydantic models for input (UserCreate) and output (UserResponse). This lets you accept certain fields when creating but exclude sensitive ones (like passwords) from responses. FastAPI's response_model parameter handles the filtering automatically.

Dependency Injection & Middleware

Dependency Injection

FastAPI's Depends() lets you declare reusable logic (DB connections, auth checks, settings) that gets injected into route handlers automatically. Think of it as plugging in a component.

Dependencies
from fastapi import Depends, HTTPException, Header

# Reusable dependency
async def get_current_user(
    authorization: str = Header(...)
):
    token = authorization.replace("Bearer ", "")
    user = verify_jwt_token(token)
    if not user:
        raise HTTPException(401)
    return user

# DB session dependency
def get_db():
    db = SessionLocal()
    try:
        yield db         # like context manager
    finally:
        db.close()       # always runs, even on error

# Both dependencies injected automatically
@app.get("/profile")
async def get_profile(
    current_user = Depends(get_current_user),
    db = Depends(get_db)
):
    return db.query(User).filter(...).first()

Middleware

Middleware runs before every request and after every response. Perfect for logging, CORS, rate limiting, and adding response headers.

Middleware Example
import time
from fastapi import Request
from fastapi.middleware.cors import CORSMiddleware

# CORS middleware (allow browser requests)
app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://myapp.com"],
    allow_methods=["*"],
    allow_headers=["*"],
)

# Custom timing middleware
@app.middleware("http")
async def add_process_time_header(
    request: Request, call_next
):
    start = time.time()
    response = await call_next(request)
    duration = time.time() - start
    response.headers["X-Process-Time"] = str(duration)
    return response

Auto-Generated Docs & Async Support

Swagger UI (/docs)
GET /users/{user_id} Retrieve User
POST /users Create User
DEL /users/{user_id} Delete User
FastAPI generates an interactive Swagger UI at /docs and ReDoc at /redoc. You can test endpoints directly in the browser. The schema comes from your type hints — no YAML writing needed.
Async/Await Support
FastAPI is built on ASGI (Asynchronous Server Gateway Interface). Use async def for I/O-bound operations like database queries, HTTP calls, and file operations.
Sync vs Async
# Sync — blocks other requests while waiting
@app.get("/slow")
def slow_endpoint():
    result = requests.get(external_api)  # BLOCKS!
    return result.json()

# Async — other requests served while waiting
@app.get("/fast")
async def fast_endpoint():
    result = await httpx.get(external_api)  # NON-BLOCKING
    return result.json()
Use async def with async libraries (httpx, asyncpg, motor). Don't mix async def with blocking sync libraries — it defeats the purpose!

Docker — Containerization

Package your app and all its dependencies into a portable container. Run it the same way everywhere — your laptop, CI server, production cloud.

Why Docker? The "Works on My Machine" Problem

❌ Without Docker
  • "It works on my machine!" (but not prod)
  • Dev uses Python 3.10, prod has 3.8
  • Missing environment variable in staging
  • Different OS, different behavior
  • Onboarding takes days of setup
✅ With Docker
  • Build once, run anywhere — identical environment
  • Exact Python version locked in Dockerfile
  • All env vars defined in docker-compose.yml
  • Isolated filesystem, process space, network
  • New dev? docker compose up — done in minutes
📦
Containers are like shipping containers. Before standard shipping containers, moving cargo was chaotic — different shapes, sizes, handling requirements. Containers standardized everything. Now any container fits any ship or truck. Docker does the same for software — your app is packaged with everything it needs and runs on any machine that has Docker installed.

Images vs Containers

🍪
Image = cookie cutter. Container = actual cookie. You have one cookie cutter (image) but can make many cookies from it (containers). Each cookie is independent — you can eat one without affecting the others. The cookie cutter itself never changes.
Docker Image
  • ● Read-only template / blueprint
  • ● Made of layers (each Dockerfile instruction = 1 layer)
  • ● Stored in registries (Docker Hub, ECR)
  • ● Identified by name:tag (e.g., nginx:1.25)
  • ● Built with docker build
  • ● Shared, versioned, cached
Docker Container
  • ● Running instance of an image
  • ● Has a writable layer on top of image layers
  • ● Isolated: own filesystem, network, process space
  • ● Can be started, stopped, paused, deleted
  • ● Created with docker run
  • ● Ephemeral — data lost on delete (use volumes!)

Image Layer Architecture

Writable Layer (Container layer)
↑ only this part is writable
Layer 4: COPY app/ /app/ (3 MB)
Layer 3: RUN pip install requirements (45 MB)
Layer 2: RUN apt-get update (18 MB)
Layer 1: FROM python:3.11-slim (130 MB)
↓ read-only layers (cached, shared between containers)

Dockerfile Anatomy

A Dockerfile is a recipe for building your image. Each instruction creates a new layer. Hover each line to understand what it does.

Dockerfile (Production-ready FastAPI)
1# ──── Stage 1: Builder ────


2FROM python:3.11-slim AS builderBase image — slim = minimal OS, no extras


3WORKDIR /appSet working directory inside container


4ENV PYTHONDONTWRITEBYTECODE=1 \Env vars — skip .pyc files, no buffering


5    PYTHONUNBUFFERED=1


6COPY requirements.txt .Copy requirements FIRST (cache this layer)


7RUN pip install --no-cache-dir -r requirements.txtInstall deps — cached unless requirements.txt changes


8# ──── Stage 2: Runner ────


9FROM python:3.11-slim AS runnerFresh slim image — no build tools!


10WORKDIR /app


11RUN adduser --system --no-create-home appuserSecurity: create non-root user


12COPY --from=builder /usr/local/lib /usr/local/libCopy installed packages from builder stage


13COPY . .Copy application source code last


14USER appuserRun as non-root for security


15EXPOSE 8000Document which port app listens on


16CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]Default command to run on container start
Multi-Stage Build benefit:
Stage 1 (builder) has pip, build tools, compiler etc. Stage 2 (runner) only copies the installed libraries — no pip, no build tools. Final image is much smaller and more secure. This is the industry standard for production containers.

Docker CLI Commands

Click any command card to expand its explanation.

docker build -t myapp:1.0 .
Build an image from the Dockerfile in current directory (.). -t tags it as myapp:1.0. Use --no-cache to rebuild all layers fresh.
docker run -p 8000:8000 myapp:1.0
Run a container from the image. -p host:container maps ports. Add -d for detached (background) mode. --name mycontainer gives it a name.
docker ps -a
List containers. Without -a, shows only running containers. With -a, shows all including stopped ones. Shows container ID, image, status, ports.
docker logs -f mycontainer
Stream container logs. -f = follow (like tail -f). --tail 100 to see last 100 lines. Essential for debugging.
docker exec -it mycontainer bash
Run a command inside a running container. -it = interactive terminal. Opens bash shell inside the container — great for debugging.
docker push myrepo/myapp:1.0
Push image to a registry. First do docker login. Prerequisite: tag image as username/image:tag or full registry URL for ECR/GCR.
docker pull nginx:latest
Download an image from a registry. docker run pulls automatically if image not found locally, but explicit pull is useful in CI pipelines.
docker system prune -a
Remove unused resources — stopped containers, unused images, networks. Frees disk space. -a removes all unused images, not just dangling ones. Use with care!

Docker Compose — Multi-Container Apps

🎻
Docker Compose is like an orchestra conductor. Each musician (container) plays their part, but the conductor (Compose) tells them when to start, what to play, and how to coordinate. One file defines the whole ensemble.

      - "8000:8000"

      POSTGRES_PASSWORD: pass

      - "6379:6379"

volumes:
  postgres_data: {}                     # named volume, survives container restarts

Start all services in background
docker compose down -v
Stop + remove containers and volumes
docker compose logs -f api
Follow logs of the api service

Volumes & Networking

Volumes — Persistent Data
Containers are ephemeral — data is lost when deleted. Volumes persist data outside the container lifecycle.
Named Volume: Managed by Docker
volumes:
- mydata:/var/lib/data
Bind Mount: Map host path
volumes:
- ./src:/app/src
Use named volumes for databases in production. Use bind mounts for development hot-reload.
Networking — Container Communication
Containers in the same Docker network can reach each other by service name — no IP addresses needed.
# api container connects to db:
DATABASE_URL=postgresql://
db
/mydb
# Not localhost! Service name "db"
  • bridge — default, isolated containers on same host
  • host — container uses host's network directly
  • overlay — spans multiple Docker hosts (Swarm)
  • none — no networking
Section 04

Kubernetes (K8s) — Container Orchestration

When you have hundreds of containers across many machines, you need something to manage them all. That's Kubernetes — the air traffic control for your containers.

Why Kubernetes?

✈️
Kubernetes is like an airport's air traffic control system. Docker gives you planes (containers). But running hundreds of planes without coordination is chaos — who lands where? What if a runway closes? Air traffic control (K8s) coordinates everything: where each plane goes, reroutes if a runway fails, and adds more runways when traffic spikes.
🔄
Self-Healing
Container crashed? K8s restarts it. Node died? K8s reschedules containers on healthy nodes. No manual intervention needed.
📈
Auto-Scaling
Traffic spike at 9am? K8s automatically scales from 2 to 20 replicas. Traffic drops at night? Scale back to 2. Based on CPU, memory, or custom metrics.
🚀
Rolling Deployments
Deploy new version with zero downtime. K8s gradually replaces old pods with new ones. If new version crashes, it automatically rolls back.

Cluster Architecture

Control Plane Components
  • API Server — front-end to the cluster; all kubectl commands go here
  • etcd — distributed key-value store; the "brain" storing all cluster state
  • Scheduler — decides which node to place new pods on
  • Controller Manager — runs controllers that watch state and reconcile
Worker Node Components
  • kubelet — agent on each node; ensures containers in pods are running
  • kube-proxy — maintains network rules; enables Service routing
  • Container runtime — actually runs containers (containerd, CRI-O)
  • Pods — smallest deployable unit; group of 1+ containers

Core Kubernetes Objects

Pod — Smallest Deployable Unit

🏠
A Pod is a dorm room. Multiple roommates (containers) share the same room (network namespace and storage volumes). They can talk to each other via localhost.
  • ● Usually 1 container per Pod (with optional sidecars)
  • ● Ephemeral — gets a new IP when restarted
  • ● Never manage Pods directly — use Deployments
  • ● All containers in a Pod share network + volumes
pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app-pod
  labels:
    app: my-app
spec:
  containers:
  - name: api
    image: myrepo/my-app:1.0
    ports:
    - containerPort: 8000
    resources:
      requests:
        cpu: "250m"    # 0.25 CPU cores
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "256Mi"

Deployment — Manages Pod Replicas

👔
A Deployment is like an HR manager. You tell it "I need 3 workers at all times." If a worker quits (pod dies), the HR manager hires a replacement immediately.
  • ● Maintains desired number of Pod replicas
  • ● Handles rolling updates and rollbacks
  • ● Manages a ReplicaSet (which manages Pods)
  • ● This is what you use 99% of the time
deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3              # keep 3 pods running
  selector:
    matchLabels:
      app: my-app
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # max extra pods during update
      maxUnavailable: 0     # never bring pod count below 3
  template:              # pod template
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: api
        image: myrepo/my-app:1.0
        ports:
        - containerPort: 8000

Service — Stable Network Endpoint

📞
A Service is a phone number that always reaches the right department. Even if employees change (pods restart), the number stays the same. The Service knows which pods are currently "in" and routes calls to them.
ClusterIP — internal only (default)
NodePort — expose on each node's IP
LoadBalancer — cloud LB (AWS ELB etc)
service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  type: ClusterIP     # or NodePort / LoadBalancer
  selector:
    app: my-app       # routes to pods with this label
  ports:
  - port: 80          # Service port
    targetPort: 8000  # Pod port
    protocol: TCP

# Other services reach this via DNS:
# my-app-service.default.svc.cluster.local

ConfigMap — Non-sensitive Configuration

Store configuration separate from container images. Change config without rebuilding images. Inject as environment variables or files.

ConfigMap vs Secret:
ConfigMaps = plain text (database URLs, feature flags, log levels). Secrets = base64-encoded sensitive data (passwords, API keys, TLS certs).
configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "INFO"
  MAX_CONNECTIONS: "100"
  FEATURE_FLAG: "true"
---
# In Deployment spec.containers:
envFrom:
- configMapRef:
    name: app-config    # inject all as env vars

Secrets — Sensitive Data

Store passwords, API keys, and certificates. Base64 encoded (NOT encrypted by default — enable encryption at rest in production!)

⚠️ Important:
K8s Secrets are base64-encoded, not encrypted. For production, use Sealed Secrets, AWS Secrets Manager, or HashiCorp Vault with a CSI driver.
secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  # Values must be base64 encoded
  # echo -n "mysecretpassword" | base64
  DB_PASSWORD: bXlzZWNyZXRwYXNzd29yZA==
  API_KEY: c3VwZXJzZWNyZXRrZXk=
---
# In container spec:
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-secret
      key: DB_PASSWORD

kubectl — The Kubernetes CLI

kubectl apply -f deployment.yaml
Create or update resources from a YAML file. The go-to command for deploying anything. Use -R -f ./k8s/ to apply a whole directory.
kubectl get pods -n production
List pods in the production namespace. Add -w to watch in real time. Add -o wide to see node placement. kubectl get all shows everything.
kubectl describe pod my-pod
Get detailed info about a pod — events, container status, resource usage. First command to run when troubleshooting a failing pod. Shows reasons for CrashLoopBackOff, ImagePullError, etc.
kubectl logs -f my-pod -c api
Stream logs from the api container in my-pod. --previous to see logs from a crashed container. --tail=100 for last 100 lines.
kubectl exec -it my-pod -- bash
Open interactive shell inside a running pod. Like docker exec but for K8s. Useful for debugging — run commands inside the container.
kubectl scale deploy my-app --replicas=5
Manually scale a Deployment to 5 replicas. For production, prefer HorizontalPodAutoscaler (HPA) for automatic scaling based on metrics.
kubectl rollout undo deploy/my-app
Roll back to the previous deployment version. K8s keeps revision history. Use --to-revision=2 to roll back to a specific version. kubectl rollout status deploy/my-app to check progress.
kubectl port-forward svc/my-app 8080:80
Forward local port 8080 to service port 80. Access localhost:8080 to reach the service. Essential for testing without exposing services externally.

Scaling & Rolling Updates

Horizontal Pod Autoscaler (HPA)
Automatically scales replicas based on CPU usage, memory, or custom metrics (e.g., requests per second).
hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef:
    kind: Deployment
    name: my-app
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
Rolling Update Strategy
K8s updates pods gradually — never taking all down at once. Zero-downtime deployments by default.
Updating from v1 → v2 (3 replicas):
Start ● v1 ● v1 ● v1
Step 1 ● v1 ● v1 ● v1 + v2
Step 2 ● v1 ● v1 ● v2 ● v2
Step 3 ● v1 ● v2 ● v2 ● v2
Done ● v2 ● v2 ● v2

Ingress & Load Balancing

An Ingress is a smart HTTP router that sits in front of your Services and routes traffic based on hostname or URL path.

ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: api.myapp.com               # route by hostname
    http:
      paths:
      - path: /v1                        # route by path
        backend:
          service:
            name: api-service
            port: { number: 80 }
      - path: /admin
        backend:
          service:
            name: admin-service
            port: { number: 80 }
  tls:                                  # HTTPS with cert-manager
  - hosts: [api.myapp.com]
    secretName: api-tls-cert

1. What does "idempotent" mean in the context of HTTP methods?
○ The request can only be made once
○ Making the same request multiple times has the same effect as making it once
○ The request has no response body
○ The request must include authentication headers
GET, PUT, DELETE, and PATCH are idempotent — calling them N times gives the same result as calling once. POST is NOT idempotent — calling it twice creates two resources.
2. Which HTTP status code should a REST API return when a new resource is successfully created via POST?
○ 200 OK
○ 201 Created
○ 202 Accepted
○ 204 No Content
201 Created is the correct response for successful resource creation. It should include a Location header pointing to the new resource's URL. 204 is for successful DELETE with no body to return.
3. In a JWT, which part contains the user data (claims)?
○ The Header (first segment)
○ The Payload (second segment)
○ The Signature (third segment)
○ JWTs don't contain user data
The Payload (second segment) contains claims — user data like sub (user ID), name, role, and exp (expiry). Remember: the payload is only base64-encoded, NOT encrypted, so never put passwords in it!
4. In FastAPI, what happens if you declare a function parameter with a type hint of int but a client sends a string?
○ FastAPI crashes with an unhandled exception
○ FastAPI silently converts the string to None
○ FastAPI automatically returns a 422 Unprocessable Entity response
○ FastAPI tries to cast the string to int and succeeds
FastAPI uses Pydantic for automatic validation. If the type doesn't match, it returns 422 Unprocessable Entity with a detailed error message showing exactly which field failed and why. This happens before your code runs!
5. What is the key difference between a Docker Image and a Docker Container?
○ Images are for Linux, containers are for any OS
○ An image is a read-only blueprint; a container is a running instance of that image
○ Containers are stored in registries; images run on machines
○ There is no difference — they're the same thing
Image = cookie cutter (read-only blueprint with all layers). Container = actual cookie (running instance with a writable layer on top). One image can produce many containers. A container adds a writable layer to the read-only image layers.
6. Why should you copy requirements.txt and run pip install BEFORE copying your application code in a Dockerfile?
○ It makes the image smaller
○ It allows Docker to cache the dependency installation layer, rebuilding faster when only your code changes
○ Python requires packages before code
○ It's just a convention, it doesn't matter
Docker caches each layer. If requirements.txt hasn't changed, Docker uses the cached pip install layer — saving minutes on every build. If you copy all code first, any code change invalidates the pip install cache.
7. What is the role of etcd in a Kubernetes cluster?
○ It routes external traffic to pods
○ It schedules pods onto worker nodes
○ It is a distributed key-value store that holds the entire cluster state
○ It manages Docker images on each node
etcd is the "brain" of Kubernetes — a distributed, consistent key-value store that holds ALL cluster state: what pods exist, what nodes are healthy, what the desired configuration is. If etcd goes down, the cluster can't function. Always back it up!
8. What Kubernetes object should you use to ensure 3 replicas of a Pod are always running?
○ Pod directly (3 separate Pod YAMLs)
○ Service
○ Deployment
○ ConfigMap
A Deployment manages a ReplicaSet which ensures the desired number of pod replicas are always running. If a pod crashes, the Deployment controller creates a new one. Deployments also handle rolling updates and rollbacks.
9. What does Helm provide for Kubernetes that raw YAML files don't?
○ Better security
○ Templating, versioning, and package management — install/upgrade/rollback whole applications
○ Faster pod scheduling
○ Container image compression
Helm is like npm/apt for Kubernetes. A Helm chart packages all your K8s YAML files with templating (so one chart works for dev/staging/prod with different values). It tracks releases, lets you upgrade with one command, and rollback if something breaks.
10. Which API style uses Protocol Buffers (binary format) over HTTP/2 and is best suited for high-performance microservice communication?
○ REST
○ GraphQL
○ gRPC
○ SOAP
gRPC uses Protocol Buffers (binary, 3-10x smaller than JSON) over HTTP/2 (multiplexed connections). It's up to 10x faster than REST for internal microservice calls. Downside: not natively browser-compatible, requires .proto schema files.

Top Interview Questions

Most commonly asked questions in backend/DevOps interviews. Click to reveal answers.

APIs

What's the difference between REST and RESTful?▼
REST is an architectural style with 6 constraints (stateless, uniform interface, etc.). RESTful is a term for an API that follows those constraints. Technically, most "REST APIs" are RESTful — they follow the REST principles over HTTP. REST can technically work over any protocol.
When would you use GraphQL over REST?▼
Use GraphQL when: (1) Different clients need different data shapes (mobile vs web), (2) You have complex, nested data relationships, (3) You want to prevent over-fetching/under-fetching, (4) You're rapidly iterating on a product. Avoid GraphQL when caching is critical or when your team is small and REST is sufficient.
Explain JWT vs Session-based auth▼
Session-based: Server stores session ID in memory/DB → stateful. JWT: Client holds the token → stateless. JWTs scale better (no shared session store) but can't be invalidated before expiry without a denylist. Sessions can be revoked instantly but require shared state between server instances.
What is CORS and how does it work?▼
CORS (Cross-Origin Resource Sharing) is a browser security mechanism that restricts web pages from making requests to a different domain than the one serving the page. The browser sends an Origin header; the server responds with Access-Control-Allow-Origin. FastAPI handles CORS via CORSMiddleware. Non-browser clients (curl, Postman) are not affected by CORS.

Docker & Kubernetes

What happens when a Pod crashes in K8s?▼
If managed by a Deployment: kubelet detects the failure, the Deployment controller sees the actual state (2 pods) doesn't match desired state (3 pods), and schedules a new pod. If it keeps crashing, it enters CrashLoopBackOff with exponential backoff. Check logs with kubectl logs --previous.
How do you achieve zero-downtime deployments in K8s?▼
Use RollingUpdate strategy with maxUnavailable:0 (never reduce pod count below desired). Add readinessProbes so K8s only sends traffic to healthy pods. Set proper terminationGracePeriodSeconds so the old pod can finish in-flight requests. Also configure preStop hooks for graceful shutdown.
What's the difference between CMD and ENTRYPOINT in Dockerfile?▼
ENTRYPOINT defines the executable that always runs. CMD provides default arguments to ENTRYPOINT (or a default command if no ENTRYPOINT). CMD can be overridden at runtime (docker run myimage bash), ENTRYPOINT cannot (without --entrypoint). Best practice: ENTRYPOINT for the main command, CMD for default arguments.
ConfigMap vs Secret — when to use each?▼
ConfigMap: non-sensitive config (log levels, feature flags, URLs, port numbers). Secret: sensitive data (passwords, API keys, TLS certs). Secrets are base64-encoded (not encrypted!) by default — for real security, enable encryption at rest and use tools like Sealed Secrets or AWS Secrets Manager with CSI driver.

FastAPI

What is ASGI and how does FastAPI use it?▼
ASGI (Asynchronous Server Gateway Interface) is the async evolution of WSGI. It allows servers to handle multiple requests concurrently with async/await. FastAPI is an ASGI framework (built on Starlette). You run it with an ASGI server like uvicorn or gunicorn+uvicorn workers in production.
How does FastAPI dependency injection work?▼
Depends() is a special FastAPI mechanism. When a route function has a parameter with Depends(some_function), FastAPI calls that function first and injects its return value. Dependencies can have their own dependencies (nested DI). Generator dependencies (with yield) support cleanup code — perfect for DB sessions.
When should you use async def vs def in FastAPI?▼
Use async def when: using async libraries (asyncpg, httpx, motor, aiofiles). Use regular def when: using blocking/sync libraries (psycopg2, requests, boto3) — FastAPI runs sync endpoints in a thread pool automatically to avoid blocking the event loop. Never use blocking calls inside async def!

Ecosystem

What are the 3 pillars of observability?▼
1. Metrics (Prometheus/Grafana) — Quantitative measurements over time: latency, error rate, throughput. 2. Logs (ELK/Loki) — Discrete events with context: error messages, request details. 3. Traces (Jaeger/Tempo) — Request flows across services, showing latency at each step. Together they answer: what's wrong, what happened, and why is it slow?
What is a Helm chart and why use it?▼
A Helm chart is a package of K8s YAML templates with configurable values. Instead of maintaining separate YAML files for dev/staging/prod, one chart + different values.yaml covers all. Helm also tracks release history (helm rollback), and Helm Hub (Artifact Hub) has pre-built charts for postgres, redis, nginx, etc.
What problem does a service mesh solve?▼
In microservices, each service needs: retries, circuit breaking, mTLS, tracing, load balancing, rate limiting. Without a mesh, each service implements these in code. A service mesh (Istio, Linkerd) moves this logic into sidecar proxies — your app code stays simple, infrastructure handles the cross-cutting concerns.
