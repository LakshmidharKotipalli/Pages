Section 1

Transformers

The engine that powers nearly every modern LLM. The Transformer architecture, introduced in 2017, changed everything about how machines process language.

The Big Analogy: Imagine you're reading a sentence and trying to understand a word. You don't just look at that word in isolation — you glance back at other words for context. A Transformer does exactly this, but for every word simultaneously, looking at all other words to build a richer understanding.

What is a Transformer?

A Transformer is a neural network architecture made of stacked layers. Each layer has two key parts: an Attention mechanism and a Feed-Forward Network.

Introduced in the 2017 paper "Attention Is All You Need", Transformers replaced older RNNs by processing all tokens at the same time (in parallel) instead of one by one.

Encoder vs Decoder

Encoder: Reads and understands the full input. Used in models like BERT. Great for classification, search.

Decoder: Generates output token by token. Used in GPT, LLaMA. Great for text generation.

Encoder-Decoder: Reads input then generates output. Used in T5, translation models.

Query, Key, and Value — Explained Simply

🔍

Query (Q)

"What am I looking for?" — The current token's question. Like typing a search query.

🏷️

Key (K)

"What do I offer?" — Every other token advertises its content. Like book index keywords.

📦

Value (V)

"Here's my actual information." — The content retrieved when a key matches a query.

How it works: The model computes a dot product between each Query and every Key to get a similarity score. High scores → more attention. Those scores are softmax-normalized to get weights. Then a weighted sum of Values is returned as the output for that position. Formula: Attention(Q,K,V) = softmax(QKᵀ / √d_k) · V
Why "Multi-Head" Attention?

Instead of running attention once, the model runs it multiple times in parallel — each time with different Q, K, V weight matrices. Each "head" learns to focus on a different type of relationship:

  • Head 1 might learn syntax (which words are verbs vs nouns)
  • Head 2 might learn coreference ("it" refers to "the cat")
  • Head 3 might learn semantic roles (who did what to whom)

All head outputs are concatenated and projected into a single vector. GPT-3 uses 96 attention heads!

Positional Encoding — How the Model Knows Word Order

Attention treats all tokens equally — it has no built-in sense of "this word comes before that one." Positional encoding fixes this by adding position information to each token's embedding.


Absolute Positional Encoding (original Transformers): A unique vector is added to each position using sine and cosine waves at different frequencies. Position 1 gets one pattern, position 2 gets another, etc.


RoPE (Rotary Positional Encoding): Used in modern models like LLaMA and GPT-NeoX. Instead of adding position vectors, it rotates the Q and K vectors by an angle proportional to their position. This allows the model to generalize better to longer sequences than it was trained on.

Section 2

Tokenization

Before an LLM can process text, it must convert it into numbers. Tokenization is that conversion step — splitting text into small units called tokens, then mapping them to integer IDs.

The Pizza Analogy: Think of tokenization like cutting a pizza. You don't eat it whole, and you don't cut it atom by atom. You cut it into slices — but the slices aren't always equal. "running" might be one token, but "unbelievable" might be cut into "un" + "believ" + "able". The model eats slices, not letters or whole sentences.

Fun Fact: The word "unbelievable" might tokenize into 3–4 tokens: un believ able. A single emoji like 🤖 might be 3 tokens! Numbers like 12345 may be split as 1, 23, 45. This is why LLMs sometimes struggle with counting letters in words — they never "see" individual characters!

How BPE Works — Step by Step

Byte Pair Encoding (BPE) is the most common tokenization algorithm. Here's how it builds its vocabulary by merging frequent character pairs:

1

low | lower | new | newer
3
Next most frequent: "lo"+"w" = 4 times. Merge!
low | lower | new | newer
4
Repeat until vocabulary target reached (e.g. 50,000 merges for GPT-2)
low | lower | new | newer

Special Tokens

[CLS]

Classification token. Added at start. BERT uses its embedding to represent the whole sentence for classification tasks.

[SEP]

Separator token. Used to separate two sentences (e.g., question + answer in BERT-style models).

<EOS>

End-of-sequence token. Signals the model to stop generating. GPT models use this to end responses.

[PAD]

Padding token. Used to make sequences the same length in a batch. Model learns to ignore it.

<BOS>

Beginning-of-sequence token. Signals the start of a new document or conversation to the model.

[UNK]

Unknown token. Fallback for characters not in vocabulary. Modern BPE avoids this by using byte-level encoding.

Section 3

Context Window

The context window is the model's working memory — the maximum number of tokens it can "see" at once. Everything outside it is invisible.

The Whiteboard Analogy: Imagine a whiteboard that can only hold 4,096 words. Once it's full, to write more you must erase from the beginning. The model only has access to what's on the whiteboard — it can't remember what was erased.

Why Context Window Matters

  • Can't summarize a 500-page book all at once
  • Long chat histories get truncated (model "forgets" early messages)
  • Bigger context = more RAM/VRAM needed
  • Attention cost scales with context squared (O(n²))
  • Token count determines API cost

KV Cache — Making Inference Faster

What is the KV Cache?

During generation, the model computes Key (K) and Value (V) vectors for every token. Without caching, it would recompute these for ALL previous tokens on every new step — incredibly wasteful. The KV cache stores them so they can be reused.

Without KV Cache

To generate token #100, the model recomputes K and V for tokens 1–99 from scratch. Every step gets slower and slower.

With KV Cache

K and V for tokens 1–99 are stored in memory. To generate token #100, only compute K and V for token #100. Constant time per step!

KV Cache Memory Visualization (per layer)
Trade-off: KV Cache saves compute time but costs GPU memory. A 128K context window with a large model can use tens of gigabytes just for the cache. This is why deploying large LLMs with long contexts requires expensive hardware.
Section 4

Sampling Strategies

After the Transformer processes input, it outputs a probability distribution over every token in its vocabulary. Sampling is how we pick the next token from that distribution.

The Music Analogy: Imagine a musician improvising. With very strict rules (low temperature), they only play "safe" notes — predictable but boring. With total freedom (high temperature), they might play anything — surprising but possibly chaotic. Temperature lets you tune this dial.

Decoding Strategies Compared

Greedy Decoding

Always picks the single highest-probability token. Deterministic — same input always gives same output.

Pros: Fast, consistent
Cons: Repetitive, boring, can get stuck in loops

Beam Search

Keeps the top-K most promising sequences at each step. Explores multiple paths simultaneously and picks the best.

Pros: Better than greedy for structured output
Cons: Still repetitive in open-ended generation

Temperature Sampling

Divides logits by T before softmax. T<1 → sharper (more confident). T>1 → flatter (more random). T=0 → greedy.

Pros: Simple, controllable
Cons: Very high T can produce gibberish

Top-P (Nucleus) Sampling

Picks from the smallest set of tokens whose cumulative probability ≥ P. The "nucleus" adapts — smaller when the model is confident, larger when uncertain.

Pros: Adapts to confidence, diverse output
Cons: Can still be unpredictable

Section 5

Embeddings

How does a model "understand" that "king" and "queen" are related, or that "Paris" is to "France" as "Tokyo" is to "Japan"? The answer is embeddings — rich numerical representations of meaning.

The GPS Analogy: Think of embeddings as GPS coordinates, but in 768 or 4096 dimensions instead of just latitude/longitude. Words that mean similar things get coordinates close together. "Happy" and "joyful" would be neighbors. "King" and "Queen" would be close, but both far from "pizza".

1

Text → Token ID

The word "king" maps to token ID 4581 in the vocabulary

2

Token ID → Row Lookup

Row 4581 of the embedding matrix is retrieved. This matrix has shape [vocab_size × embed_dim] — e.g., [50,000 × 768]

3

Vector Retrieved

A 768-dimensional vector is returned — a unique "fingerprint" for "king" that encodes its meaning

4

Positional Encoding Added

Position vector is added to tell the model "this is the 3rd token in the sequence"

Vector Arithmetic — The Famous Examples

vec("king") − vec("man") + vec("woman")
≈ nearest neighbor:
→ vec("queen") 👑
vec("Paris") − vec("France") + vec("Japan")
≈ nearest neighbor:
→ vec("Tokyo") 🗼
vec("fast") − vec("slow") + vec("cold")
≈ nearest neighbor:
→ vec("hot") 🌡️

This works because embeddings learn to encode relationships as directions in vector space. "male → female" and "capital → country" become consistent geometric directions.

How is Similarity Measured? (Cosine Similarity)

The standard way to measure how "close" two embeddings are is cosine similarity — measuring the angle between two vectors.

  • Cosine = 1.0: Vectors point in the same direction → identical meaning
  • Cosine = 0.0: Vectors are perpendicular → unrelated meaning
  • Cosine = -1.0: Vectors point opposite directions → opposite meaning

Unlike Euclidean distance, cosine similarity doesn't care about magnitude — only direction. This is important because a 100-word document and a 10-word document about the same topic should be similar.

Section 6

Training

How does a model go from random weights to being able to answer questions, write code, and hold conversations? Training happens in three distinct phases.

The School Analogy:
Pre-training = Reading every book, article, and website ever written — absorbing all human knowledge.
Fine-tuning = Studying for a specific exam — focusing on being a helpful assistant.
RLHF = Getting a teacher who gives you stars when your answers are great and corrections when they're not.

What happens in Pre-training?

The model is trained on a massive dataset (trillions of tokens from the internet, books, code) to predict the next token. Given "The cat sat on the", the model should predict "mat".

This is called self-supervised learning — no human labels needed! The text itself provides the labels.

Scale

  • GPT-3: Trained on ~300B tokens
  • LLaMA 3: Trained on ~15 trillion tokens
  • Training can cost millions of dollars
  • Requires thousands of GPUs running for weeks
  • The model learns grammar, facts, reasoning, code — all from next-token prediction!
Next-Token Prediction — The Training Signal
Input: The cat sat on the
Predict: mat ← This is the label
Loss: How wrong was the prediction? → Update all weights slightly to do better next time

What is Supervised Fine-Tuning?

After pre-training, the model knows a lot but isn't helpful yet — it might just continue text instead of answering questions. SFT trains it on curated examples of (prompt → ideal response) pairs written by humans.

Example: Given "What is the capital of France?", the ideal response is "The capital of France is Paris." The model adjusts its weights to match these response patterns.

Why Fine-Tuning Works

The model already has all the knowledge from pre-training — SFT just teaches it how to use that knowledge in a helpful way. Think of it as narrowing a wide beam of light into a focused laser.

Fine-tuning uses far less data (thousands to millions of examples, not trillions) and is much cheaper than pre-training.

SFT Training Examples
USER PROMPT
Explain photosynthesis in simple terms
IDEAL RESPONSE
Photosynthesis is the process plants use to make their own food. They take in sunlight, water, and carbon dioxide, and convert them into glucose (sugar) and oxygen. Think of it like a plant's personal solar-powered kitchen!

What is RLHF?

Reinforcement Learning from Human Feedback. After SFT, humans compare pairs of model responses and rate which is better. These preferences train a reward model that scores responses.

Then the main LLM is trained using PPO (Proximal Policy Optimization) to maximize the reward model's score — making responses more helpful, harmless, and honest.

Section 7

Inference — Autoregressive Generation

When you send a message to an LLM, here's exactly what happens. Tokens are generated one at a time, each building on the last — this is called autoregressive generation.

The Typewriter Analogy: Imagine a typewriter that can only type one letter at a time, and each new letter depends on everything typed before it. The model never writes a full word in one go — it types character by character (or token by token), feeding what it just typed back into itself to decide the next character.

The Decode Loop — Step by Step

1

Tokenize the prompt

Your input text is tokenized into a sequence of integer IDs: [464, 2883, 7586, ...]

2

Forward pass through the Transformer

All prompt tokens pass through every layer simultaneously. The model produces a logit vector — one score per vocabulary token — for the last position.

3

Apply sampling strategy

Logits → softmax → probabilities. Apply temperature, top-k, or top-p. Sample one token from the resulting distribution.

4

Append new token and repeat

The new token is appended to the sequence. The entire sequence (prompt + generated tokens so far) is fed back in for the next forward pass.

5

Stop when done

Generation stops when an <EOS> token is generated, a max length is hit, or a stop sequence appears.

Why is generation slow? Each new token requires a full forward pass through all transformer layers — potentially 32 to 96 layers deep. Generating 1,000 tokens means 1,000 full forward passes. This is why GPU is so important, and why KV Cache (Section 3) is critical for speed — it avoids recomputing keys and values for all previous tokens on each step.

Common Misconceptions

Myth vs Reality

There's a lot of confusion about how LLMs actually work. Let's clear it up.

MYTH: LLMs understand language the same way humans do
People assume LLMs "comprehend" meaning the way we do, with consciousness and intentionality.
REALITY: LLMs are extremely sophisticated pattern matchers
They predict the statistically most likely next token based on patterns learned from text. There's no proven "understanding" in a human sense — just very powerful statistical modeling.
MYTH: LLMs search the internet in real time
Many users assume ChatGPT is Googling your question when it answers.
REALITY: Knowledge is frozen at training cutoff
The base LLM has no internet access. Its knowledge is baked in from training data. Some products (like GPT-4 with browsing) add a separate tool to search the web — but that's an add-on, not the LLM itself.
MYTH: Bigger models always give better answers
Surely a 70B model always beats a 7B model, right?
REALITY: Smaller fine-tuned models often outperform larger base models
A 7B model fine-tuned for medical Q&A can outperform a general 70B model on medical questions. Task-specific fine-tuning often matters more than raw size.
MYTH: LLMs "decide" what to say next
It feels like the AI is thinking and deciding, like a human would.
REALITY: It's all probability distributions
The model outputs a vector of probabilities over its vocabulary and randomly samples from it. There's no deliberation — just matrix multiplications and sampling from a distribution.
