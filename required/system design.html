👤 Client

The user's browser or mobile app. Sends HTTP/S requests. For performance, it uses keep-alive connections and caches static assets locally. Modern clients also use HTTP/2 (multiplexed streams) or HTTP/3 (QUIC protocol).

  • HTTP/2: Multiple requests in one TCP connection (no head-of-line blocking)
  • HTTP/3: UDP-based, faster mobile handshakes
  • Service Workers: Offline caching in PWAs

Scalability: Vertical vs Horizontal

🍽️
Restaurant analogy: Vertical scaling = hire a faster, stronger chef (upgrade your server's CPU/RAM). Horizontal scaling = open more restaurant locations (add more servers). Vertical has limits — no single chef can cook 10,000 meals/minute. Horizontal scales infinitely but requires coordination.

📈 Vertical Scaling (Scale Up)

  • Add more CPU, RAM, or disk to one machine
  • Simple — no application changes needed
  • Hard upper limit: largest EC2 instance is ~448 vCPUs
  • Single point of failure
  • Good for: databases, stateful services short-term
  • Cost: exponential as you go higher

📊 Horizontal Scaling (Scale Out)

  • Add more machines; distribute load across them
  • Requires stateless architecture (no server-side sessions)
  • No theoretical limit — Netflix runs 100,000s of instances
  • Fault tolerant — one node fails, others continue
  • Requires load balancer + service discovery
  • Cost: linear — double servers = double capacity

Load Balancing Algorithms

🎯
The restaurant host analogy: A host at a busy restaurant decides which table (server) to seat you at. They could just cycle through tables (Round Robin), pick the least-busy waiter (Least Connections), or always send your family to the same section (Consistent Hashing — so your waiter knows your preferences).

Round Robin

Requests are distributed equally in order: Server 1 → Server 2 → Server 3 → Server 1 → ...

Weighted Round Robin: If Server 1 is twice as powerful, give it 2 requests for every 1 to Server 2.

✅ Simple. ✅ Even distribution for uniform requests. ❌ Ignores server load. ❌ Long-running requests can pile up on one server.

Least Connections

Send each new request to the server with the fewest active connections. Ideal when requests have highly variable processing time (e.g., some API calls take 5ms, others take 5 seconds).

✅ Adapts to actual load. ✅ Great for long-lived connections. ❌ Slightly higher LB overhead (must track connections). ❌ Needs session persistence layer for stateful apps.

Consistent Hashing

Imagine a clock (ring). Servers sit at positions on the ring. A request's hash maps to a position, and goes to the next clockwise server. Adding/removing a server only redistributes ~1/N of requests, not all of them.

✅ Minimal reshuffling when servers change. ✅ Cache locality — same user always hits same server. ✅ Used by Cassandra, DynamoDB, Akamai CDN. Virtual nodes (vnodes) fix uneven distribution.

IP Hash

Hash the client's IP address to determine which server handles their requests. Same client = same server (session stickiness).

✅ Easy session persistence. ✅ No shared session store needed. ❌ Uneven if many users behind same NAT IP. ❌ Server changes redistribute all users from affected IPs.

Caching Strategies

🧠
Caching is like a waiter's memory. If you always order "the usual," the waiter remembers without checking the menu. A cache stores frequently accessed data in fast memory so you don't hit the slow database every time. This can reduce latency from 100ms → 1ms.

Write-Through Cache

Write to cache AND database simultaneously. Cache is always consistent. Slightly slower writes (both must succeed). Use when: read-heavy, consistency critical. Consistent

Write-Back (Write-Behind)

Write to cache only; database is updated asynchronously. Faster writes. Risk: data loss on cache crash before DB write. Use when: write-heavy, can tolerate brief inconsistency. Risk: data loss

Cache-Aside (Lazy Loading)

Application checks cache first. On miss, reads DB and populates cache. Most common pattern (Redis). Cache only holds what's actually needed. Most Common

Eviction Policies

  • LRU — Least Recently Used (most common)
  • LFU — Least Frequently Used
  • TTL — Expire after fixed time
  • FIFO — First in, first out
  • Random — Redis option for low overhead
Redis vs Memcached: Redis supports rich data structures (lists, sets, sorted sets, hashes), persistence, pub/sub, Lua scripting, and cluster mode. Memcached is simpler, faster for pure key-value caching. Choose Redis for 99% of use cases in 2025.

📦 Database Sharding — What, Why, and How ▼

Sharding splits a large database into smaller, faster pieces called shards, each stored on a different server. Like splitting a phone book into A-M and N-Z books stored in different libraries.

  • Horizontal sharding: Split rows across servers (e.g., user IDs 1-1M on shard 1, 1M-2M on shard 2)
  • Vertical sharding: Split columns/tables (user data on one DB, order data on another)
  • Key-based (hash) sharding: Hash(user_id) % num_shards — even distribution, hard to add shards
  • Range-based sharding: Alphabetical or date ranges — easy to add shards, risk of hotspots
  • Directory-based sharding: Lookup table tracks where each record lives — flexible, extra hop

Problems: Cross-shard joins are expensive. Rebalancing is hard. Hotspots (celebrity problem — one shard gets 90% of traffic). Solutions: virtual shards, consistent hashing, application-level distributed transactions.

🔄 Replication — Primary/Replica, Multi-Primary ▼

Primary-Replica (Master-Slave): One write node, multiple read replicas. All writes go to primary, replicated asynchronously to replicas. Reads can go to replicas (read scaling). Risk: replication lag means replicas may be slightly behind.

Multi-Primary (Multi-Master): Multiple nodes accept writes. Requires conflict resolution (last-write-wins, vector clocks, CRDTs). Used by CockroachDB, Google Spanner. Harder to build but great availability.

Leaderless Replication (Cassandra, DynamoDB): Any node accepts writes. Uses quorum reads/writes (W + R > N for consistency). No single point of failure.

CAP Theorem

🏦
Bank analogy: Imagine a bank with branches. It can't be simultaneously: always open (Available), always showing the exact same balance across branches (Consistent), AND survive a phone line outage between branches (Partition Tolerant). During a network partition, it must pick two of three.

CP (Consistency + Partition)

Returns error or timeout during partition. Examples: HBase, Zookeeper, MongoDB (by default). Good for: financial ledgers, inventory.

AP (Availability + Partition)

Returns possibly stale data but never fails. Examples: Cassandra, DynamoDB, CouchDB. Good for: social feeds, DNS, shopping carts.

CA (Consistency + Availability)

Only possible without partitions — unrealistic in distributed systems. Single-node databases (SQLite). Real world: CA systems sacrifice P, which means they're not truly distributed.

PACELC Extension: CAP only describes behavior during partitions. PACELC also covers the trade-off between Latency and Consistency when the network is working normally. Most real systems make both trade-offs.

Message Queues & Event Streaming

🍣
Kafka = conveyor belt sushi. Producers put dishes on the belt (publish events). Consumers pick what they want when they're ready. The belt keeps moving — it doesn't wait. Multiple consumer groups can read the same belt independently. Messages are retained so a slow consumer can catch up.

Apache Kafka

  • Distributed event streaming platform
  • Topics → Partitions → Offsets
  • Retention: replay messages days/weeks later
  • Throughput: millions of messages/second
  • Consumer groups: parallel processing
  • Use: event sourcing, audit logs, stream processing

RabbitMQ / AWS SQS

  • Message broker — point-to-point or pub/sub
  • Messages deleted after consumption
  • AMQP protocol, routing with exchanges
  • Dead letter queues for failed messages
  • SQS: fully managed, FIFO or standard
  • Use: task queues, RPC, email sending

Fault Tolerance: Circuit Breakers, Retries & Timeouts

💡
Circuit breaker = electrical circuit breaker. When a downstream service fails repeatedly, the circuit "opens" — your service stops calling it immediately (fast failure) instead of waiting and cascading failures. After a timeout, it "half-opens" and tests one request. If it succeeds, circuit closes.

Circuit Breaker States

  • Closed: Normal — requests flow through
  • Open: Failing — fast-fail all requests
  • Half-Open: Test — let one request through to check recovery

Retry with Backoff

  • Exponential backoff: wait 1s, 2s, 4s, 8s...
  • Jitter: add random noise to prevent thundering herd
  • Max retries: don't retry indefinitely
  • Idempotency: safe to retry only if side-effect-free

Timeouts & Bulkheads

  • Timeout: Don't wait forever — fail fast
  • Bulkhead: Isolate thread pools per service so one slow service doesn't starve others
  • Rate limiting: Token bucket, leaky bucket algorithms

📥 Data Ingestion

Sources: Databases (Postgres, MySQL), data lakes (S3, GCS), streaming (Kafka, Kinesis), APIs, web scraping. Formats: CSV, Parquet (columnar — 5-10x smaller), Avro (schema evolution), ORC.
Tools: Apache Spark for batch, Apache Flink for stream, Airbyte for ELT, dbt for transformations.
Key considerations: Data quality (nulls, duplicates, outliers), schema validation, data lineage tracking, PII handling.

Feature Stores

🛒
Feature store = grocery store with meal prep. The offline store pre-chops vegetables in bulk (batch feature computation). The online store keeps ready-to-cook bags at the counter (low-latency feature serving at inference time). Same vegetables, two speeds.

Offline Store

Historical feature data for training. Stored in data warehouse (BigQuery, Snowflake, S3 + Parquet). Supports time-travel queries to prevent data leakage. Computed via batch jobs (nightly or hourly).

Online Store

Low-latency (<10ms) feature serving for real-time inference. Backed by Redis, DynamoDB, or Cassandra. Features are materialized from the offline store. Critical for fraud detection, recommendations, search ranking.

Feast (Open Source)

Define features once, use in training and serving. Supports BigQuery (offline) + Redis (online). Feature registry in YAML. Prevents training/serving skew — same code path computes both. FOSS, widely used in enterprises.

Tecton (Managed)

Enterprise feature platform. Handles streaming features natively (Kafka → real-time feature updates). Built-in monitoring, backfills, SLAs. More expensive but turnkey. Used by Lyft, Verizon, Capital One.

Model Training Infrastructure

🖥️ Distributed Training with Ray & Horovod ▼

Why distributed? Training GPT-3 (175B params) on a single A100 GPU would take ~355 years. Distribute across 1,000 GPUs and it takes weeks.

Data Parallelism: Split data across GPUs, each GPU has a full model copy. Gradients are averaged and synced after each batch (AllReduce). Horovod uses NCCL for GPU-to-GPU communication via NVLink.

Model Parallelism: Split the model across GPUs (different layers on different GPUs). Required when model > single GPU memory. Pipeline parallelism staggers computation to minimize GPU idle time.

Tensor Parallelism: Split individual weight matrices across GPUs. Megatron-LM uses this to train 540B parameter models. Used by NVIDIA for training frontier models.

Ray Train: Python-first distributed training. Works with PyTorch, TensorFlow, Hugging Face. Handles fault tolerance — if a node crashes, Ray restarts it without losing the whole run.

📝 Model Registries: MLflow, W&B, DVC ▼

MLflow: Tracks experiments (parameters, metrics, artifacts). Model registry with staging/production stages. REST API for model serving. Open source. Used at Databricks.

Weights & Biases (W&B): Best-in-class visualization. Real-time loss curves, gradient histograms, GPU utilization. Team collaboration. Sweeps for hyperparameter search. Artifacts for dataset/model versioning.

DVC (Data Version Control): Git for large files. Track datasets and models as if they were code. DVC pipelines are reproducible. Works with S3, GCS, Azure Blob. Integrates with any CI/CD.

Model Serving & Inference Systems

TorchServe

PyTorch's official model server. REST + gRPC APIs. Handles model versioning, A/B testing, batching. Custom handlers for preprocessing. Management API for hot-swapping models without downtime.

NVIDIA Triton Inference Server

Supports PyTorch, TensorFlow, ONNX, TensorRT, vLLM backends in one server. Dynamic batching. Multi-GPU, multi-model serving. Concurrent model execution. Industry standard for high-throughput GPU serving.

BentoML

Package ML models as production services. Bentofile defines dependencies. Auto-generates Docker images. Supports batching, adaptive batching. Cloud deployment (BentoCloud) or self-hosted.

vLLM

The gold standard for LLM serving (2023-2025). PagedAttention memory management. Continuous batching. 24x higher throughput than naive serving. OpenAI-compatible API. Supports quantization (GPTQ, AWQ).

LLM Serving at Scale — Deep Dive

📚
KV Cache = a librarian's index cards. When processing your prompt, the model computes keys and values for each token. These are expensive to recompute. The KV cache stores them so generating each new token only needs to look up the index card, not re-read all the books.

PagedAttention (vLLM's Core Innovation)

Traditional LLM serving pre-allocates a contiguous memory block for the max sequence length per request, wasting 60-80% of GPU memory on padding. PagedAttention stores KV cache in non-contiguous pages (like OS virtual memory), eliminating fragmentation.

Result: vLLM can serve 24x more concurrent requests than HuggingFace TGI on the same GPU. Prefix caching reuses KV pages for identical prompt prefixes (system prompts) across users — zero recomputation overhead.

Continuous Batching

Old way (static batching): Batch 8 requests. Shortest finishes in 10 steps, longest in 100 steps. 8 GPUs sit idle for 90 steps waiting for the slowest request. Terrible GPU utilization.

Continuous batching: When a sequence finishes, immediately insert a new request into that "slot." The batch size stays full. Used by vLLM, Orca. Achieves near-100% GPU utilization under load.

Speculative Decoding

Analogy: An intern drafts an email, the senior editor reviews and corrects it in one pass instead of writing word-by-word.

A small draft model (7B) generates K tokens cheaply. The large target model (70B) verifies all K tokens in one forward pass (same cost as generating 1 token). If tokens are correct, accept all — K tokens for the cost of 1. Rejected tokens are regenerated. Net speedup: 2-3x for typical text with a good draft model.

Quantization for LLMs

Reduce model precision to save memory and increase throughput:

  • FP16/BF16: Standard — half precision, ~50% memory vs FP32
  • INT8 (LLM.int8): Quantize weights to 8-bit — 2x memory reduction, ~5% quality loss
  • INT4 (GPTQ, AWQ): 4-bit weights — 4x reduction, ~10% quality loss. Run 70B model on 2x A100
  • GGUF (llama.cpp): 2-4 bit quantization for CPU inference

AWQ (Activation-aware Weight Quantization) outperforms GPTQ by protecting weights that affect activations most — better quality at same compression.

Recommendation System Architecture

Stage 1: Candidate Generation

Narrow 1B items → 1,000 candidates. Use collaborative filtering (matrix factorization), content-based (item embeddings), or two-tower models. Must be fast (ANN search in vector DB).

Stage 2: Ranking

Score 1,000 candidates → top 100. Use a larger model (GBDT, deep neural net) with user context, item features, engagement signals. Latency budget: <50ms.

Stage 3: Re-Ranking & Filtering

Apply business rules (diversity, freshness, safety filters, sponsored content injection). Final 10-50 items shown to user. Also handle cold-start (new user/item) with fallback strategies.

Real-Time Inference

  • Latency: <100ms
  • Trigger: HTTP request
  • Compute: pre-loaded model in memory
  • Examples: fraud detection, ad ranking, recommendations
  • Tools: TorchServe, Triton, vLLM, BentoML
  • Challenge: GPU cost at low traffic

Batch Inference

  • Latency: hours (acceptable)
  • Trigger: cron job, event
  • Compute: spin up cluster, process, spin down
  • Examples: nightly churn scores, weekly recommendations email
  • Tools: Spark, Ray Batch, SageMaker Batch Transform
  • Advantage: much cheaper GPU utilization
🔄 Topic 3

MLOps & LLMOps

Shipping models is the easy part. Keeping them working in production — that's the hard part MLOps solves.

🛫
MLOps is like aviation maintenance. A pilot (data scientist) flies the plane (trains the model). But ground crews (MLOps engineers) handle maintenance, refueling, flight logs, safety checks, and scheduling. Without them, planes fall out of the sky. Without MLOps, models degrade silently in production.

MLOps Maturity Levels

Level 0 — Manual

Scientists train models in notebooks. Manual deployment. No tracking. No automation. Typical at startups or early ML teams. Bottleneck: every retraining is a human crisis.

Level 1 — ML Pipeline

Automated training pipeline. CT (continuous training) triggered by data changes. Model registry. Basic monitoring. Experiment tracking with MLflow or W&B. Deployment still partly manual.

Level 2 — CI/CD for ML

Full CI/CD: code change triggers test → train → validate → deploy. Feature store in use. Automated A/B testing. Production monitoring with automatic retraining triggers. Canary deployments standard.

CI/CD for ML Models

Continuous Integration — What to Test

  • Unit tests: Data validation (schema, nulls, ranges), feature engineering functions
  • Model tests: Does accuracy stay above threshold? Are predictions deterministic?
  • Training tests: Can the model overfit on a small batch? (sanity check)
  • Infrastructure tests: Does the serving container start? Can it handle the expected QPS?
  • Data integrity: Great Expectations, Pandera for schema validation

Deployment Strategies

  • Blue-Green: Run old and new version simultaneously. Switch traffic instantly. Easy rollback.
  • Canary: Send 5% of traffic to new model. If metrics stable, gradually increase. Roll back if degradation detected.
  • Shadow (Dark Launch): New model processes real traffic but responses are discarded. Compare offline. Zero user impact.
  • A/B Testing: Statistically split users. Compare business metrics (CTR, revenue) between model versions.

Continuous Training Triggers

  • Schedule-based: Retrain every week — simple but ignores data drift
  • Data-triggered: Retrain when N new labeled samples are collected
  • Performance-triggered: Retrain when model accuracy drops below threshold
  • Drift-triggered: Retrain when data distribution drift is detected (PSI, KL divergence)

ML Model Monitoring in Production

Data Drift

Input feature distribution shifts over time. Example: a fraud model trained on 2023 data sees 2025 transaction patterns it's never seen. Detection: PSI (Population Stability Index), KL/JS divergence, Wasserstein distance, Kolmogorov-Smirnov test.

Causes: Seasonal changes, new user segments, product changes

Concept Drift

The relationship between features and labels changes. Example: "email contains 'free'" used to strongly predict spam; now it's often legitimate marketing. Harder to detect — requires labeled data or proxy metrics.

More dangerous: Silent, invisible, causes gradual degradation

Prediction Monitoring

Track prediction distribution, confidence scores, class frequencies. Alert if prediction entropy suddenly changes. Use Evidently AI, WhyLabs, or Arize for automated drift reports.

Business Metrics

The ultimate ground truth. Track CTR, conversion rate, revenue per user, churn rate. ML metrics (AUC, F1) don't always correlate with business metrics — always instrument both.

LLMOps — Managing LLM Apps in Production

LLMOps = MLOps + Prompt Engineering + Evaluation + Safety + Cost Management. LLMs are non-deterministic, expensive to run, and hard to evaluate. Traditional ML metrics (accuracy, F1) don't work — you need semantic evaluation.

Prompt Versioning

Treat prompts like code — version control in Git. Use tools like LangSmith, PromptLayer, or Langfuse to track which prompt version produced which output. Never change a production prompt without testing. A/B test prompt versions like you A/B test model versions.

LLM Evaluation Frameworks

  • RAGAS: Faithfulness, Answer Relevancy, Context Precision, Context Recall for RAG
  • LangSmith: Tracing + eval datasets + annotation queues
  • Promptfoo: YAML-based test suites, red-teaming
  • DeepEval: 15+ built-in metrics, CI integration

Cost Optimization

  • Caching: Semantic cache (GPTCache) for similar queries
  • Prompt compression: LLMLingua reduces prompts 3-20x
  • Model routing: Route simple queries to GPT-4o-mini, complex to GPT-4o
  • Batching: Batch async requests for 50% cost reduction
  • Local models: vLLM + Llama-3 for non-sensitive workloads

Guardrails & Safety

  • Input guardrails: PII detection, prompt injection detection
  • Output guardrails: Toxicity, hallucination, off-topic detection
  • Tools: NVIDIA NeMo Guardrails, LlamaGuard, Guardrails AI
  • Patterns: Input → sanitize → LLM → validate → output

LLM Evaluation with RAGAS

from ragas import evaluate
from ragas.metrics import (
    faithfulness,         # Answer grounded in context?
    answer_relevancy,     # Answer addresses the question?
    context_precision,    # Retrieved chunks relevant?
    context_recall        # All needed info retrieved?
)
from datasets import Dataset

# Your RAG evaluation dataset
eval_data = {
    "question": ["What is the refund policy?"],
    "answer": ["Refunds processed within 7 days."],
    "contexts": [["Our policy: refunds in 5-7 business days..."]],
    "ground_truth": ["Refunds take 5-7 business days."]
}
dataset = Dataset.from_dict(eval_data)

# Run evaluation
result = evaluate(
    dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall]
)
# {'faithfulness': 0.92, 'answer_relevancy': 0.88, ...}
print(result)

Retrieval Strategies: Dense, Sparse, Hybrid

Dense Retrieval

Use embedding models (text-embedding-3-large, BGE, E5) to convert queries and documents to vectors. Find nearest neighbors via ANN (HNSW, IVF). Understands semantics — "car" matches "automobile."

Good at: Semantic matching, paraphrases

Bad at: Exact keyword matching, rare terms, IDs

Sparse Retrieval (BM25)

Classic TF-IDF based. Fast, no GPU. Exact keyword matching. Still king for: product codes, names, exact phrases. BM25 is the default in Elasticsearch, OpenSearch, Weaviate (hybrid mode).

Good at: Exact matches, rare keywords

Bad at: Synonyms, context, semantics

Hybrid + Reranking

Run both dense and sparse, combine scores (RRF — Reciprocal Rank Fusion). Then apply a cross-encoder reranker (Cohere Rerank, BGE-Reranker) on top-50 results → final top-5. Best accuracy, higher latency.

Best accuracy — standard for production RAG 2025

Building a RAG Pipeline with LangChain

from langchain.document_loaders import DirectoryLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import Chroma
from langchain.chains import create_retrieval_chain
from langchain.chains.combine_documents import create_stuff_documents_chain
from langchain_core.prompts import ChatPromptTemplate

# 1. Load & chunk documents
loader = DirectoryLoader("./docs", glob="**/*.pdf")
docs = loader.load()
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000, chunk_overlap=200
)
chunks = splitter.split_documents(docs)

# 2. Embed & store in vector DB
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(chunks, embeddings)
retriever = vectorstore.as_retriever(
    search_type="mmr",         # Max Marginal Relevance — diverse results
    search_kwargs={"k": 5}
)

# 3. Build RAG chain
llm = ChatOpenAI(model="gpt-4o", temperature=0)
prompt = ChatPromptTemplate.from_template("""
Answer based only on context. If unsure, say so.

Context: {context}
Question: {input}
""")
chain = create_retrieval_chain(
    retriever,
    create_stuff_documents_chain(llm, prompt)
)

# 4. Query
response = chain.invoke({"input": "What is the refund policy?"})
print(response["answer"])

AI Application Architecture Patterns

🔗
RAG Pattern
Click to see details

Retrieve → Augment → Generate. Best for knowledge bases, documentation search, enterprise Q&A. No retraining needed. Knowledge always fresh. Works with any LLM.

🤖
Agent Pattern
Click to see details

LLM as orchestrator: plan → use tools → observe → repeat. Tools: search, code execution, APIs, databases. Best for complex multi-step tasks. Risk: error propagation, cost.

🔀
Router Pattern
Click to see details

LLM classifies intent, routes to specialized handlers. Simple queries → fast model. Complex → powerful model. RAG queries → retrieval. Code → code interpreter. Reduces cost and latency.

⛓️
Chain Pattern
Click to see details

Sequential LLM calls where output of one is input to next. Extract → Validate → Summarize → Format. More reliable than one big prompt. Each step can be tested independently.

🗳️
Ensemble Pattern
Click to see details

Run same query through multiple models, vote on best answer (Self-Consistency). Or generate N candidates with high temperature, select best with low-temp evaluator. Higher accuracy, higher cost.

🧩
Tool Use Pattern
Click to see details

LLM decides which function to call. Function executes in your code. Result returned to LLM. Enables grounding — calculations, lookups, real-time data. Core of all agentic systems (Claude, GPT-4, Gemini all support this natively).

🔗 URL Shortener (e.g., bit.ly)

Scale: 100M URLs shortened/day, 10B redirects/day. Read-heavy (100:1 read/write ratio).

Core API: POST /shorten → {long_url} → returns short_code. GET /:code → 301/302 redirect.

Key design choices:

  • Short code generation: Base62 encoding of auto-increment DB ID (a-zA-Z0-9 = 62 chars). 6 chars = 62^6 = 56 billion unique URLs. Or MD5/SHA256 hash + take first 6 chars (collision risk — use counter suffix).
  • 301 vs 302: 301 = permanent, browser caches → less traffic to your servers but you lose analytics. 302 = temporary, every redirect hits your server → full analytics. Choose 302 for click tracking.
  • Database: Simple KV store (Redis for hot URLs, Cassandra/DynamoDB for durability). Schema: {short_code → long_url, created_at, user_id, expiry}.
  • Scaling reads: Redis cache in front of DB (99% cache hit rate for popular links). CDN for globally distributed redirects.
  • Analytics: Write click events to Kafka → Flink/Spark for real-time stats → OLAP DB (ClickHouse) for dashboards.

Common Architecture Patterns

Fan-Out Pattern

One event triggers writes to many destinations. Example: Tweet → write to 1M followers' timelines. Fan-out on write = fast reads but slow/expensive writes. Fan-out on read = slow reads but cheap writes. Hybrid based on follower count is the industry solution.

CQRS (Command Query Responsibility Segregation)

Separate read model from write model. Writes → command handler → event store. Reads → optimized read model (denormalized, cached). Allows independent scaling. Read model can be rebuilt from event history.

Event Sourcing

Store sequence of events instead of current state. Bank balance = sum of all transactions, not a single balance field. Audit log by default. Time-travel debugging. Replay events to rebuild state. Used by financial systems, Kafka-based architectures.

Pub-Sub Pattern

Publishers emit events without knowing subscribers. Subscribers filter by topic. Decoupled, scalable, extensible (add new subscribers without modifying publisher). Kafka, SNS, Google Pub/Sub. Trade-off: eventual delivery, no ordering guarantee across partitions.

Kubernetes for AI Workloads

🚢
Kubernetes is like a shipping port manager. Docker containers are shipping containers (standardized boxes of your app). Kubernetes decides which ship (node) gets which containers, restarts fallen containers, scales them up when demand rises, and routes traffic intelligently.
⚙️ Core Kubernetes Concepts for AI Engineers ▼
  • Pod: Smallest deployable unit. One or more containers sharing network + storage. Think of it as one instance of your model server.
  • Deployment: Manages replica sets. Declare "I want 3 pods running model-server:v2" — K8s ensures it.
  • Service: Stable network endpoint for a set of pods (pods come and go, service IP stays constant).
  • Ingress: Routes external HTTP traffic to services. Like an nginx config but managed by K8s.
  • PersistentVolume (PV): Network-attached storage for pods. Models are often loaded from PV at startup.
  • GPU resource requests: nvidia.com/gpu: 1 in the container spec — K8s scheduler places pod only on nodes with GPUs.
  • Node selectors / tolerations: Pin GPU workloads to GPU node pools. Keep CPU workloads on cheaper nodes.
  • HPA (Horizontal Pod Autoscaler): Scale pods based on CPU, memory, or custom metrics (e.g., GPU utilization, queue depth).
  • Karpenter / Cluster Autoscaler: Automatically add/remove nodes. Critical for burst training or variable inference load.
🎮 GPU Provisioning & Optimization ▼
  • GPU types: A100 (80GB) — best for large model training. H100 (80GB) — 3x A100 speed, transformer optimized. A10G (24GB) — inference, cheaper. T4 (16GB) — budget inference. L4 — best price/perf for inference 2024-25.
  • Multi-GPU setups: NVLink for GPU-to-GPU bandwidth (600 GB/s) vs PCIe (32 GB/s). 8×A100 node = DGX A100, standard for training frontier models.
  • GPU sharing: NVIDIA MIG (Multi-Instance GPU) — partition A100 into 7 independent instances. Time-slicing for development. vLLM handles multi-GPU model parallelism automatically.
  • MFU (Model FLOP Utilization): Ratio of actual FLOPS to theoretical max. Good training achieves 30-55% MFU. vLLM inference achieves 50-80% GPU memory utilization vs 20-40% for naive serving.
💰 Spot Instances & Cost Optimization for ML ▼
  • Spot/Preemptible instances: 60-90% cheaper than on-demand but can be interrupted with 2 minutes notice. Perfect for training (save checkpoints every N steps) — NOT for serving.
  • Training best practices: Checkpoint to S3 every 100 steps. Use spot with on-demand fallback (AWS mixed instance policy). With PyTorch DDP + checkpointing, interruption costs only a few minutes of compute.
  • Serving: On-demand or reserved instances. Reserve 1-year for baseline load (40% discount). Spot for burst traffic with queue-based auto-scaling.
  • Savings Plans / Committed Use Discounts: AWS/GCP allow 1-3 year commitments for 40-60% discount. Good for consistent ML workloads.
  • Right-sizing: Don't run a 70B model on 8×A100 if 2×A100 handles your QPS. Profile first with load testing (Locust, k6).

Networking Basics for AI Engineers

VPC (Virtual Private Cloud)

Private network in the cloud. Your resources (EC2, RDS) live here, isolated from internet. Subnets divide VPC into public (internet-facing) and private (internal only). NAT gateway allows private subnet resources to call external APIs without being directly reachable.

CDN for AI

Service Mesh (Istio)

For microservice architectures. Handles: mTLS (encrypted service-to-service), traffic management, circuit breaking, observability — without changing application code. Sidecar proxy (Envoy) injected per pod. Used at scale for ML microservices.

Q1/15 A Cassandra database returns data that might be slightly outdated but is always available. Which CAP property combination is this?
✅ Correct! Cassandra is an AP system — it prioritizes availability and partition tolerance. It uses eventual consistency, meaning data will eventually converge but reads may briefly return stale values. CA systems can't exist in distributed networks (you can't avoid partitions).
Question 1 of 15
Q2/15 vLLM's PagedAttention achieves its massive throughput improvement primarily by doing what?
✅ Correct! Traditional LLM serving pre-allocates contiguous memory blocks per sequence (wasting 60-80% via padding). PagedAttention uses virtual memory-style paging — KV cache stored in non-contiguous blocks. This enables near-optimal memory utilization and allows vLLM to serve 24x more concurrent requests.
Question 2 of 15
Q3/15 You're designing a URL shortener that needs to track clicks. Should the redirect use HTTP 301 or 302, and why?
✅ Correct! 301 is permanent — browsers cache it and go directly to the destination, bypassing your servers and losing click data. 302 is temporary — every click goes through your server, enabling full analytics. Trade-off: 302 means more load on your servers. bit.ly uses 301 for performance, but analytics-focused shorteners use 302.
Question 3 of 15
Q4/15 What is the key advantage of consistent hashing over simple modulo hashing when servers are added or removed?
✅ Correct! With modulo hashing (key % N), adding one server changes the mapping for nearly all keys — causing a massive cache miss storm. Consistent hashing places servers on a ring; adding a server only reassigns ~1/N keys (just those that fall between the new server and its predecessor). This is why Cassandra, DynamoDB, and CDNs use it.
Question 4 of 15
Q5/15 In MLOps, what is "concept drift" and how does it differ from "data drift"?
✅ Correct! Data drift: your model sees different input patterns than training (e.g., new device types). Concept drift: the world changes so the mapping from inputs to outputs changes (e.g., "free" in emails no longer predicts spam). Concept drift is harder to detect — it requires ground truth labels or proxy signals. Both require monitoring and retraining.
Question 5 of 15
Q6/15 What is the primary purpose of a Feature Store's "offline store" vs "online store"?
✅ Correct! The offline store (BigQuery, Parquet on S3) stores historical feature data for training — it supports time-travel queries to prevent data leakage. The online store (Redis, DynamoDB) serves features at inference time with sub-10ms latency. The key benefit: same feature computation logic used in both, preventing training-serving skew.
Question 6 of 15
Q7/15 In speculative decoding, what role does the "draft model" play?
✅ Correct! The draft model (small, fast — e.g., 7B) proposes K tokens cheaply. The large model (slow, accurate — e.g., 70B) verifies all K tokens in ONE forward pass (same compute as generating just 1 token). Accepted tokens are kept; rejected tokens are regenerated. Net result: 2-3x speedup on typical text outputs.
Question 7 of 15
Q8/15 Twitter/X uses a hybrid fan-out strategy. For a celebrity with 50 million followers, which approach is used?
✅ Correct! Fan-out on write to 50M followers would create a massive write spike and use 50M × tweet_size of memory. Instead, celebrity tweets are flagged. When a user's timeline loads, the system pulls regular user tweets (pre-computed) PLUS celebrity tweets they follow (computed at read time). This is the hybrid approach used in production.
Question 8 of 15
Q9/15 Which fine-tuning method allows you to fine-tune a 70B model on a consumer GPU (e.g., 2×A100 80GB)?
✅ Correct! QLoRA (Quantized LoRA) quantizes the base model to 4-bit (reducing 70B model from ~140GB to ~35GB), then trains LoRA adapter matrices in higher precision. This makes it possible to fine-tune a 70B model on 2×A100 80GB GPUs. Regular LoRA still needs FP16 base model weights, requiring more VRAM. QLoRA was introduced in the 2023 paper by Dettmers et al.
Question 9 of 15
Q10/15 RAGAS evaluates RAG systems with multiple metrics. What does "Faithfulness" measure?
✅ Correct! RAGAS Faithfulness measures whether the LLM's answer is supported by the retrieved context — detecting hallucinations. A score of 1.0 means every claim in the answer can be found in the context. The other key metrics: Answer Relevancy (does the answer address the question?), Context Precision (are retrieved chunks relevant?), Context Recall (was all needed info retrieved?).

Q11/15 In the RADIO framework for system design interviews, what should you do FIRST before drawing any diagrams?
✅ Correct! Requirements gathering (the R in RADIO) is step one and interviewers specifically test for this. Ask: How many DAU? Read/write ratio? Latency SLA? Global or single region? Strong or eventual consistency needed? Many candidates jump to drawing boxes — this is a red flag. Spend 5 minutes on requirements before drawing anything.
Question 11 of 15
Q12/15 What is the "thundering herd" problem in caching, and how is it typically solved?
✅ Correct! The thundering herd: a cached item expires, 10,000 simultaneous requests all get a miss, all query the database at once — potentially crashing it. Solutions: (1) Mutex/locking — first request recomputes, others wait. (2) Probabilistic early expiration (XFetch) — randomly refresh slightly before TTL. (3) Cache stampede prevention via Lua scripts in Redis (atomic get-or-set).
Question 12 of 15
Q13/15 What is Kubernetes' HPA (Horizontal Pod Autoscaler) and when would you use it for ML serving?
✅ Correct! HPA watches metrics and scales replicas between min and max limits. For ML serving: use CPU/GPU utilization as trigger (e.g., scale up when GPU util > 70%). For LLM serving: use custom metrics like request queue depth or TTFT (time to first token). Combine with Karpenter to auto-provision new GPU nodes when needed.
Question 13 of 15
Q14/15 In hybrid retrieval for RAG, what does RRF (Reciprocal Rank Fusion) do?
✅ Correct! RRF formula: score = Σ 1/(k + rank_i) for each retrieval system's ranking. It's robust to score scale differences between dense (cosine similarity 0-1) and sparse (BM25 0-∞) systems — you don't need to normalize scores. Simply combine ranks. Then apply a cross-encoder reranker on the fused top-50 for final ordering. This pipeline achieves the best RAG retrieval accuracy in 2025.

✅ Correct! AI Engineer is the fastest-growing tech role in 2024-2026. They build products using LLMs — RAG pipelines, agents, evaluation systems, prompt engineering. They need strong software engineering skills more than deep ML theory. ML Engineers focus on the model itself: training loops, optimization algorithms, CUDA, distributed training. Both roles are distinct and in high demand.
