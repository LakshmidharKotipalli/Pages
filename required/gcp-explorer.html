What is Google Cloud Platform?

Google Cloud Platform (GCP) is Google's suite of cloud computing services that runs on the same infrastructure Google uses internally for its end-user products like Google Search, Gmail, and YouTube. It offers compute, storage, AI, databases, networking, and developer tools — all accessible via the internet.

Think of it this way

GCP is like a massive city of specialized shops — a compute shop (Cloud Run, GKE), a storage warehouse (Cloud Storage), an AI research lab (Vertex AI), a database district (Cloud SQL, Firestore, Bigtable), and a security headquarters (IAM). You rent exactly what you need, only pay for what you use, and Google maintains all the infrastructure.

Document AI

Google Cloud's intelligent document processing platform. It uses machine learning to automatically read, understand, and extract structured data from any document — PDFs, scanned images, forms, invoices, contracts, and more.

OCR Form Parsing Entity Extraction Gemini-powered GA + Preview

Think of Document AI as a super-smart intern who never gets tired. You hand them a stack of 10,000 invoices, and instead of manually typing every supplier name, invoice number, and total into a spreadsheet (which would take weeks), they scan each document in milliseconds and fill in a perfectly structured spreadsheet for you — with nearly perfect accuracy.

Processor Types

🔤
General OCR Processor
Converts printed or handwritten text in any document into machine-readable text. Supports 60+ languages. The foundation for all other processors.
📝
Form Parser
Detects key-value pairs in forms (e.g., "Name: John Doe", "Date: 01/15/2025"). Returns structured field-value pairs without any ML training needed.
🧾
Invoice Parser
Pre-trained specifically for invoices. Extracts supplier name, invoice number, line items, totals, tax amounts, and payment terms automatically.
🪪

Specialized parsers for US Driver's License, Passport, and national ID cards. Extracts name, DOB, address, ID number, and validates against known formats.
🗂️
Layout Parser (Gemini)
New in 2025/2026 — powered by Gemini Flash LLM. Understands complex document layouts: tables, lists, headers, sections. Returns chunked output perfect for RAG pipelines.
🔬
Custom Extractor
Train your own document processor using Document AI Workbench. Label just 50-100 sample documents and the platform trains a custom entity extractor for your specific documents.

Key Features

  • Pre-trained models — No ML expertise needed for most use cases. Just call the API.
  • Human-in-the-loop (HITL) — Route low-confidence extractions to human reviewers before they enter your database.
  • Document AI Warehouse — Store, search, and manage extracted document data at scale.
  • Batch processing — Process thousands of documents asynchronously via Cloud Storage triggers.
  • Confidence scores — Every extracted field comes with a confidence score (0.0–1.0), so you know what to trust.

Common Use Cases

📬 Accounts payable automation 🏥 Medical record digitization 🏦 KYC / ID verification 📃 Contract analysis 📦 Customs & logistics forms 🧑‍💼 HR onboarding documents 🏠 Mortgage processing 📚 Document RAG pipelines

Quick Code Example

Process an invoice with Document AI
Python 
from google.cloud import documentai_v1 as documentai

# Initialize the Document AI client
client = documentai.DocumentProcessorServiceClient()

# Specify your processor (invoice parser)
processor_name = client.processor_path(
    project="my-project",
    location="us",
    processor="my-invoice-processor-id"
)

# Load the PDF document
with open("invoice.pdf", "rb") as f:
    content = f.read()

# Create the processing request
request = documentai.ProcessRequest(
    name=processor_name,
    raw_document=documentai.RawDocument(
        content=content,
        mime_type="application/pdf"
    )
)

# Process the document
result = client.process_document(request=request)
document = result.document

# Extract entities
for entity in document.entities:
    print(f"{entity.type_}: {entity.mention_text} (confidence: {entity.confidence:.2f})")
# Output: supplier_name: Acme Corp (confidence: 0.98)
# Output: invoice_id: INV-2024-001 (confidence: 0.99)
# Output: total_amount: $1,250.00 (confidence: 0.97)

Cloud Storage

Google's infinitely scalable object storage service. Store any amount of data — from a single file to exabytes — and access it from anywhere in the world. It's the backbone of most GCP architectures, used by virtually every other GCP service.

Object Storage Buckets 4 Storage Classes Lifecycle Rules 99.999999999% Durability
Analogy

Cloud Storage is like a massive filing cabinet warehouse with infinite drawers. Each drawer is a "bucket" — a container for your files (called "objects"). Every file has a unique name, and you can retrieve any file instantly from anywhere in the world. Unlike a real filing cabinet, this one never runs out of space, never loses a file (11 nines of durability!), and you only pay for the space you actually use.

Core Concepts

🪣
Buckets
The top-level container for objects. Bucket names are globally unique across all of Google Cloud (like domain names). Each bucket has a location (region or multi-region) and a default storage class.
📁
Objects
Individual files stored in a bucket. Can be any type: images, videos, CSVs, ML models, backups, code artifacts. Objects are immutable — to "update" a file, you upload a new version. Supports object versioning.
🔗
Object URLs
Every object has a URL: gs://bucket-name/path/to/file.jpg. You can also generate signed URLs that give temporary public access to private objects — without making the whole bucket public.

Lifecycle Rules automate storage class transitions! You don't have to manually move objects. Set a rule like: "After 30 days, move to Nearline. After 90 days, move to Coldline. After 365 days, move to Archive." Cloud Storage does it automatically, cutting your storage costs without any manual effort.

Access Control Options

  • IAM Bucket Policies — Recommended. Grant roles (Storage Object Viewer, Storage Admin) to users or service accounts using IAM.
  • ACLs (Access Control Lists) — Fine-grained per-object permissions. Legacy approach, IAM is preferred.
  • Signed URLs — Generate a time-limited URL (e.g., valid for 1 hour) that lets anyone download a specific private object.
  • Public Access Prevention — Enable at org level to prevent anyone from accidentally making buckets public.
Common Cloud Storage operations
bash
# Create a bucket
gcloud storage buckets create gs://my-bucket \
  --location=us-central1 \
  --default-storage-class=STANDARD

# Upload a file
gcloud storage cp local-file.jpg gs://my-bucket/images/

# Download a file
gcloud storage cp gs://my-bucket/images/photo.jpg .

# Set lifecycle rule (JSON file)
gcloud storage buckets update gs://my-bucket \
  --lifecycle-file=lifecycle.json

# Generate a signed URL (valid 1 hour)
gcloud storage sign-url gs://my-bucket/private.pdf \
  --duration=1h \
  --service-account=sa@project.iam.gserviceaccount.com

BigQuery

Google's serverless, highly scalable data warehouse. BigQuery lets you analyze petabytes of data using standard SQL — and get results in seconds. There are no servers to manage, no indexes to tune, and no capacity planning. You just write SQL and BigQuery figures out how to run it at scale.

Serverless SQL Petabyte-scale Columnar Storage BigQuery ML Streaming
Analogy

BigQuery is like a magical library that reads columns, not rows. Imagine a library with 1 billion rows of customer data. A normal database reads each row one by one to find what you need. BigQuery reads only the columns you asked for, in parallel across thousands of machines simultaneously — so a query on 1 TB of data might finish in 5 seconds instead of 5 hours.

Why BigQuery is Fast — Columnar Storage Explained

❌ Row-based storage (traditional DB)
Row 1: [John | 25 | NYC | $50k]
Row 2: [Jane | 30 | LA | $70k]
Row 3: [Bob | 22 | SF | $60k]
To get the average salary, the DB must read every column of every row — even though you only need the salary column. Wastes 75% of I/O.
✅ Columnar storage (BigQuery)
Name col: [John, Jane, Bob]
Age col: [25, 30, 22]
Salary col: [$50k, $70k, $60k]
To get average salary, BigQuery reads only the salary column. 75% less I/O. Also compresses beautifully (similar values next to each other). Thousands of servers do this in parallel.

Key Concepts

  • Datasets — A group of tables within a project. Like a "schema" or "database" in traditional SQL.
  • Tables — Hold your data. Can be native BQ tables, external tables (CSV in Cloud Storage), or views.
  • Partitioning — Split a huge table by date column so queries only scan relevant partitions. Saves money and time.
  • Clustering — Sort data by a column (e.g., country). Queries filtering by that column skip irrelevant blocks automatically.
  • Slots — BQ's unit of compute. 1 slot = a virtual CPU. BQ auto-scales slots; you can reserve them for predictable cost.
  • Streaming inserts — Insert rows in real-time for up-to-the-second analytics dashboards. Data available for queries within seconds.

BigQuery ML — Train ML Models in SQL!

BigQuery ML lets you train and deploy ML models directly in BigQuery using SQL — no Python or TensorFlow required!
Train a classification model in SQL
SQL
-- Train a logistic regression model

OPTIONS(model_type='logistic_reg',
        input_label_cols=['churned']) AS

-- Predict churn on new customers
SELECT * FROM
ML.PREDICT(MODEL `my_dataset.churn_model`,
  (SELECT * FROM `my_dataset.new_customers`));

Cloud Functions

Google's event-driven serverless compute service. Write a single function in your language of choice, attach it to a trigger (HTTP request, Cloud Storage upload, Pub/Sub message, Firestore change), and it runs automatically — with zero server management and zero idle cost.

FaaS Event-driven 2nd Gen (Cloud Run-based) Serverless
Analogy

Cloud Functions is like a vending machine. The machine (your function) sits idle, consuming no electricity. When you press a button (trigger), it performs one specific action (your code runs), dispenses the item, then goes back to idle. Each button (trigger type) does a different thing: HTTP button, Pub/Sub button, Storage button, Firestore button — all using the same machine concept.

Trigger Types

🌐
HTTP Trigger
Function runs when its URL receives a GET/POST request. Most common — used to build simple REST APIs or webhooks.
📨
Pub/Sub Trigger
Function runs when a message is published to a Pub/Sub topic. Perfect for event-driven pipelines and background processing.
🗄️
Cloud Storage Trigger
Function runs when a file is uploaded, updated, or deleted from a bucket. Ideal for image resizing, virus scanning, or triggering ML processing.
🔥
Firestore Trigger
Function runs when a Firestore document is created, updated, or deleted. Used for sending notifications or updating denormalized data.

This means you get all the benefits of Cloud Run — longer timeouts, more memory, true concurrency — while keeping the simpler "write one function" developer experience.

Example: Auto-resize images on upload

Cloud Function triggered by Cloud Storage upload
Python
import functions_framework
from google.cloud import storage
from PIL import Image
import io

# This function fires whenever a new image is uploaded to the bucket
@functions_framework.cloud_event
def resize_image(cloud_event):
    data = cloud_event.data
    bucket_name = data["bucket"]
    file_name = data["name"]

    if not file_name.endswith((".jpg", ".png")):
        return  # Skip non-images

    client = storage.Client()
    bucket = client.bucket(bucket_name)
    blob = bucket.blob(file_name)

    # Download, resize, re-upload as thumbnail
    img_bytes = blob.download_as_bytes()
    img = Image.open(io.BytesIO(img_bytes))
    img.thumbnail((300, 300))

    thumb_blob = bucket.blob(f"thumbnails/{file_name}")
    buf = io.BytesIO()
    img.save(buf, format="JPEG")
    thumb_blob.upload_from_string(buf.getvalue(), content_type="image/jpeg")
    print(f"Thumbnail created: thumbnails/{file_name}")

Cloud Pub/Sub

Google's fully managed publish-subscribe messaging service. It's the glue that connects different parts of your application — one service publishes events, and multiple other services receive and react to them independently. Handles millions of messages per second globally.

Messaging Streaming Global At-least-once delivery Ordering
Analogy

Think of Pub/Sub like a YouTube channel. A YouTuber (publisher) posts a new video to their channel (topic). Every subscriber (subscription) who follows that channel automatically gets notified. If 3 different apps "subscribe" to the same "new order" topic, all 3 receive the message independently — the warehouse system, the email notification system, and the analytics system can all react to the same event without being directly connected to each other.

GCP Databases — Which One to Use?

GCP offers 6+ managed database services, each optimized for a different use case. Picking the right one is one of the most important architecture decisions you'll make.

Analogy

Think of databases like different kinds of storage in a restaurant kitchen: Cloud SQL is the organized refrigerator (structured, labeled, relational). Firestore is a flexible pantry with labeled jars (document-based, flexible schema). Bigtable is a massive industrial cold storage (billions of rows, high throughput). Spanner is a synchronized multi-location inventory system (globally distributed, always consistent). Memorystore is the countertop (in-memory, instant access).

Interactive Database Selector

1. What's your data structure?
🗃️
Cloud SQL
Relational

Fully managed MySQL, PostgreSQL, and SQL Server. Google handles backups, patches, failover, and scaling — you just write SQL. Best for: traditional web apps, ERP systems, and migrating from on-premise databases.

  • Up to 96 vCPUs, 624 GB RAM per instance
  • Automated daily backups with point-in-time recovery
  • High availability with automatic failover (99.95% SLA)
  • Read replicas for horizontal read scaling
Use when: You need SQL and ACID transactions, migrating a MySQL/PostgreSQL app, or building a traditional web application.
🔥
Cloud Firestore
NoSQL Document

Serverless, scalable NoSQL document database. Data is stored in documents (like JSON objects) organized in collections. Supports real-time listeners — your app UI updates automatically when data changes. Great for mobile/web apps.

  • Flexible schema — each document can have different fields
  • Real-time sync for mobile/web apps (Firebase SDKs)
  • Offline support — apps work without internet, sync when reconnected
  • Strong consistency within a document, eventual across collections
Use when: Building mobile apps, user profiles, product catalogs, real-time chat — anything needing flexible schema and real-time sync.
⚙️
Cloud Bigtable
NoSQL Wide-Column

Petabyte-scale, low-latency NoSQL database for high-throughput workloads. The same technology that powers Google Search indexing, Google Analytics, and Gmail. Designed for data sizes from 10TB to petabytes. Not suitable for small datasets.

  • Single-digit millisecond latency at any scale
  • Millions of read/write operations per second
  • Integrates with Apache Beam, Dataflow, BigQuery
  • No SQL support, no transactions across rows
Use when: IoT sensor data, time-series analytics, financial tick data, ad-tech, or any workload with 10TB+ and millions of operations/second.
🌍
Cloud Spanner
Global SQL

Google's "holy grail" database — globally distributed, strongly consistent, relational. It gives you the best of both worlds: SQL with full ACID transactions AND horizontal scaling across regions. Used for banking, global inventory, gaming leaderboards.

  • 99.999% availability SLA (five 9s) — almost no downtime ever
  • External consistency — strongest possible consistency guarantee
  • Standard SQL with full joins, transactions, and foreign keys
  • Expensive — starts at ~$0.90/hour per node. Not for small projects.
Use when: Global financial transactions, banking systems, multi-region inventory, gaming backends where consistency and scale are both critical.
💎
AlloyDB for PostgreSQL
Managed PostgreSQL

Google's premium fully managed PostgreSQL — 4x faster than standard PostgreSQL, 100% compatible with existing Postgres apps. Uses a disaggregated storage/compute architecture. Ideal when Cloud SQL isn't fast enough but you still want PostgreSQL compatibility. Includes AI-powered query advisors.

⚡
Memorystore (Redis / Valkey)
In-Memory Cache

Fully managed Redis (and Valkey, the open-source Redis fork) service. Sub-millisecond data access. Used for caching frequently-accessed data (so your database doesn't get hammered), session storage, leaderboards, and rate limiting. Think of it as a blazing-fast scratchpad next to your main database.

Google Kubernetes Engine (GKE)

Google's managed Kubernetes service. Kubernetes (K8s) is the world's most popular container orchestration system — and Google invented it! GKE takes the complexity out of running Kubernetes by managing the control plane, upgrades, and node provisioning for you.

Kubernetes Container Orchestration Autopilot Mode Standard Mode Multi-cluster
Analogy

GKE is like a shipping port with an automated crane operator. Your containers (Docker images) are like shipping containers — standardized boxes that hold your app. Pods are groups of containers on a pallet. Nodes are the ships that carry pallets. The cluster is the entire port. The Kubernetes control plane is the automated crane operator — it decides where each container goes, starts new ships (nodes) when the port gets busy, and replaces broken containers automatically.

Standard vs Autopilot Mode

⚙️ Standard Mode

You manage the nodes (VMs). Choose machine type, number of nodes, and configure everything. More control, more responsibility. You're billed for node VMs even when they're idle.

  • Full control over node configuration
  • Custom machine types and GPUs
  • Node pools with different configurations
  • You manage node upgrades and security patches
  • Billed for all node VMs even when idle
🤖 Autopilot Mode (Recommended)

Google manages the nodes for you. You just deploy your workloads (Pods). GCP automatically provisions and scales nodes, applies security patches, and optimizes cluster configuration. Billed only for Pod resource requests — no wasted capacity.

  • No node management — GCP handles everything
  • Pay only for what Pods actually request
  • Automatic security hardening and patches
  • Built-in node auto-provisioning
  • Less control over node-level configuration

Core Kubernetes Concepts on GKE

  • Pod: The smallest deployable unit. One or more containers that share a network namespace and storage. Think: one task unit.
  • Deployment: Manages a replicated set of identical Pods. Handles rolling updates and rollbacks automatically.
  • Service: Gives a stable IP address and DNS name to a set of Pods (since Pods are ephemeral — they come and go).
  • Ingress: Routes external HTTP/HTTPS traffic to the right Service. Like a front door with routing rules for your cluster.
  • HPA (Horizontal Pod Autoscaler): Automatically adds/removes Pod replicas based on CPU, memory, or custom metrics.

Cloud Build + Artifact Registry

GCP's CI/CD backbone. Cloud Build is a serverless build system that runs your pipelines (build, test, push) without managing any servers. Artifact Registry is the package manager for everything it produces — container images, npm packages, Maven JARs, Python wheels, and more.

CI/CD Serverless Build Container Registry Multi-format Binary Authorization
Analogy

Think of Cloud Build as an automated assembly line in a factory. When a new part arrives (code commit), the assembly line automatically: tests the part quality (run tests), assembles it into a finished product (build Docker image), wraps it for shipping (push to Artifact Registry), and delivers it to the store (deploy to Cloud Run or GKE). Artifact Registry is the warehouse where all packaged products are stored, versioned, and ready to ship.

steps:
  # Step 1: Run unit tests
  - name: 'python:3.12'
    entrypoint: pip
    args: ['install', '-r', 'requirements.txt']
  - name: 'python:3.12'
    entrypoint: pytest
    args: ['tests/', '-v']

  # Step 2: Build Docker image
  - name: 'gcr.io/cloud-builders/docker'
    args:
      - build
      - '-t'
      - 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA'
      - '.'

  # Step 3: Push to Artifact Registry
  - name: 'gcr.io/cloud-builders/docker'
    args:
      - push
      - 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA'

  # Step 4: Deploy to Cloud Run
  - name: 'gcr.io/google.com/cloudsdktool/cloud-sdk'
    entrypoint: gcloud
    args:
      - run
      - deploy
      - my-service
      - '--image=us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA'
      - '--region=us-central1'
      - '--platform=managed'

substitutions:
  _REGION: us-central1

options:
  logging: CLOUD_LOGGING_ONLY
📦 Artifact Registry

The successor to Container Registry. Stores and manages build artifacts in one central, secure place.

  • Multi-format: Docker images, Helm charts, npm, Maven, PyPI, APT/YUM packages
  • Vulnerability scanning: Automatically scans container images for CVEs
  • Binary Authorization: Only allow verified, signed images to be deployed to GKE/Cloud Run
  • Regional repositories: Store artifacts close to your deployment region for faster pulls
🔗 Integrations

Cloud Build integrates natively with the entire GCP ecosystem:

  • GitHub / GitLab: Triggers on push, pull request, or tag events
  • Cloud KMS: Sign build artifacts for supply chain security
  • Cloud Deploy: Progressive delivery to GKE and Cloud Run with approval gates
  • Pub/Sub: Publish build status events for custom notifications and workflows

IAM & Security

Identity and Access Management (IAM) is the security backbone of every GCP project. It answers the fundamental question: "Who can do what on which resource?" Understanding IAM is critical before building anything on GCP — get it wrong and you either lock yourself out or expose your resources to the world.

IAM Service Accounts Roles Secret Manager Cloud KMS VPC Service Controls
Analogy

IAM is like a company's access badge system. Every employee (principal) has a badge (identity) with a job title (role). Their job title determines what rooms (resources) they can enter and what they can do there. A "Viewer" badge lets you look around. An "Editor" badge lets you make changes. An "Owner" badge gives you master access. Security guards (GCP policy engine) check your badge every time you try to do anything.

The 3 Components of IAM

👤 WHO — Principals (Identities)
  • Google Account — A real person's Google identity (email). Used for console access.
  • Google Group — A collection of Google accounts. Grant roles to a group instead of individuals for easier management.
  • Service Account — A non-human identity for applications, VMs, and Cloud Run services. Your app's "identity card" when it calls GCP APIs.
  • Workspace Domain — All users in a Google Workspace organization (e.g., everyone at company.com).
🎯 WHAT — Roles (Permissions)
  • Primitive Roles — Owner, Editor, Viewer. Very broad — avoid using in production. Give too much access.
  • Predefined Roles — Google-managed roles for specific services (e.g., roles/storage.objectViewer). Use these in most cases.
  • Custom Roles — You define exactly which permissions to include. Maximum least-privilege control. More management overhead.
🎯 WHERE — Resources & Policy Hierarchy

IAM policies are attached at different levels. Higher-level policies are inherited by everything below them — you cannot grant less than what a parent already grants.

🏢
Organization — Top level. Policies here apply to everything in the company.
📁
Folders — Group related projects (e.g., "Production", "Development"). Useful for department-level access.
📦
Projects — The main unit of organization in GCP. Most access control happens here.
🪣
Resources — Individual GCS buckets, BigQuery datasets, etc. Finest-grained access control.

Security Services Beyond IAM

🔐
Secret Manager
Store API keys, database passwords, and certificates as versioned secrets. Your app fetches secrets at runtime instead of baking them into code or environment variables. Fully audited — every access is logged. Integrated with Cloud Run, GKE, and Cloud Functions.
# Access a secret from Python
from google.cloud import secretmanager
client = secretmanager.SecretManagerServiceClient()
secret = client.access_secret_version(
    name="projects/123/secrets/db-password/versions/latest"
)
db_password = secret.payload.data.decode("UTF-8")
🔑
Cloud KMS (Key Management Service)
Manage cryptographic keys for encrypting your data. Create, rotate, and disable encryption keys without ever seeing the key material. GCP uses customer-managed encryption keys (CMEK) to let you control encryption of BigQuery, Cloud Storage, GKE, and more.
🛡️
Security Command Center
GCP's centralized security dashboard. Continuously scans your GCP resources for misconfigurations (e.g., public Cloud Storage buckets, overly permissive firewall rules), vulnerabilities in VMs, and threat detections (crypto-mining, unusual API access). Shows a risk score and remediation steps.
Security Best Practice: Always follow the Principle of Least Privilege — grant the minimum permissions needed to do a job, nothing more. Use Service Accounts with specific predefined roles, not the Editor or Owner primitive roles.
